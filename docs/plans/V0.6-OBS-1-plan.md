## 1. Context & Current State

### 1.1 What observability the gateway already ships

`docs/OBSERVABILITY.md` and the wiring in `cmd/server/main.go`
(`:601-615`) give operators three pillars:

| Pillar | Surface | Where |
|---|---|---|
| Structured audit | JSON events to stdout/file/HTTP | `internal/audit/*` |
| Metrics | `/metrics` (Prometheus) on the **data-plane** mux | `cmd/server/main.go:602`, `internal/metrics/metrics.go` |
| Tracing | OTLP/Jaeger/stdout exporters | `cfg.Tracing.Enabled` at `main.go:613-615` |

What is missing is a **runtime-introspection** pillar — CPU, heap,
mutex, block, goroutine, allocs, threadcreate, and execution-trace
profiles. Today, an operator hitting a regression in production (e.g.
latency creep after a `max_part_buffer` change landed in V0.6-PERF-1,
or an allocation spike introduced by the retryer's backoff/jitter
table in V0.6-PERF-2) has only metrics and logs. Root-causing those
symptoms to a function in `internal/crypto/engine.go` or a
goroutine-leak in the MPU manifest decrypt path requires rebuilding
the binary with `net/http/pprof` temporarily imported — a change-
management hit under any sane SRE policy.

### 1.2 What the admin subsystem already provides

V0.6-CFG-1 delivered precisely the primitive this plan needs:

- A separate listener (`internal/admin/server.go:66-122`) disabled by
  default, bound to `127.0.0.1:8081` (`config.go:560-569`), with
  optional TLS (`AdminTLSConfig`, `config.go:422-427`).
- Bearer-token authentication with constant-time compare and a
  file-based token source that re-reads on every request
  (`admin/auth.go:15-52`, `admin/server.go:144-161`).
- A rate limiter gated by `RateLimit.RequestsPerMinute`
  (`admin/ratelimit.go`, default 30 rpm).
- A context flag (`ctxKeyAdmin`) and reusable predicate
  `admin.IsAdminRequest(r)` (`admin/server.go:31-36`).
- Public mux accessor (`Server.Mux()`) already used by the rotation
  handlers (`cmd/server/main.go:681`) and the MPU admin routes
  (`:688`).

The admin subsystem is **the** place to mount pprof. Mounting on the
data-plane mux (`/debug/pprof/*`) would be a security and capacity
regression: pprof handlers block goroutines (CPU/trace profiles run
for up to `seconds=`, default 30 s, holding the writer), carry no
authentication, and expose the binary's in-memory structure (goroutine
stacks, heap histograms, symbol names) to any S3 client. `main.go:601`
is therefore **off-limits**.

### 1.4 Relevant Go runtime semantics

`net/http/pprof` registers **via package init** when imported — the
infamous underscore import `_ "net/http/pprof"` side-effect-mounts
profiles onto `http.DefaultServeMux`. We will **not** rely on that
global. Instead, the supported mechanism is:

- `runtime/pprof` — the profile engines themselves.
- `net/http/pprof` exports **handler constructors** that take no
  implicit registration: `pprof.Index`, `pprof.Cmdline`,
  `pprof.Profile`, `pprof.Symbol`, `pprof.Trace`, and
  `pprof.Handler(name)` which returns an `http.Handler` backed by the
  named `runtime/pprof.Profile` (heap, goroutine, allocs, block,
  mutex, threadcreate).

This is documented in Plotka, *Efficient Go* (O'Reilly, 2022) Ch. 9
["CPU Optimizations"](https://learning.oreilly.com/library/view/-/9781098105709/)
and in Bodner, *Learning Go, 2nd Edition* (O'Reilly, 2024) Ch. 14
["The Context and Runtime"](https://learning.oreilly.com/library/view/-/9781098139285/).
Adkins et al., *Building Secure and Reliable Systems* (O'Reilly, 2020)
Ch. 15 ["Debugging and Investigating Systems"](https://learning.oreilly.com/library/view/-/9781492083115/)
explicitly calls out "debugging interfaces must be authenticated,
authorized, and audited, or they become a privilege-escalation
vector". That is the north star for this plan.

### 1.5 Performance / safety properties of pprof in production

From Plotka Ch. 9 and Kennedy, *Ultimate Go: Advanced Concepts*
(Pearson, 2024) "[Profiling in Production](https://learning.oreilly.com/videos/-/9780135339503/)":

- **CPU profile**: ~5 % steady-state overhead during the sampling
  window (default 30 s, configurable via `seconds=`). Negligible
  steady-state cost when not sampling.
- **Heap profile**: snapshot — O(heap-object-count) to serialise,
  low steady-state cost (Go's allocator samples at one per
  `runtime.MemProfileRate`, default 512 KiB).
- **Goroutine profile**: snapshot — cheap unless goroutine count is
  pathological (then it's exactly the data you need).
- **Block / mutex profiles**: require `runtime.SetBlockProfileRate` /
  `runtime.SetMutexProfileFraction` to be non-zero; both default to
  off. Enabling them has a non-trivial steady-state cost (block ~1 %,
  mutex ~2-5 % depending on contention). We will expose these as
  opt-in knobs, not always-on.
- **Execution trace** (`/debug/pprof/trace?seconds=N`): highest-
  overhead tool (10-30 % during sampling window) but invaluable for
  scheduler/GC analysis. Must be rate-limited by the admin RPM cap
  already in place.

### 1.6 Symbol table / binary stripping risk

Go binaries built with `-ldflags="-s -w"` strip symbols, which
severely degrades the quality of pprof output (function names become
addresses). The gateway's `Dockerfile` and `Dockerfile.fips` need
auditing for this — it is a **precondition to documenting a profiling
runbook that actually works**. Handled in Phase E.

## 2. Design Goals & Non-Goals

### Goals

1. Land a production-safe, admin-gated profiling surface at
   `/admin/debug/pprof/*` that is:
   - **Disabled by default.** Presence of the routes is gated by
     `admin.profiling.enabled: true`.
   - **Requires `admin.enabled: true`.** Profiling cannot be turned
     on without the admin listener — there is no other listener it
     could safely attach to.
   - **Authenticated by the existing bearer token**, rate-limited by
     the existing admin rate-limiter, optionally TLS-terminated by
     the existing admin TLS block. Zero new auth surface.
   - **Audited.** Every profile fetch emits an `audit` event with
     endpoint, remote IP (if available), duration, and success flag,
     via `AuditLogger.LogAccessWithMetadata`.
   - **Instrumented.** A bounded-cardinality counter
     `s3_gateway_admin_pprof_requests_total{endpoint,outcome}` is
     registered so operators can see profile-fetch activity alongside
     normal admin traffic in Grafana.
2. Support the seven standard pprof endpoints (`index`, `cmdline`,
   `profile`, `symbol`, `trace`, plus named profiles `heap`,
   `goroutine`, `allocs`, `block`, `mutex`, `threadcreate`) with the
   same URL shapes Go operators already know.
3. Treat block- and mutex-profiling rates as config-driven knobs
   (`admin.profiling.block_rate`, `admin.profiling.mutex_fraction`),
   both defaulting to 0 (disabled). The profiling endpoints return
   empty profiles when the rates are 0 — which matches Go's runtime
   behaviour and needs operator documentation, not a code change.
4. Fail-closed on misconfiguration: if `admin.profiling.enabled` is
   true but `admin.enabled` is false, `LoadConfig` returns a clear
   error at startup. If `profiling.enabled` is true and the admin
   listener is bound to a non-loopback address without TLS, startup
   fails with a fix-forward message (same policy V0.6-CFG-1 applies
   to admin itself — see `config.go` validation block).
5. Ship an operator runbook (`docs/OBSERVABILITY.md` §"Runtime
   Profiling") with three recipes:
   - `go tool pprof -http=:0 https://...` against the live gateway.
   - `curl | flamegraph.pl` pipeline for airgapped environments.
   - Goroutine-leak snapshot workflow.

### Non-Goals

- **Continuous profiling (Pyroscope / Parca / Polar Signals).**
  Out of scope for v0.6. The endpoints land here; pull-based
  continuous scraping is an opinionated operational choice filed as
  follow-up *V0.6-OBS-1-followup-pyroscope* (§8).
- **Exposing pprof on the data-plane listener.** Explicitly
  forbidden. See §1.2.
- **Separate auth for profiling.** The admin bearer token is the
  admin bearer token. Splitting it (e.g. a "profiler" role) would
  imply an RBAC story the admin subsystem does not have. If / when
  RBAC lands (v0.7 candidate), the pprof routes trivially inherit.
- **`runtime/debug.SetGCPercent` / `runtime.SetMaxThreads` knobs via
  admin HTTP.** Those are one-way-door knobs on a live process and
  deserve their own plan. Not in OBS-1.
- **A web UI.** `go tool pprof -http` already provides one; we do
  not need to serve flamegraphs ourselves.
- **Pprof for the FIPS build.** The FIPS build (`-tags=fips`,
  `Dockerfile.fips`) must **not** regress — pprof does not depend on
  `crypto/*` and is safe under GOFIPS140. Validated by tests under
  both build tags.

## 3. Configuration Surface

### 3.1 New `AdminProfilingConfig` stanza

Added to `internal/config/config.go` next to the existing
`AdminRateLimitConfig`:

```go
// AdminProfilingConfig controls the /admin/debug/pprof/* routes.
//
// Disabled by default. When enabled, requires AdminConfig.Enabled
// and (on non-loopback addresses) AdminTLSConfig.Enabled. The admin
// bearer token and rate limit apply; no new auth surface is added.
type AdminProfilingConfig struct {
    // Enabled mounts /admin/debug/pprof/* on the admin mux.
    Enabled bool `yaml:"enabled" env:"ADMIN_PROFILING_ENABLED"`

    // BlockRate is passed to runtime.SetBlockProfileRate at startup.
    // 0 (default) disables block profiling. Positive values sample
    // one event per N ns of blocking; 1 samples all (expensive).
    BlockRate int `yaml:"block_rate" env:"ADMIN_PROFILING_BLOCK_RATE"`

    // MutexFraction is passed to runtime.SetMutexProfileFraction.
    // 0 (default) disables mutex profiling. Positive values sample
    // 1/N contention events.
    MutexFraction int `yaml:"mutex_fraction" env:"ADMIN_PROFILING_MUTEX_FRACTION"`

    // MaxConcurrentProfiles bounds in-flight profile/trace requests.
    // Defaults to 2. A semaphore inside the profiling handler rejects
    // additional requests with 429 until one finishes.
    MaxConcurrentProfiles int `yaml:"max_concurrent_profiles" env:"ADMIN_PROFILING_MAX_CONCURRENT"`

    // MaxProfileSeconds caps the `seconds=` query parameter for
    // /profile and /trace. Defaults to 60. Values above cap are
    // rejected with 400.
    MaxProfileSeconds int `yaml:"max_profile_seconds" env:"ADMIN_PROFILING_MAX_SECONDS"`
}
```

`AdminConfig` gains one field:

```go
type AdminConfig struct {
    // ... existing ...
    Profiling AdminProfilingConfig `yaml:"profiling"`
}
```

Defaults (set in `LoadConfig` next to the existing `Admin` block at
`config.go:560-569`):

```go
Admin: AdminConfig{
    Enabled: false,
    Address: "127.0.0.1:8081",
    Auth:    AdminAuthConfig{Type: "bearer"},
    RateLimit: AdminRateLimitConfig{RequestsPerMinute: 30},
    Profiling: AdminProfilingConfig{
        Enabled:               false,
        BlockRate:             0,
        MutexFraction:         0,
        MaxConcurrentProfiles: 2,
        MaxProfileSeconds:     60,
    },
},
```

### 3.2 Validation rules (added to `Config.Validate`)

1. `Profiling.Enabled && !Admin.Enabled` → error
   `"admin.profiling requires admin.enabled"`.
2. `Profiling.Enabled && Admin.Address` is **not** loopback (not
   `127.0.0.1`, not `::1`, not `localhost`) `&& !Admin.TLS.Enabled`
   → error `"admin.profiling on non-loopback requires admin.tls.enabled"`.
3. `Profiling.BlockRate < 0` → error.
4. `Profiling.MutexFraction < 0` → error.
5. `Profiling.MaxConcurrentProfiles < 1` → normalise to 1
   (warn-log at startup).
6. `Profiling.MaxProfileSeconds < 1 || > 600` → error.

### 3.3 `config.yaml.example` addition

Under the admin block (CFG-1 will land `admin:` in the example; if it
has not by the time OBS-1 implements, add a minimal admin stub here):

```yaml
admin:
  enabled: false
  address: "127.0.0.1:8081"
  # ... existing keys ...
  profiling:
    enabled: false         # Mount /admin/debug/pprof/* when true.
    block_rate: 0          # 0 = disabled. See runtime.SetBlockProfileRate.
    mutex_fraction: 0      # 0 = disabled. See runtime.SetMutexProfileFraction.
    max_concurrent_profiles: 2
    max_profile_seconds: 60
```

## 4. Implementation Phases

### Phase A — Profiling handler package (`internal/admin/profiling.go`)

Create a new file `internal/admin/profiling.go` that owns all pprof
wiring. No other package imports `net/http/pprof` — we keep the
global `http.DefaultServeMux` pristine.

Exported symbols:

```go
// RegisterPprofRoutes mounts profiling handlers on mux under /debug/pprof.
// It is a no-op when cfg.Enabled is false.
// The caller is responsible for the authn / rate-limit middleware
// (the admin Server already chains those).
func RegisterPprofRoutes(mux *http.ServeMux, cfg config.AdminProfilingConfig,
    metrics ProfilingMetrics, audit ProfilingAudit, logger *logrus.Logger)

// ApplyRuntimeProfilingRates calls runtime.SetBlockProfileRate and
// runtime.SetMutexProfileFraction once at startup per cfg. Separate
// from RegisterPprofRoutes because runtime state lives at the process
// level, not per-mux.
func ApplyRuntimeProfilingRates(cfg config.AdminProfilingConfig, logger *logrus.Logger)
```

Internal structure:

- A semaphore `chan struct{}` of capacity `cfg.MaxConcurrentProfiles`.
  The `/profile` and `/trace` handlers acquire before delegating;
  acquire-fail returns 429 with `Retry-After: 1`.
- A `seconds` query-parameter validator that rejects values outside
  `[1, cfg.MaxProfileSeconds]` with 400.
- Per-endpoint wrapping that:
  1. Asserts `admin.IsAdminRequest(r)` — defence in depth; if false,
     emit a critical log and return 500 (indicates a wiring bug).
  2. Records start time.
  3. Delegates to the upstream pprof handler.
  4. Increments
     `s3_gateway_admin_pprof_requests_total{endpoint,outcome}`
     where outcome ∈ `{"ok","busy","bad_request","error"}`.
  5. Emits audit event `EventTypePprofFetch` with metadata
     `{endpoint, duration_ms, bytes_written}`.

Route table mounted (rooted at `/debug/pprof` — admin-server mux is
already the admin-side root):

| Path | Upstream handler |
|---|---|
| `/debug/pprof/` | `pprof.Index` |
| `/debug/pprof/cmdline` | `pprof.Cmdline` |
| `/debug/pprof/profile` | `pprof.Profile` (CPU, `seconds=`) |
| `/debug/pprof/symbol` | `pprof.Symbol` |
| `/debug/pprof/trace` | `pprof.Trace` (`seconds=`) |
| `/debug/pprof/heap` | `pprof.Handler("heap").ServeHTTP` |
| `/debug/pprof/goroutine` | `pprof.Handler("goroutine").ServeHTTP` |
| `/debug/pprof/allocs` | `pprof.Handler("allocs").ServeHTTP` |
| `/debug/pprof/block` | `pprof.Handler("block").ServeHTTP` |
| `/debug/pprof/mutex` | `pprof.Handler("mutex").ServeHTTP` |
| `/debug/pprof/threadcreate` | `pprof.Handler("threadcreate").ServeHTTP` |

Exit criteria: package compiles; `go vet ./...`,
`go vet -tags=fips ./...`, `go vet -tags=hsm ./...` clean; unit tests
for `RegisterPprofRoutes` with `cfg.Enabled=false` (no routes
mounted) and `=true` (11 routes mounted, each returns 200 under a
valid token against an `httptest.Server`).

### Phase C — Wire-up in `cmd/server/main.go`

Insert after the existing `admin.RegisterMPUAdminRoutes` call
(around `main.go:688-691`), guarded by the nested flag:

```go
if cfg.Admin.Profiling.Enabled {
    admin.ApplyRuntimeProfilingRates(cfg.Admin.Profiling, logger)
    admin.RegisterPprofRoutes(
        adminServer.Mux(),
        cfg.Admin.Profiling,
        adminProfilingMetricsAdapter{m: m},
        adminProfilingAuditAdapter{a: auditLogger},

        "block_rate":     cfg.Admin.Profiling.BlockRate,

        "max_seconds":    cfg.Admin.Profiling.MaxProfileSeconds,
        "max_concurrent": cfg.Admin.Profiling.MaxConcurrentProfiles,

```

## 5. Security Review Checklist

Every item must be green before the task is marked DONE.

- [ ] Profiling endpoints only mount on the admin listener. A unit
      test asserts that `internal/api` (the data-plane router) has
      zero handlers under `/debug/pprof/*`.
- [ ] `admin.profiling.enabled=true` without `admin.enabled=true`
      fails at `LoadConfig`.
- [ ] `admin.profiling.enabled=true` on non-loopback without TLS
      fails at `LoadConfig`.
- [ ] Bearer-auth middleware fires **before** the pprof handler
      chain — verified by hitting `/admin/debug/pprof/heap` with no
      `Authorization` header and getting 401 (not a profile).
- [ ] Rate limiter fires — verified as integration test in Phase D.
- [ ] `MaxConcurrentProfiles` bounds in-flight long-running profiles
      so an operator mistake cannot wedge the admin listener.
- [ ] `MaxProfileSeconds` bounds CPU / trace sampling windows so an
      attacker (with valid token) cannot pin a core indefinitely.
- [ ] Audit event emitted on every fetch, including the remote-addr
      where available — satisfies Adkins Ch. 15's "auditable debug
      interface" mandate.
- [ ] No new global state: the only runtime-level side-effect is
      `runtime.SetBlockProfileRate` / `runtime.SetMutexProfileFraction`,
      both explicitly documented in the config.
- [ ] FIPS build path unchanged (pprof does not call into
      `crypto/*`).

## 6. Backwards / Forwards Compatibility

- **Backwards.** Zero. Profiling is disabled by default; a config
  silent about `admin.profiling` defaults to `enabled: false`. Every
  existing deployment continues to behave identically.
- **Forwards.** The handler package `internal/admin/profiling.go` is
  a closed implementation detail. A future RBAC story (v0.7
  candidate) can wrap `RegisterPprofRoutes` with a role check without
  breaking callers. A future continuous-profiling integration
  (Pyroscope / Parca) can drive the existing endpoints via pull — no
  handler changes needed.

## 9. Open Questions for the Implementer

1. Should `MaxProfileSeconds` default to `60` (gentle) or `30`
   (matches Go default)? Recommendation: **60**, because MinIO /
   Garage conformance jobs routinely run > 30 s and a single CPU
   profile should be able to span one of those jobs.
2. Should we emit **per-endpoint** metrics, or a single
   `endpoint="pprof"` label with endpoint as a metric axis? Current
   plan: per-endpoint, explicit whitelist — keeps cardinality
   bounded and makes PromQL aggregation obvious.
3. Should block / mutex rates be **hot-reloadable** (SIGHUP path
   already exists in CFG-1's config reloader)? Current plan: **no**
   for v0.6. Changing them at runtime is safe but surprising in log
   output; file a follow-up if operators ask.
