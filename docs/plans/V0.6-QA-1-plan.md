**Existing committed baselines** (single scope, not per-provider):

- `docs/perf/v0.6-perf-2-baseline.txt` and

**What is missing**

1. **No baseline corpus.** Six completed plans (SEC-1, SEC-3,
   PERF-1, PERF-2, S3-1, S3-3) each said "feeds QA-1" and
   *stopped there*. Their `benchstat` deltas are parked without
   a destination. Today there is no directory you can point at
   and say "this is the v0.6 performance envelope".
2. **No provider stratification.** The only per-provider split
   anywhere is the load-test `Makefile` targets whose output is
   a free-form log. Operators cannot answer "does MinIO behave
   the same as Garage under PERF-1's streaming path?".
3. **No regression detection.** There is no job that compares a
   new benchmark run against a committed baseline and fails
   (nightly) or warns (PR). PERF-1's ≥ 50 % / ≥ 30 % gates and
   PERF-2's retry-metric claims live on inside unit tests, but
   once merged the invariant is not re-checked.
4. **Two benchmark gaps for "feeds QA-1" items that never
   shipped their benchmarks**:
   - `BenchmarkUploadPartCopy_Chunked_64KiB_Range`,
     `_Legacy_16MiB_Range`, `_Plaintext_Passthrough_1GiB`
     (`docs/plans/V0.6-S3-1-plan.md:351`) — not present in
     `internal/s3/client_bench_test.go`.
   - `BenchmarkMPUDecryptReader_100MiB` — PERF-1 shipped the
     encrypt side only (`mpu_encrypter_bench_test.go`); the
     decrypt side is symmetrical and benchstat-worthy.
5. **No docs page.** `docs/ROADMAP.md:119` promises one;
   `docs/PERFORMANCE.md` does not exist.
6. **No SLO baseline.** ADR 0010 defers circuit-breaker design
   to QA-1 data; PERF-2 open-question Q2
   (`V0.6-PERF-2-plan.md:1094-1096`) likewise. Both need a
   published p50 / p95 / p99 per operation per provider.

### 1.2 Why this is worth doing now

- Every future perf or security change otherwise re-litigates
  "did this regress?" qualitatively. The six "feeds QA-1"
  deferrals become eight when S3-3 and SEC-3 are re-run.
- ADR 0010 cannot progress to a circuit-breaker v0.7 decision
  without tail-latency evidence.
- Roadmap line `docs/ROADMAP.md:119` is the only outstanding
  Testing & Quality item for v0.6 closure other than QA-2.

## 2. Goals and Non-Goals

### 2.1 Goals

1. **Reproducible methodology.** Document a single procedure
   that produces the numbers; codify it in `make` targets and CI.
2. **Committed micro-benchmark baseline** in
   `docs/perf/v0.6-qa-1/micro-baseline.txt`, covering every
   `Benchmark*` function listed in §1.1 plus the two new ones
   in §5.2.
3. **Committed per-provider macro baseline** in
   `docs/perf/v0.6-qa-1/macro-<provider>.json`, one file per
   local provider (`minio`, `garage`, `rustfs`, `seaweedfs`).
   JSON (not log) so CI and the docs renderer can parse it.
4. **Nightly CI gate** that re-runs both micro and macro,
   compares via `benchstat` (micro) and a custom statistical
   comparator (macro), and fails the nightly job if any
   tracked statistic regresses by more than the threshold
   declared in §6.1. PR gate is *advisory only* (warn comment,
   never fails) because GitHub runners are noisy (see §3.3).
5. **A published docs page** at `docs/PERFORMANCE.md` that
   renders the baseline table per provider and links to the CI
   artefacts. Referenced from `README.md`, `docs/ROADMAP.md`,
   and ADR 0010.
6. **Two new benchmarks** (§5.2) to close the two benchmark
   gaps identified in §1.1 item 4.
7. **An SLO annex** (§8) that publishes p50 / p95 / p99
   numbers per operation per provider, suitable for operator
   sizing decisions and future ADR references.

### 2.2 Non-Goals (explicit; each with rationale)

- **External / paid-backend baselines in the initial delivery.**
  AWS / B2 / Wasabi / Hetzner have non-deterministic cross-region
  latency and billing implications. They are opt-in via env
  vars and populate a separate `docs/perf/v0.6-qa-1/macro-<provider>.json`
  only when nightly credentials are configured. Not part of
  DoD because secrets availability is outside this task's
  control.
- **New benchmarks for ops other than those listed in §5.**
  A perf suite that tries to cover every S3 verb loses focus.
  We fix the scope at: envelope encryption, MPU encrypt /
  decrypt streaming, PutObject with and without faults,
  UploadPartCopy, and the two load tests (range-read,
  multipart upload). Tagging / ObjectLock / Presign are out.
- **Continuous profiling / flamegraph archiving.** That is
  `/admin/debug/pprof` territory (V0.6-OBS-1) and belongs to
  a future V0.7-OBS task; QA-1 only *points at* it in the
  runbook when a regression fires (§9.2).
- **Absolute number-hunting ("gateway must hit X MB/s").**
  CI runners drift; any absolute threshold becomes a flake
  generator. QA-1 detects **deltas against the committed
  baseline for the same runner class**, not absolute values.
- **Mutation testing.** That is V0.6-QA-2.

## 3. Methodology

### 3.1 Micro-benchmarks (Go `testing.B`)

Adopt the recipe in *Efficient Go* (Plotka, 2022), ch. 8
"Benchmarking" — §"Too-High Variance", §"Tips and Tricks for
Microbenchmarking", §"Running Benchmarks for Different Inputs"
([https://learning.oreilly.com/library/view/-/9781098105709/](https://learning.oreilly.com/library/view/-/9781098105709/)).

Canonical invocation (per `internal/.../*_bench_test.go`):

```bash
go test -run='^$' \
        -bench='^Benchmark' \
        -benchmem \
        -benchtime=10s \
        -count=10 \
        -cpu=4 \
        -timeout=30m \
        ./internal/crypto/... ./internal/s3/... \
    > docs/perf/v0.6-qa-1/micro-<platform>.txt
```

Rationale for the parameters:

| Flag | Value | Why |
|---|---|---|
| `-benchtime=10s` | 10 s per op | Settles goroutine pool / GC; matches PERF-2's convention. |
| `-count=10` | 10 runs | Minimum `benchstat` asks for to compute a confidence interval. Odd-vs-even noise from CPU thermal boost averages out. |
| `-cpu=4` | Pinned 4 | Removes GOMAXPROCS drift between laptops and CI. |
| `-run='^$'` | No unit tests | Benchmarks only; no accidental unit-test overhead. |
| `-timeout=30m` | Headroom | Full suite is ≈ 18 min with `-count=10`; 30 min leaves slack. |

Comparison: `benchstat old.txt new.txt` → fail if any row's
geomean delta exceeds the thresholds in §6.1.

**Noise discipline** (ch. 8 §"Too-High Variance"):
- CI runner is pinned to a single `runs-on: ubuntu-latest`
  image family; QA-1 does **not** cross-compare across runner
  classes. Baseline files carry a header comment naming the
  runner + Go version; regeneration on runner migration is a
  manual `make bench-baseline` step.
- `-race` is **off** for benchmarks; `-race` inflates CPU 2-5×
  and destroys signal (Plotka §"Tips and Tricks").
- `GOGC=off` not used. We measure *with* GC because that is
  what production sees.

### 3.2 Macro / soak tests (in-process gateway, real backend)

Reuse `test/conformance/load_test.go`, already wired to
Testcontainers-Go via the `test/provider/` registry. Extend
the existing `SOAK_*` env vars with a structured-output toggle
(§5.1) so the load tests emit JSON alongside their log lines.

Fixed macro fixture for the baseline (matches PERF-1's 100 MiB
convention in `docs/issues/v0.6-issues.md:848` and PERF-2's
request-rate convention):

| Env var | Baseline value |
|---|---|
| `SOAK_WORKERS` | 10 |
| `SOAK_DURATION` | 60s |
| `SOAK_QPS` | 25 |
| `SOAK_OBJECT_SIZE` | 52428800 (50 MiB) |
| `SOAK_CHUNK_SIZE` | 65536 (64 KiB, engine default) |
| `SOAK_PART_SIZE` | 10485760 (10 MiB) |

**Per-provider isolation.** Each provider runs in its own
Testcontainer on its own port (`test/provider/minio.go`,
`garage.go`, `rustfs.go`, `seaweedfs.go`), so co-location does
not contaminate results. The conformance harness already
supports `GATEWAY_TEST_SKIP_*=1` (see `Makefile:60-66`) for
narrowing; QA-1 uses these to run one provider at a time to
avoid container-count CPU contention on the runner.

**Metrics captured per run**:
- Wall-clock throughput (MB/s) — from byte-count / duration.
- Per-request latency histogram (nanoseconds). QA-1 adds
  p50 / p95 / p99 to the existing summary log via
  `gonum.org/v1/gonum/stat` (already transitively available)
  or by hand with a sort-and-index (no new dependency).
- CPU seconds — from `runtime.NumCPU()` * wall × observed
  load (read from `/proc/self/stat` on Linux; skip on Darwin).
- Heap high-water mark — `runtime.ReadMemStats`/`HeapInuse`
  sampled every 500 ms, max retained.
- Retry counts — read from Prometheus registry after the run
  via `prometheus.DefaultGatherer.Gather()` (PERF-2 metrics
  already registered there).

*Statistical reasoning*: we prefer p95/p99 over mean+stddev
because request latency is usually log-normal, not Gaussian —
see *Latency* (Pekka Enberg, Manning 2025), ch. 3 "Measuring
Latency" ([https://learning.oreilly.com/library/view/-/9781633438088/](https://learning.oreilly.com/library/view/-/9781633438088/)).
And we report p99 explicitly, not just p95, per *Site
Reliability Engineering* (Beyer et al., 2016), ch. 6 "Monitoring
Distributed Systems" — §"The Four Golden Signals"
([https://learning.oreilly.com/library/view/-/9781491929117/](https://learning.oreilly.com/library/view/-/9781491929117/))
— because p99 is what dominates tail-aware retry policy
tuning (see ADR 0010 context).

### 3.3 Why PR-gate is advisory, nightly is strict

*Pro .NET Benchmarking* (Akinshin, Apress 2019), ch. 6
"Analysis Methodology" — §"Noise and Outliers"
([https://learning.oreilly.com/library/view/-/9781484249413/](https://learning.oreilly.com/library/view/-/9781484249413/))
— documents that shared-tenant CI runners produce per-benchmark
standard deviations of 5-20 %, enough to produce false-positives
on any single-run comparison. The nightly job runs the full
10-count harness (≈ 18 min micro + 4×≈90 s macro = ≈ 25 min)
which is long enough that benchstat's t-test filters out most
runner noise. The PR gate runs a *quick* 3-count variant
(≈ 6 min micro only) whose sole purpose is to *flag* obviously
catastrophic regressions; it never fails, only posts a delta
comment.

### 3.4 How we avoid "benchmarks lie"

*Efficient Go* ch. 7 §"Benchmarks Lie" lists three hazards we
address:

1. **Compiler elimination of measured work.** Every new
   benchmark includes a `b.ReportAllocs()` and sinks its
   result into an exported `runtime.KeepAlive`-backed
   package-level sink (mirroring the pattern in
   `engine_bench_test.go` already).
2. **Non-deterministic input sizes.** All benchmarks either
   take a fixed `int` size parameter (as `_5MiB`/`_100MiB`)
   or use `b.SetBytes(int64(plainLen))` so `MB/s` is
   normalised.
3. **Warm-up not accounted for.** `-benchtime=10s` plus
   `b.ResetTimer()` after any setup block; the existing
   benchmarks already follow this — QA-1 enforces it in a
   lint check (§5.4).

### 4.1 Why a `v0.6-qa-1/` subdirectory

Per-milestone archiving mirrors `docs/perf/v0.6-perf-2-*.txt`.
v0.7 will add `docs/perf/v0.7-qa-1/` with updated numbers, and
the `docs/PERFORMANCE.md` page picks the latest by convention.
This is explicitly *not* a single-source-of-truth `current.json`
style layout — version pinning is deliberate so ADR references
remain stable.

## 5.2 Two new benchmarks (detail)

### `BenchmarkMPUDecryptReader_100MiB`

Symmetric to `BenchmarkMPUEncryptReader_100MiB`
(`internal/crypto/mpu_encrypter_bench_test.go:19-21`). Fixture:
build a valid 100 MiB ciphertext once outside the timed loop;
inside the loop, reconstruct a `NewMPUDecryptReader` from it
and `io.Copy(io.Discard, r)`. `b.SetBytes(100 * 1024 * 1024)`.

### `BenchmarkUploadPartCopy_*`

Lives in a new `internal/s3/upload_part_copy_bench_test.go`
because the existing `client_bench_test.go` is scoped to
`s3.Client` calls; UploadPartCopy runs through the
proxy-client layer. Fixture: in-memory `httptest.Server`
serving a deterministic 1 GiB source object (per-request
stream, not materialised). Three variants match the
sub-table in `V0.6-S3-1-plan.md:351`:

- `_Chunked_64KiB_Range` — chunked-AEAD source, 64 KiB range
  per `UploadPartCopy` request, 16 sequential parts.
- `_Legacy_16MiB_Range` — legacy-AEAD source, 16 MiB range,
  1 part (hits the `MaxLegacyCopySourceBytes` ceiling).
- `_Plaintext_Passthrough_1GiB` — plaintext source, single
  backend-native `UploadPartCopy` (no gateway rewrite); the
  fastest of the three and the baseline for "what native
  looks like".

All three `b.SetBytes` on the payload size so throughput in
MB/s comes out of `go test -bench`.

## 6. Regression Thresholds

### 6.1 Threshold table

| Metric class | Tracked statistic | Regress = fail nightly if |
|---|---|---|
| Micro ns/op | `benchstat` geomean p < 0.05 | Δ > +15 % |
| Micro B/op | `benchstat` geomean p < 0.05 | Δ > +20 % |
| Micro allocs/op | `benchstat` geomean p < 0.05 | Δ > 0 (any new allocation) |
| Macro throughput | `throughput_mbps` | Δ < −15 % |
| Macro latency p95 | `latency_ns.p95` | Δ > +20 % |
| Macro latency p99 | `latency_ns.p99` | Δ > +25 % (tail noisier) |
| Macro heap HWM | `heap_inuse_max_bytes` | Δ > +25 % |
| Macro errors | `errors` | any non-zero where baseline = 0 |
| Macro retries_total | `retries_total` | Δ > +50 % (informational-only; retry noise is high) |

Rationale: the asymmetric allocs/op threshold (any new alloc
fails) mirrors PERF-1's heap-bound posture — a new allocation
in `NewMPUPartEncryptReader`'s hot path silently defeats the
streaming rewrite; we want that to trip immediately. Other
thresholds are generous enough to survive runner noise (see
§3.3).

### 6.2 Manual override

A PR author who knows a regression is intentional (e.g. a
security hardening that costs throughput) files the baseline
update in the same PR:

```
make bench-macro-minio bench-macro-garage bench-macro-rustfs bench-macro-seaweedfs
git add docs/perf/v0.6-qa-1/macro-*.json
```

Plus a CHANGELOG entry explaining the delta. The nightly then
runs green against the updated baseline. This is the same
workflow as committing `coverage.out`-derived thresholds.

## 7. Tests

### 7.1 Unit tests (Phase A)

- `TestSummaryRecord_Percentiles` — golden test for p50/p95/p99
  on a synthetic latency slice (sizes: 100, 10000; values:
  linear, log-normal).
- `TestSummaryRecord_HeapSampler` — sampler goroutine starts,
  records a value ≥ baseline, stops on ctx cancel within
  100 ms.
- `TestSummaryRecord_PromReader` — empty registry ⇒ 0 retries;
  pre-populated `retries_total` counter ⇒ correct reading.
- `TestSummaryRecord_JSONSchema` — marshalled object validates
  against the §4.2 schema (schema file committed under
  `test/conformance/testdata/macro.schema.json`; validator is
  `github.com/santhosh-tekuri/jsonschema/v5` — zero-dep, test-
  scope only).

### 7.2 Benchmarks-of-benchmarks

Every new benchmark is also exercised under
`go test -run=^Test -tags=conformance` in a dedicated
`*_bench_smoke_test.go` companion that calls the same fixture
with a 1 µs ceiling to prove it terminates and produces a
non-empty stream. This catches the class of "my benchmark
loop never executes" bugs that `go test -bench` silently hides.

### 7.3 Nightly workflow self-test (Phase D)

- `scripts/bench-compare_test.sh` (shell unit tests via
  [`bats-core`](https://github.com/bats-core/bats-core), which
  is `apt-install`-able on `ubuntu-latest`):
  - Golden "same as baseline" ⇒ exit 0.
  - Golden "throughput −20 % minio" ⇒ exit 1.
  - Golden "latency p99 +30 % garage" ⇒ exit 1.
  - Golden "allocs +1 on one micro row" ⇒ exit 1.
  - Golden "errors 3 where baseline 0" ⇒ exit 1.

### 7.4 PR advisory workflow self-test (Phase E)

The advisory job includes a deliberate `|| true` on the
compare step; QA-1 ships a synthetic `ci/perf-smoke-branch`
with a micro-regression pre-landed, verified manually at
merge time that the PR comment appears. Documented in
`docs/PERFORMANCE.md`.

### 7.5 Full-suite proofs of non-regression

- `go test ./... -race -short` green.
- `go test -tags=conformance -race ./test/conformance/...` green
  against MinIO (PR gate parity).
- `make bench-baseline` on a developer laptop produces all
  five artefacts, `diff` against committed is empty the
  second time it is run on the same machine within 1 hour
  (thermal-stability sanity check; not a strict test, an
  informational one).

## 9. Risks and Mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| 1 | `ubuntu-latest` runner noise produces nightly false-positives | H | M | 10× count on nightly + benchstat t-test; thresholds asymmetric and tuned on first month of data; issues filed with `perf-regression` label are triaged, not auto-ack'd |
| 2 | Committed baseline goes stale after unrelated Go-runtime upgrade | M | L | Baseline file header pins `go_version`; CI warns (non-failing) when `go.mod`'s Go version differs from the baseline's recorded version; updating is one `make bench-baseline` commit |
| 3 | Testcontainers-Go spin-up contention on runner consumes > 6 GB RAM for 4 parallel providers | M | M | Phase C runs macros **serially**; Phase D matrix jobs each run on a fresh runner (each job = one provider, one container at a time) |
| 4 | Committed large JSON files bloat the repo | L | L | Each macro JSON ≈ 2 KB; four files per milestone ≈ 8 KB; micro .txt ≈ 12 KB; total < 25 KB per milestone. Git LFS unnecessary |
| 5 | External providers (AWS/B2/Wasabi) drift separately, invalidate cross-provider comparisons | M | L | External providers are opt-in via secrets; their JSON files live alongside but are marked `"runner": "github-nightly-external"`; the docs table distinguishes local vs external rows |
| 6 | Developer laptops cannot reproduce the numbers (Apple silicon vs amd64) | H | L | Header records `GOARCH`/`GOOS`; `docs/PERFORMANCE.md` states "baseline is from linux/amd64 CI only; laptop runs are informational"; developers comparing laptop deltas just use their own previous run as the baseline |
| 7 | New benchmark accidentally elides its timed work via compiler inlining | L | M | `scripts/bench-lint.sh` (Phase B) + the smoke test in §7.2; both fail PR CI |
| 8 | `benchstat` (`golang.org/x/perf/cmd/benchstat`) tool drift breaks the comparator | L | L | Pin version in `go.mod`'s `tools.go` and in the workflow `go install ...@v0.0.0-YYYYMMDD-...` step |

## 12. Open Questions

- **Q1.** Should the PR advisory comment include a link to the
  failing job's pprof artefact? *Proposed:* no, not in QA-1 —
  pprof capture on every PR-time benchmark run doubles the
  wall clock. File a V0.7-OBS task if operators ask.
- **Q2.** Where should the baseline go after v0.7 — overwrite
  `v0.6-qa-1/` in place, or carry forward `v0.7-qa-1/`?
  *Proposed:* carry forward. ADR references must remain
  stable; overwriting silently invalidates "ADR 0010 cites
  p99 MinIO = X ms".
- **Q3.** Do we publish the runner class in the docs page?
  *Proposed:* yes, in the methodology section. Operators
  reading p99 numbers need to know they are from
  `ubuntu-latest` (2 vCPU, 7 GB) — real-world MinIO on
  c6i.large will often be faster.
- **Q4.** Should the nightly also run external providers
  (AWS/B2/Wasabi/Hetzner) when credentials are configured?
  *Proposed:* separate opt-in matrix job, disabled by default.
  Credentials-gated exactly as
  `.github/workflows/conformance.yml:100-142` already does.
  Tracked but not blocking for QA-1's initial delivery.
- **Q5.** Should we adopt `benchstat -filter` to exclude rows
  with p > 0.1? *Proposed:* yes, add the flag in
  `scripts/bench-compare.sh`. Quiets benchmarks whose run-to-
  run variance is too high to draw a conclusion (*Efficient
  Go* ch. 8 §"Too-High Variance" recommends this explicitly).

```bash
# Prereqs: Docker, go1.25+, benchstat, jq.
go install golang.org/x/perf/cmd/benchstat@latest
make bench-baseline

# Inspect deltas vs committed:
benchstat \
  docs/perf/v0.6-qa-1/micro-baseline.txt \
  /tmp/s3-encryption-gateway-micro-new.txt
```

### B. Trigger the nightly manually

```bash
gh workflow run performance-baseline.yml --ref main
gh run watch
```

### C. Update the committed baseline after an intentional regression

```bash
# After merging a PR that deliberately changes perf:
make bench-baseline
git add docs/perf/v0.6-qa-1/
git commit -m "perf: refresh QA-1 baseline after <feature>"
```
