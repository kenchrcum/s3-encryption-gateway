## 1. Background & Design Rationale

### 1.1 Problem Statement

The gateway currently operates in two modes:

| Mode | Config | Behaviour |
|------|--------|-----------|
| **Single-bucket** | `proxied_bucket` set | Validates SigV4 if `backend.access_key` is configured; falls through for all other requests |
| **Multi-bucket / passthrough** | `backend.use_client_credentials: true` | Forwards client credentials directly to the backend; V4 requests rejected (host header mismatch) |

In both modes, a request that lacks an `Authorization` header or `AWSAccessKeyId` query parameter reaches the handler with a valid backend S3 client, no auth check performed. This is CWE-306.

### 1.2 Target Design

The gateway manages its own **credential store** — a list of (access-key, secret-key) pairs. Every inbound request must:

1. Present an access key (extracted via existing `ExtractCredentials`).
2. Have that access key recognised by the credential store.
3. Have its signature validated against the corresponding stored secret (V4 HMAC or V2/query-param HMAC).

The **backend** scope (single vs. multi-bucket) is determined solely by the backend credentials configured in `backend.access_key` / `backend.secret_key` — not by a mode flag. `proxied_bucket` survives as an optional bucket-level filter. `backend.use_client_credentials` is removed entirely.

### 1.3 Literature Backing

The design follows the token-store pattern described in *API Security in Action* (Neil Madden, Manning 2021), applied to HMAC-based credential validation:

- A **credential store** is the canonical source of truth for which clients are authorised.
- Each request is **authenticated** against the store before any business logic runs.
- The store is consulted in **constant time** to prevent timing oracles (cf. `crypto/hmac.Equal`).
- Credentials are **injected at deploy time** via environment variables or mounted secrets — never committed in plaintext.

## 2. Architecture Overview

```
Client ──► [AuthMiddleware] ──► [Handler] ──► Backend S3
              │
              ├─ ExtractCredentials(r)          (existing, unchanged)
              ├─ store.Lookup(accessKey)         (new)
              ├─ ValidateSignature(r, secret)    (reused + extended)
              └─ on failure → 403 / 401 (S3 error XML)
```

The `AuthMiddleware` is inserted into the HTTP chain **before** any handler is called. The handler's `getS3Client` is simplified: it assumes the middleware has already authenticated the request, and always returns the configured backend client.

**Files:** `internal/config/config.go`

#### Steps

1. Add `GatewayCredential` struct (as above) to `config.go`.

### Phase 2 — CredentialStore Interface & StaticCredentialStore

**Files:** `internal/api/credential_store.go` (new)

#### Interface

```go
package api

// CredentialStore is a read-only lookup of gateway-managed credentials.
// Implementations must be safe for concurrent use.
type CredentialStore interface {
    // Lookup returns the secret key for the given access key, or
    // ErrUnknownAccessKey if the key is not registered.
    Lookup(accessKey string) (secretKey string, label string, err error)
}
```

#### StaticCredentialStore

```go
// StaticCredentialStore is an in-memory credential store loaded at startup.
// Keys are compared using a constant-time map lookup (the map key is the
// access key; the value comparison in ValidateSignature is constant-time).
type StaticCredentialStore struct {
    // m maps access key → (secretKey, label).
    m map[string]credentialEntry
}

type credentialEntry struct {
    secretKey string
    label     string
}

func NewStaticCredentialStore(creds []config.GatewayCredential) (*StaticCredentialStore, error) {
    if len(creds) == 0 {
        return nil, fmt.Errorf("credential store requires at least one credential")
    }
    m := make(map[string]credentialEntry, len(creds))
    for _, c := range creds {
        if c.AccessKey == "" || c.SecretKey == "" {
            return nil, fmt.Errorf("credential entry is missing access key or secret key")
        }
        m[c.AccessKey] = credentialEntry{secretKey: c.SecretKey, label: c.Label}
    }
    return &StaticCredentialStore{m: m}, nil
}

func (s *StaticCredentialStore) Lookup(accessKey string) (string, string, error) {
    entry, ok := s.m[accessKey]
    if !ok {
        return "", "", ErrUnknownAccessKey
    }
    return entry.secretKey, entry.label, nil
}
```

**Implementation notes:**
- Map access is O(1) but the Go map lookup for strings is **not** constant-time. This is acceptable: timing on the access-key lookup leaks whether the key is known, but an attacker can enumerate access keys trivially via the error response (`InvalidAccessKeyId` vs `SignatureDoesNotMatch`) anyway — this is standard AWS S3 behaviour. Constant-time comparison is mandatory only for the **secret key** itself (used in HMAC computation).
- Add unit tests: known key → returns secret; unknown key → `ErrUnknownAccessKey`; empty store → construction error.

### Phase 3 — AuthMiddleware

**Files:** `internal/api/middleware.go` (add to existing file) or `internal/api/auth_middleware.go` (new file)

#### Design

```go
// AuthMiddleware returns an HTTP middleware that validates every request
// against the credential store before passing it to next.
//
// Auth flow:
//  1. ExtractCredentials — extract access key from request.
//  2. store.Lookup       — check access key is known.
//  3. ValidateSignature  — verify HMAC (V4 or V2) using stored secret.
//  4. Attach resolved label to request context for audit logging.
//  5. Call next; on any failure return S3-formatted error.
func AuthMiddleware(store CredentialStore, clockSkew time.Duration, logger *logrus.Logger) func(http.Handler) http.Handler
```

#### Detailed logic

```
request arrives
  │
  ├─ ExtractCredentials(r)
  │     ├─ no credentials → return ErrMissingCredentials (→ AccessDenied 403)
  │     └─ creds extracted (access key always present; secret key may be empty for V4 header)
  │
  ├─ store.Lookup(creds.AccessKey)
  │     └─ not found → return ErrUnknownAccessKey (→ InvalidAccessKeyId 403)
  │
  ├─ IsSignatureV4Request(r)?          ← checks "AWS4-HMAC-SHA256" prefix ONLY
  │     ├─ yes → ValidateSignatureV4(r, storedSecret, clockSkew)
  │     │           └─ fail → return ErrSignatureMismatch (→ SignatureDoesNotMatch 403)
  │     └─ no  → IsSignatureV2Request(r)?   ← checks "AWS " prefix OR AWSAccessKeyId param
  │                 ├─ yes → ValidateSignatureV2(r, storedSecret)
  │                 │           └─ fail → return ErrSignatureMismatch (→ SignatureDoesNotMatch 403)
  │                 └─ no  → return ErrMissingCredentials (→ AccessDenied 403)
  │
  ├─ attach credential label to context (for audit log)
  └─ call next(w, r)
```

**Key constraint:** the auth middleware must run **before** the handler reads the request body or makes any backend call. Insert it as the outermost application-level middleware (inside rate-limit, tracing, and logging middlewares, outside handler routing).

### Phase 4 — ValidateSignatureV2 (new) & IsSignatureV2Request predicate

**Files:** `internal/api/auth.go`, `internal/api/credentials.go`

The existing `ValidateSignatureV4` covers SigV4 header auth and presigned V4. The new `ValidateSignatureV2` covers:

- **Query-parameter style** (`AWSAccessKeyId=...&AWSSecretAccessKey=...&Signature=...`): This is a non-standard extension used by some S3-compatible SDKs and the existing `use_client_credentials` mode. The signature covers `StringToSign = METHOD\n\nContent-Type\nDate\n/bucket/key`.
- **Authorization header V2** (`AWS ACCESS_KEY:SIGNATURE`): Older SDKs and some tools still use this.

```go
// ValidateSignatureV2 validates an AWS Signature Version 2 request.
// It supports both the Authorization header and query-parameter styles.
// secretKey is the stored secret for the access key identified in the request.
func ValidateSignatureV2(r *http.Request, secretKey string) error
```

**Implementation notes:**
- StringToSign = `HTTP-Verb\n\nContent-Type\nDate\nCanonicalizedAmzHeaders\nCanonicalizedResource`
- The HMAC is `Base64(HMAC-SHA1(secretKey, StringToSign))`.
- Replay protection: check the `Date` header (or `Expires` query param) is within a configurable window (default: same `ClockSkewTolerance`).
- Use `hmac.Equal` for the final comparison.
- If neither V4 nor V2 signature is present but an access key was extracted (e.g. bare query-param `AWSAccessKeyId` without `Signature`), return `ErrMissingCredentials`.

#### Bug fix: `IsSignatureV4Request` in `credentials.go`

**IMPORTANT:** The existing `IsSignatureV4Request` in `internal/api/credentials.go` returns `true` for **both** the `"AWS4-HMAC-SHA256"` prefix (SigV4) **and** the `"AWS "` prefix (SigV2 Authorization header). This means the Phase 3 dispatch `IsSignatureV4Request(r)? yes→V4, no→V2` would incorrectly route V2 Auth-header requests into V4 validation.

Two changes are required in `internal/api/credentials.go`:

1. **Fix `IsSignatureV4Request`** — restrict to `"AWS4-HMAC-SHA256"` prefix only:

```go
// IsSignatureV4Request reports whether the request carries a SigV4
// Authorization header ("AWS4-HMAC-SHA256 ...") or SigV4 query parameters
// (X-Amz-Algorithm=AWS4-HMAC-SHA256).
func IsSignatureV4Request(r *http.Request) bool {
    auth := r.Header.Get("Authorization")
    return strings.HasPrefix(auth, "AWS4-HMAC-SHA256") ||
        r.URL.Query().Get("X-Amz-Algorithm") == "AWS4-HMAC-SHA256"
}
```

2. **Add `IsSignatureV2Request`** — detects V2 Auth-header and V2 query-param style:

```go
// IsSignatureV2Request reports whether the request carries a SigV2
// Authorization header ("AWS ACCESS_KEY:SIG") or V2 query parameters
// (AWSAccessKeyId + Signature).
func IsSignatureV2Request(r *http.Request) bool {
    auth := r.Header.Get("Authorization")
    if strings.HasPrefix(auth, "AWS ") {
        return true
    }
    q := r.URL.Query()
    return q.Get("AWSAccessKeyId") != "" && q.Get("Signature") != ""
}
```

These two predicates together form the complete dispatch in `AuthMiddleware` (see Phase 3).

### Phase 5 — Refactor `getS3Client` in `handlers.go`

**Files:** `internal/api/handlers.go`

The function `getS3Client` (lines 580–670) is radically simplified:

**Before:** Complex branching over `UseClientCredentials`, V4/V2 detection, per-request credential extraction.

**After:**

```go
// getS3Client returns the configured backend S3 client.
// Authentication has already been validated by AuthMiddleware before this
// point; this function only needs to return the pre-configured client.
func (h *Handler) getS3Client(r *http.Request) (s3.Client, error) {
    if h.s3Client != nil {
        return h.s3Client, nil
    }
    if h.clientFactory != nil {
        return h.clientFactory.GetClient()
    }
    return nil, fmt.Errorf("no S3 client available")
}
```

All credential-extraction and signature-validation logic is removed from `getS3Client`. The `ErrSigV4NotSupportedWithPassthrough` sentinel becomes unused and can be removed.

### Phase 6 — `proxied_bucket` as Optional Bucket Filter

**Files:** `internal/api/handlers.go` (or a new `bucket_filter_middleware.go`)

`proxied_bucket` is retained as a non-breaking optional filter. If set, a middleware inserted **after** `AuthMiddleware` rejects requests targeting a different bucket with `AccessDenied`. If not set, all authenticated requests pass.

**Before creating a new symbol:** check whether the existing `middleware.BucketValidationMiddleware` in `cmd/server/main.go` (already present in the outer middleware chain) already covers this behaviour. If it does, Phase 6 may only need to:

1. Confirm `BucketValidationMiddleware` is positioned **after** `AuthMiddleware` in the chain (authentication must always precede bucket filtering).
2. Extend it to handle the no-op case when `proxied_bucket` is empty, rather than creating a parallel `BucketFilterMiddleware`.

If the existing middleware does not provide equivalent behaviour, define a new symbol:

```go
// BucketFilterMiddleware, when proxiedBucket is non-empty, rejects any
// authenticated request targeting a bucket other than proxiedBucket.
// When proxiedBucket is empty, it is a no-op pass-through.
func BucketFilterMiddleware(proxiedBucket string) func(http.Handler) http.Handler
```

### Phase 7 — Wire into `cmd/server/main.go`

**Files:** `cmd/server/main.go`

1. Call `cfg.ResolvedCredentials()` to obtain the credential slice (secrets populated from env vars).
2. Construct `StaticCredentialStore` from the resolved credentials; fail-fast on error.
3. **Zero the resolved credential slice immediately after store construction** — consistent with the existing `zeroBytes(activePassword)` pattern used for the backend password:

```go
resolved := cfg.ResolvedCredentials()
store, err := api.NewStaticCredentialStore(resolved)
if err != nil {
    log.Fatal(err)
}
// Zero plaintext secrets from the transient slice; the store holds its own copy.
for i := range resolved {
    resolved[i].SecretKey = strings.Repeat("\x00", len(resolved[i].SecretKey))
}
resolved = nil
```

4. Insert `AuthMiddleware(store, cfg.Auth.ClockSkewTolerance, logger)` at the correct position in the middleware chain.
5. Insert `BucketFilterMiddleware(cfg.ProxiedBucket)` after auth middleware.
6. Remove all `UseClientCredentials` wiring.

### Phase 8 — Helm Chart Updates

**Files:**

#### values.yaml — credential shape

The chart uses the `configValue` pattern for every config leaf (`{value: "", valueFrom: {secretKeyRef: ...}}`). Credentials must follow this same pattern. Do **not** use a Go-layer `secretKeyEnv` indirection in `values.yaml` — the `valueFrom` wiring happens entirely in the Helm template.

Use **indexed environment variable names** so the Go config resolver can map them deterministically (`GW_CRED_0_ACCESS_KEY` / `GW_CRED_0_SECRET_KEY`, `GW_CRED_1_ACCESS_KEY` / `GW_CRED_1_SECRET_KEY`, …):

```yaml
config:
  auth:
    clockSkewTolerance:
      value: "5m"
    # Gateway credential store — at least one entry required.
    # accessKey and secretKey follow the configValue pattern (value/valueFrom).
    # label is a plain string (no secret — used only for audit logs).
    credentials:
      - accessKey:
          value: ""
          valueFrom:
            secretKeyRef:
              name: ""
              key: ""
        secretKey:
          value: ""           # Dev only — use valueFrom in production
          valueFrom:
            secretKeyRef:
              name: ""
              key: ""
        label: "default"
```

#### deployment.yaml — env var emission

The template must:

1. **Remove the existing `$useClientCreds` conditional block** (≈15 lines controlling `BACKEND_ACCESS_KEY`, `BACKEND_SECRET_KEY`, and `BACKEND_USE_CLIENT_CREDENTIALS` env vars). This block becomes dead code once `UseClientCredentials` is removed.

2. **Emit indexed env vars** for each credential entry using a `range` loop:

```yaml
{{- range $i, $cred := .Values.config.auth.credentials }}
- name: {{ printf "GW_CRED_%d_ACCESS_KEY" $i }}
  {{- include "s3-encryption-gateway.envVar" $cred.accessKey | nindent 2 }}
- name: {{ printf "GW_CRED_%d_SECRET_KEY" $i }}
  {{- include "s3-encryption-gateway.envVar" $cred.secretKey | nindent 2 }}
{{- end }}
```

   The `s3-encryption-gateway.envVar` helper already handles `value` vs `valueFrom` branching — reuse it, do not duplicate the logic.

3. The Go `loadFromEnv` implementation (Phase 1) must read `GW_CRED_N_SECRET_KEY` for each indexed credential to resolve secrets; the `SecretKeyEnv` field on `GatewayCredential` is **not** used by Helm (it remains available for bare-metal / non-Helm deployments where operators set env vars manually).

#### values.schema.json additions

The `clockSkewTolerance` leaf and each `accessKey`/`secretKey` leaf in the credential items must use the `configValue` ref pattern — not plain `"type": "string"`:

```json
"auth": {
  "type": "object",
  "properties": {
    "clockSkewTolerance": {
      "allOf": [{ "$ref": "#/$defs/configValue" }]
    },
    "credentials": {
      "type": "array",
      "minItems": 1,
      "items": {
        "type": "object",
        "required": ["accessKey"],
        "properties": {
          "accessKey":  { "allOf": [{ "$ref": "#/$defs/configValue" }] },
          "secretKey":  { "allOf": [{ "$ref": "#/$defs/configValue" }] },
          "label":      { "type": "string" }
        }
      }
    }
  }
}
```

## 5. Migration Path for Existing Deployments

### 5.1 Pre-V1.0 (unauthenticated / `use_client_credentials: true`)

Operators currently relying on unauthenticated access must add an `auth.credentials` block before upgrading to the release that ships V1.0-AUTH-1.

**Step-by-step:**

```bash
# 1. Generate a strong gateway access key and secret key
export GW_ACCESS_KEY=$(openssl rand -hex 16)
export GW_SECRET_KEY=$(openssl rand -hex 32)

# 2. Update the gateway config (or Helm values) to add the credential
# 3. Configure all clients to use GW_ACCESS_KEY / GW_SECRET_KEY
# 4. Deploy the updated gateway
# 5. Verify: curl (unauthenticated) returns AccessDenied
```

### 5.2 Pre-V1.0 (`use_client_credentials: true`)

The gateway in this mode forwarded client credentials to the backend. Operators must:

1. Remove `backend.use_client_credentials: true` from config.
2. Add `auth.credentials` with the same key pairs the clients were using.
3. Ensure `backend.access_key` / `backend.secret_key` are configured with credentials that have the required backend scope.

The backend credential scope (which buckets the backend key can access) now controls multi-bucket vs. single-bucket behaviour.

### 5.3 Pre-V1.0 (`proxied_bucket` set)

No migration required. `proxied_bucket` continues to work as an optional bucket filter. Just add `auth.credentials`.

## 6. Backward Compatibility Summary

| Feature | Status in V1.0-AUTH-1 |
|---------|----------------------|
| `proxied_bucket` | **Retained** — optional bucket filter |
| `backend.use_client_credentials` | **Removed** — startup error if present in config |
| Unauthenticated access | **Blocked** — any request without valid credentials returns 403 |
| SigV4 header auth | **Supported** — validated against credential store |
| SigV4 presigned URLs | **Supported** — validated against credential store |
| SigV2 query-param auth | **Supported** — validated against credential store (new) |
| SigV2 Authorization header | **Supported** — validated against credential store (new) |
| Multiple gateway credentials | **Supported** — any configured credential is accepted |
