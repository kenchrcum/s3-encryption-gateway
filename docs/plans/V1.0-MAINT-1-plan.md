## 1. Context & Motivation

Three v1.0 security issues introduced **breaking changes** to the on-disk
format of encrypted objects.  Objects written by earlier versions of the
gateway carry none of these new markers.  The gateway's decrypt path
maintains a dual-read window for each change — old objects can still be
read — but this window must eventually close to harden the security
posture.

| Issue | Change introduced | New metadata marker | Legacy condition (absence means…) |
|---|---|---|---|
| **SEC-2** | Per-chunk IV derivation switched from XOR to HKDF-SHA256 | `x-amz-meta-enc-iv-deriv = "hkdf-sha256"` | Marker absent/empty → XOR path (deprecated, remove in v3.0) |
| **SEC-4** | AAD fallback now gated by an explicit marker | `x-amz-meta-enc-legacy-no-aad = "true"` | Marker absent → no-AAD fallback blocked for new objects; pre-SEC-4 objects that were written without AAD will fail to decrypt unless this marker is back-tagged. No-AAD fallback path deprecated, remove in v3.0 (same policy as XOR-IV and fallback-v1) |
| **SEC-27** | Metadata-fallback encrypt path changed from outer-AEAD-wrapped to streaming (v2) | `x-amz-meta-encryption-fallback-version = "2"` | Marker absent/`"1"` → legacy outer-AEAD path (`decryptFallbackV1`, deprecated, remove in v3.0) |

The migration tool (`s3eg-migrate`) gives operators the ability to
**bring all existing objects up to the current format** in a single
offline pass, after which the legacy read paths can eventually be
retired.

### Supported gateway versions & upgrade paths

The breaking changes land in **v0.7.0**.  v0.6.x (specifically v0.6.4,
the final patch of the v0.6 line) retains full dual-read support for all
three legacy formats **and** ships a backport of the migration tool so
that operators can migrate before upgrading.

The tool is therefore designed to run against **either** gateway version:

| Scenario | Gateway version when tool runs | What the tool reads | What the tool writes |
|---|---|---|---|
| **Option A — migrate-before-upgrade** | v0.6.4 | Legacy objects (XOR-IV / no-AAD / fallback-v1) | Modern objects (HKDF / AAD / fallback-v2) — already readable by v0.6.4 via its dual-read window |
| **Option B — migrate-after-upgrade** | v0.7.0 | Legacy objects still in the bucket | Modern objects — the gateway can already read legacy objects in v0.7.0 via its backward-compat window; migration closes that window permanently |

**Operator upgrade sequences:**

```
Option A (safest — zero-downtime migration):
  1. Upgrade gateway to v0.6.4
  2. Run: s3eg-migrate --gateway-version 0.6.4 --migration-class all
  3. Verify: s3eg-migrate --dry-run  →  0 legacy objects
  4. Upgrade gateway to v0.7.0

Option B (simpler — acceptable when downtime window exists or
          bucket is not write-heavy):
  1. Upgrade gateway to v0.7.0
  2. Run: s3eg-migrate --gateway-version 0.7.0 --migration-class all
  3. Verify: s3eg-migrate --dry-run  →  0 legacy objects
```

Both options leave the bucket in an identical fully-modern state.
Option A is preferred for production deployments with continuous writes
because the gateway never serves partially-migrated objects under v0.7.0
semantics.  Option B is acceptable for environments with a maintenance
window or read-mostly buckets.

> **Key invariant:** the tool never writes an object in a format that
> the currently-running gateway cannot immediately read back.  In Option A
> the migrated (modern) format is readable by v0.6.4.  In Option B the
> gateway is already at v0.7.0 which reads both legacy and modern.  This
> invariant is enforced by the `--gateway-version` flag (see §4 Phase 5).

## 2. Legacy Object Recognition

This section is the core analytical output of the plan.  It defines
the exact metadata fingerprint that identifies each legacy class and
tells the tool how to handle each one.

### 2.1 Object classes

Every encrypted object carries `x-amz-meta-encrypted = "true"` (or the
compact form `x-amz-meta-e = "true"`).  Objects lacking this marker are
**plaintext** and are outside this tool's scope (in-place encryption is
a separate future issue).

Within encrypted objects, the combination of metadata markers determines
the object's class.  The tool must **inspect the HEAD metadata** of each
object and classify it before deciding whether to migrate.

```
              ┌──────────────────────────────────────────────────────────┐
              │           Encrypted Object (x-amz-meta-encrypted=true)  │
              └───────────────────────────┬──────────────────────────────┘
                                          │
                    ┌─────────────────────┴──────────────────────┐
                    │                                            │
             x-amz-meta-                              x-amz-meta-
          encryption-fallback                     encryption-fallback
               absent                                  = "true"
                    │                                            │
          ┌─────────┴──────────┐                  ┌─────────────┴───────────┐
          │                    │                  │                         │
    x-amz-meta-          x-amz-meta-       fallback-version       fallback-version
  enc-iv-deriv=       enc-legacy-no-aad     absent / "1"              = "2"
   "hkdf-sha256"        = "true"                  │                         │
          │                    │            CLASS C: SEC-27           already
     MODERN                CLASS B:          legacy outer-          migrated
  (all flags set)           SEC-4            AEAD fallback         (no action)
                           legacy no-AAD           │
                                          ┌────────┴────────┐
                                          │                 │
                                  enc-iv-deriv=        enc-iv-deriv=
                                  "hkdf-sha256"         absent/""
                                       │                     │
                                  C_HKDF              C_XOR (also
                                (half-migrated)        CLASS A)
```

#### CLASS A — XOR-IV / no HKDF (SEC-2 legacy)
- **Detection:** `x-amz-meta-enc-iv-deriv` is **absent or empty**.
- **Scope:** chunked objects only (objects without `x-amz-meta-enc-chunked-format` use the non-chunked path and are unaffected by SEC-2).
- **Action:** decrypt with XOR path → re-encrypt with HKDF path → write back with `MetaIVDerivation = "hkdf-sha256"`.
- **Interacts with SEC-4:** if the object also lacks `x-amz-meta-enc-legacy-no-aad`, the legacy AAD fallback is blocked by SEC-4.  For objects written before AAD was introduced at all, the operator must first back-tag them with SEC-4's marker (see §5.3).

#### CLASS B — no-AAD legacy (SEC-4 legacy)
- **Detection:** `x-amz-meta-enc-legacy-no-aad = "true"`.
- **Scope:** non-chunked (single-AEAD) objects written before AAD was added.
- **Action:** decrypt via the no-AAD path (permitted because the marker is present) → re-encrypt with AAD → write back **without** the `x-amz-meta-enc-legacy-no-aad` marker.
- **Definition of done for this class:** the marker is absent on the migrated object; subsequent decryptions use the full AAD path.

#### CLASS C — legacy metadata-fallback (SEC-27 legacy)
- **Detection:** `x-amz-meta-encryption-fallback = "true"` AND `x-amz-meta-encryption-fallback-version` is **absent or `"1"`**.
- **Sub-classes:**
  - **C_XOR:** also lacks `x-amz-meta-enc-iv-deriv` → needs both SEC-2 and SEC-27 migration.
  - **C_HKDF:** has `x-amz-meta-enc-iv-deriv = "hkdf-sha256"` but fallback-version is still `"1"` → needs SEC-27 migration only.
- **Action:** decrypt via `decryptFallbackV1` (outer-AEAD path) → re-encrypt via `encryptChunkedWithMetadataFallback` (new streaming path) → write back with `MetaFallbackVersion = "2"`.

#### MODERN — no migration needed
- **Detection:** chunked objects have `enc-iv-deriv = "hkdf-sha256"`;
  non-chunked objects lack `enc-legacy-no-aad`;
  fallback objects have `fallback-version = "2"`.
- **Action:** skip (idempotency guarantee).

### 2.2 Metadata key reference

| Constant | Metadata key | Relevant class |
|---|---|---|
| `MetaEncrypted` | `x-amz-meta-encrypted` | All — presence gating |
| `MetaChunkedFormat` | `x-amz-meta-enc-chunked-format` | A, C — chunked-only check |
| `MetaIVDerivation` | `x-amz-meta-enc-iv-deriv` | A, C_XOR — SEC-2 legacy |
| `MetaLegacyNoAAD` | `x-amz-meta-enc-legacy-no-aad` | B — SEC-4 legacy |
| `MetaFallbackMode` | `x-amz-meta-encryption-fallback` | C — SEC-27 legacy |
| `MetaFallbackVersion` | `x-amz-meta-encryption-fallback-version` | C — version discriminator |
| `MetaFallbackPointer` | `x-amz-meta-encryption-fallback-ptr` | C — companion-object location |
| `MetaWrappedKeyCiphertext` | `x-amz-meta-encryption-wrapped-key` | All KMS-mode objects |
| `MetaKMSKeyID` | `x-amz-meta-encryption-kms-id` | All KMS-mode objects |

### 2.3 Classification algorithm (pseudo-Go)

```go
type ObjectClass int

const (
    ClassModern     ObjectClass = iota // No migration needed
    ClassA_XOR                         // SEC-2: XOR IV derivation
    ClassB_NoAAD                       // SEC-4: no-AAD legacy
    ClassC_Fallback_XOR                // SEC-27 + SEC-2: v1 fallback + XOR IV
    ClassC_Fallback_HKDF               // SEC-27: v1 fallback, HKDF IV already
    ClassPlaintext                     // Not encrypted; skip
    ClassUnknown                       // Cannot determine; log and skip
)

func ClassifyObject(meta map[string]string) ObjectClass {
    if meta[MetaEncrypted] != "true" && meta["x-amz-meta-e"] != "true" {
        return ClassPlaintext
    }

    isFallback   := meta[MetaFallbackMode] == "true"
    fallbackVer  := meta[MetaFallbackVersion]  // "" or "1" = legacy; "2" = new
    ivDeriv      := meta[MetaIVDerivation]     // "" = XOR; "hkdf-sha256" = new
    legacyNoAAD  := meta[MetaLegacyNoAAD] == "true"
    isChunked    := meta[MetaChunkedFormat] == "true"

    if isFallback {
        if fallbackVer == "2" {
            return ClassModern // fallback v2, check other flags below
        }
        // fallback v1 (absent or "1")
        if ivDeriv == "" {
            return ClassC_Fallback_XOR
        }
        return ClassC_Fallback_HKDF
    }

    if legacyNoAAD {
        return ClassB_NoAAD
    }

    if isChunked && ivDeriv == "" {
        return ClassA_XOR
    }

    return ClassModern
}
```

> **Note:** CLASS B (`legacyNoAAD`) and CLASS A (`XOR-IV`) are not
> mutually exclusive; a very old chunked object could theoretically
> require both migrations.  The worker applies them in sequence:
> SEC-4 migration (decrypt no-AAD → re-encrypt with AAD) first,
> then SEC-2 migration (XOR → HKDF) if the resulting re-encrypted
> object would still be chunked without HKDF.  In practice, the
> gateway only wrote `MetaLegacyNoAAD` on non-chunked objects, so
> this combination is not expected; the code handles it defensively.

### Phase 3 — Core migration engine (`internal/migrate/migrator.go`, `worker.go`)

**Scope:** the main orchestration loop and per-object worker.

#### 3a — `Migrator` struct

```go
type Migrator struct {
    S3Client       s3iface.Client           // GetObject, PutObject, ListObjectsV2, HeadObject
    SourceEngine   crypto.EncryptionEngine  // configured with OLD key/format settings
    TargetEngine   crypto.EncryptionEngine  // configured with NEW key/format settings
    GatewayVersion semver.Version           // running gateway version; governs format compat checks
    Workers        int
    StateFile      string
    DryRun         bool
    Verify         bool                     // enable post-write verification
    VerifyDelay    time.Duration
}
```

`GatewayVersion` is parsed from `--gateway-version` and used to:
1. Validate at startup that the target format is readable by the declared
   gateway version (e.g. refuse to write fallback-v2 if the gateway is
   declared as < v0.6.4).
2. Include the version in the state file for auditability.
3. Gate the `backfill-legacy-no-aad` sub-command: on v0.7.0+ the no-AAD
   fallback is blocked by default, so backfilling is mandatory before any
   CLASS B object can be migrated; the tool enforces this automatically.

- [ ] `Migrate(ctx context.Context, bucket, prefix string) error`
  1. `LoadOrCreate` state from `StateFile`.
  2. Paginate `ListObjectsV2(bucket, prefix, StartAfter=state.Checkpoint)`.
  3. For each object: classify via `HeadObject` → dispatch to worker pool.
  4. On ctx cancellation: drain pool, save state, return gracefully.
  5. On completion: print final `Stats` report.
- [ ] Worker goroutine (`worker.go`):
  ```go
  func (m *Migrator) migrateObject(ctx context.Context, bucket, key string, class ObjectClass) error
  ```
  Steps (see §3.1 data flow):
  1. `GetObject` with full metadata.
  2. `SourceEngine.Decrypt(body, metadata)` — uses dual-read window (XOR / no-AAD / fallback-v1 as needed).
  3. `TargetEngine.Encrypt(plaintext, originalUserMetadata)` — produces HKDF/AAD/fallback-v2 object.
  4. `PutObject` (same key, new metadata).
  5. If `m.Verify`: call `verify.Verify(ctx, bucket, key, plaintext)`.
  6. `state.MarkDone(key)`.
- [ ] Error handling:
  - Single-object failure: log error, `state.MarkFailed(key, err)`, continue.
  - Context cancellation: drain, save, return.
  - S3 connectivity failure (5xx, timeout): retry with exponential backoff (max 3 retries); then fail the object.
- [ ] Back-pressure: use a `chan struct{}` semaphore sized to `Workers` to cap concurrency.

#### 3b — Per-class migration logic

Each class requires a distinct decrypt/re-encrypt recipe:

**CLASS A (XOR-IV):**
- Source: chunked object; `SourceEngine` must NOT set `MetaIVDerivation` — use the XOR decrypt path.
- Target: `TargetEngine.Encrypt` will set `MetaIVDerivation = "hkdf-sha256"` automatically on the new object.
- Implementation note: the existing `chunkedDecryptReader` already dispatches on the absent flag → XOR path.  No special configuration needed; the source engine uses the production engine as-is.

**CLASS B (no-AAD):**
- Source: non-chunked object; `MetaLegacyNoAAD = "true"` is present → the `gcm.Open(nil, iv, ciphertext, nil)` fallback in `engine.Decrypt` is already guarded by this flag.  Decrypt proceeds normally.
- Target: `TargetEngine.Encrypt` writes the new object **without** `MetaLegacyNoAAD`.
- Post-condition: the new object will decrypt only via the AAD path; the no-AAD fallback is permanently blocked.

**CLASS C_XOR (fallback + XOR):**
- Source: `SourceEngine` must call `decryptFallbackV1` (outer-AEAD path).  Dispatched automatically by `isFallbackMode` + absent `MetaFallbackVersion`.
- After decryption, re-encrypt with `TargetEngine.Encrypt` which emits fallback-v2 + HKDF.
- Note: the companion object (if `MetaFallbackPointer` is set) must also be deleted after successful migration.

**CLASS C_HKDF (fallback only):**
- Same as C_XOR except the IV derivation is already HKDF; only the outer-AEAD wrapper needs to be replaced with v2.

#### 3c — Companion object cleanup

For fallback objects where metadata is stored in a companion S3 object
(referenced by `MetaFallbackPointer`):
- After successful PutObject of the migrated object, issue `DeleteObject` on the companion key.
- This is best-effort: failure to delete the companion is logged but does not fail the migration of the primary object.
- The new object either embeds the manifest inline (≤ 1.8 KiB) or writes its own companion under the new format; it is not dependent on the old companion object.

### Phase 4 — Post-write verification (`internal/migrate/verify.go`)

**Scope:** end-to-end correctness guarantee for each migrated object.

- [ ] `Verify(ctx, s3, engine, bucket, key string, expectedPlaintext []byte) error`:
  1. `GetObject(bucket, key)` — reads back the just-written ciphertext.
  2. `engine.Decrypt(body, metadata)` — decrypts with the target engine.
  3. `sha256.Sum256(decrypted)` vs `sha256.Sum256(expectedPlaintext)` — constant-time comparison.
  4. Mismatch → return `ErrVerifyMismatch`.
- [ ] `ErrVerifyMismatch` sentinel error; worker logs it as a critical failure and does NOT mark the object as done.
- [ ] For large objects (> configured `VerifyMaxBytes`, default 256 MiB), skip full comparison and instead verify only metadata correctness (new class markers present, old class markers absent) + first/last 4 KiB hash check.
- [ ] Unit test: inject a PutObject stub that corrupts 1 byte; verify `ErrVerifyMismatch` is returned.

### Phase 5 — CLI (`cmd/migrate/main.go`)

**Scope:** operator-facing binary using Cobra or `flag`.

```
s3eg-migrate \
  --config gateway.yaml \           # gateway config file (source of engine params)
  --gateway-version 0.6.4 \         # REQUIRED: version of the currently-running gateway
                                    # accepts: 0.6.4 | 0.7.0  (validated at startup)
  --source-key-version 1 \          # optional: force old KEK version for source decrypt
  --target-key-version 2 \          # optional: force new KEK version for target encrypt
  --bucket mybucket \
  --prefix backups/ \               # optional; default = migrate entire bucket
  --workers 4 \                     # default 4
  --dry-run \                       # scan only; no writes
  --verify \                        # enable post-write hash verification (default: on)
  --no-verify \                     # disable verification (faster, less safe)
  --state-file migration.json \     # resume state file (default: <bucket>-migration.json)
  --migration-class all \           # which classes to migrate; comma-separated
                                    # values: all | sec2 | sec4 | sec27
  --verify-delay 500ms \            # pause before verify read (default 0; set to 500ms-2s
                                    # for distributed MinIO to tolerate eventual consistency)
  --log-level info \                # info | debug | warn
  --output json                     # text (default) | json (machine-readable progress)
```

- [ ] `--gateway-version`: **required**; accepted values `0.6.4` and `0.7.0`.  At startup the
      tool validates:
  - `0.6.4`: all three legacy formats can be read AND the modern format (HKDF /
    AAD / fallback-v2) can be read back immediately — ✅ safe to migrate.
  - `0.7.0`: legacy formats are still readable via the backward-compat dual-read
    window; modern format is the default write path — ✅ safe to migrate.
  - Any other value: tool exits with code `1` and a clear error message listing
    the supported versions.
- [ ] `--dry-run`: list and classify objects; report counts per class; no writes; exit 0.
- [ ] `--migration-class`: allows limiting scope (e.g. `sec4` only migrates CLASS B objects).
- [ ] `--state-file`: defaults to `<bucket>[-<prefix-hash>]-<gateway-version>-migration.json`
      in the current directory.  Embedding the gateway version in the filename prevents
      accidentally resuming an Option A state file under an Option B run.
- [ ] Progress: when running in a TTY, emit a live progress bar using a lightweight library or
      simple carriage-return output.  When not a TTY (CI, scripts), emit structured JSON or
      newline-delimited text lines.
- [ ] Exit codes:
  - `0` — all eligible objects migrated (or already modern).
  - `1` — fatal error (S3 unavailable, config invalid, unsupported `--gateway-version`, etc.).
  - `2` — partial migration: some objects failed; state file contains the failed list.
- [ ] `--config` parsing reuses `internal/config` package to construct `EncryptionEngine`
      instances (both source and target).
- [ ] Print a summary table on exit that includes the gateway version context:
  ```
  Migration complete
    Gateway version : 0.6.4  (Option A — migrate before upgrade)
    Bucket          : mybucket
    Prefix          : backups/
  Total objects scanned: 12 345
    Already modern (skipped): 10 000
    Plaintext (skipped):          42
    Migrated (CLASS A / SEC-2):  1 200
    Migrated (CLASS B / SEC-4):    500
    Migrated (CLASS C / SEC-27):   600
    Failed:                          3
  State file: mybucket-backups-0.6.4-migration.json
  Failed objects: (see state file)
  Next step: run --dry-run to confirm 0 legacy objects, then upgrade to v0.7.0
  ```

### Phase 6 — Dry-run report (`internal/migrate/dryrun.go`)

**Scope:** a classification-only pass that is always safe to run.

- [ ] `DryRunScan(ctx, s3, bucket, prefix string, workers int) (DryRunReport, error)`

## 5. Special Cases & Edge Conditions

### 5.0 Gateway version compatibility matrix

The tool's behaviour depends on which gateway version is declared via
`--gateway-version`.  The following table is the authoritative reference;
it is also validated programmatically at startup.

| `--gateway-version` | Can read (SourceEngine) | Can write (TargetEngine) | Notes |
|---|---|---|---|
| `0.6.4` | XOR-IV ✅, no-AAD ✅, fallback-v1 ✅, HKDF ✅, AAD ✅, fallback-v2 ✅ | HKDF ✅, AAD ✅, fallback-v2 ✅ | Full dual-read window; modern write path |
| `0.7.0` | XOR-IV ✅ (compat window), no-AAD ✅ (compat window), fallback-v1 ✅ (compat window), HKDF ✅, AAD ✅, fallback-v2 ✅ | HKDF ✅, AAD ✅, fallback-v2 ✅ | Breaking changes active; compat window still open for reads |

**Source engine selection logic** (`internal/migrate/migrator.go`):
- For both versions, `SourceEngine` is the production `crypto.EncryptionEngine`
  from `internal/crypto`; no special configuration needed.  The engine's existing
  dispatch logic (`deriveChunkIV`, `MetaLegacyNoAAD` guard, `decryptFallbackV1`
  / `decryptFallbackV2` dispatch) handles all legacy read paths automatically.
- The `--gateway-version` flag does NOT change how the source engine decrypts;
  it governs startup validation and the state file version tag only.

**Target engine selection logic:**
- For both `0.6.4` and `0.7.0` the target engine is configured identically:
  the production `crypto.EncryptionEngine` with its default (modern) settings.
  The resulting objects carry `enc-iv-deriv=hkdf-sha256`, no `legacy-no-aad`,
  and `fallback-version=2` — readable by both v0.6.4 and v0.7.0.

**Startup validation steps** (fail with exit code `1` if any check fails):
1. Parse `--gateway-version`; reject unknown values.
2. Probe the S3 endpoint with a `HeadBucket` call; fail if unreachable.
3. Attempt to construct both SourceEngine and TargetEngine from `--config`;
   fail if any required key version is missing.
4. If `--gateway-version 0.7.0` and any CLASS B objects are present in a
   dry scan of the first page: emit a `slog.Warn` reminding the operator
   that `backfill-legacy-no-aad` may be needed.

### 5.1 KEY ROTATION vs. FORMAT MIGRATION

This tool handles two orthogonal migration dimensions:

1. **Format migration** (this plan's primary concern): XOR→HKDF (SEC-2), no-AAD→AAD (SEC-4), fallback-v1→v2 (SEC-27).
2. **Key rotation** (the tool's secondary use case per the issue spec): re-encrypting from an old KEK version to a new KEK version.

Both can be combined in a single pass.  When `--source-key-version N` and `--target-key-version M` are supplied together with format migration flags, the worker decrypts with the old key AND old format, then re-encrypts with the new key AND new format in a single GetObject/PutObject round trip.

### 5.2 MPU (multipart upload) objects

Objects uploaded via encrypted multipart upload (V1.0-SEC-25) have their own manifest format (`x-amz-meta-encryption-mpu`).  These are **out of scope** for this tool in v1.0 because:
- SEC-2 / SEC-4 / SEC-27 breaking changes predate the MPU encryption feature.
- No legacy MPU-encrypted objects in the XOR/no-AAD/fallback-v1 format exist.

The classifier must recognise MPU objects (presence of `x-amz-meta-encryption-mpu`) and return `ClassModern` to skip them safely.  A future plan will address MPU-specific migration if needed.

### 5.3 Back-tagging SEC-4 objects with no marker

The most subtle edge case: objects written **before both** AAD was introduced **and** before SEC-4 added the `MetaLegacyNoAAD` flag have **neither** the AAD commitment **nor** the legacy marker.  After SEC-4 was deployed, these objects will fail to decrypt entirely (the no-AAD fallback is blocked because the flag is absent).

This scenario requires a **two-step process**:
1. **Identify** objects that fail to decrypt with the current engine (Decrypt returns an error).
2. **Back-tag** them: issue a `CopyObject` with the same key and add `MetaLegacyNoAAD = "true"` to their metadata (this does NOT re-encrypt; it only adds the marker).

Only then can the migration tool's CLASS B path proceed (decrypt no-AAD → re-encrypt with AAD → remove marker).

The CLI exposes this as a sub-command:
```
s3eg-migrate backfill-legacy-no-aad \
  --bucket mybucket \
  --prefix backups/ \
  --workers 4
```
This sub-command:
1. Attempts `Decrypt` on each object using the full-AAD path.
2. On failure, attempts `Decrypt` with the no-AAD path (directly, without the guard).
3. If the no-AAD path succeeds, issues a `CopyObject` adding `MetaLegacyNoAAD = "true"`.
4. Reports the number of objects back-tagged.

**Security note:** this sub-command is privilege-escalating in the sense that it back-tags objects, so it should be guarded behind a `--confirm-backfill` flag and emit a prominent warning.

### 5.4 Fallback companion objects

When a fallback object's manifest exceeds the metadata size limit, the manifest is stored in a companion S3 object at the key stored in `MetaFallbackPointer`.  During migration:
1. The migrator reads `MetaFallbackPointer` from the primary object's metadata.
2. It fetches the companion object to reconstruct the full manifest for decryption.
3. After successful PutObject of the migrated primary, it deletes the companion.
4. The new primary object either embeds its manifest inline or creates its own companion under the new naming scheme.

### 5.5 S3 consistency model

AWS S3 provides strong read-after-write consistency for all object operations (PUTs, DELETEs,
lists) as of December 2020.  A `GetObject` immediately after `PutObject` will reflect the new
object on AWS S3 and on single-node MinIO.

**Distributed MinIO caveat:** in distributed-erasure-code MinIO deployments, strong consistency
is not guaranteed under network partition scenarios.  The verify `GetObject` may transiently
return the previous object version.  Operators running distributed MinIO should set
`--verify-delay <duration>` (default `0`; recommended `500ms`–`2s` for distributed MinIO) to
introduce a short pause between `PutObject` and the verify read.  The migration tool accepts this
flag and emits a `slog.Warn` if verify fails, prompting a manual re-check rather than silently
marking the object as migrated.

### 5.6 Large objects and memory

The migration tool must never accumulate an entire object in memory:
- The `GetObject` response body is streamed directly through `SourceEngine.Decrypt`.
- The `Decrypt` output reader is streamed directly through `TargetEngine.Encrypt`.
- The `Encrypt` output reader is streamed directly to `PutObject`.
- For verification, the plaintext is held in memory only if it is ≤ `VerifyMaxBytes` (default 256 MiB); for larger objects, a streaming hash comparison is used.

> **Exception:** `decryptFallbackV1` materialises the entire ciphertext in memory (as noted in the SEC-27 issue).  For CLASS C objects, peak memory is ≈ 2× object size during migration.  Operators should account for this when choosing worker count for buckets containing large fallback-v1 objects.

## 6. Safety Mechanisms

| Mechanism | Implementation |
|---|---|
| **Dry-run mode** | `--dry-run`: no `PutObject` calls; report only |
| **Verify-after-write** | `--verify` (default on): GetObject + Decrypt + hash compare after each PutObject |
| **Idempotency** | State checkpoint prevents re-migrating already-migrated objects |
| **Resume** | State file survives crashes; re-run continues from checkpoint |
| **Atomic S3 overwrite** | S3 `PutObject` on the same key is atomic; no temporary objects |
| **Verify delay** | `--verify-delay` (default `0`) adds a pause before the verify `GetObject`; recommended for distributed MinIO deployments |
| **Companion cleanup** | Companion objects deleted only after successful primary write |
| **Error isolation** | Single-object failure does not abort the run; logged and continued |
| **No rollback** | S3 has no transactions; document in runbook.  State file lists migrated keys; recovery is manual re-migration of failed keys with the old engine still available. |
| **Backfill guard** | `backfill-legacy-no-aad` sub-command requires `--confirm-backfill` flag |
| **Memory bounds** | Streaming pipeline; full materialisation only for fallback-v1 (documented) |

## 8. Risks & Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Migrating an object that fails to decrypt (corrupt ciphertext) | High | Dry-run step surfaces these before any writes. Object-level error isolation: migration continues, object logged to `Failed` list. |
| Back-tagging an object that was not actually no-AAD legacy (false positive) | High | `backfill-legacy-no-aad` only tags if the no-AAD decrypt succeeds AND the AAD decrypt fails. Requires `--confirm-backfill`. |
| PutObject succeeds but GetObject in verify returns stale data | Low | S3 strong read-after-write consistency (since 2020) for AWS S3 and single-node MinIO. Use `--verify-delay` for distributed MinIO. |
| Operator runs tool against wrong gateway version (e.g. declares `0.6.4` but gateway is already `0.7.0`) | Medium | `--gateway-version` is required and validated. State file embeds the declared version; a mismatch on resume causes exit `1`. Runbook instructs operators to verify gateway version before running. |
| Operator upgrades gateway to `0.7.0` before migration, then tries to resume an Option A state file | Low | State file version tag mismatch is detected at startup; operator is prompted to start a fresh run with `--gateway-version 0.7.0`. All previously migrated objects are skipped via idempotency (already `ClassModern`). |
| Worker count too high → S3 rate limiting (429) | Medium | Default 4 workers. Retry with backoff on 429 / 503 responses. Document recommended worker counts per backend. |
| Memory spike during CLASS C fallback-v1 migration of large objects | Medium | Document ≈ 2× object-size peak; recommend reducing `--workers` for buckets with large fallback objects. |
| Key version mismatch (source key not available) | High | CLI validates that all required key versions are loaded before starting. Worker fails the object (not the run) if unwrap fails. |
| Interrupted migration leaves a companion object orphaned | Low | Companion deletion is idempotent; a re-run of the migration for that object will attempt the delete again. |
