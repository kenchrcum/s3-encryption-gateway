# V0.6-SEC-3 — Encrypted Multipart Uploads — Implementation Plan

Status: Draft
Owner: Crypto / API
Priority: P0
Labels: `milestone:v0.6`, `area:crypto`, `area:s3`, `area:security`

> Scope: close the plaintext-at-rest hole for multipart uploads, documented
> in ADR 0002 and flagged in `README.md:700`. Implementation follows
> ADR 0009 (*Encrypted Multipart Uploads with Per-Upload DEK and
> Finalization Manifest*). This plan lays out a seven-phase rollout,
> identifies every call site to change, lists the tests that gate each
> phase, and records the primary risks.

---

## 1. Context & Current State

### 1.1 The gap in code

- **`handleUploadPart`** (`internal/api/handlers.go:2211-2295`) has a
  dead `if uploadID == ""` branch. The multipart router at
  `internal/api/handlers.go:107` guarantees `uploadID` is non-empty for
  every request reaching this handler, so the encryption branch is
  unreachable. Every part is buffered with `io.ReadAll` (`:2278`) and
  forwarded **verbatim** to the backend.
- **`handleCreateMultipartUpload`** (`:1922-2006`) forwards to
  `s3Client.CreateMultipartUpload` (`:1973`). No DEK generated, no
  state stashed, no wrap.
- **`handleCompleteMultipartUpload`** (`:2318-2429`) forwards part list
  to `s3Client.CompleteMultipartUpload` (`:2393`). No manifest written.
- **`handleAbortMultipartUpload`** (`:2432-2487`) pure pass-through.
- **`handleUploadPartCopy`** (`internal/api/upload_part_copy.go:89+`)
  writes plaintext parts per ADR 0006 §"Destination parts are plaintext
  (ADR 0002)."

### 1.2 Building blocks that already exist

- **Chunked AEAD framing** (`internal/crypto/chunked.go`,
  `encryptChunked` at `internal/crypto/engine.go:793+`) produces
  per-chunk `ciphertext || tag` with a JSON manifest. The **framing is
  reusable**; only the IV schedule changes for MPU.
- **Envelope encryption** (`KeyManager.WrapKey` /
  `UnwrapKey`, `internal/crypto/keymanager.go:24-50`) already binds one
  wrapped DEK to a metadata context — natural fit for one DEK per
  multipart upload.
- **Fallback metadata object pointer**
  (`MetaFallbackMode`/`MetaFallbackPointer`,
  `internal/crypto/engine.go:51-52`,
  `encryptChunkedWithMetadataFallback` at `:961+`) provides an
  existing mechanism for oversized manifests.
- **Ranged decrypt** (`engine.DecryptRange`,
  `internal/crypto/engine.go:1263`; handler at
  `internal/api/handlers.go:834`) already skips chunks outside
  requested ranges. Polymorphising `DecryptRange` over a `Manifest`
  interface covers both single-PUT chunked and MPU-chunked paths.
- **HKDF** is already on the dependency tree
  (`golang.org/x/crypto/hkdf`, pulled in indirectly; if absent, it's
  a stdlib-level dep with zero risk).
- **Key rotation state machine** (ADR 0007, `internal/admin/`,
  `internal/crypto/rotation_state.go`, `RotatableKeyManager`) already
  tracks in-flight wraps — extensible to track in-flight MPUs.
- **`Server.DisableMultipartUploads`** (`internal/config/config.go:172`)
  remains as the fail-closed escape hatch.

### 1.3 Test coverage today

- Unit: XML parsing fuzz + part-list validation
  (`internal/api/handlers_test.go:976,1047`). **No encryption
  assertions** on multipart flows.
- Integration: `TestMultipartUpload_ProviderInterop`
  (`test/provider_test.go:275-712`) round-trips a multipart upload
  against live providers but **does not verify ciphertext at rest**.
- `UploadPartCopy` unit tests
  (`internal/api/upload_part_copy_test.go`) cover source-classification
  and hard-refusal paths from ADR 0006.

## 2. Non-Goals

- **On-disk format compatibility with AWS Encryption SDK.** Out of
  scope; future ADR.
- **Additional `StateStore` backends (DynamoDB / etcd / backend-
  object).** Only `ValkeyStateStore` ships in v0.6. The
  `StateStore` interface is retained for future freedom, not
  immediate pluggability. Backend-object state is explicitly
  rejected — see ADR 0009 §Alternatives E.
- **Changing default `EncryptMultipartUploads`.** Default stays
  `false` in v0.6 for soak. v0.7 flips default to `true` after a
  release of production exposure.
- **Per-part client integrity (`x-amz-content-sha256`).** Deferred to
  v0.7.
- **Cryptographic rollback protection for Valkey state.** Deferred
  to v0.7 hardening (see ADR 0009 §Future Considerations #5).
- **Valkey Cluster / Sentinel-specific deployment guidance.**
  Documented in runbook; `go-redis/v9` handles both transparently
  and the gateway config takes a single connection string.

## 3. Design Summary (authoritative reference: ADR 0009)

- **Per-upload DEK** generated on `CreateMultipartUpload`, wrapped via
  the bucket's `KeyManager`, persisted in a **Valkey hash** keyed
  `mpu:<base64url(sha256(uploadId))>` with a 7-day TTL refreshed
  on every `UploadPart`. TLS 1.3 required on the Valkey
  connection in production.
- **Per-part, per-chunk IV** via
  `HKDF-Expand(DEK, salt=sha256(uploadId), info=iv_prefix || be32(part) || be32(chunk))`,
  truncated to 12 bytes.
- **Chunked AEAD per part** — each part produces
  `plain + chunkCount * 16` backend bytes; concatenation of parts is
  decryptable by the gateway.
- **Finalization manifest** written at `CompleteMultipartUpload`
  either inline in `x-amz-meta-encryption-manifest` (≤ 1.8 KiB) or as
  a companion object via the existing fallback-pointer mechanism.
- **Abort** deletes the Valkey key (`DEL mpu:<h>`); TTL expiry
  cleans up orphans automatically.
- **Ranged GET** uses the manifest's per-part layout to translate
  plaintext offsets to backend byte offsets and reconstructs IVs via
  the HKDF schedule.
- **Opt-in per bucket** via `PolicyConfig.EncryptMultipartUploads`
  (default `false`).
- **Fail-closed escape hatch:** deployments that cannot run Valkey
  must use `Server.DisableMultipartUploads=true` (5 GiB object
  ceiling, no state needed).

## 4. Implementation Phases

### Phase A — Crypto primitives (IV derivation, manifest type)

Scope: pure library work, no handler or state-store touchpoints. Fully
unit-testable in isolation.

- [ ] Add `internal/crypto/mpu_iv.go`:
      `DeriveMultipartIV(dek, uploadIDHash, ivPrefix, part, chunk) [12]byte`
      using `golang.org/x/crypto/hkdf` with SHA-256.
- [ ] Add `internal/crypto/mpu_manifest.go`:
      `MultipartManifest` struct (version, algorithm, chunk_size,
      iv_prefix, wrapped_dek, kms_key_id, kms_provider, parts
      list, original_etag), JSON marshal/unmarshal, size-budget
      predicate (`FitsInline(limit int) bool`).
- [ ] Add a polymorphism shim: `Manifest` interface with
      `ChunkLayout(plainOffset int64) (part int, chunk int,
      intraChunk int64, err error)` implemented by both the existing
      chunked manifest and the new multipart manifest.
- [ ] Add `internal/crypto/mpu_iv_test.go`:
    - Known-answer test (KAT) for a fixed `(DEK, uploadId, prefix,
      part, chunk)` tuple. Locks the derivation against accidental
      change. Regenerating the KAT requires a conscious update and a
      matching CHANGELOG entry.
    - IV-uniqueness test: exhaustive across
      part ∈ {1, 2, 10000}, chunk ∈ {0, 1, 100, 65535} for a fixed
      DEK. No collisions.
    - Different-uploadId test: same (part, chunk) with different
      `uploadIDHash` → different IVs.
    - Length invariant: output is always 12 bytes.
- [ ] Add `internal/crypto/mpu_manifest_test.go`:
    - Round-trip JSON marshal/unmarshal.
    - Size-budget edge cases: 29 parts fits, 31 parts does not
      (calibrated against 1.8 KiB cap).
    - Version-tag test: unknown version → error.

**Gate:** `-race` clean; `-tags=fips` clean; `-tags=hsm` clean; KAT
locked into tree.

### Phase B — Valkey state store

Scope: the `StateStore` interface and the `ValkeyStateStore`
implementation. No handler changes yet. Adds one new Go dependency
(`github.com/redis/go-redis/v9`) and one new infrastructure
dependency documented in operator guides.

- [ ] Add `github.com/redis/go-redis/v9` to `go.mod`. Verify it
      builds under `-tags=fips` and `-tags=hsm` (pure-Go client,
      no cgo — should be clean).
- [ ] Add config stanza `internal/config/config.go`:
      ```yaml
      multipart_state:
        valkey:
          addr: "valkey.internal:6379"
          username: ""              # Valkey 6.0+ ACL username (optional)
          password_env: "VALKEY_PASSWORD"  # env var name; not the literal password
          db: 0
          tls:
            enabled: true           # default true; required in production
            ca_file: "/etc/gateway/valkey-ca.pem"
            cert_file: "/etc/gateway/valkey-client.pem"
            key_file: "/etc/gateway/valkey-client-key.pem"
            insecure_skip_verify: false
            min_version: "1.3"      # TLS 1.3 default
          insecure_allow_plaintext: false  # dev-only; startup warning + metric
          ttl_seconds: 604800       # 7 days
          dial_timeout: "2s"
          read_timeout: "1s"
          write_timeout: "1s"
          pool_size: 16
          min_idle_conns: 2
      ```
      Fail-closed at startup when
      `EncryptMultipartUploads=true` is set on any bucket policy
      but `multipart_state.valkey.addr` is unset.
- [ ] Add `internal/mpu/state.go`:
    - `UploadState` struct and `PartRecord` struct (see ADR 0009
      §State Record Schema). Fields include `PolicySnapshot`.
    - `StateStore` interface:
      ```go
      type StateStore interface {
          Create(ctx context.Context, state *UploadState) error
          Get(ctx context.Context, uploadID string) (*UploadState, error)
          AppendPart(ctx context.Context, uploadID string, part PartRecord) error
          Delete(ctx context.Context, uploadID string) error
          HealthCheck(ctx context.Context) error
          Close() error
      }
      ```
    - `ValkeyStateStore` implementation backed by
      `*redis.Client` (or `*redis.ClusterClient`; detected from
      the address form). Operations:
        - `Create`: `HSETNX mpu:<h> meta <json>` +
          `EXPIRE mpu:<h> 604800` — atomic via `MULTI/EXEC` or
          a two-line Lua script. Fails `ErrUploadAlreadyExists`
          if the key exists.
        - `Get`: `HGETALL mpu:<h>` → parse `meta` + all
          `part:*` fields into `UploadState`.
        - `AppendPart`: `HSET mpu:<h> part:<N> <json>` +
          `EXPIRE mpu:<h> 604800` in a pipeline.
        - `Delete`: `DEL mpu:<h>`.
        - `HealthCheck`: `PING`, 1 s timeout.
    - Sentinel errors: `ErrUploadNotFound`,
      `ErrUploadAlreadyExists`, `ErrStateUnavailable` (wraps
      connection errors for caller pattern-matching).
    - `NewValkeyStateStore(ctx, cfg) (*ValkeyStateStore, error)`
      performs a `HealthCheck` on construction and fails closed
      if Valkey is unreachable. Startup-failure is the desired
      behaviour: a misconfigured Valkey should not silently
      downgrade to disabled MPU encryption.
- [ ] Add `internal/mpu/state_test.go`:
    - **Uses miniredis** (`github.com/alicebob/miniredis/v2`) as
      the in-process Valkey-protocol test fixture — no
      containers needed for unit tests.
    - Round-trip: `Create` → `AppendPart` × 3 → `Get` → `Delete`.
    - TTL refresh: after `AppendPart`, assert TTL ≈ 604800
      (miniredis exposes `FastForward` to test expiry).
    - Wrapped-DEK is not plaintext in the Valkey value (verify
      by inspecting the raw `meta` field).
    - `Create` on existing key → `ErrUploadAlreadyExists`.
    - `Get` / `AppendPart` / `Delete` on missing key →
      `ErrUploadNotFound`.
    - Concurrent `AppendPart` for distinct part numbers:
      both survive, final `Get` shows both parts. Miniredis
      serialises commands so this validates the state-machine
      semantics; a second test against a real Valkey container
      lives in Phase G integration tests.
    - `HealthCheck` fails on a closed miniredis.
    - Graceful `Close`.
- [ ] Add `internal/mpu/valkey_tls_test.go`:
    - Startup fails when
      `EncryptMultipartUploads=true` + Valkey TLS disabled +
      `insecure_allow_plaintext=false`.
    - Startup emits a warning + sets
      `gateway_mpu_valkey_insecure=1` gauge when
      `insecure_allow_plaintext=true`.

**Gate:** `-race` clean; unit coverage ≥ 85% on `internal/mpu/`.
Integration coverage with a real Valkey container lives in
Phase G.

### Phase C — Handler wiring (happy path)

Scope: thread the DEK + state through Create / Upload / Complete /
Abort. `EncryptMultipartUploads=false` default ensures no behaviour
change for existing deployments.

- [ ] Modify `internal/config/policy.go`:
      add `PolicyConfig.EncryptMultipartUploads bool` (default
      `false`).  Add `BucketEncryptsMultipart(bucket string) bool`
      helper mirroring the existing `BucketRequiresEncryption`.
- [ ] Modify `internal/api/handlers.go`:
    - `Handler` struct gains a `mpuStateStore mpu.StateStore` field
      and a `mpuEnabled` policy check helper.
    - `handleCreateMultipartUpload` (`:1922-2006`):
      when `BucketEncryptsMultipart` returns `true`:
      1. Generate 32-byte DEK and 12-byte `iv_prefix`.
      2. `keyManager.WrapKey(ctx, DEK, metadata)`.
      3. Call `s3Client.CreateMultipartUpload` → backend uploadId.
      4. `mpuStateStore.Create(&UploadState{...})`.
      5. Return the backend uploadId unchanged.
      On failure between (3) and (4): call
      `s3Client.AbortMultipartUpload` and surface 500.
    - `handleUploadPart` (`:2211-2295`):
      replace the dead branch. If state exists for `uploadID`:
      1. Stream `r.Body` through an `mpuChunkedEncrypter` that
         uses `DeriveMultipartIV` with the part number.
      2. PUT ciphertext to backend via `s3Client.UploadPart`.
      3. `mpuStateStore.AppendPart(uploadID,
         PartRecord{N, EncLen, Chunks, PlainLen, ETag})`.
      If state does not exist (legacy bucket policy): existing
      pass-through path.
    - `handleCompleteMultipartUpload` (`:2318-2429`):
      if state exists:
      1. Load state.
      2. Validate client-supplied part list against state.parts
         (same part numbers, same ETags).
      3. Build `MultipartManifest`.
      4. Size-check: fits inline → set metadata; else → write
         fallback companion object + set
         `MetaFallbackPointer`.
      5. Call `s3Client.CompleteMultipartUpload` with the full
         metadata set on the final object. **Manifest write order:
         metadata is supplied to Complete, so the backend atomically
         associates manifest with object.**
      6. `mpuStateStore.Delete(uploadID)`.
      On failure between (4) and (5): retry (5) with backoff; if
      persistently fails, abort backend upload and emit audit event.
    - `handleAbortMultipartUpload` (`:2432-2487`):
      if state exists: call `s3Client.AbortMultipartUpload` then
      `mpuStateStore.Delete(uploadID)`. Both errors are reported;
      state delete failure emits an audit event (`mpu.abort.orphan`)
      but does not fail the client request.
- [ ] Modify `cmd/server/main.go`:
      construct `ValkeyStateStore` alongside the existing
      `crypto.EncryptionEngine`; inject into `Handler`. Fail
      closed at startup if any bucket policy enables
      `EncryptMultipartUploads` but Valkey is unreachable or
      misconfigured. Register `StateStore.HealthCheck` in the
      existing `/readyz` aggregation so Kubernetes readiness
      reflects Valkey health.

**Gate:** all existing tests pass; new unit tests for each handler
branch (`EncryptMultipartUploads=true` and `false`); `-race` clean;
`-tags=fips` clean; `-tags=hsm` clean.

### Phase D — GET path & range decoding

Scope: extend decrypt to consume multipart manifests.

- [ ] Modify `internal/crypto/engine.go`:
    - `DecryptRange` accepts a `Manifest` interface (new sealed
      type) instead of raw chunked manifest bytes.
    - Add `manifestFromMetadata(meta)` helper that sniffs
      `x-amz-meta-encryption-mpu` vs. `x-amz-meta-encryption-chunked`
      and returns the right concrete type, resolving fallback
      pointers as needed.
- [ ] Modify `internal/api/handlers.go` `handleGetObject`:
    - On GET, if metadata indicates MPU encryption, look up the
      manifest, call `DecryptRange` with it (for ranged requests)
      or a full-stream `Decrypt` (for non-ranged).
- [ ] Modify `internal/crypto/range_optimization.go`:
    - `CalculateEncryptedRangeForPlaintextRange` accepts the
      polymorphic `Manifest` and emits the backend byte range by
      summing `parts[i].enc_len`.
- [ ] Tests:
    - `TestMPU_GetObject_Full` — full GET round-trip.
    - `TestMPU_GetObject_Range_WithinOnePart` — range in part 1.
    - `TestMPU_GetObject_Range_CrossingPartBoundary` — range
      spanning parts 1–2.
    - `TestMPU_GetObject_Range_EndOfObject` — range up to EOF.
    - `TestMPU_GetObject_Tamper_ReorderParts` — swap part 1 and 2
      backend bytes out-of-band; GET must fail with 500 +
      audit event (tag verification failure).
    - `TestMPU_GetObject_Tamper_TruncateLastChunk` — truncate the
      final chunk of the final part; GET must fail.

**Gate:** above tests green under `-race`; no regression in existing
single-PUT chunked range tests (`internal/crypto/range_decrypt_test.go`,
`internal/api/handlers_test.go` range group).

### Phase E — UploadPartCopy into encrypted MPU

Scope: align ADR 0006's dispatch with encrypted destinations.

- [ ] Modify `internal/api/upload_part_copy.go`:
    - Detect `mpuEnabled` for destination bucket; if true,
      the three source classes (plaintext / chunked / legacy) each
      re-encrypt through the destination's per-upload DEK schedule.
      The plaintext-source fast path (backend-native
      `UploadPartCopy`) **is disabled for encrypted MPU
      destinations** — document inline with cross-reference to ADR
      0009 §Consequences.
    - `PolicyConfig.RequireEncryption=true` hard-refusal
      (ADR 0006 §Error Codes) is now automatically satisfied when
      `EncryptMultipartUploads=true` on the destination; remove the
      refusal path when the destination MPU is encrypted and log an
      info event.
- [ ] Tests (unit):
    - `TestUploadPartCopy_MPU_PlaintextSource_EncryptedDest`
      — re-encryption round-trip.
    - `TestUploadPartCopy_MPU_ChunkedSource_EncryptedDest_WithRange`
      — source chunked, destination MPU, with copy-source range.
    - `TestUploadPartCopy_MPU_LegacySource_EncryptedDest` — legacy
      fallback re-encrypted.

**Gate:** ADR 0006 unit suite remains green; new tests added.

## 6. Risks & Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| IV derivation bug → nonce reuse → catastrophic confidentiality loss | Critical | KAT test locked in tree; code review by two crypto reviewers; `TestMPU_GetObject_Tamper_ReorderParts` exercises derivation boundary. |
| Valkey outage blocks all multipart operations | High | Document as required infrastructure; recommend Valkey Sentinel or Cluster for HA; `/readyz` reflects Valkey health so Kubernetes can failover traffic. Single-PUT encryption and GET of complete objects remain unaffected. Emergency fallback: operator sets `DisableMultipartUploads=true` to degrade gracefully. |
| Valkey state leaks wrapped DEKs | High | Wrapped DEKs are ciphertext; recovering plaintext requires compromising the `KeyManager`. TLS 1.3 required on the wire. Startup fail-closed if TLS is disabled without explicit `insecure_allow_plaintext=true` escape hatch. |
| Valkey AOF loss window (≤ 1 s of writes) | Medium | Documented behaviour. Deterministic IV schedule makes retries byte-identical; client's Complete-validation against recorded `etag` catches the mismatch and triggers retry of missing parts. For stronger durability, operators configure `appendfsync always` at a latency cost. |
| Multi-replica gateway races on the same uploadId | Medium | Valkey atomic primitives (`HSETNX`, `MULTI/EXEC`) serialise concurrent Create and AppendPart. `TestEncryptedMPU_MultiReplica_Concurrent` asserts correctness end-to-end. No sticky-session routing required. |
| Manifest write vs. backend Complete ordering crash window | High | Manifest supplied **as metadata** in the Complete call → atomic with respect to the backend. No inter-call window. |
| S3 `x-amz-meta-*` header size limits differ across backends | Medium | 1.8 KiB inline budget has a ~200-byte headroom vs. AWS's 2 KB limit. Integration test matrix asserts real-backend behaviour. Any backend rejecting our headers triggers automatic fallback-object path. |
| Opt-in flag flipped mid-flight for an in-flight upload | Low | Handler captures policy snapshot at `CreateMultipartUpload` and stores it in the state record's `policy_snapshot` field; subsequent `UploadPart`/`Complete` use the captured policy, not the live one. Test asserts this. |
| Rotation of KEK during an in-flight MPU | Low | State record carries the KEK version; unwrap on GET uses that version. ADR 0007 rotation "commit" iterates `SCAN mpu:*` and waits for each active upload to complete, abort, or be re-wrapped. |
| Valkey client library CVE / churn | Low | `github.com/redis/go-redis/v9` is widely used, actively maintained. Pin a minor version, track advisories via existing Renovate config (`renovate.json`). |

## 7. Known Limitations (documented, not blockers)

- **Valkey required for multipart encryption.** Documented as
  a first-class infrastructure requirement in README, helm
  chart, and docker-compose example. Deployments that cannot
  run Valkey use `DisableMultipartUploads=true` (5 GiB
  single-PUT ceiling).
- **AOF loss window ≤ 1 s** with default Valkey persistence —
  deterministic IV schedule makes client retries byte-identical.
  Documented.
- **Cryptographic rollback protection** deferred to v0.7 (see
  ADR 0009 §Security Considerations and §Future Considerations).
- **Default `EncryptMultipartUploads=false`** in v0.6 — operators
  must opt-in per bucket. v0.7 flips default after a soak period.
