# V1.0-S3-2 — S3 API Feature Parity Plan

**Status:** Planning  
**Priority:** P1  
**Owner:** API  
**Labels:** `milestone:v1.0`, `area:compat`, `area:api`

---

## 1. Objective

Achieve feature parity with the most common S3-compatible server implementations
(AWS S3, MinIO, Ceph RGW, Garage, Wasabi) by implementing all widely-used S3
API operations that are currently missing from the gateway's HTTP router.

The gateway's role as a **transparent encryption proxy** dictates a strict
classification: operations that touch object *data* must be intercepted and
handled in the gateway; operations that are purely metadata or configuration
records can be safely **proxied verbatim** to the backend.

## 4. Implementation Strategy

### 4.1 Proxy Verbatim Pattern (Majority Case)

For all operations that do not touch object *data or encryption metadata*, the
implementation is a **signed reverse-proxy forwarding call**. The existing
`forwardSignatureV4Request` helper in `internal/api/handlers.go` already
implements authenticated passthrough.

The pattern for each new operation:

```go
func (h *Handler) handleGetBucketPolicy(w http.ResponseWriter, r *http.Request) {
    vars   := mux.Vars(r)
    bucket := vars["bucket"]
    start  := time.Now()
    
    resp, err := h.forwardSignatureV4Request(r)
    if err != nil {
        // translate & write S3 error
        h.writeS3ClientError(w, r, err, "GetBucketPolicy", bucket, "", start)
        return
    }
    defer resp.Body.Close()
    copyProxyResponse(w, resp)
    h.metrics.RecordHTTPRequest(r.Context(), "GetBucketPolicy", r.URL.Path, resp.StatusCode, time.Since(start), 0)
}
```

A shared `copyProxyResponse(w, *http.Response)` helper copies status code,
headers (filtered), and body. This should be extracted/centralised to avoid
duplication across the ~35 new handlers.

### 4.2 DeleteBucket — Guarded Proxy

`DELETE /{bucket}` must be proxied but should perform a local audit event and
optionally refuse if the bucket is referenced in the gateway's policy config
(prevent orphaned policy entries). The handler:

1. Checks if `policyManager` has a policy referencing the bucket → logs a warning
   (does not block — policy enforcement is operator's responsibility).
2. Forwards to backend.
3. On success: emits audit event `bucket_deleted`.

### 4.3 Bucket Encryption Subresource — Proxy Verbatim

`GET|PUT|DELETE /{bucket}?encryption` controls **provider-side SSE** (at-rest
encryption on the backend storage node). This is entirely orthogonal to the
gateway's per-object client-side encryption: the gateway may sit in front of a
backend that independently has SSE-S3 or SSE-KMS configured, and that is the
operator's concern, not the gateway's.

The gateway therefore **proxies these requests verbatim** — identical to how it
handles ACLs, lifecycle rules, or bucket policies. The gateway neither inspects
nor modifies the request body or the response.

**Rationale:** The `?encryption` subresource describes what the *backend storage
layer* does with data at rest. The gateway's per-object encryption is a
separate, additive layer applied before bytes reach the backend. Intercepting or
masking the backend's SSE configuration would prevent operators from managing
their backend independently and would misrepresent which layer is responsible
for what.

### 4.4 SelectObjectContent — Blocked (501)

`POST /{bucket}/{key}?select` operates on decrypted object content with a SQL-
like expression evaluated server-side. Since the gateway cannot push a streaming
query evaluator between decrypt and return, this operation cannot be safely
proxied. Return `501 NotImplemented` with an explanation in the error body.
This is explicitly **out of scope** for v1.0 and is recorded as a known
limitation.

### 4.5 CORS Preflight (`OPTIONS`) — Gateway-Handled

`OPTIONS` requests for CORS preflight should be handled at the gateway level by
forwarding to the backend and reflecting its CORS headers back. This enables
browser-based S3 SDKs to function correctly without requiring additional CORS
configuration on the gateway.

### 4.6 ListBuckets (Root `/`) — Proxy Verbatim

`GET /` with no bucket path segment is ListBuckets. The root route currently
falls through to a 404. This must be registered on the root router **before**
the S3 subrouter prefix.

## 6. Routing Considerations

### Route Ordering — Subresource Query Parameters

Gorilla Mux matches routes in registration order. Query-parameter routes for
subresources must be registered **before** the generic `GET /{bucket}` and
`GET /{bucket}/{key}` catch-alls. The existing code already follows this
pattern for `?uploads`, `?retention`, etc.

New bucket-level subresources must follow the existing pattern:

```go
// Register BEFORE generic s3Router.HandleFunc("/{bucket}", ...)
s3Router.HandleFunc("/{bucket}", h.handleGetBucketPolicy).Methods("GET").Queries("policy", "")
s3Router.HandleFunc("/{bucket}", h.handlePutBucketPolicy).Methods("PUT").Queries("policy", "")
s3Router.HandleFunc("/{bucket}", h.handleDeleteBucketPolicy).Methods("DELETE").Queries("policy", "")
// ... etc.
```

### Root-Level Route — ListBuckets

`GET /` must be registered on the **root router** (`r`), not the S3 subrouter,
to prevent the `/` prefix from matching before the path is correctly identified
as an empty bucket name:

```go
r.HandleFunc("/", h.handleListBuckets).Methods("GET")
```

### OPTIONS Method

CORS preflight (`OPTIONS`) must be registered for both bucket-level and
object-level routes:

```go
s3Router.HandleFunc("/{bucket}", h.handleCORSPreflight).Methods("OPTIONS")
s3Router.HandleFunc("/{bucket:[^/]+}/{key:.+}", h.handleCORSPreflight).Methods("OPTIONS")
```

## 7. Proxy Helper Design

```go
// copyProxyResponse copies an upstream HTTP response to the client writer.
// It strips hop-by-hop headers and sets the correct status code.
func copyProxyResponse(w http.ResponseWriter, resp *http.Response) {
    hopByHop := map[string]bool{
        "Connection": true, "Keep-Alive": true, "Proxy-Authenticate": true,
        "Proxy-Authorization": true, "Te": true, "Trailers": true,
        "Transfer-Encoding": true, "Upgrade": true,
    }
    for k, vv := range resp.Header {
        if !hopByHop[k] {
            for _, v := range vv {
                w.Header().Add(k, v)
            }
        }
    }
    w.WriteHeader(resp.StatusCode)
    io.Copy(w, resp.Body) //nolint:errcheck
}

// handlePassthrough is the generic passthrough handler body.
// Callers pass the S3 operation name for metrics and audit.
func (h *Handler) handlePassthrough(w http.ResponseWriter, r *http.Request, operation, bucket, key string) {
    start := time.Now()
    resp, err := h.forwardSignatureV4Request(r)
    if err != nil {
        h.writeS3ClientError(w, r, err, operation, bucket, key, start)
        return
    }
    defer resp.Body.Close()
    copyProxyResponse(w, resp)
    h.metrics.RecordHTTPRequest(r.Context(), operation, r.URL.Path, resp.StatusCode, time.Since(start), 0)
    if h.auditLogger != nil {
        h.auditLogger.Log(audit.Event{
            Operation: operation, Bucket: bucket, Key: key,
            StatusCode: resp.StatusCode, Duration: time.Since(start),
        })
    }
}
```

## 8. Testing Strategy

Tests for all new operations must follow the three-tier taxonomy from
`docs/TESTING.md`. A test lives in **exactly one** tier.

---

### 8.1 Tier 1 — Unit Tests (no build tag; run on every `go test ./...`)

**Location:** `internal/api/handlers_test.go` (and `internal/api/utils_test.go`
for the new helpers).

**Approach:** Table-driven tests using an `httptest.Server` as a stub backend.
Each test constructs an HTTP request, routes it through the handler under test,
and asserts on the recorded response (status code, headers, body).

**Coverage required for every new handler:**

| Test case | What it verifies |
|---|---|
| `_Proxy_ForwardsRequest` | Gateway makes exactly one upstream call; response status/body are preserved |
| `_Proxy_BackendError` | 5xx from backend → gateway returns S3-shaped error XML; no panic |
| `_Proxy_MissingBucket` | Route variable `bucket` always present; handler does not panic on empty string |
| `_Proxy_HeadersFiltered` | Hop-by-hop headers (`Connection`, `Transfer-Encoding`, etc.) are stripped |

**Additional handler-specific cases:**

| Handler | Extra unit tests |
|---|---|
| `handleDeleteBucket` | Audit event emitted on success; warning logged when policy refs bucket |
| `handleListBuckets` | Root `/` route resolves correctly; response XML is forwarded verbatim |
| `handleGetBucketEncryption` / `handlePutBucketEncryption` / `handleDeleteBucketEncryption` | Standard proxy cases (same 4 base cases); no special interception — backend SSE config is the backend's concern |
| `handleSelectObjectContent` | Returns `501 NotImplemented` with S3 error XML body |
| `handleCORSPreflight` | `OPTIONS` → backend CORS headers reflected; `Access-Control-*` headers present |
| `copyProxyResponse` | Status code, body, and safe headers copied; hop-by-hop stripped |
| `handlePassthrough` | Metric `RecordHTTPRequest` called once with correct operation name; audit event logged |

**New capability bits required:**  
New `Capabilities` constants must be added to `test/provider/capabilities.go`
(see §8.2). Unit tests that assert capability-gated behaviour must compile with
no build tag and must not start Docker.

## 10. Known Limitations (Explicit Out-of-Scope for v1.0)

| Operation | Reason |
|---|---|
| `SelectObjectContent` | Requires server-side SQL evaluation on decrypted data — not feasible in a proxy model |
| `WriteGetObjectResponse` | S3 Object Lambda integration — proxy model incompatible |
| S3 Batch Operations API | Separate control-plane API (`s3control`) — out of scope |
| Accelerate endpoint | DNS-level routing feature — not applicable to a proxy |
