# V0.6-S3-2 — Object Lock / Retention / Legal Hold — Implementation Plan

Status: Draft
Owner: API

Labels: `milestone:v0.6`, `area:s3`, `area:security`, `area:compliance`

## 1. Context & Current State

The audit performed on 2026-04-17 (see `/docs/plans/V0.6-S3-2-audit-2026-04-17`
in this plan's §11 references for the full report) established that
Object Lock is **entirely unimplemented** in the gateway today, despite
a line in `docs/S3_API_IMPLEMENTATION.md:335-338` claiming
"Legal hold / Retention: Pass through to backend". The documentation
claim is **aspirational and not supported by code**. Concretely:

- **Six subresource endpoints are unrouted.** The route table at
  `internal/api/handlers.go:90-118` does not match `?retention`,
  `?legal-hold`, or `?object-lock`. Requests for these subresources
  fall through to generic PUT/GET handlers, which will:
  - **encrypt the retention-config XML body as if it were an object
    payload** on `PUT /{b}/{k}?retention` — storing gibberish ciphertext
    at `/{b}/{k}` with subresource qualifier silently ignored, and
  - return unexpected 404/NotImplemented errors on the bucket-level
    endpoints.
- **All Object-Lock request headers are silently stripped on PUT.**
  The metadata-extraction loop at `internal/api/handlers.go:1100-1108`
  whitelists only `x-amz-meta-*`. Any of the six canonical headers
  (`x-amz-object-lock-mode`,
  `x-amz-object-lock-retain-until-date`,
  `x-amz-object-lock-legal-hold`,
  `x-amz-bypass-governance-retention`,
  `x-amz-object-lock-enabled-for-bucket`,
  `x-amz-object-lock-token`) are dropped before
  `s3Client.PutObject` (`handlers.go:1292`). The `Client.PutObject`
  signature (`internal/s3/client.go:27`) has **no parameter** for
  lock/retention. The AWS SDK v2 `PutObjectInput` supports
  `ObjectLockMode`, `ObjectLockRetainUntilDate`, and
  `ObjectLockLegalHoldStatus` fields, but these are never populated
  (`internal/s3/client.go:282-293`). A caller who issues a
  well-formed PUT with retention headers, expecting compliance
  coverage, receives **an unlocked object**. This is worse than
  "not supported" — it is silent non-compliance.
- **CopyObject and UploadPartCopy drop lock headers identically.**
  `handleCopyObject` (`handlers.go:2542-2735`) reads only
  `x-amz-tagging`; the destination metadata loop at
  `:2633-2640` whitelists `x-amz-meta-*` plus the standard-header set.
  UploadPartCopy (`internal/api/upload_part_copy.go:89-401`) inspects
  only `x-amz-copy-source` and `-source-range`. No lock field on the
  SDK request is set.
- **GET/HEAD do not surface Object-Lock response headers.** The SDK
  `HeadObjectOutput` exposes `ObjectLockMode`, `ObjectLockRetainUntilDate`,
  and `ObjectLockLegalHoldStatus`; `s3Client.HeadObject`
  (`internal/s3/client.go:418-451`) reads none of them. Similarly for
  GET at `internal/s3/client.go:306-386`. Clients cannot query lock
  state through the gateway.
- **`Client` interface has no Retention / LegalHold methods.** See
  `internal/s3/client.go:25-44`. No `RetentionConfig`, `LegalHold`, or
  `ObjectLockConfiguration` types exist.
- **No admin authentication path exists.** Grep of
  `internal/api/`, `internal/config/`, and `cmd/server/` finds zero
  matches for "admin" or "bypass". `x-amz-bypass-governance-retention`
  therefore has no mechanism by which to gate its use.
- **Encryption–lock interaction is unexamined.** No ADR, comment, or
  design doc acknowledges that locks in this gateway necessarily
  apply to the **ciphertext** (what the backend stores) rather than
  the plaintext (what the caller thinks they are locking). This
  matters because:
  1. Key rotation and re-encryption (handlers.go:902-905 and
     `docs/adr/0003-kms-implementation-scope-v05.md`) assume objects
     can be rewritten. A compliance-mode lock makes the ciphertext
     unrewritable until the retention period expires — **silently
     breaking the rotation workflow** for locked objects.
  2. Retention guarantees *byte durability*, not *readable-data
     durability*. If the wrapping KEK for a locked ciphertext is
     rotated out of the available set, the locked bytes become
     unrecoverable. The caller's mental model ("lock my plaintext for
     7 years") is incorrect without explicit guidance.
- **Signature V4 pass-through is a latent partial pass-through.** The
  `forwardSignatureV4Request` path (`handlers.go:189-332`) copies all
  headers verbatim at `:256-262`, so SigV4 requests against
  **unencrypted** buckets would accidentally convey Object-Lock
  headers. This is incidental, not by design, and the behaviour for
  encrypted PUTs is the non-forward path described above.

The audit report's classification of these gaps (see §11 ref 1) is the
baseline this plan treats as the pre-change state.

## 2. Scope Boundary with V0.6-CFG-1

The only S3 header in the Object-Lock surface that requires
caller-identity-aware authorization is
`x-amz-bypass-governance-retention`. AWS S3 gates it on the
`s3:BypassGovernanceRetention` IAM permission; the gateway has no
equivalent admin/IAM layer today (the audit confirms zero matches for
"admin" / "bypass" across `internal/api/`, `internal/config/`, and
`cmd/server/`).

This plan **does not** introduce an admin-authorization mechanism.
V0.6-CFG-1 ("Admin API for Key Rotation") is the designated owner of
that primitive and will arrive with:

- a config stanza describing admin principals / authentication,
- a reusable `(*Handler).isAdmin(r)` check,
- audit event plumbing for admin decisions.

Until V0.6-CFG-1 ships, this plan adopts a fail-closed stance: any
request carrying `x-amz-bypass-governance-retention` is refused with
`403 AccessDenied` and an audit event with
`reason=admin_authorization_not_implemented`. This is the correct
default per *IT Auditing* (Kegerreis et al., 2019) Ch. 8
§"Account Management" [8] — admin surfaces must default-deny until
the authorization mechanism exists.

When V0.6-CFG-1 lands, a small follow-up
(**V0.6-S3-2-followup** in §10) will:
1. Replace the unconditional refusal with the `isAdmin` check,
2. Honour the per-bucket `PolicyConfig.DisallowLockBypass` opt-out
   introduced here, and
3. Add the conditional-forward test cases deferred in §6.

Governance-mode retention is still fully implemented by this plan —
only the *bypass* path is deferred. Callers who want to reduce a
governance-mode retention before V0.6-CFG-1 must issue the operation
directly against the backend. This is documented in
`docs/S3_API_IMPLEMENTATION.md` per Phase G.

Priority remains **P2** as originally tracked. The silent-drop
framing in §1 is noted for operator awareness but does not promote
priority within this plan.

## 3. Design Goals & Non-Goals

### Goals

1. **No silent drops.** Every Object-Lock request header either
   reaches the backend in the correct SDK field, or the request is
   rejected with an explicit S3 error that makes the gap visible to
   clients.
2. **All six subresource endpoints routed.** PutObjectRetention /
   GetObjectRetention / PutObjectLegalHold / GetObjectLegalHold /
   PutObjectLockConfiguration / GetObjectLockConfiguration have
   dedicated handlers that validate XML bodies and forward to the
   backend.
3. **Headers on PUT / CopyObject / UploadPartCopy propagate
   end-to-end.** Add `ObjectLockMode`, `ObjectLockRetainUntilDate`,
   and `ObjectLockLegalHoldStatus` to the `Client` interface's
   PutObject and CopyObject paths so the re-encryption-mediated write
   preserves caller-supplied lock state.
4. **GET / HEAD surface backend lock state.** Extract the three SDK
   output fields and include them as response headers so clients can
   observe retention timestamps and legal-hold status through the
   gateway.
5. **Fail-closed bypass refusal.**
   `x-amz-bypass-governance-retention` is unconditionally refused
   with `403 AccessDenied` + audit event. Admin-authorized forwarding
   is scoped to V0.6-CFG-1. **Never silently dropped.**
6. **Provider-support matrix documented.** Per-backend capability
   table with CI-ready test tags for MinIO, Ceph RGW, AWS S3,
   Backblaze B2, Garage, and Cloudflare R2. Unsupported providers
   return `NotImplemented` with a pointer to the matrix.
7. **ADR 0007** documents the ciphertext-locking semantics and the
   interaction with key rotation.
8. **Scaffolding for V0.6-CFG-1.**
   `PolicyConfig.DisallowLockBypass` is introduced as a
   forward-compatible per-bucket opt-out flag so when V0.6-CFG-1
   wires admin-gated bypass it can consume the flag without a
   second config migration. This plan validates the flag parses
   correctly but does not branch on it (bypass is unconditional
   refusal regardless). Pattern consistent with the
   `PolicyConfig.RequireEncryption` field introduced in V0.6-S3-1.

### Non-Goals

- **Gateway-side WORM enforcement.** The gateway does not attempt to
  re-implement retention; it delegates to the backend entirely. If the
  backend does not support Object Lock, the gateway returns
  `NotImplemented` — it does not try to simulate WORM locally.
  Rationale: client-side WORM by an encrypting proxy is a research
  problem (roughly, cryptographically-enforced retention via
  time-locked key release) out of scope for this milestone.
- **Changing the encryption model to lock plaintext.** Locks apply to
  ciphertext. A future ADR could explore plaintext-manifest locking,
  but that is a much larger design change.
- **Admin authorization for `x-amz-bypass-governance-retention`.**
  Scoped to V0.6-CFG-1. This plan refuses the header unconditionally
  until the admin primitive exists. See §2 for the boundary rationale.
- **Per-object IAM / granular caller identity for any header.** Not
  in v0.6 at all; a v1.0 concern.
- **Default-on enablement.** Object Lock routing and header handling
  are shipped enabled by default (no feature flag) because the
  alternative — silently dropping headers — is the worse failure mode.
  Per-bucket enable/disable is governed by the backend's own
  Object-Lock configuration.
- **Retroactive locking of existing objects.** Per the S3 contract,
  only new versions can receive lock state via PUT-time headers; the
  gateway follows this.

## 4. Target Request / Response Semantics

### 4.1 Subresource routing

Six new handlers registered in `internal/api/handlers.go:90-118`
ordering matters — add **above** the generic PUT at `:112` so query
parameters match first:

```go
// Object Lock subresources (object-level)
s3Router.HandleFunc("/{bucket:[^/]+}/{key:.+}", h.handleGetObjectRetention).
    Methods("GET").Queries("retention", "")
s3Router.HandleFunc("/{bucket:[^/]+}/{key:.+}", h.handlePutObjectRetention).
    Methods("PUT").Queries("retention", "")
s3Router.HandleFunc("/{bucket:[^/]+}/{key:.+}", h.handleGetObjectLegalHold).
    Methods("GET").Queries("legal-hold", "")
s3Router.HandleFunc("/{bucket:[^/]+}/{key:.+}", h.handlePutObjectLegalHold).
    Methods("PUT").Queries("legal-hold", "")

// Object Lock configuration (bucket-level)
s3Router.HandleFunc("/{bucket}", h.handleGetObjectLockConfiguration).
    Methods("GET").Queries("object-lock", "")
s3Router.HandleFunc("/{bucket}", h.handlePutObjectLockConfiguration).
    Methods("PUT").Queries("object-lock", "")
```

Each handler:

1. Reads the XML body (for PUT) with `io.LimitReader(..., maxXMLSize)`
   mirroring the multipart XML parser at `handlers.go:1988`.
2. Validates the XML using stdlib `encoding/xml` with strict mode and
   explicit struct tags (no anonymous empty interfaces). Attacker-
   controlled XML is parsed with the same posture applied in V0.6-S3-1
   to `CompleteMultipartUpload`.
3. Invokes the corresponding new `Client` interface method (§4.3).
4. Forwards the backend response verbatim for GET. For PUT, returns
   `200 OK` with no body per the S3 contract.

### 4.2 Request-header propagation on PUT / Copy / UploadPartCopy

Three headers are extracted on every object-write path and passed
through the `Client` layer via a new `ObjectLockInput` struct:

| Header | SDK field | Validation |
|---|---|---|
| `x-amz-object-lock-mode` | `ObjectLockMode` | Must be `GOVERNANCE` or `COMPLIANCE`; 400 InvalidArgument otherwise. |
| `x-amz-object-lock-retain-until-date` | `ObjectLockRetainUntilDate` | RFC 3339 timestamp; 400 InvalidArgument otherwise. Must be in the future. |
| `x-amz-object-lock-legal-hold` | `ObjectLockLegalHoldStatus` | Must be `ON` or `OFF`; 400 InvalidArgument otherwise. |

Integration points:

- **`handlePutObject`** (`handlers.go:1044-1308`): extract after the
  metadata loop at `:1108`; pass to the modified `PutObject` call at
  `:1292`.
- **`handleCopyObject`** (`handlers.go:2542-2735`): extract after
  `x-amz-tagging` read at `:2561`; the destination `PutObject` call at
  `:2706` receives the lock input. Source-side lock state is
  **not** automatically inherited by the destination — this mirrors
  the AWS S3 behaviour (lock state on copy is destination-explicit).
- **`handleUploadPartCopy`** (`upload_part_copy.go:89-401`): locks
  **cannot attach to individual parts**; they attach at
  `CompleteMultipartUpload`. This plan therefore extends
  `handleCompleteMultipartUpload` (`handlers.go:2315-...`) to read the
  three lock headers and pass them to
  `s3Client.CompleteMultipartUpload`.

### 4.3 `Client` interface extension

```go
// internal/s3/client.go

type ObjectLockInput struct {
    Mode               string    // "GOVERNANCE" | "COMPLIANCE" | ""
    RetainUntilDate    *time.Time
    LegalHoldStatus    string    // "ON" | "OFF" | ""
}

// New methods on Client interface:
// Note: PutObjectRetention takes no bypassGovernance flag today because
// bypass is refused at the handler layer before reaching this method (see
// §4.5). V0.6-CFG-1 will add the flag here once admin-authz exists.
PutObjectRetention(ctx context.Context, bucket, key string,
    versionID *string, retention *RetentionConfig) error
GetObjectRetention(ctx context.Context, bucket, key string,
    versionID *string) (*RetentionConfig, error)
PutObjectLegalHold(ctx context.Context, bucket, key string,
    versionID *string, status string) error
GetObjectLegalHold(ctx context.Context, bucket, key string,
    versionID *string) (string, error)
PutObjectLockConfiguration(ctx context.Context, bucket string,
    config *ObjectLockConfiguration) error
GetObjectLockConfiguration(ctx context.Context, bucket string) (*ObjectLockConfiguration, error)

// Modified methods (adding *ObjectLockInput parameter):
PutObject(ctx context.Context, bucket, key string, reader io.Reader,
    metadata map[string]string, contentLength *int64, tagging string,
    lock *ObjectLockInput) error
CopyObject(ctx context.Context, dstBucket, dstKey string, srcBucket, srcKey string,
    srcVersionID *string, metadata map[string]string,
    lock *ObjectLockInput) (string, map[string]string, error)
CompleteMultipartUpload(ctx context.Context, bucket, key, uploadID string,
    parts []CompletedPart, lock *ObjectLockInput) (string, error)

// Supporting types:
type RetentionConfig struct {
    Mode            string    // "GOVERNANCE" | "COMPLIANCE"
    RetainUntilDate time.Time
}

type ObjectLockConfiguration struct {
    ObjectLockEnabled string         // "Enabled"
    Rule              *LockRule
}

type LockRule struct {
    DefaultRetention *DefaultRetention
}

type DefaultRetention struct {
    Mode  string // "GOVERNANCE" | "COMPLIANCE"
    Days  *int32
    Years *int32 // exactly one of Days/Years
}
```

Signature changes to existing methods are a breaking change to the
`Client` interface. A migration strategy is described in §5 Phase A.

### 4.4 GET/HEAD response-header surfacing

`s3Client.HeadObject` (`internal/s3/client.go:418-451`) and
`s3Client.GetObject` (`:306-386`) are modified to copy the three lock
fields from the SDK output into the metadata map with canonical
response-header names:

```go
if result.ObjectLockMode != "" {
    metadata["x-amz-object-lock-mode"] = string(result.ObjectLockMode)
}
if result.ObjectLockRetainUntilDate != nil {
    metadata["x-amz-object-lock-retain-until-date"] =
        result.ObjectLockRetainUntilDate.UTC().Format(time.RFC3339)
}
if result.ObjectLockLegalHoldStatus != "" {
    metadata["x-amz-object-lock-legal-hold"] = string(result.ObjectLockLegalHoldStatus)
}
```

`handleHeadObject` (`handlers.go:1389-1467`) and `handleGetObject`
already loop over the metadata map writing response headers; no change
needed at the handler layer once the client layer populates the map.
`isEncryptionMetadata` (used to filter in the response-writing loops)
correctly excludes these keys because they do not start with
`x-amz-meta-encryption-`.

### 4.5 `x-amz-bypass-governance-retention` — fail-closed refusal

Until V0.6-CFG-1 introduces admin authorization (see §2), every
request carrying this header is refused with `403 AccessDenied` and
an audit event with `reason=admin_authorization_not_implemented`.

The header is parsed in **every path that might otherwise honour
it** — PutObjectRetention, DeleteObject, DeleteObjects — so the
refusal is consistent regardless of entry point. Silent drop is
never acceptable; the caller is always informed.

```go
// internal/api/object_lock.go (sketch)
if r.Header.Get("x-amz-bypass-governance-retention") == "true" {
    s3Err := &S3Error{
        Code:       "AccessDenied",
        Message:    "Governance retention bypass is not enabled on this gateway.",
        Resource:   r.URL.Path,
        HTTPStatus: http.StatusForbidden,
    }
    s3Err.WriteXML(w)
    if h.auditLogger != nil {
        h.auditLogger.LogAccessWithMetadata(
            "bypass_governance_refused", bucket, key,
            getClientIP(r), r.UserAgent(), getRequestID(r),
            false, fmt.Errorf("bypass not authorised"), time.Since(start),
            map[string]interface{}{
                "reason": "admin_authorization_not_implemented",
                "note":   "deferred to V0.6-CFG-1",
            },
        )
    }
    return
}
```

When V0.6-CFG-1 lands, the refusal block is replaced by a call to
`h.isAdmin(r)` plus a `PolicyManager.BucketDisallowsLockBypass` check.
The audit event shape stays the same — only the outcome branches
differ. This keeps the dashboard continuity and the follow-up
migration trivial.

### 4.6 Per-bucket policy scaffolding (forward-compatibility)

`internal/config/policy.go`'s `PolicyConfig` gets one new field:

```go
// DisallowLockBypass, when true, will refuse
// x-amz-bypass-governance-retention for matching buckets even when
// the caller is otherwise admin-authorized.
//
// This flag is INTRODUCED in V0.6-S3-2 as forward-compatibility
// scaffolding; its gating semantics are ACTIVATED in V0.6-CFG-1 when
// the admin-authorization check lands. Until then, every bypass
// request is refused unconditionally regardless of this flag (§4.5).
//
// Consistent with RequireEncryption introduced in V0.6-S3-1 / ADR 0006.
DisallowLockBypass bool `yaml:"disallow_lock_bypass,omitempty"`
```

`PolicyManager` gets a companion `BucketDisallowsLockBypass(bucket string) bool`
that returns the flag value for test assertions but is not yet
consulted on the request path. V0.6-CFG-1 will wire the consult.

Parsing and policy-reloader coverage is validated in this plan so
V0.6-CFG-1 arrives to a working policy surface.

### 4.8 Error surface

| Condition | HTTP | Code | Notes |
|---|---|---|---|
| Malformed retention / legal-hold XML body | 400 | `MalformedXML` | Reuse XML validator posture from `handlers.go:1982-2114`. |
| Invalid `x-amz-object-lock-mode` value | 400 | `InvalidArgument` | Not `GOVERNANCE`/`COMPLIANCE`. |
| Malformed retain-until date | 400 | `InvalidArgument` | Non-RFC3339. |
| Retain-until in the past | 400 | `InvalidArgument` | Mirrors S3 contract. |
| `x-amz-bypass-governance-retention: true` on any request | 403 | `AccessDenied` | Unconditional in this plan; audit event `reason=admin_authorization_not_implemented`. V0.6-CFG-1 replaces with the admin-gated decision. |
| Backend rejects lock operation (unsupported provider) | 501 | `NotImplemented` | Point to provider matrix. |
| Backend is MinIO/Ceph and bucket not created with Object-Lock enabled | 409 | `InvalidRequest` | `ObjectLockConfigurationNotFoundError` from backend, translated. |

## 5. Work Breakdown

### Phase A — `Client` interface migration
- A1. Add `ObjectLockInput`, `RetentionConfig`,
  `ObjectLockConfiguration`, `LockRule`, `DefaultRetention` types
  to `internal/s3/client.go`.
- A2. Add the six new methods and modify `PutObject`, `CopyObject`,
  `CompleteMultipartUpload` signatures to take `*ObjectLockInput`.
  `nil` means "no lock applied" and preserves current behaviour.
- A3. Implement on `s3Client` via the AWS SDK V2 input fields. For
  the subresource ops use the SDK's `PutObjectRetentionInput`,
  `PutObjectLegalHoldInput`, `PutObjectLockConfigurationInput` and
  their Get counterparts.
- A4. Implement on `ProxyClient` (`internal/s3/proxy_client.go`) via
  HTTP forwarding — same pattern as the V0.6-S3-1
  `UploadPartCopy` forwarding at `proxy_client.go:210-288`.
- A5. Update every caller of the three modified methods to pass `nil`
  for `*ObjectLockInput` unless lock is explicitly requested.
  Approximately 14 call sites per the V0.6-S3-1 audit; each is a
  minimal mechanical change.

### Phase B — Object write-path integration
- B1. `handlePutObject` (`handlers.go:1044-1308`): extract the three
  lock headers into `ObjectLockInput`; validate per §4.2; pass to
  `s3Client.PutObject`.
- B2. `handleCopyObject` (`handlers.go:2542-2735`): same, pass to
  destination `PutObject`.
- B3. `handleCompleteMultipartUpload` (`handlers.go:2315-...`): extract
  lock headers; pass to `s3Client.CompleteMultipartUpload`.
- B4. `handleUploadPartCopy` (`upload_part_copy.go`): no change —
  parts don't carry lock state.

### Phase C — Read-path response surfacing
- C1. Extend `s3Client.HeadObject` and `s3Client.GetObject` to
  extract the three SDK output fields into the returned metadata
  map.
- C2. Verify `isEncryptionMetadata` filter at the response-writing
  loops does not accidentally filter these keys.
- C3. Update `ProxyClient` GetObject / HeadObject to parse the same
  headers from the backend response.

### Phase D — Subresource endpoint handlers
- D1. New file `internal/api/object_lock.go` with six handlers —
  mirroring the single-file layout used for `upload_part_copy.go`
  in V0.6-S3-1.
- D2. XML body validation with stdlib `encoding/xml`; strict
  decoder; 100 KiB max body size (retention and lock-config XML
  are small fixed structures).
- D3. Route registration added to `handlers.go:90-118` as shown in
  §4.1.

### Phase E — Bypass refusal & V0.6-CFG-1 scaffolding
- E1. Central helper
  `refuseBypassGovernanceRetention(w, r, h, bucket, key, start)` in
  `internal/api/object_lock.go` that writes the 403 + audit event
  shape from §4.5. Called from **every** path where the header
  could be honoured: PutObjectRetention, DeleteObject,
  DeleteObjects.
- E2. `PolicyConfig.DisallowLockBypass` +
  `PolicyManager.BucketDisallowsLockBypass` added as scaffolding
  (parses & validates; not yet consulted on the request path —
  consumed by V0.6-CFG-1).
- E3. Test coverage for the parse / policy-reloader paths so
  V0.6-CFG-1 arrives to a working knob.

**Explicitly out of scope for Phase E:** `AdminConfig`,
`(*Handler).isAdmin`, principal extraction, shared-secret
authentication. These are V0.6-CFG-1 deliverables.

Per §6.

### Phase G — Documentation
- G1. ADR 0007
  `docs/adr/0007-object-lock-ciphertext-semantics.md` documenting:
  - Locks apply to ciphertext.
  - Interaction with key rotation (locked objects cannot be
    rewritten; rotation worker must skip and audit).
  - Plaintext-durability caveat: retention guarantees byte
    persistence, not readable-data persistence. Losing the KEK for
    a locked object means the caller has a locked, unreadable blob.
  - Provider-support matrix (authoritative copy).

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Breaking `Client` interface changes break out-of-tree consumers. | The interface is already internal (`internal/s3/`). No public API is exposed. Changes are contained. |
| Lock headers silently stripped in existing deployments that pretend compliance. | This is the **current** state — ship the fix default-on so operators cannot regress into it. Release-note the behaviour change prominently. |
| Deferring admin-gated bypass means operators cannot reduce governance-mode retention via the gateway. | Documented in `docs/S3_API_IMPLEMENTATION.md` per Phase G: operators must target the backend directly for governance-decrease operations until V0.6-CFG-1 lands. This is the strictly-safer stance per *IT Auditing* (Kegerreis et al., 2019) Ch. 8 §"Account Management" [8] — admin surfaces must default-deny until the authorization mechanism exists. |
| Follow-up migration in V0.6-CFG-1 regresses an already-shipped behaviour. | The refusal response shape (XML body + audit event) is frozen in this plan and becomes the "deny branch" of the V0.6-CFG-1 admin check. Dashboard / audit-log queries remain stable. The scaffolding test `TestPolicyConfig_DisallowLockBypass_ParsedButNotConsulted` explicitly flips in V0.6-CFG-1, making the cutover visible in the test diff. |
| Compliance-mode objects cannot be rotated → silent accumulation of un-rotated KEK references. | Emit `gateway_rotation_skipped_locked_total` metric + per-object log; operators monitor. Documented in KEY_ROTATION_RUNBOOK per Phase G4. A future proposal (v1.0) could introduce a "rotation anchoring" scheme where the KEK version used on a locked object is held in escrow with matching retention — out of scope here. |
| Ciphertext-locking surprises users who assumed plaintext-locking. | ADR 0007 + S3_API_IMPLEMENTATION.md explicit section. *Security Engineering* Ch. 26.5 [2] on evidence-chain clarity: if operators do not know what is locked, the lock is evidentially worthless. |
| XML parsing edge cases (XXE, billion-laughs). | stdlib `encoding/xml` with `Strict=true`, `AutoClose=nil`, and an `io.LimitReader` — same posture as the V0.6-S3-1 multipart XML parser at `handlers.go:1982-2114`. Fuzz test in §6.4. |
| SDK response field types differ across AWS SDK minor versions. | Pin to the current `aws-sdk-go-v2/service/s3` version in go.mod; verify on dependabot bumps via the existing `TestObjectLock_*` suite. |
| Per-backend support differences not caught by tests. | Provider matrix is CI-enforced via build tags; PRs adding new providers must declare their matrix row. |

## 10. Follow-ups / Out-of-scope items surfaced during planning

1. **V0.6-S3-2-followup — admin-gated bypass (owned by V0.6-CFG-1).**
   When V0.6-CFG-1 lands the `(*Handler).isAdmin` primitive, a small
   follow-up replaces the unconditional `refuseBypassGovernanceRetention`
   helper (§4.5) with an admin check that consults
   `PolicyManager.BucketDisallowsLockBypass` as the second-line
   defence. The deferred test names are already enumerated in §6.1.
   Estimated follow-up effort: 0.5 d after V0.6-CFG-1 primitives exist.
2. **Locked-object-aware rotation worker.** This plan emits the
   skip metric and documents the skip behaviour, but the worker
   itself needs its own work-item to properly implement a retry
   queue with "awaiting retention expiry" as a first-class state.
3. **Rotation anchoring / KEK escrow.** A design for guaranteeing
   readable-data durability over retention windows — likely an
   escrow of all KEK versions used by any locked object, refusing
   to retire a KEK version until all locked objects referencing it
   have expired. ADR 0007 flags this as a v1.0 concern.
4. **Conditional retention update (governance-mode decrease).**
   AWS S3 allows reducing a governance-mode retention with
   `x-amz-bypass-governance-retention`. Without admin auth in v0.6,
   callers must target the backend directly for this operation;
   documented in `docs/S3_API_IMPLEMENTATION.md` per Phase G. The
   V0.6-S3-2-followup above restores gateway-mediated reduction.
5. **Cross-region replication with Object Lock.** Out of scope for
   v0.6; would require a coordinated design with V0.6-OPS-1.
