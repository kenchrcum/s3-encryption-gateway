## 1. Context & Current State

### 1.1 What the chart validates today

| Layer | Mechanism | File | Coverage |
|---|---|---|---|
| Static lint | `helm lint` | built-in | YAML syntax, Chart.yaml required fields, template render success |
| Template-time guards | `{{- fail ... -}}` | `templates/validate.yaml` | 5 guards (track/valkey, ingress mutex, weighted sum=100, etc.) |
| CI smoke tests | `scripts/test-chart.sh` + `tests/ci-values.yaml` | shell | 3 render scenarios; grep-based assertions |
| Progressive delivery tests | `scripts/test-progressive-delivery.sh` | shell | 27 render + guard-rail tests (OPS-1) |
| `values.schema.json` | **missing** | — | **nothing** |

`helm lint` with a missing schema silently succeeds on *any* values
combination that renders. That means:

1. A typo like `config.encriptoin.password.value` renders a chart
   that simply omits the env var — the pod then crashes in a hot
   loop at startup. No lint-time signal.
2. Setting `replicaCount: "2"` (string instead of int) works in
   some template contexts and fails with a cryptic sprig error in
   others. No lint-time signal.
3. A user migrating from single-password mode to KMS who sets
   `keyManager.enabled.value: "true"` but forgets
   `cosmian.endpoint.value` gets a 500-loop at startup with a
   misleading error — again, no lint-time signal.
4. OPS-1's guards run only at template time. A CI job that runs
   `helm lint -f values.bad.yaml` without `--strict` *does not
   fire the guards at all* (fail is triggered during render, not
   during lint unless lint is invoked with a values file that
   reaches the template).

### 1.2 Why `values.schema.json` is the right layer

Helm's documented mechanism for values validation is a
JSONSchema draft-07 (`$schema:
"http://json-schema.org/draft-07/schema#"`) document placed at
`values.schema.json` next to `Chart.yaml`. Helm:

- Validates `values.yaml` against the schema during `helm lint`.
- Validates the *merged* values (base + `-f overlay.yaml` + `--set`)
  during `helm install`, `helm upgrade`, and `helm template`.
- Fails fast, **client-side**, with JSON-path-prefixed error
  messages ("validation error: replicaCount: Invalid type.
  Expected: integer, given: string").
- Makes the chart self-describing for IDEs (VS Code Helm, IntelliJ
  Kubernetes plugin) and for downstream tooling (`kubeconform`,
  `chart-testing`, `helm-schema-gen`).

Matt Butcher, Matt Farina & Josh Dolitsky, *Learning Helm*
(O'Reilly, 2021), Ch. 6 "Developing Templates" §"Validating Values
with JSON Schema" is the canonical reference; Andrew Block & Austin
Dewey, *Managing Kubernetes Resources Using Helm, 2nd Edition*
(Packt, 2022), Ch. 5 covers the conditional-validation patterns
(`if/then/else`, `dependencies`, `oneOf`) we need.

### 1.4 Out-of-scope (not covered by this plan)

- Server-side admission validation (a ValidatingWebhook or Kyverno
  policy) — that would catch runtime misuse but requires a
  separate admission controller; defer to v0.7. A follow-up is
  filed in §8.
- Schema for the Valkey subchart (`valkey.*`) — the subchart has
  its own schema; we whitelist the top-level block with
  `additionalProperties: true` and leave sub-validation to the
  subchart's own schema (which Helm already runs recursively when
  the subchart ships one).
- Auto-generated schema as sole source of truth. We use
  `helm values-schema-json` (or the newer `helm-schema` plugin)
  only as a **drift detector** in CI, never as the authoritative
  schema — the hand-written schema is richer than any generator
  can produce (conditional rules, pattern constraints, descriptions).

## 2. Stateful Invariants the Schema Must Encode

These are the non-trivial rules that a naïve "types-only" schema
would miss. Each has a corresponding OPS-1 template-time guard that
stays in place as the last line of defence; the schema surfaces the
error one stage earlier.

### I1. Track ⇒ external Valkey

`track` set to a non-empty string implies:
- `valkey.enabled` MUST be `false` (per-release subchart does not
  share state), AND
- `config.multipartState.valkey.addr.value` OR
  `config.multipartState.valkey.addr.valueFrom` MUST be set.

Schema encoding:
```json
{
  "allOf": [
    {
      "if":   { "properties": { "track": { "enum": ["blue", "green", "stable", "canary"] } }, "required": ["track"] },
      "then": {
        "properties": {
          "valkey": { "properties": { "enabled": { "const": false } } },
          "config": { "properties": { "multipartState": { "properties": { "valkey": { "properties": { "addr": {
            "anyOf": [
              { "properties": { "value":     { "type": "string", "minLength": 1 } }, "required": ["value"] },
              { "properties": { "valueFrom": { "type": "object", "minProperties": 1 } }, "required": ["valueFrom"] }
            ]
          }}}}}}}
        }
      }
    }
  ]
}
```

### I2. Mutually exclusive ingress paths

`ingress.enabled: true` AND `ingress.traefik.enabled: true` is
invalid. Encoded as `not` + `allOf`:
```json
{ "not": { "allOf": [
  { "properties": { "ingress": { "properties": { "enabled":         { "const": true } }, "required": ["enabled"] } } },
  { "properties": { "ingress": { "properties": { "traefik": { "properties": { "enabled": { "const": true } }, "required": ["enabled"] } } } } }
]}}
```

### I3. Weighted services require traefik.enabled

`ingress.traefik.weighted.enabled: true` implies
`ingress.traefik.enabled: true`. (OPS-1 guard #4.)

### I4. Weighted services weight sum = 100

JSONSchema draft-07 cannot express "sum of array field equals N"
declaratively. **Decision:** keep this as a template-time guard
only (OPS-1 guard #5); the schema adds a `minItems: 2` + per-item
`weight: { type: integer, minimum: 0, maximum: 100 }` so typos are
caught early, but the sum check remains in
`templates/validate.yaml`. Documented in §7 as an accepted
limitation.

### I5. KeyManager enabled ⇒ provider-specific required fields

`config.encryption.keyManager.enabled.value: "true"` implies:
- `config.encryption.keyManager.provider.value` in
  `["cosmian", "hsm-stub"]` (today's supported providers — extend
  the enum as new providers land), AND
- If `provider: cosmian`, then
  `cosmian.endpoint.value` OR `cosmian.endpoint.valueFrom` must
  be set, AND
- `cosmian.keys.value` OR `cosmian.keys.valueFrom` must be set.

### I6. `useClientCredentials` toggles backend creds

`config.backend.useClientCredentials.value: "true"` makes
`backend.accessKey` + `backend.secretKey` optional (already the
case in the template); schema documents this with a description
but does not enforce exclusion — operators may still want to set
default creds as a fallback. Documented choice.

### I7. TLS required when `useCertManager: false` AND `tls.enabled: true`

If `config.tls.enabled.value: "true"` AND
`config.tls.useCertManager.value: "false"`, then `certFile` AND
`keyFile` (value or valueFrom) must be set. Schema-encoded with
nested `if/then`.

### I8. FIPS overlay compatibility

The schema MUST NOT set `additionalProperties: false` at the root
or at `podAnnotations`, `extraEnv`, `image`, `resources`, because
`values.fips.yaml` (and user overlays) add arbitrary compliance
annotations and env vars. Fine-grained `additionalProperties:
false` is applied **only** at structured sub-trees where typos
are the real risk (`config.backend.*`, `config.encryption.*`,
`ingress.traefik.weighted.sticky.*`).

### I9. `valueFrom` is a sealed union

Every `{value, valueFrom}` pair must have exactly one of
`secretKeyRef | configMapKeyRef | fieldRef | resourceFieldRef`
set *inside* `valueFrom`. Encoded as `oneOf` per leaf.

### I10. Readiness and liveness probe exclusivity

If `livenessProbe.httpGet` is set, `livenessProbe.exec` and
`livenessProbe.tcpSocket` MUST be unset. (Standard k8s probe
semantics — schema enforces client-side.)

## 3. Phased Rollout

### Phase A — Skeleton schema + coverage baseline (0.5 d)

- [ ] Create `helm/s3-encryption-gateway/values.schema.json` with
      `$schema` draft-07 header, `type: object`, top-level
      `properties` block for every group listed in §1.3 with `type`
      and `description` only (no constraints yet).
- [ ] Every leaf carries a `description` pulled from the inline
      comment in `values.yaml`. Makes IDE tooltips work day-one.
- [ ] Run `helm lint helm/s3-encryption-gateway` — must still pass.
- [ ] Run `helm lint helm/s3-encryption-gateway -f tests/ci-values.yaml` —
      must still pass.
- [ ] Run `helm lint helm/s3-encryption-gateway -f values.fips.yaml` —
      must still pass.
- [ ] Run `helm lint` with each of the 6 example values files
      under `examples/` — must still pass.

Exit criteria: schema is present, zero false-positives across
existing overlays, zero schema-driven rejections.

### Phase B — Type constraints and enums (1.0 d)

- [ ] Add `type`, `minimum`/`maximum`, `pattern`, `enum` to every
      leaf. Examples:
      - `replicaCount: { type: integer, minimum: 0, maximum: 1000 }`
      - `image.pullPolicy: { enum: ["Always", "IfNotPresent", "Never"] }`
      - `service.type: { enum: ["ClusterIP", "NodePort", "LoadBalancer", "ExternalName"] }`
      - `service.port: { type: integer, minimum: 1, maximum: 65535 }`
      - `track: { enum: ["", "blue", "green", "stable", "canary"] }`
      - `config.logLevel.value: { enum: ["debug", "info", "warn", "error"] }`
      - `config.listenAddr.value: { type: string, pattern: "^:?[0-9]+$|^[0-9a-zA-Z\\.-]+:[0-9]+$" }`
      - Duration strings (`15s`, `5m`, `2160h`): `{ type: string, pattern: "^[0-9]+(ns|us|µs|ms|s|m|h)$" }`
      - Byte sizes (`1048576`): `{ type: string, pattern: "^[0-9]+$" }` (env vars come as strings).
- [ ] Apply `additionalProperties: false` selectively per I8.
- [ ] Golden-file negative tests: for each constraint, add a
      `tests/schema/bad-<rule>.yaml` that MUST be rejected.

Exit criteria: 100 % of leaf keys have a constrained `type`; ~30
negative tests in CI.

### Phase C — Sealed `valueFrom` union (0.5 d)

- [ ] Define a reusable `$defs` entry:
      ```json
      "$defs": {
        "valueFromRef": {
          "type": "object",
          "oneOf": [
            { "required": ["secretKeyRef"],      "properties": { "secretKeyRef":      { "$ref": "#/$defs/keyRef" } } },
            { "required": ["configMapKeyRef"],   "properties": { "configMapKeyRef":   { "$ref": "#/$defs/keyRef" } } },
            { "required": ["fieldRef"],          "properties": { "fieldRef":          { "$ref": "#/$defs/fieldRef" } } },
            { "required": ["resourceFieldRef"],  "properties": { "resourceFieldRef":  { "$ref": "#/$defs/resourceFieldRef" } } }
          ],
          "additionalProperties": false
        }
      }
      ```
- [ ] Define a `configValue` $def:
      `oneOf: [ { has value }, { has valueFrom (matching $defs/valueFromRef) } ]`.
      Allow empty `valueFrom: {}` as "unset" to preserve today's
      defaults in `values.yaml` (the empty `{}` satisfies the
      schema when `value` is present, because the schema branches
      on which key is non-empty).
- [ ] Migrate every `{value, valueFrom}` pair under `config.*` to
      `$ref: "#/$defs/configValue"`. ~80 call-sites.

Exit criteria: schema compile size is ≤ 2000 lines via $defs
reuse; every `{value, valueFrom}` leaf validated uniformly.

### Phase D — Conditional invariants I1–I10 (1.5 d)

- [ ] Encode I1 (track ⇒ external Valkey).
- [ ] Encode I2 (ingress mutex).
- [ ] Encode I3 (weighted ⇒ traefik.enabled).
- [ ] Encode I5 (KeyManager provider-specific required fields).
- [ ] Encode I7 (TLS cert presence).
- [ ] Encode I10 (probe mutual exclusion).
- [ ] Each invariant ships with a paired positive test

accepted as template-time; 20 paired tests pass; `helm lint` on
OPS-1 example files still clean.

### Phase F — CI gating (0.5 d)

Extend `.github/workflows/helm-test.yml`:

- [ ] Job `lint-base`: `helm lint helm/s3-encryption-gateway`.
- [ ] Job `lint-overlays`: matrix over
      `[values.prod.yaml, values.dev.yaml, values.fips.yaml, tests/ci-values.yaml, examples/*.yaml]`
      — each must `helm lint` cleanly.
- [ ] Job `schema-negative`: for every file under
      `tests/schema/bad-*.yaml`, assert
      `helm lint` exits **non-zero** with the expected error
      substring. Uses a small shell harness
      (`tests/schema/run-negative.sh`).
- [ ] Job `schema-drift`: runs
      `helm plugin install https://github.com/dadav/helm-schema` (or
      the `helm values-schema-json` equivalent) against
      `values.yaml`, diffs the generated schema skeleton against
      the hand-written schema on a **key-presence basis only**
      (not content) — fails if the hand-written schema is missing
      a key that exists in values.yaml. Prevents forgotten keys.
- [ ] Job `render-overlays`: `helm template` each overlay, pipe
      through `kubeconform -summary -schema-location default
      -schema-location 'https://raw.githubusercontent.com/datreeio/CRDs-catalog/main/{{.Group}}/{{.ResourceKind}}_{{.ResourceAPIVersion}}.json'`
      — catches k8s-level schema errors the Helm schema can't see
      (bad container port, unknown field in Deployment).
- [ ] Update `scripts/test-chart.sh` to invoke `helm lint` with
      the schema present; add a new test 4 that attempts to
      install the chart with a **known-bad** value file and
      asserts failure.

Exit criteria: CI red on any of:
- missing key in schema (drift),
- accidental schema relaxation (negative tests pass where they
  shouldn't),
- overlay lint failure,
- rendered manifest failing `kubeconform`.

| File | Rule violated | Expected error substring |

non-zero AND stderr must contain the expected substring.

### 5.3 Drift test

`helm plugin install https://github.com/losisin/helm-values-schema-json`
(or `dadav/helm-schema`) is run in CI; emits a machine-generated
schema; a small `scripts/schema-drift.sh` compares the **set of
JSON-path leaves** between generated and hand-written schemas and
fails if hand-written is missing any.

### 5.4 Render-level validation

`helm template` → `kubeconform` for each overlay. This catches:
- resource kinds/versions the gateway chart uses that the cluster
  does not support;
- field typos in `deployment.yaml` that render valid YAML but
  invalid k8s objects.

### 5.5 Snapshot tests

For `values.prod.yaml` and `values.dev.yaml`, a snapshot of the
rendered manifest list (`helm template | kustomize cfg grep`-
filtered to kind + name) is committed under
`tests/snapshots/`. CI fails on diff unless the snapshot is
regenerated (forcing an intentional review).

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Schema is too strict and breaks valid user overlays relying on undocumented fields. | `additionalProperties: false` applied *only* at structured sub-trees (§I8); root `additionalProperties` stays `true`; pre-release dry-run against a corpus of community values files pulled from `docs/examples/`. |
| Schema and `values.yaml` drift as new fields are added. | `schema-drift` CI job (§5.3) fails the build when a key appears in `values.yaml` but not the schema. Enforced, not aspirational. |
| Schema bloats to unreadable size. | Heavy use of `$defs` (reused `configValue`, `valueFromRef`, `keyRef`, `probe`, `resourceBlock`). Target ≤ 2000 lines. |
| Draft-07 cannot express "sum of array = N" (I4 weighted weights). | Template-time guard in `validate.yaml` stays. Documented in §2.I4. |
| `helm lint --strict` becomes a trap because schema validation failures look like k8s failures. | README adds a FAQ: "If `helm lint` fails with a JSON-path error, consult the description in `values.schema.json` at that path; the description has the fix." |
| Generator-produced drift schema disagrees with hand-written due to `$defs` reuse. | Drift check compares **leaf paths only**, not structures. Explicitly scripted in `scripts/schema-drift.sh`. |
| Existing community forks have values files that use typos we'll now reject. | Release notes flag OPS-2 as an "enhanced validation" minor bump; major typo fixes are documented in CHANGELOG.md; users can temporarily bypass with `--disable-openapi-validation` (but template guards still fire). |
| Subchart (Valkey) schema conflicts with ours. | Top-level `valkey: { type: object, additionalProperties: true }`; delegate to subchart's own schema. |
| SEC-2 FIPS overlay rejected because schema requires a FIPS key it doesn't have. | Schema designed so **no** value is required at the root — every required-ness is *conditional on another field being set*. FIPS overlay asserts no new conditions. |
| `helm install` on a cluster with an older Helm (≤3.4, which predates schema support) would silently skip validation. | Chart.yaml's `kubeVersion` minimum implicitly scopes Helm ≥3.5; doc note in README. |

## Appendix A. Schema skeleton sketch

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "https://github.com/kenchrcum/s3-encryption-gateway/helm/values.schema.json",
  "title": "s3-encryption-gateway Helm values",
  "description": "Validates values.yaml and overlays for the S3 Encryption Gateway Helm chart.",
  "type": "object",
  "additionalProperties": true,
  "properties": {
    "replicaCount": {
      "type": "integer",
      "minimum": 0,
      "maximum": 1000,
      "description": "Number of gateway pod replicas. Set to 0 to pause the deployment. Prod overlays set this to >=3."
    },
    "track": {
      "type": "string",
      "enum": ["", "blue", "green", "stable", "canary"],
      "description": "Deployment track label for blue/green and canary topologies. Empty string (default) renders no track label. See docs/OPS_DEPLOYMENT.md §2."
    },
    "image": {
      "type": "object",
      "additionalProperties": false,
      "required": ["repository"],
      "properties": {
        "repository": { "type": "string", "minLength": 1 },
        "tag":        { "type": "string" },
        "pullPolicy": { "enum": ["Always", "IfNotPresent", "Never"] }
      }
    },
    "config": { "$ref": "#/$defs/config" },
    "ingress": { "$ref": "#/$defs/ingress" }
  },
  "allOf": [
    { "$ref": "#/$defs/invariantTrackRequiresExternalValkey" },
    { "$ref": "#/$defs/invariantIngressMutex" },
    { "$ref": "#/$defs/invariantWeightedRequiresTraefik" },
    { "$ref": "#/$defs/invariantKeyManagerProviderRequired" },
    { "$ref": "#/$defs/invariantTLSCertRequired" }
  ],
  "$defs": {
    "configValue": {
      "type": "object",
      "properties": {
        "value":     { "type": ["string", "null"] },
        "valueFrom": { "$ref": "#/$defs/valueFromRef" }
      },
      "additionalProperties": false
    },
    "valueFromRef": {
      "type": "object",
      "oneOf": [
        { "type": "object", "maxProperties": 0 },
        { "required": ["secretKeyRef"] },
        { "required": ["configMapKeyRef"] },
        { "required": ["fieldRef"] },
        { "required": ["resourceFieldRef"] }
      ]
    }
  }
}
```

(Full schema written during Phase A–D; ~1400 lines with $defs
reuse.)

## Appendix B. Commands cheat-sheet (for the implementer)

```bash
# Phase A: create skeleton
$EDITOR helm/s3-encryption-gateway/values.schema.json

# Verify baseline lint is green
helm lint helm/s3-encryption-gateway
helm lint helm/s3-encryption-gateway -f helm/s3-encryption-gateway/values.fips.yaml
for f in helm/s3-encryption-gateway/examples/*.yaml; do
  helm lint helm/s3-encryption-gateway -f "$f" || echo "FAIL: $f"
done

# Phase F: negative harness local run
bash helm/s3-encryption-gateway/tests/schema/run-negative.sh

# Phase F: drift check local run
helm plugin install https://github.com/losisin/helm-values-schema-json || true
bash helm/s3-encryption-gateway/scripts/schema-drift.sh

# Phase E: render snapshots
helm template gw helm/s3-encryption-gateway -f helm/s3-encryption-gateway/values.prod.yaml \
  --set config.backend.accessKey.value=stub --set config.backend.secretKey.value=stub \
  --set config.encryption.password.value=stub \
  | yq 'select(.kind != null) | .kind + " " + .metadata.name' \
  | sort > helm/s3-encryption-gateway/tests/snapshots/values.prod.txt
```
