# V0.6-CFG-1 — Admin API for Key Rotation (Drain + Cutover) — Implementation Plan

Status: Draft
Owner: Config / Crypto
Priority: P1
Labels: `milestone:v0.6`, `area:kms`, `area:config`, `area:observability`

> Scope clarified relative to the one-line tracker entry at
> `docs/issues/v0.6-issues.md:560`. This plan is sequenced **before**
> V0.6-S3-2 so that the shared admin primitive (config stanza, separate
> admin listener, bearer-token auth, `(*Handler).isAdmin(r)`) exists when
> S3-2 needs it. Scope is strictly:
>
> 1. Invent the minimum admin subsystem the gateway lacks today
>    (§4 Phase A). No pprof, no diagnostic endpoints — V0.6-OBS-1 will be
>    planned and implemented separately and may consume the primitive
>    later.
> 2. Deliver a **metadata-only drain + cutover** rotation flow via a new
>    `/admin/kms/rotate/*` endpoint family (§4 Phase B–D). No bulk object
>    re-encryption. Existing objects continue to decrypt via the adapter's
>    dual-read window; new writes wrap DEKs with the new active key
>    version.
> 3. Extend the `KeyManager` contract with an **optional**
>    `RotatableKeyManager` extension interface (§4 Phase B) so the base
>    contract from V0.6-SEC-1 / ADR 0004 remains stable and the HSM stub
>    does not have to lie about capabilities.
>
> A new ADR (0007) will be filed alongside this plan documenting the
> metadata-only cutover semantics and the extension-interface pattern.

## 1. Context & Current State

### 1.1 There is no admin surface

- Route registration (`internal/api/handlers.go:89-118`) mounts only
  `/health`, `/ready`, `/live`, and the S3 gateway routes. No `/admin/*`
  subrouter, no admin middleware, no pprof. Server bootstrap in
  `cmd/server/main.go:557` wires `/metrics` then delegates to
  `handler.RegisterRoutes`.
- No admin authentication exists. `internal/api/auth.go` and
  `internal/api/credentials.go` implement SigV4/SigV2 for S3 **data-plane**
  requests only. `internal/middleware/` ships
  `bucket_validation`, `logging`, `recovery`, `security`, `tracing` — none
  of them authenticate.
- `Config` (`internal/config/config.go:19-35`) has no `Admin` stanza. The
  V0.6-S3-2 plan (see `docs/plans/V0.6-S3-2-plan.md:109-146`) already
  declares `(*Handler).isAdmin(r)` as a CFG-1 deliverable and forward-
  references an unimplemented admin flag. This plan is that implementation.

### 1.2 Key-versioning is already per-object; rotation is a metadata-only
### operation (critical)

- `internal/crypto/keymanager.go:72-80` defines `KeyEnvelope` with
  `KeyID`, `KeyVersion int`, `Provider`, `Ciphertext`, `CreatedAt`.
- The engine tags every encrypted object with those fields on write
  (`internal/crypto/engine.go:487-498` single-AEAD, `:903-914` chunked,
  `:1019-1028` fallback-chunked) via the metadata constants
  `MetaKeyVersion`, `MetaKMSKeyID`, `MetaKMSProvider`,
  `MetaWrappedKeyCiphertext`.
- On read, the engine **trusts the envelope metadata** and calls
  `kmsManager.UnwrapKey(ctx, env, metadata)` — there is no engine-level
  "active read version" (`internal/crypto/engine.go:579-611` single-AEAD,
  `:1124-1138` chunked, `:1287-1301` range).
- The chunked manifest (`internal/crypto/chunked.go:32-41`) intentionally
  carries **no** key-id or key-version — one DEK per object, all chunks
  share it. No per-chunk envelope tagging is required.

**Consequence.** Rotation does not require re-encrypting any object. It
is a two-step state change: (a) drain writes in flight under the old
active wrapping key; (b) flip the adapter's notion of "active version"
so the next `WrapKey` picks the new key. Reads of already-encrypted
objects work unchanged because the KMS retains old wrapping keys and
the per-object metadata names its own envelope. This is exactly the
envelope-encryption rotation pattern described in Wong, *Real-World
Cryptography* Ch. 9 §9.2.3 ("rotating keys without rotating ciphertext")
[1] and NIST SP 800-57 Pt. 1 §8.2.4 on cryptoperiod transitions as
summarised in Aumasson, *Serious Cryptography* §10 [2].

### 1.3 Adapter versioning state

- **Memory** (`internal/crypto/keymanager_memory.go`):
  `map[int][]byte keys` + `activeVersion int` (lines 49-56). Candidate
  lookup for decrypt iterates all versions (lines 209-224). **No exported
  mutator** — new versions can only be seeded via `WithMemoryVersions`
  at construction.
- **Cosmian KMIP** (`internal/crypto/keymanager_cosmian.go`): models
  `Keys []KMIPKeyReference` + `DualReadWindow int` (lines 25-32). Active
  is always `opts.Keys[0]` (line 150). Decrypt tries up to
  `DualReadWindow+1` candidates (lines 195-227). Again no exported
  rotate/promote method.
- **HSM stub** (`internal/crypto/keymanager_hsm_stub.go`): returns
  `ErrProviderUnavailable` for every method (as expected per ADR 0004).

## 2. Design Goals & Non-Goals

### Goals

1. Land a reusable admin subsystem (config, separate listener, bearer-
   token authn, `isAdmin(r)` helper) that V0.6-S3-2 and the future
   V0.6-OBS-1 can consume without rework.
2. Expose a drain + cutover workflow via four verbs on a single
   `/admin/kms/rotate` resource: `POST /start`, `GET /status`,
   `POST /commit`, `POST /abort`. Optional idempotency via a
   client-supplied rotation-id.
3. Metadata-only cutover: no reads of existing objects, no rewrite of
   ciphertexts, no bucket scans. The adapter's dual-read window
   (Cosmian `DualReadWindow`, memory `candidateVersions`) remains the
   grace-window primitive.
4. Treat rotation as a **long-running admin operation** with explicit
   state transitions (`idle → draining → ready_for_cutover →
   committing → committed | aborted`) and structured audit at every
   transition, per Adkins et al., *Building Secure and Reliable Systems*
   Ch. 9 ("Implementing Change") and Ch. 14 ("Deploying Code") [3].
5. Fail closed on adapter capability gaps: if the configured KeyManager
   does not implement `RotatableKeyManager`, the endpoints return
   `501 NotImplemented` with an actionable message.

### Non-Goals

- **Bulk re-encryption of existing objects.** Filed as follow-up
  *V0.6-CFG-1-followup-rewrap* (§10). Requires listing / pagination /
  throttling / resume semantics; out of scope for v0.6.
- **Changes to the envelope format or chunked manifest.** Existing
  metadata keys suffice.
- **pprof / profiling endpoints.** V0.6-OBS-1 will plan and land those;
  this plan does not mount pprof even though the admin mux is the
  natural place.
- **mTLS admin auth.** Bearer-token on a separate listener is sufficient
  for v0.6 (per user decision in planning Q1). mTLS remains an option
  for v0.7.
- **Piggybacking on SigV4 with a reserved admin access key.** Rejected
  for conflating data- and control-plane credentials (BSRS Ch. 5 ,
  "Limiting the blast radius") [3].
- **SIGHUP-triggered rotation.** Rotation is strictly admin-API-driven;
  `RotationPolicyConfig.GraceWindow` is reused as the drain-deadline
  knob but rotation itself is not triggered by config reload. This
  matches the "explicit operator action" principle from
  Beyer et al., *The Site Reliability Workbook* Ch. 16 [4].

## 3. Target HTTP Contract

### 3.1 Configuration surface (`internal/config/config.go`)

New `Admin` stanza on `Config`:

```yaml
admin:
  enabled: false            # default: off (opt-in)
  address: "127.0.0.1:8081" # separate listener; default: loopback
  tls:
    enabled: true           # forbidden=false when address is loopback-only
    cert_file: ""
    key_file: ""
  auth:
    type: "bearer"          # only "bearer" in v0.6
    token_file: ""          # file path; 0600, never inline
    # token: ""             # inline only permitted when ADMIN_ALLOW_INLINE_TOKEN=1 (dev)
  rate_limit:
    requests_per_minute: 30 # per-source-IP; defense in depth
```

Validation rules (in `Config.Validate`):

- If `admin.enabled=true` and `admin.address` is not loopback
  (`127.0.0.1`/`::1`), `admin.tls.enabled` must be `true`.
- Exactly one of `token_file` / `token` must be set; inline requires
  `ADMIN_ALLOW_INLINE_TOKEN=1`.
- Token minimum length 32 bytes decoded from hex/base64 (reject short
  tokens at startup).
- `admin.address` must differ from the data-plane `listen_addr`.

Hot-reload: admin stanza additions are forbidden mid-reload (added to
the guard at `config.go:918-970`) — the listener is only ever
started/stopped at process start.

### 3.2 Admin endpoint family

All routes are mounted on the separate admin mux. All require a valid
`Authorization: Bearer <token>` header (constant-time comparison, per
BSRS Ch. 7 on timing side-channels [3]). All responses are JSON.

| Verb | Path | Semantics |
|---|---|---|
| `POST` | `/admin/kms/rotate/start` | Initiate drain. Body: `{"target_version": int?, "grace_window_seconds": int?, "rotation_id": string?}`. 200 → `{state:"draining", rotation_id, started_at, ...}`. 409 if another rotation is in flight (unless the same `rotation_id` — idempotent). |
| `GET`  | `/admin/kms/rotate/status` | Returns `{state, rotation_id, started_at, in_flight_wraps, drain_deadline, current_version, target_version}`. Safe to poll. |
| `POST` | `/admin/kms/rotate/commit` | Requires state ∈ `{draining, ready_for_cutover}`. If still draining with in-flight > 0 past deadline → `409 Conflict` unless `{"force": true}`. Performs the adapter cutover and flips `state → committed`. |
| `POST` | `/admin/kms/rotate/abort` | Cancel from `{draining, ready_for_cutover}`. No-op from `committed`. Body: `{"reason": string}`. |

### 3.3 State machine

```
           start(target?)
   idle ─────────────▶ draining
                          │
   in_flight_wraps == 0   │   OR   deadline elapsed
                          ▼
                  ready_for_cutover
                          │
                       commit
                          ▼
                     committing (short) ─▶ committed
                          │
                       abort
                          ▼
                      aborted
```

Invariants (checked in tests):

- `state` transitions only forward, except `{draining,
  ready_for_cutover} → aborted`.
- A `committed` or `aborted` operation's terminal record is retained
  in-memory for a bounded window (`admin.retention` — 24 h default)
  purely for status polling; no persistence.
- Exactly one rotation is active at a time. Process-local mutex; this
  is acceptable because gateway replicas should be rotated
  independently (see §7 risk #3).

### 3.4 Error surface

| HTTP | Code | Cause |
|---|---|---|
| 401 | `Unauthorized` | Missing / malformed / invalid bearer |
| 403 | `Forbidden` | Valid token but source IP outside `admin.allow_cidrs` (future knob; in v0.6 the listener binding alone gates this) |
| 409 | `Conflict` | Rotation already in flight; or commit attempted with `in_flight > 0` and no `force` |
| 412 | `PreconditionFailed` | Admin subsystem disabled at runtime but endpoint reached (should be impossible; defensive) |
| 429 | `TooManyRequests` | Rate limit tripped |
| 501 | `NotImplemented` | Configured KeyManager does not implement `RotatableKeyManager` (e.g. HSM stub) |
| 503 | `ServiceUnavailable` | KMS `HealthCheck` failed at commit time |

All error bodies carry `{error:{code, message, rotation_id?}}`.

### Phase B — `RotatableKeyManager` extension interface

- [ ] Add to `internal/crypto/keymanager.go`:

```go
// RotatableKeyManager is an optional extension implemented by
// adapters that support runtime rotation. Adapters that cannot rotate
// (e.g. current HSM stub) intentionally do not implement it; callers
// type-assert and return ErrRotationNotSupported otherwise.
type RotatableKeyManager interface {
    KeyManager

    // PrepareRotation returns the adapter's view of the rotation
    // target (e.g. the next version it is willing to promote). If
    // the adapter can't choose without operator input it returns
    // ErrRotationAmbiguous; the admin endpoint must then require
    // target_version in the request body.
    PrepareRotation(ctx context.Context, target *int) (RotationPlan, error)

    // PromoteActiveVersion atomically advances the adapter's active
    // wrapping key to the previously-prepared target. Must be safe
    // to call concurrently with WrapKey/UnwrapKey: callers that
    // started before Promote observe old version; callers that
    // started after observe new. No ciphertext is rewritten.
    PromoteActiveVersion(ctx context.Context, plan RotationPlan) error
}

type RotationPlan struct {
    CurrentVersion int
    TargetVersion  int
    ProviderData   map[string]string // opaque adapter-specific hints
}

var (
    ErrRotationNotSupported = errors.New("key rotation not supported by adapter")
    ErrRotationAmbiguous    = errors.New("target_version required for this adapter")
    ErrRotationConflict     = errors.New("rotation already in progress or invalid state")
)
```

- [ ] Memory adapter (`keymanager_memory.go`): implement by extending
  the existing `keys` map and `activeVersion` field. New version's
  master-key material is pulled from a new `AddVersion(ctx,
  version int, material []byte)` method (separate from Promote so
  operators can stage material independently).
- [ ] Cosmian adapter (`keymanager_cosmian.go`): implement by
  reordering `opts.Keys` so the new reference becomes index 0.
  Adding new key references is still a config-file operation — the
  adapter's `PrepareRotation` reports which configured reference is
  eligible for promotion (next-higher version number not yet
  promoted).
- [ ] HSM stub: intentionally does **not** implement the interface.
  Admin endpoint returns 501 `NotImplemented` with message `"hsm
  adapter does not support runtime rotation in v0.6"` and references
  ADR 0004.
- [ ] All new methods get conformance-suite entries in
  `internal/crypto/conformance_test.go` (V0.6-SEC-1 precedent):
  `TestConformance_Rotation_RoundTrip`,
  `TestConformance_Rotation_Concurrency`,
  `TestConformance_Rotation_UnsupportedAdapter`.

### Phase C — Drain-tracking primitive

- [ ] Add `internal/crypto/rotation_state.go`:

```go
type RotationState struct {
    mu             sync.Mutex
    state          State // enum: Idle, Draining, ReadyForCutover, Committing, Committed, Aborted
    rotationID     string
    startedAt      time.Time
    plan           RotationPlan
    graceDeadline  time.Time
    inFlightWraps  atomic.Int64
    done           chan struct{}
    lastError      error
}
```

- [ ] Wrap `WrapKey` in the engine (`internal/crypto/engine.go:374-387,
  824-837, 950-960`) with `rotationState.beginWrap()` /
  `endWrap()` counters. Reads (`UnwrapKey`) are **not** tracked —
  the whole design relies on reads continuing unhindered during
  rotation.
- [ ] Drain goroutine: from `Draining` state, polls `inFlightWraps` at
  100 ms intervals; transitions to `ReadyForCutover` when it hits
  0, or when `graceDeadline` elapses (operator then decides whether
  to force-commit).
- [ ] Expose `RotationState.Snapshot()` returning a defensive copy for
  the `/status` endpoint — never hand out the struct directly.

### Phase D — Admin handlers for rotation

- [ ] New file `internal/api/admin_rotation.go`:
  - `handleRotateStart` — type-assert `engine.KeyManager()` to
    `RotatableKeyManager`; 501 if not implemented. Call
    `PrepareRotation`; persist plan on `rotationState`; transition
    to `Draining`; emit audit event `key_rotation.start`.
  - `handleRotateStatus` — marshal `Snapshot()`.
  - `handleRotateCommit` — require state ∈ `{Draining (with
    in_flight==0 or force), ReadyForCutover}`; call

    and emit `key_rotation.commit`; on failure → `Aborted` with
    error, emit `key_rotation.failure`.
  - `handleRotateAbort` — transition from `{Draining,
    ReadyForCutover}` to `Aborted`; emit `key_rotation.abort`.
- [ ] Register all four routes on the admin mux with bearer-auth and
  rate-limit middleware applied.
- [ ] Audit payloads use `LogAccessWithMetadata("key_rotation", …)`
  with `{step, rotation_id, actor:"admin-bearer", old_version,
  target_version, provider, result, duration_ms, in_flight_at_commit}`.
  This matches the shape the V0.6-S3-2 plan established for admin
  decisions (`docs/plans/V0.6-S3-2-plan.md:389-411`).
- [ ] New metrics in `internal/metrics/metrics.go`:
  - `kms_active_key_version{provider}` (gauge, updated at startup
    and after commit).
  - `kms_rotation_operations_total{step, result}` counter
    (`step ∈ {start, commit, abort}`).
  - `kms_rotation_duration_seconds{step}` histogram.
  - `kms_rotation_in_flight_wraps` gauge exported continuously
    (scrape-driven).

## 7. Risks & Mitigations

| # | Risk | Mitigation |
|---|---|---|
| 1 | **Commit races a long-running wrap that started under old version.** The old wrap completes after cutover with the *previous* envelope tag — correct behaviour, but if the operator interprets `in_flight_wraps==0` before clicking commit, a concurrent request might still slip in. | `beginWrap` reads the current `state`; if state∈`{Draining, ReadyForCutover}`, it still uses the pre-commit adapter — cutover only affects the *next* `WrapKey` after `PromoteActiveVersion` returns. `PromoteActiveVersion` is documented (and tested) to be atomic w.r.t. concurrent wraps. This is the canonical "epoch-based rotation" pattern in Wong, *Real-World Cryptography* Ch. 9 [1]. |
| 2 | **Bearer token leaks via logs / panics.** | Never log the token; redact `Authorization` headers in request-log middleware (already done for SigV4 creds per `internal/middleware/logging.go`); emit only SHA-256 fingerprint if diagnostic needed. Per BSRS Ch. 7 on "Design for Least Privilege" [3]. |
| 3 | **Replica divergence.** Gateway is typically deployed N-replica; rotation flips state process-locally. Replica A commits while replica B is still draining → inconsistent view. | Document (in `docs/ADMIN_API.md`) that operators must call `start` + `commit` against each replica independently (load-balancer-bypass), or drain the fleet via canary. Add a footgun check: `/admin/kms/rotate/status` includes `instance_id` and `replica_count` hints (if available from the config); rotation via a shared load balancer URL is a documented anti-pattern. Longer-term (v0.7) use a distributed lock; for v0.6 a bounded drain + documented procedure is acceptable per Beyer, *SRE Workbook* Ch. 16 on progressive rollouts [4]. |
| 4 | **Operator force-commits with in-flight wraps > 0.** | Those wraps complete against the pre-commit adapter (still the correct envelope), so no data loss. The `force` path emits a warning-level audit event and increments a dedicated metric label `kms_rotation_operations_total{step="commit",result="forced"}`. |
| 5 | **Token file rotated at runtime.** | Token source is read on every request via a `tokenSource func() []byte`; file-watcher reloads atomically. Covered by a unit test that rewrites the file mid-run and re-validates. |
| 6 | **Adapter `PromoteActiveVersion` fails partway (e.g. KMIP network flap).** | Handler sets state → `Aborted`, records `lastError`, emits failure audit + metric. Active version unchanged; no ciphertext was rewritten, so no cleanup needed. Caller retries by re-issuing `start`. |
| 7 | **Admin listener accidentally exposed (binding to `0.0.0.0`).** | Validation at startup forces `tls.enabled=true` when the address is not loopback. Also emits a structured warning at WARN level naming the bound address. |
| 8 | **HSM adapter operators expect rotation to work.** | Endpoint returns 501 with link to ADR 0004 (`"hsm adapter contract"`) and ADR 0007. Documented in `docs/ADMIN_API.md`. |
| 9 | **Memory adapter `AddVersion` accepts arbitrary material.** | `AddVersion` validates length (32 bytes for AES-256-GCM), zero-check, and is gated behind the admin endpoint so the bearer token is the authorization boundary. Per NIST SP 800-57 Pt. 1 §5.1, master key material must never be reused across versions [2]. |
| 10 | **Rate-limit bypass via many source IPs.** | Default rate-limit is per-IP but a global cap of 120 rpm applies at the listener. Admin listener default binding is loopback-only, so the real mitigation is network topology. |

## 10. Follow-ups / Out-of-scope items

1. **V0.6-CFG-1-followup-rewrap** — bulk re-encryption endpoint
   (`POST /admin/kms/rewrap`) that iterates buckets/prefixes and
   re-wraps per-object DEKs under the new active version. Requires
   listing pagination, throughput caps, resume tokens, per-bucket
   progress tracking. File at v0.7 planning.
2. **V0.6-CFG-1-followup-mtls** — mTLS client-cert admin authentication
   as an alternative to bearer. Needs a cert-reload story and a
   documented CA expectations.
3. **V0.6-CFG-1-followup-distributed-lock** — coordinated rotation
   across gateway replicas. Likely backed by Redis/etcd or an external
   lock service. Removes the §7 risk #3 operator burden.
4. **V0.6-OBS-1** — pprof behind the admin mux. Now unblocked by this
   plan; OBS-1 authors should mount `net/http/pprof` on the existing
   admin listener rather than a third listener.
5. **V0.6-S3-2** — object-lock passthrough admin-authz hooks consume
   the `isAdmin(r)` helper delivered here. No CFG-1 code change
   required; S3-2 merely wires it through.
6. **CLI companion (`gatewayctl kms rotate …`)** — thin CLI that
   drives the admin API; out of scope for this plan but should land
   alongside the v0.6 release notes.
