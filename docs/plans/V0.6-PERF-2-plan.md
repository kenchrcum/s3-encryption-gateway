## 1. Context & Current State

### 1.1 What the S3 client currently does

Retry behaviour on the S3 backend is entirely inherited from the
AWS SDK Go V2 defaults baked into
`internal/s3/client.go:207-227`:

```go
awsCfg, err := awsconfig.LoadDefaultConfig(context.Background(),
    awsconfig.WithRegion(region),
    awsconfig.WithCredentialsProvider(credentials.NewStaticCredentialsProvider(...)),
    awsconfig.WithRequestChecksumCalculation(aws.RequestChecksumCalculationWhenRequired),
    awsconfig.WithResponseChecksumValidation(aws.ResponseChecksumValidationWhenRequired),
)
```

No `WithRetryer`, no `WithRetryMode`, no `WithRetryMaxAttempts`,
no custom `aws.Retryer` implementation, no per-operation override.
The SDK therefore uses the default `retry.Standard` retryer:

| SDK default | Value |
|---|---|
| Retry mode | `aws.RetryModeStandard` |
| Max attempts | 3 (1 original + 2 retries) |
| Max backoff | 20 s |
| Backoff algorithm | Exponential, jittered (uniform, half of computed) |
| Retry-token bucket | 500 tokens, 5 per retryable failure |
| Retryable errors | SDK-classified (throttling, 5xx, network, transient DNS) |

For the gateway this means:

1. **No operator control.** When a backend (MinIO, Garage, Hetzner,
   Wasabi, Backblaze) is transiently overloaded, there is no way to
   raise attempts, widen backoff, or tighten timeouts without a code
   change.
2. **No per-operation differentiation.** `PutObject` (an
   idempotency-sensitive data-plane write) retries with the same
   policy as `HeadObject` (trivially idempotent, safe to retry
   aggressively).
3. **No observability.** The SDK records no gateway-facing metric
   for retries. Operators cannot answer "is the gateway hammering
   the backend?" without packet captures or backend-side logs.
4. **No budget enforcement.** Under a thundering-herd event the
   SDK's token bucket protects the backend to some extent, but the
   gateway has no visibility into bucket depletion and no way to
   convert retry exhaustion into a clean `503 Slow Down` to the
   client.
5. **No idempotency safeguards for non-idempotent verbs.** The SDK
   happily retries `PutObject` on a 5xx response even if the request
   *succeeded* and only the response was lost. For most object
   storage this is benign (PUT is idempotent on the same key), but
   for the following verbs the stock behaviour is unsafe or
   wasteful:
   - `CompleteMultipartUpload` — not idempotent across retries if
     the first call completed; the second call returns
     `NoSuchUpload` and the caller sees a spurious error. See
     `internal/s3/client.go:700-727`.
   - `DeleteObjects` (batch) — idempotent per key, but the SDK
     currently rewinds the entire request body via the Content-MD5
     middleware (`client.go:1096-1128`), which reallocates on every
     attempt. PERF-1 style pooling would help here.
   - `CopyObject` — idempotent on the destination key, but an
     already-succeeded copy returns a fresh ETag on retry, so
     application-level `If-Match` patterns see a drift.
6. **No integration with audit logging or tracing.** Retries are
   invisible to `internal/audit/*` and to OpenTelemetry spans emitted
   at `client.go:308-315`. A slow request that internally retried
   twice appears as a single slow span with no attempt breakdown.

### 1.2 What the gateway already has in this space

- **Audit sink retry loop** (`internal/audit/sink.go:122-152`) —
  naive `for i := 0; i <= retryCount; i++` with
  `time.Sleep(backoff * 2^i)`. No jitter, no context cancel, no
  observability. Known rough spot, not a reference implementation.
  Flagged here so we do not accidentally extend it; PERF-2 is
  scoped to the S3 backend client.
- **`SeekableBody` wrapper** (`internal/s3/seekable_body.go`,
  introduced by V0.6-PERF-1 Phase D) — already handles the SigV4
  seek-for-retry contract. Works correctly with any retryer; no
  change needed here.
- **Deterministic MPU ciphertext** (`internal/crypto/mpu_iv.go:38`,
  `mpu_encrypter.go:23`) — retries produce byte-identical
  ciphertext. This is the property that makes `UploadPart` safe to
  retry aggressively; we exploit it in §4.2.
- **Prometheus metrics infrastructure**
  (`internal/metrics/metrics.go`) — existing `s3OperationsTotal`,
  `s3OperationDuration`, `s3OperationErrors` counters are the
  natural home for the new retry counters.

### 1.3 Why this has not been fixed yet

- v0.5 and earlier had a single-tenant, single-backend deployment
  profile; SDK defaults were "good enough".
- v0.6 introduces the multi-provider conformance matrix
  (V0.6-QA-4): MinIO, Garage, AWS, Wasabi, Hetzner, Backblaze B2 —
  each with different transient-error characteristics (Garage
  returns 503 more readily on write contention; Hetzner rate-limits
  at the HTTP-429 layer with `Retry-After`; Wasabi occasionally
  returns `InternalError` on cold paths). A single SDK default
  cannot serve all six well.
- V0.6-QA-1 (performance baselines) will start reporting tail
  latency per provider; operator-visible retry metrics are a
  prerequisite for diagnosing what those baselines show.

### 1.4 Prior art inside the codebase

- `internal/config/config.go:167-193` (`ServerConfig`) — established
  pattern for typed knobs with `yaml` + `env` tags and a
  `Default<Name>` sibling constant. We mirror this for
  `BackendRetryConfig`.
- `internal/config/config.go:227-236` (`SinkConfig.RetryCount /
  RetryBackoff`) — existing naming precedent; we avoid re-using
  these identifiers on a different struct to prevent operator
  confusion.
- `internal/s3/client.go:173-264` (`ClientFactory`) — the single
  construction site for per-request S3 clients. One change here
  plumbs retry config through every caller.
- `docs/plans/V0.6-PERF-1-plan.md:680-810` — plan template and
  DoD discipline we will mirror line-for-line.
- `docs/plans/V0.6-CFG-1-plan.md` — precedent for an observability-
  -and-metrics-heavy configuration feature.

## 2. Goals & Non-Goals

### Goals

1. **Operator-configurable retry policy** for the S3 backend
   client. Expose at minimum:
   - retry mode (`standard` | `adaptive` | `off`),
   - max attempts (global default),
   - max backoff ceiling,
   - initial backoff (algorithmic only; the `retry.Standard`
     retryer does not expose this, so it is implemented via a
     custom retryer — see §4.2),
   - per-operation override map (e.g. `PutObject: 1` to disable
     retries on a specific verb).
2. **Idempotency safeguards** for non-idempotent verbs. Concretely:
   - `CompleteMultipartUpload` defaults to `max_attempts: 1`
     (no automatic retry; caller-initiated retry is still
     possible and safe given MPU determinism per §1.2).
   - `DeleteObject` / `DeleteObjects` retry safely (idempotent by
     S3 semantics).
   - `CopyObject` retry is gated on a capability flag
     `server.retry.safe_copy_object` (default `true`; operators
     who rely on `If-Match` contracts can disable).
3. **Observable retries.** Every retry attempt emits:
   - a Prometheus counter increment
     (`s3_backend_retries_total{operation,reason}`),
   - a Prometheus histogram observation of attempt count per
     logical request (`s3_backend_attempts`),
   - an audit event when attempts ≥ 2 on a data-plane verb,
   - an OpenTelemetry span event on the outer operation span
     (not a new span; retries are attempts *within* one logical
     operation).
4. **Deterministic, measurable default improvement** over the SDK
   default, demonstrated on the conformance matrix:
   - median request duration unchanged (< 2 %),
   - p99.9 under induced fault load (10 % injected 503) at least
     20 % lower than the SDK default,
   - total retry bandwidth under a thundering-herd simulation at
     least 30 % lower than the SDK default (credit: full-jitter
     decorrelated backoff, per *DDIA 2E* §8.2 and Titmus §9.3).
5. **No regression.** All existing unit, integration, and
   conformance tests pass unchanged. Default behaviour with an
   empty config stanza is at-least-as-resilient as the current
   SDK default (max attempts ≥ 3, exponential backoff, jitter).
6. **No API / on-disk format change.** No client-visible HTTP
   protocol change; no manifest or ciphertext layout change; no
   admin API change.

### Non-Goals

- **Circuit breakers.** Half-open / open / closed state machines,
  health-based short-circuiting — out of scope. Tracked separately
  under a future milestone if real-world data shows they are
  needed. Reference: *Designing Distributed Systems, 2nd Ed.* ch. 3.
- **Client-side rate limiting of gateway→backend traffic.** Rate
  limiting is a separate concern governed by `internal/middleware`;
  PERF-2 only governs retries that have already been emitted.
- **Hedged / speculative requests.** Parallel "race the backends"
  patterns are incompatible with our single-backend proxy model.
- **Retry on crypto / policy rejection.** AEAD tag failures,
  policy denials, admin-API 401/403 must never be retried. The
  classifier in §4.3 is hard-coded to never mark these retryable.
- **Changes to the audit-sink retry loop.** Known rough spot
  (§1.2), but its call path, failure semantics, and SLO are
  different. Scoped to its own future issue.
- **Changes to the Cosmian / KMS client retry.** Out of scope.
  Opens a follow-up ticket if needed.

### Phase B — `BackendRetryConfig` type & plumbing

**Touch:** `internal/config/config.go`,
`internal/config/config_test.go`,
`config.yaml.example`.

**New types:**

```go
// BackendRetryConfig governs retries emitted by the S3 backend
// client. All fields optional; zero values fall back to safe
// defaults (see DefaultBackendRetry* constants below).
type BackendRetryConfig struct {
    // Mode selects the retryer implementation.
    //   "standard" (default) — exponential backoff with jitter
    //   "adaptive"           — token bucket + adaptive rate limit
    //   "off"                — no retries (single attempt only)
    Mode string `yaml:"mode" env:"BACKEND_RETRY_MODE"`

    // MaxAttempts is the total number of attempts (including the
    // first). Must be >= 1. Default: 3.
    MaxAttempts int `yaml:"max_attempts" env:"BACKEND_RETRY_MAX_ATTEMPTS"`

    // InitialBackoff is the base delay before the first retry.
    // Default: 100 ms.
    InitialBackoff time.Duration `yaml:"initial_backoff" env:"BACKEND_RETRY_INITIAL_BACKOFF"`

    // MaxBackoff caps the per-attempt delay. Default: 20 s
    // (matches SDK default).
    MaxBackoff time.Duration `yaml:"max_backoff" env:"BACKEND_RETRY_MAX_BACKOFF"`

    // Jitter selects the jitter algorithm.
    //   "full"          (default) — uniform [0, computed]
    //   "decorrelated"            — AWS Architecture Blog algorithm
    //   "equal"                   — half computed + uniform half
    //   "none"                    — no jitter (debug only; unsafe
    //                               under contention).
    Jitter string `yaml:"jitter" env:"BACKEND_RETRY_JITTER"`

    // PerOperation overrides MaxAttempts per S3 API verb. Keys are
    // canonical SDK operation names: "PutObject", "GetObject",
    // "HeadObject", "UploadPart", "CompleteMultipartUpload",
    // "CopyObject", "UploadPartCopy", "DeleteObject",
    // "DeleteObjects", "ListObjectsV2", "ListParts", etc.
    // A value of 1 disables retries for that operation.
    PerOperation map[string]int `yaml:"per_operation"`

    // SafeCopyObject enables retrying CopyObject. Some applications
    // require If-Match/If-None-Match contracts that break across a
    // retried copy; disable here to opt out. Default: true.
    SafeCopyObject *bool `yaml:"safe_copy_object" env:"BACKEND_RETRY_SAFE_COPY_OBJECT"`
}

const (
    DefaultBackendRetryMode           = "standard"
    DefaultBackendRetryMaxAttempts    = 3
    DefaultBackendRetryInitialBackoff = 100 * time.Millisecond
    DefaultBackendRetryMaxBackoff     = 20 * time.Second
    DefaultBackendRetryJitter         = "full"
)
```

Add to `BackendConfig`:

```go
type BackendConfig struct {
    // ... existing fields ...
    Retry BackendRetryConfig `yaml:"retry"`
}
```

Validation rules (enforced in a new
`BackendRetryConfig.Validate() error`):

- `Mode ∈ {"standard", "adaptive", "off", ""}` (empty ⇒ default).
- `MaxAttempts ≥ 1`; values > 10 are rejected (operator foot-gun).
- `InitialBackoff ≥ 1 ms`; `MaxBackoff ≥ InitialBackoff`; both
  ≤ 5 min.
- `Jitter ∈ {"full", "decorrelated", "equal", "none", ""}`.
- `PerOperation` keys must match a whitelist of known SDK
  operation names (reject typos loudly).
- `SafeCopyObject` default is `true` if nil.

**Tests:**
- `TestBackendRetryConfig_Defaults` — zero-value config yields
  the documented defaults after normalisation.
- `TestBackendRetryConfig_Validate` — table-driven: every invalid
  combination yields a precise error message citing the offending
  field and allowed values.
- `TestBackendRetryConfig_EnvOverrides` — env vars override YAML;
  conflict precedence matches the rest of the project
  (env > yaml).
- `TestBackendConfig_RoundTrip` — YAML round-trip preserves every
  field.

**No behavioural change lands in this phase — it is pure
plumbing.** `Server.Retry` is parsed, validated, and reachable
but unused by Phase C.

### Phase C — Custom `Retryer` + classifier

**Touch:** `internal/s3/retry.go` (new),
`internal/s3/retry_test.go` (new),
`internal/s3/client.go` (plumb only; no behaviour change yet).

**New package-internal types:**

```go
// retryer implements aws.Retryer with gateway-specific policy.
type retryer struct {
    inner         aws.Retryer           // delegated: retry.Standard or retry.AdaptiveMode
    cfg           config.BackendRetryConfig
    perOp         map[string]int        // lowered keys
    classifier    errorClassifier
    clock         clock                 // time.Now(); injectable for tests
    rand          *rand.Rand            // jitter RNG; injectable for tests
    backoffCalc   backoffCalculator     // full / decorrelated / equal / none
    onAttempt     func(op string, attempt int, err error)
    onGiveUp      func(op string, attempts int, err error)
}

type errorClassifier interface {
    // Classify returns the reason label (for metrics) and whether
    // the error is retryable under this policy. Returns
    // (reason="", retryable=false) for non-retryable errors.
    Classify(op string, attempt int, err error) (reason string, retryable bool)
}
```

**Implementation steps:**

1. **Delegation.** `retryer.IsErrorRetryable(err error) bool`
   invokes the SDK's `retry.Standard.IsErrorRetryable` first,
   then layers gateway-specific opt-outs:
   - `CompleteMultipartUpload` → always false (non-idempotent
     per §1.2);
   - `CopyObject` → false if `cfg.SafeCopyObject == false`;
   - any error whose chain contains `crypto.ErrInvalidEnvelope`,
     `crypto.ErrUnwrapFailed`, policy-denied, admin-auth-failed
     → false (hard-coded safety);
   - everything else → delegate to inner.
2. **Backoff.** `retryer.RetryDelay(attempt int, err error)
   (time.Duration, error)` implements:
   - `full-jitter`: `delay = rand.Int63n(cap)`, where
     `cap = min(maxBackoff, initial * 2^attempt)`. Reference
     implementation cited from the AWS Architecture Blog and
     *DDIA 2E* §8.2.
   - `decorrelated-jitter`: `delay = min(maxBackoff, rand.Int63n(3
     * prev_delay) + initial)`. Canonical AWS formulation; *DDIA
     2E* §8.2 proves tighter tail than full-jitter under
     contention.
   - `equal-jitter`: `delay = cap/2 + rand.Int63n(cap/2)`.
   - `none`: `delay = min(maxBackoff, initial * 2^attempt)`
     (debug only; the validator warns on this choice).
3. **Max attempts.** `retryer.MaxAttempts() int` returns the
   per-operation override if set, else the global.
4. **Retry-After honouring.** For HTTP 429 / 503 responses that
   include `Retry-After`, the computed backoff is `max(delay,
   retryAfter)`. Parsed from SDK's error types via
   `smithyhttp.ResponseError`. Bounded above by `MaxBackoff * 2`
   (never sleep longer than twice the cap).
5. **Context cancellation.** All sleeps go through
   `clock.SleepContext(ctx, d)` — returns `ctx.Err()` as soon as
   the request context is cancelled. No naked `time.Sleep`.

**Classifier (`errorClassifier`):**

```go
// defaultClassifier returns reasons for the metric label:
//   "timeout", "conn_reset", "throttle_429", "throttle_503",
//   "internal_500", "dns", "tls", "sdk_generic", "non_retryable".
type defaultClassifier struct{}
```

Reason labels are a closed set — validated by a test — so the
Prometheus label cardinality never surprises operators.

**AdaptiveMode option.** When `cfg.Mode == "adaptive"`, `inner`
is a `retry.NewAdaptiveMode()` rather than `retry.NewStandard()`.
AdaptiveMode adds client-side rate limiting via a token bucket
that shrinks on `ThrottlingException` — useful under known-hot
load. Default remains `standard`.

**Tests:**
- `TestRetryer_StandardHappyPath` — no error, no retries,
  single attempt.
- `TestRetryer_ThreeAttemptsOnThrottle429` — injected 429 ×2;
  third attempt succeeds; `onAttempt` called twice; backoff
  between attempts observable via injected clock.
- `TestRetryer_ContextCanceledMidSleep` — cancel the context
  while retryer is sleeping; `RetryDelay`'s sleep returns
  `context.Canceled` promptly; no further attempts.
- `TestRetryer_CompleteMultipartUpload_NeverRetries` — explicit
  per-op override verified.
- `TestRetryer_CopyObject_GatedBySafeCopyObject` — both
  branches.
- `TestRetryer_CryptoErrorNeverRetried` — `ErrInvalidEnvelope`,
  `ErrUnwrapFailed`, tamper errors all short-circuit to
  non-retryable.
- `TestRetryer_RetryAfterHeaderHonoured` — 503 with
  `Retry-After: 5`; next attempt gated on ≥ 5 s.
- `TestRetryer_JitterAlgorithms` — golden values per algorithm
  given a fixed RNG seed.
- `TestRetryer_MaxAttemptsCapsGiveUp` — per-op override of 1
  means exactly one attempt.
- `TestRetryer_AdaptiveMode_Smoke` — mode selection wires
  `retry.AdaptiveMode`, happy-path round-trip passes.
- `TestRetryer_RaceSafe` — run 100 parallel
  `IsErrorRetryable` / `RetryDelay` calls under `-race`; no
  data race (internal `rand.Rand` is locked).

Still no behavioural change in `client.go` — the new retryer is
constructed but not yet wired.

### Phase D — Wire retryer into `ClientFactory`

**Touch:** `internal/s3/client.go` (`GetClientWithCredentials`,
`ClientFactory`), `internal/s3/client_test.go`,
`cmd/server/main.go` (pass `backendCfg.Retry` into factory
construction — it's already reachable via the parent config).

**Changes:**

1. `ClientFactory` gains an explicit retry config pointer:

   ```go
   type ClientFactory struct {
       baseConfig  *config.BackendConfig
       retryConfig config.BackendRetryConfig // normalised
       metrics     *metrics.Metrics          // injected; optional
       logger      *logrus.Logger            // injected; optional
   }

   func NewClientFactory(cfg *config.BackendConfig, opts ...ClientFactoryOption) *ClientFactory
   ```

   New options: `WithMetrics`, `WithLogger`. Defaults preserve
   current behaviour (no metrics, no logger) so external callers
   need not change.

2. `GetClientWithCredentials` installs the retryer:

   ```go
   awsCfg, err := awsconfig.LoadDefaultConfig(context.Background(),
       awsconfig.WithRegion(region),
       awsconfig.WithCredentialsProvider(...),
       awsconfig.WithRetryer(func() aws.Retryer {
           return newRetryer(f.retryConfig, newMetricsSink(f.metrics), f.logger)
       }),
       awsconfig.WithRequestChecksumCalculation(aws.RequestChecksumCalculationWhenRequired),
       awsconfig.WithResponseChecksumValidation(aws.ResponseChecksumValidationWhenRequired),
   )
   ```

   Note the function wrapper: `WithRetryer` expects a factory so
   each request context gets a fresh retry-state (fresh token
   bucket, no cross-request interference).

3. `cfg.Retry.Mode == "off"` installs `aws.NopRetryer{}` directly
   (skipping the custom retryer entirely). Documented behaviour:
   *absolutely* one attempt, no adaptive rate limit, no retry
   metrics. Intended for debug / conformance test isolation.

4. The new `metricsSink` type converts `onAttempt` /
   `onGiveUp` callbacks into `metrics.RecordBackendRetry(...)`
   calls. See Phase E.

**Tests:**
- Existing `TestClientFactory_*` pass unchanged.
- `TestClientFactory_RetryerInstalled` — construct a client,
  trigger a synthetic retryable error via the fault
  round-tripper, assert `MaxAttempts` attempts were made and

- `TestClientFactory_RetryMode_Off` — single attempt; no retry

- `TestClientFactory_PerOperationOverride_DisablesPutObject` —

### Phase E — Metrics, audit, tracing

**Touch:** `internal/metrics/metrics.go`,
`internal/audit/events.go` (new audit event type),
`internal/s3/client.go` (tracing span events).

**New Prometheus metrics:**

```go
// s3BackendRetriesTotal counts retry attempts per operation &
// classifier reason.
s3BackendRetriesTotal *prometheus.CounterVec
// labels: ["operation", "reason", "mode"]

// s3BackendAttemptsPerRequest is a histogram of total attempts
// made per logical backend request (≥ 1).
s3BackendAttemptsPerRequest *prometheus.HistogramVec
// labels: ["operation"]
// buckets: linear 1..10

// s3BackendRetryGiveUpsTotal counts operations that exhausted
// MaxAttempts.
s3BackendRetryGiveUpsTotal *prometheus.CounterVec
// labels: ["operation", "final_reason"]
```

## 4. Technical Deep-Dives

### 4.1 Why not just set `WithRetryMaxAttempts` and call it done?

`WithRetryMaxAttempts` only adjusts a single integer; the SDK's
`retry.Standard` still drives algorithm, token bucket, and
classifier. None of the goals in §2 beyond #1 are reachable via
SDK-only configuration. A custom `aws.Retryer` is the minimum
surface that satisfies the observability and idempotency-
safeguards requirements; it is also the AWS-recommended
extension point (see SDK docs referenced in §1.5).

### 4.2 Backoff algorithms in detail

Let `attempt` be the 1-indexed attempt number (first retry is
`attempt=1`). Let `B = InitialBackoff`, `C = MaxBackoff`. All
formulas clamp the final value to `C`.

- **Full jitter** (AWS Arch Blog, *DDIA 2E* §8.2):
  `delay = rand.Int63n(min(C, B * 2^attempt))`
  Tight under light contention; proven best against a thundering
  herd of N clients when N is unknown.
- **Decorrelated jitter** (AWS Arch Blog; preferred under high
  contention):
  `delay = min(C, rand.Int63n(3*prev) + B)`, where `prev` is the
  previous attempt's delay (or `B` on the first retry). Needs
  one integer of per-request state. Tighter tail than full
  jitter when the expected-retry count > 3.
- **Equal jitter** (popular, middling tail):
  `delay = cap/2 + rand.Int63n(cap/2)` with `cap = min(C, B *
  2^attempt)`.
- **None** (debug only): `delay = min(C, B * 2^attempt)`;
  marked unsafe in the validator's warning text.

All four algorithms share the same `prev_delay` recording and
the same `SleepContext` path. The jitter choice is a single
`switch` in `backoffCalculator.Next(attempt, prev) time.Duration`.

### 4.3 Adaptive vs standard mode

The SDK's `retry.AdaptiveMode` wraps `retry.Standard` with a
client-side token bucket that shrinks on throttling events,
preventing the client from contributing to cascading failure.
Trade-off: adaptive mode can emit fewer total retries than
requested — sometimes fewer than `MaxAttempts` — if the bucket
is drained. This is correct behaviour under contention but can
surprise operators. The ADR documents this explicitly; the
validator emits a log-warn when both `mode: adaptive` and an
aggressive `PerOperation` override are set.

### 4.4 Classifier safety property

The classifier must never mark the following as retryable (hard
invariant, tested by a table-driven test):

- Any error whose chain (`errors.Is`) contains
  `crypto.ErrInvalidEnvelope`, `crypto.ErrUnwrapFailed`,
  `crypto.ErrKeyNotFound`, `crypto.ErrProviderUnavailable`.
- `context.Canceled`, `context.DeadlineExceeded` (the caller
  owns the deadline; retrying after deadline is wasteful).
- SDK `types.NoSuchBucket`, `types.NoSuchKey`, `types.NoSuchUpload`
  — these are definite 404s, not transient.
- `smithyhttp.ResponseError` with status `400`, `401`, `403`,
  `404`, `405`, `409`, `411`, `412`, `413`, `415`, `416`, `422`.
- Any error emitted by the admin API (pass-through not applicable
  since admin is a separate listener — defensive check).

Retryable (classifier delegates to SDK for the nuanced cases):

- `types.SlowDown`, `types.RequestLimitExceeded`,
  `types.InternalError`, `types.ServiceUnavailable`.
- HTTP `408`, `429`, `500`, `502`, `503`, `504`.
- Network errors: DNS temporary failure, `io.EOF` mid-response
  (SDK classifies as `RetryableError` already),
  `net.OpError.Temporary()`, `syscall.ECONNRESET`,
  `syscall.ECONNREFUSED`.
- TLS handshake timeouts (handshake errors from
  `crypto/tls.RecordHeaderError` are delegated; the SDK's
  existing classifier already handles these correctly).

### 4.5 Context-aware sleep

```go
type clock interface {
    Now() time.Time
    SleepContext(ctx context.Context, d time.Duration) error
}

type realClock struct{}

func (realClock) Now() time.Time { return time.Now() }
func (realClock) SleepContext(ctx context.Context, d time.Duration) error {
    if d <= 0 {
        return nil
    }
    t := time.NewTimer(d)
    defer t.Stop()
    select {
    case <-ctx.Done():
        return ctx.Err()
    case <-t.C:
        return nil
    }
}
```

The test implementation advances a virtual clock on demand;
this keeps the retryer unit tests fast and deterministic. The
pattern is lifted from `internal/crypto/rotation_state.go` which
uses the same injectable-clock pattern for rotation grace
windows.

### 4.6 Per-request retryer factory

`WithRetryer` accepts `func() aws.Retryer`, not a singleton.
Each logical S3 operation invokes the factory once and the
returned retryer is used for that operation's attempts. This
ensures:

- `prev_delay` state (for decorrelated jitter) is per-operation;
  no cross-operation contention.
- AdaptiveMode's token bucket is per-operation *not* per-client
  — which sounds wrong but is actually the SDK's design: the
  client-wide rate limit lives in the `aws.Config`'s middleware
  stack, not the retryer, so AdaptiveMode's per-retryer state
  is just the retry-budget portion. Confirmed against the SDK
  source.
- Metrics callbacks (`onAttempt`, `onGiveUp`) have access to the
  operation name via the SDK's `aws.Context` value
  `middleware.GetOperationName(ctx)` which is set at the SDK
  middleware stack entry point.

### 4.7 Interaction with V0.6-PERF-1 seekable-body wrapper

The `SeekableBody` wrapper from PERF-1 Phase D provides
byte-identical reads across `Seek(0, SeekStart)` calls. This
is exactly what the SDK's signer expects for retries: on each
attempt the body is rewound and re-signed. No change required
here; PERF-2 is orthogonal.

For the encrypted MPU path, MPU IV determinism (§1.2) guarantees
that retried `UploadPart` calls emit byte-identical ciphertext
— a stronger property than S3 actually requires, and one the
test `TestMPUEncryptReader_DeterministicCiphertext_AcrossRetries`
already covers. No new test needed.

### 4.8 Why expose `InitialBackoff` as a knob?

The SDK's `retry.Standard` does not expose the base delay — it
uses a fixed 200 ms internally. Operators running against a
local MinIO during development want 10 ms; operators running
against a cross-region AWS endpoint may want 500 ms to avoid
wasting attempts on a backend that is genuinely further away.
The custom retryer in §Phase C bypasses the SDK's internal
delay calculation entirely and uses its own `backoffCalculator`
against `InitialBackoff`, so this knob is trivially supported
once the custom retryer is in place.

## 7. Risks & Mitigations

| Risk | Likelihood | Severity | Mitigation |
|---|---|---|---|
| Custom retryer misclassifies an error as non-retryable that the SDK would have retried, reducing availability | M | H | Classifier delegates to `retry.Standard.IsErrorRetryable` by default and only adds opt-outs; table-driven tests enumerate the opt-out set; conformance suite re-run on every provider confirms no availability regression. |
| Adaptive mode's token bucket surprises operators (fewer retries than `MaxAttempts`) | M | M | ADR documents; validator emits a log-warn on aggressive `PerOperation` overrides combined with adaptive mode. |
| Metric label cardinality grows unexpectedly | L | M | Closed-set reasons; `TestRetry_MetricCardinality` enforces the bound; labels pre-allocated at registration time. |
| Context-aware sleep not honoured under high GC pressure | L | M | `time.NewTimer` + `defer Stop()` pattern, not `time.Sleep`; unit-tested via injectable clock; race-tested. |
| Per-operation override map typo silently retries wrong op | L | H | Validator rejects unknown operation names against a whitelist (closed set from SDK operation catalogue). |
| `CompleteMultipartUpload` regression for apps that rely on current auto-retry | L | M | Documented in ADR; migration path is "retry at application layer" which is also the AWS recommendation per *Engineering Resilient Systems on AWS* ch. 5. |
| Retry storm when `max_backoff` misconfigured to a small value | L | M | Validator enforces `MaxBackoff ≤ 5 min` and `≥ InitialBackoff`; default 20 s matches SDK. |
| Fault-injector RNG seed leaks into the client code path | L | L | Fault injector is test-only, gated behind a build tag (`faulty_s3` or file name suffix `_test.go`); no production binary dependency. |
| Benchmark "20 % p99.9 reduction" target not met on every fault mix | L | M | Phase F re-runs and honestly reports; if not met, thresholds are re-stated rather than the phase being declared "done" (mirrors PERF-1 discipline). |
| AdaptiveMode's internal `token bucket per-retryer` assumption wrong for some SDK minor version bump | M | L | Pin SDK version in `go.mod` (already at `v1.41.6`); CI matrix includes SDK upgrade test against a synthetic provider before release. |

## 10. Open Questions

- **Q1.** The handler comment at `handlers.go:3876-3885` points
  at PERF-2 for a streaming-checksum optimisation in
  `handleCopyObject`. That work is orthogonal to retry policy
  and was a labelling drift at the time PERF-1 landed.
  **Proposed resolution:** keep that deferral under a new
  follow-up issue V0.6-PERF-3 (or punt to v1.0); update the
  code comment to reference the correct issue during Phase F.
  **Decision needed before Phase F lands.**
- **Q2.** Should `backend.retry.mode: adaptive` be the default?
  AWS SDK guidance is "standard unless you need adaptive";
  *Engineering Resilient Systems on AWS* ch. 5 leans toward
  adaptive for multi-tenant gateways. **Proposed:** keep
  `standard` as default for parity with SDK and with the
  conservative-by-default posture of the gateway; surface
  adaptive prominently in the ADR and README. Revisit after
  V0.6-QA-1 baselines land.
- **Q3.** Should retry policy be reloadable at runtime via the
  admin API (precedent: V0.6-CFG-1)? **Proposed:** no, not in
  this milestone. Reload requires re-constructing the
  `aws.Config` which is cheap but threads a lifetime contract
  through `ClientFactory` that we do not need yet. Track as
  potential V0.7 work if operators ask.
- **Q4.** Do we expose a `backend.retry.timeout_per_attempt`
  knob (per-attempt deadline, independent of the outer request
  context)? **Proposed:** no. Per-attempt timeouts interact
  badly with SDK retry internals; operators wanting this should
  set the outer `context.WithTimeout` at the handler layer.
  Revisit if real-world data shows need.
