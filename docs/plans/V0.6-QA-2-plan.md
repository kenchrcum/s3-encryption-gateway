### 1.2 Why 80% is the right threshold

The 80% target is not arbitrary. *Software Engineering at Google*
(Winters et al., O'Reilly 2020), ch. 11 "Testing Overview" — §"Code
Coverage" — cites 80% as the practical diminishing-returns inflection
point: below it, significant gaps in the happy path or critical error
paths are likely; above 90%, coverage cost escalates while finding
fewer real bugs. The crypto-specific packages (engine, keymanager,
MPU encrypter) already sit close to or above 80%, confirming this is
a realistic target for the gateway codebase.

The V0.6 milestone defined "coverage gate ≥ 80%" in the original
planning session (2026-04-16); the V0.6-S3-2 plan at line 665
reinforces it as a milestone gate. This plan implements it.

### 1.3 Why mutation testing, and which packages

Standard line coverage has a well-documented weakness: a test can
execute a line of code without asserting anything meaningful about
its output. *Effective Software Testing* (Aniche, Manning 2022),
ch. 8 "Mutation Testing" — §"Why Mutation Testing?"  — demonstrates
that suites with 80%+ coverage can still miss logic inversions (e.g.
`>=` vs `>`, `&&` vs `||`) that a mutation tool catches immediately.

For an encryption gateway, these logic inversions are especially
dangerous:

- An off-by-one in an AEAD chunk-boundary check (`>=` vs `>`) can
  cause silent ciphertext truncation.
- A flipped Boolean in `isAdmin` authentication logic grants
  unauthenticated access to key-rotation APIs.
- An inverted `errors.Is` check in error unwrapping skips retry
  logic, leaking plaintext to the caller on transient errors.

The mutation scope in this plan targets **crypto-free critical paths**
(per the tracker stub scope) precisely because those paths are where
logic inversions do the most damage while being hardest to detect
by reading the code:

- `internal/config` — policy evaluation, key-rotation config
  validation, YAML field parsing.
- `internal/api` — routing dispatch, error-response formatting,
  SigV4 validation helpers (the auth boundary).
- `internal/s3` — retry classification (HTTP status → retryable?),
  seekable-body reconstruction.
- `internal/middleware` — rate limiting, authentication bypass
  checks, bucket validation.

The crypto **implementation** packages (`internal/crypto`) are
excluded from mutation testing (not from coverage) because Go's
`crypto/*` primitives are not meaningfully mutatable without FIPS
certification risk — mutating an AES-GCM nonce allocation to `0`
would require the test suite to perform a full FIPS functional test
that takes > 10 minutes per mutation; this is covered instead by the
existing fuzz corpus (`internal/crypto/fuzz_test.go`).

### 1.4 Tool selection: Gremlins

Several Go mutation testing tools exist:

| Tool | Stars (approx) | Actively maintained | GitHub Actions support |
|---|---|---|---|
| `go-mutesting` | ~900 | Archived 2022 | Manual |
| `Gremlins` | ~450 | Active (2024) | Native |
| `go-mutation` | ~100 | Early stage | Manual |

**Gremlins** (`github.com/go-gremlins/gremlins`) is chosen because:

1. It integrates natively with `go test` output, making it easy to
   wire into the existing CI matrix without custom harness code.
2. It uses `--mutant-status-map` output in JSON for machine-readable
   CI summaries — same pattern as QA-1's `macro-<provider>.json`.
3. It supports `--threshold-efficacy` to enforce a minimum kill-rate
   in CI (analogous to `--cover-threshold` for coverage).
4. It understands Go build tags, so `gremlins unleash --tags=fips`
   tests mutations against the FIPS crypto path without patching.
5. `go-mutesting` is archived; starting a new Go project on it is
   inadvisable (*Effective Software Testing* ch. 8 §"Choosing a
   Mutation Testing Tool" recommends active maintenance as a
   tie-breaker).

Gremlins is installed as a dev tool via `tools.go` (same pattern as
`benchstat` in QA-1), not as a runtime dependency.

## 2. Goals and Non-Goals

### 2.1 Goals

1. **PR-gate coverage enforcement.** Every PR to `main` runs
   `go test -coverprofile` and fails if project-wide statement
   coverage drops below **80%**. The threshold is configurable via
   a Makefile variable `COVERAGE_THRESHOLD` (default 80) so it can
   be tightened incrementally.
2. **Per-package coverage reports** as CI artefacts (HTML + `func`
   text) uploaded to GitHub Actions artefacts on every run.
3. **FIPS-tagged coverage profile** (`-tags=fips`) collected in
   addition to the default profile; both must meet the 80% gate.
4. **Mutation testing with Gremlins** run nightly (not PR-time — too
   slow) on the four in-scope packages (`internal/config`,
   `internal/api`, `internal/s3`, `internal/middleware`), enforcing
   a kill-rate ≥ **70%**. Advisory comment on PR if the diff
   introduces new un-killed mutants.
5. **Fill the critical coverage gaps** identified in §1.1 with new
   unit tests — not integration tests — so they run fast on every
   PR without Docker. Total new test code estimate: ~800-1200 LOC
   across five packages.
6. **Exclusion list** (`scripts/coverage-exclude.txt`, documented in
   `docs/COVERAGE.md`) declaring which files are excluded from the
   gate and why, so exclusions are explicit and auditable.
7. **Makefile targets** for local development:
   - `make coverage-gate` — runs the gate check locally.
   - `make coverage-html` — opens a browser with the HTML report.
   - `make mutation-report` — runs Gremlins on the four packages.

### 2.2 Non-Goals (explicit; each with rationale)

- **`cmd/server/main.go` in the coverage gate.** `main` functions
  require process-level integration testing (start the binary, send
  real traffic). Excluding `cmd/server` from the gate is industry
  standard — *Software Engineering at Google* ch. 11 §"Code Coverage"
  notes that coverage of process entrypoints is more meaningfully
  measured by integration/smoke tests, not unit coverage.
- **`internal/debug`** — the pprof package (V0.6-OBS-1) is an
  HTTP-endpoint registration layer with no testable logic beyond
  "handler registered". Excluded from the gate; measured but
  informational.
- **`test/provider/external.go`, `test/provider/cosmian.go`** — these
  files contain stubs for credential-gated external providers (AWS,
  Wasabi, B2, Cosmian KMIP). They are excluded because their code
  paths are only reachable with live external credentials that are
  not available in the standard unit-test environment.
- **Test-helper files** (`*_test_helpers.go`,
  `keymanager_test_helpers.go`) — go tool cover does not count
  `_test.go` files; these `.go` test-helper files (not `_test.go`)
  are excluded via the `// nolint:unused` pattern already in use.
- **Mutation testing on `internal/crypto`.** As explained in §1.3,
  this is covered by fuzz testing (`internal/crypto/fuzz_test.go`)
  and the existing FIPS profile tests; adding Gremlins here would
  produce > 5000 trivially-killed mutants with no signal value.
- **100% coverage target.** Per *Software Engineering at Google*
  ch. 11 §"Code Coverage" — "100% coverage is not the goal; it is
  a distraction from actually writing good tests."
- **Integration-test coverage counting.** Conformance tests
  (`-tags=conformance`) are not included in the coverage profile —
  they require Docker and run on a separate CI tier. The coverage
  gate runs only Tier-1 (unit) tests to ensure it is fast (< 3 min).

## 6. Regression Thresholds

| Gate | Metric | Threshold | Action on fail |
|---|---|---|---|
| Coverage (PR) | Project total statement coverage | < 80% | Fail PR job |
| Coverage FIPS (PR) | Coverage with `-tags=fips` | < 80% | Fail PR job |
| Mutation nightly | Kill rate per package | < 70% | Open `mutation-regression` issue |
| Mutation PR advisory | New un-killed mutants in diff | > 0 | Post advisory comment (never fails) |

### Rationale for 70% kill-rate

Gremlins cannot kill all mutants. Some are "equivalent mutants" —
semantically identical to the original — which are a known
theoretical limitation of mutation testing (*Effective Software
Testing* ch. 8 §"Equivalent Mutants"). A 70% kill-rate leaves room
for ~30% equivalent mutants while still ensuring the majority of
meaningful mutations are caught. This matches Gremlins' own
documentation recommendation for production codebases.

### How to update the thresholds

Coverage: change `COVERAGE_THRESHOLD` in `scripts/coverage-gate.sh`.
Mutation: change `--threshold-efficacy` in `scripts/mutation-report.sh`.
Both changes require a PR; the PR CI gate validates the new threshold.

## 10. Risks and Mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| 1 | Phase B requires understanding opaque internal packages with no existing tests | H | M | Start with `internal/api/errors.go` (simplest), build context incrementally; use the existing bench test patterns as a template |
| 2 | `internal/s3/client.go` coverage is hard to reach without a real S3 endpoint | M | M | Use `httptest.Server` with a hand-rolled S3 XML response server (pattern already established in `client_bench_test.go`); no real S3 needed |
| 3 | Gremlins is slow (mutation testing is inherently O(N) in mutant count) | H | L | Run nightly only; PR gate is advisory; limit to 4 targeted packages; use `--only-covered` flag to skip uncovered lines |
| 4 | 80% gate causes flaky failures if coverage measurement is non-deterministic | L | M | Use `-count=1` (no caching) in the coverage run; pin Go version in CI; exclude non-deterministic packages |
| 5 | New tests introduce data races | L | H | All new tests run with `-race`; test file structure follows existing `_test.go` conventions with no shared mutable state |
| 6 | `gremlins` tool version drift breaks the mutation workflow | L | L | Pin `gremlins` version in `tools.go` and in the CI `go install` step; same pattern as `benchstat` in QA-1 |
| 7 | Coverage drops below 80% after future code additions | M | M | Threshold is enforced on every PR; implementors are notified at PR time, not at nightly; they must add tests alongside new code |

## 11. Open Questions

- **Q1.** Should the coverage gate count conformance tests
  (`-tags=conformance`) in addition to unit tests? *Proposed:*
  no — conformance tests require Docker and would make the PR gate
  ~15 minutes. File a V0.7-QA task to explore a separate "full
  coverage" nightly that includes conformance test coverage.
- **Q2.** Should we adopt `go test -cover -covermode=atomic`
  (concurrent coverage) instead of `-covermode=set`? *Proposed:*
  yes, use `-covermode=atomic` since many packages use goroutines;
  `-covermode=set` can miss concurrent paths. The cost is ~5%
  slower test runs — acceptable.
- **Q3.** Should the mutation nightly job block release tags
  (like the conformance nightly does)? *Proposed:* not in QA-2.
  Gremlins is still a relatively new tool; a 70% kill-rate
  requirement on release tags risks false-positive release blocks.
  Revisit in V0.7 after 2–3 months of nightly data.
- **Q4.** What do we do with `internal/api/auth.go` which
  implements SigV4 — is this security-critical enough to be included
  in mutation testing scope? *Proposed:* yes, add it to the Gremlins
  scope explicitly in `scripts/mutation-report.sh` as a fifth
  package. The SigV4 validation logic (credential parsing, HMAC
  comparison) is precisely the kind of logic-inversion target that
  mutation testing is designed to catch.

## 14. Appendix: Canonical Invocations

### A. Run the coverage gate locally

```bash
# Prereqs: Go 1.25+
make coverage-gate                          # default 80% threshold
make coverage-gate COVERAGE_THRESHOLD=85    # raise threshold manually
make coverage-html                          # open HTML report in browser
make coverage-fips                          # FIPS coverage profile
```

### B. Run mutation testing locally

```bash
# Prereqs: gremlins installed via tools.go
go install github.com/go-gremlins/gremlins/cmd/gremlins@latest
make mutation-report                        # all four packages
make mutation-report-pkg PKG=./internal/config  # single package
```

### C. Trigger the mutation nightly manually

```bash
gh workflow run mutation.yml --ref main
gh run watch
```

### D. Update exclusion list

Edit `scripts/coverage-exclude.txt`; add one package path per line.
The script treats lines starting with `#` as comments.
```
# Reason: main() cannot be meaningfully unit-tested
cmd/server
# Reason: pprof registration only; no testable logic
internal/debug
```
