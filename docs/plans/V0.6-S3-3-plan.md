> Scope expanded relative to the one-line issue entry at
> `docs/issues/v0.6-issues.md:568`. The task ships the 10 named
> integration tests from the issue, **plus** closes two gaps flagged
> during V0.6-SEC-3 delivery: (a) the Phase-E code path

## 1. Context & Current State

### 1.1 What V0.6-S3-1 (UploadPartCopy) shipped

`internal/api/upload_part_copy.go:90-415` implements the handler with
three-way source-class dispatch (plaintext / chunked / legacy) plus
policy / cap / range guards. Unit coverage at
`internal/api/upload_part_copy_test.go` (12 tests) exercises:

- `parseCopySource`, ETag quoting, range parsing
- All three classify paths
- The two tightening DoDs from V0.6-S3-1:
  `TestUploadPartCopy_CrossBucket_ReadDenied`,
  `TestUploadPartCopy_PlaintextSource_EncryptedDestBucket_Refused`,
  `TestUploadPartCopy_PlaintextSource_NonRequiringBucket_Allowed`
- The two cap guards:
  `TestUploadPartCopy_LegacySourceExceedsCap`,
  `TestUploadPartCopy_SourceRangeExceeds5GiB`

**Gap:** no test ever exercises the handler against a real S3 backend.
All unit tests use a naive in-process mock (`mockS3Client` with in-memory
maps) that does not exercise ETag quoting drift, XML marshalling,
concurrent part assembly, or `CompleteMultipartUpload` byte ordering
against a real multipart-upload state machine.

### 1.2 What V0.6-SEC-3 (encrypted multipart) shipped

Per the SEC-3 entry at `docs/issues/v0.6-issues.md:99-456`:

- `internal/mpu/state.go` — Valkey-backed state store keyed
  `mpu:<hex(sha256(uploadID))>` with 7-day TTL
- `internal/crypto/mpu_iv.go` — HKDF-derived per-part-per-chunk IV
- `internal/crypto/mpu_manifest.go` — inline / fallback-object manifest
- `internal/crypto/mpu_encrypter.go` — part encrypt reader
- Handler wiring at `internal/api/handlers.go:2159-3300` across Create /
  UploadPart / Complete / Abort / ranged GET / full GET
- **Phase E re-encrypt at `internal/api/upload_part_copy.go:302,681-819`**
  — every source class is decrypted then re-encrypted under the
  destination MPU DEK when the destination bucket has
  `EncryptMultipartUploads=true`

Unit coverage is strong at the crypto and state-store tiers (42 tests
between `internal/crypto/mpu_*_test.go` and
`internal/mpu/state_test.go`) and at the handler tier
(`internal/api/encrypted_mpu_handler_test.go`, 16 tests covering
ranged GET, tamper detection, fail-closed paths, `/readyz` reflection,
AppendPart failure returning 503).

**Gaps:**

1. **Zero `TestUploadPartCopy_MPU_*` tests at any tier.** Phase E is
   the only part of SEC-3 with no direct unit-test anchor. The
   CHANGELOG claim that "`UploadPartCopy` into encrypted MPU
   destinations re-encrypts through the destination DEK schedule
   regardless of source class" is untested.

2. **Integration coverage for encrypted MPU is one env-gated file.**
   `test/encrypted_mpu_test.go` runs a single
   `TestIntegration_EncryptedMultipartUpload` against an
   externally-started gateway (`S3_GATEWAY_ENDPOINT` env var). It is
   not a CI job. Only one 5 MiB part is uploaded. No range-read
   assertion, no at-rest-ciphertext assertion, no tamper-detection
   assertion in the integration tier.

3. **`AppendPart` error-handling divergence.** The encrypted-MPU
   `UploadPart` path at `internal/api/handlers.go:2730-2757` fails
   closed with HTTP 503 + `mpu.valkey_unavailable` audit event when
   `mpuStateStore.AppendPart` fails. The `UploadPartCopy` Phase-E
   path at `internal/api/upload_part_copy.go:806-813` only emits a
   `logger.Warn` and proceeds. This is a latent silent-data-loss
   bug: a Valkey hiccup during copy yields a 200 OK whose part
   record is not in Valkey, and the subsequent `Complete` will
   either miss this part entirely or fail cryptic. Phase A of this
   plan aligns the two paths before the integration tests try to
   assert on them (Scope §2.3).

### 1.3 What V0.6-CFG-1 (admin + rotation) shipped

Relevant here only because per-bucket policy loading at integration
tier is a prerequisite for two of the ten tests. The pattern in
`internal/api/upload_part_copy_test.go:543-560`
(`newPolicyManagerWithRequireEncryption`) writes YAML to `t.TempDir()`
and loads via `config.NewPolicyManager().LoadPolicies([]string{path})`.
This plan extends the integration harness to accept that same
`*config.PolicyManager`.

### 1.4 Test harness current state

Summarised from `test/gateway.go`, `test/minio.go`:

- `StartGateway(t, *config.Config) *TestGateway` instantiates a handler
  with `NewHandlerWithFeatures(s3Client, engine, logger, metrics,
  keyManager=nil, cache=nil, auditLogger=nil, cfg, policyManager=nil)`
  (`test/gateway.go:86`). The **four nils** are the harness gap.
- `GetGatewayConfig()` at `test/minio.go:612-631` returns a bare
  `*config.Config` with no `MultipartState.Valkey` stanza and no
  `PolicyFiles`.
- `test/docker-compose.yml:9-16` already provisions
  `valkey/valkey:7.2.5` on `:6379`. No TLS variant.
- MinIO fixture is shared (`sync.Once` in `StartMinIOServer`) across
  all integration tests — parallel runs must namespace their bucket
  prefixes.
- SDK: the V0.6-S3-3 issue explicitly prescribes `aws-sdk-go-v2`
  (`docs/issues/v0.6-issues.md:619-621`). The existing
  `test/encrypted_mpu_test.go` already uses it.
- Build tag: `//go:build integration` (consistent with B2 / Cosmian /
  rotation_metrics files).

### 1.5 Cosmian KMS integration environment

`test/cosmian_kms_integration_test.go` runs against a Docker-Compose
Cosmian KMS server (KMIP port 5696, REST 9998) and exercises wrap /
unwrap round trips but **does not touch multipart uploads**. The
operator ask in this plan includes extending that file with a single
end-to-end encrypted-MPU smoke test so the KMS-backed
`KeyManager.WrapKey` path is exercised on the MPU DEK code path. This
is important because `PasswordKeyManager` and `cosmianKeyManager`
produce structurally different `KeyEnvelope` blobs, and the Valkey
state store round-trips the envelope as opaque JSON
(`internal/mpu/state.go:47-48`); a Cosmian-specific envelope has
never been tested through this path.

### 3.1 Harness extensions (Phase A)

**File: `test/gateway.go`** — add options pattern:

```go
type TestGatewayOption func(*testGatewayOpts)
type testGatewayOpts struct {
    policyManager *config.PolicyManager
    keyManager    crypto.KeyManager
    mpuStore      mpu.StateStore
    auditLogger   audit.Logger
}

func WithPolicyManager(pm *config.PolicyManager) TestGatewayOption { ... }
func WithKeyManager(km crypto.KeyManager)         TestGatewayOption { ... }
func WithMPUStateStore(s mpu.StateStore)          TestGatewayOption { ... }
func WithAuditLogger(a audit.Logger)              TestGatewayOption { ... }

func StartGateway(t *testing.T, cfg *config.Config, opts ...TestGatewayOption) *TestGateway {
    // ... existing body, with NewHandlerWithFeatures(s3Client, engine, logger,
    // metrics, o.keyManager, nil, o.auditLogger, cfg, o.policyManager)
    // and h.WithMPUStateStore(o.mpuStore) if non-nil.
}
```

Signature is backward-compatible (variadic). All 16 existing
`TestS3Gateway_*` tests compile unchanged.

**File: `test/mpu_fixtures.go`** (new, `//go:build integration`):

```go
func NewTestValkeyClient(t *testing.T) *redis.Client           // per-test DB
func NewTestMPUStateStore(t *testing.T) mpu.StateStore         // wraps the above
func NewTestPasswordKeyManager(t *testing.T) crypto.KeyManager // 32-char password
func NewRawBackendS3Client(t *testing.T, m *MinIOTestServer) *s3sdk.Client
func NewTestPolicyManager(t *testing.T, yaml string) *config.PolicyManager
```

Per-test Valkey DB isolation: `redis.Options.DB = hash(t.Name()) & 0xF`
(16 DBs available on default Valkey; collisions accepted since tests
use unique `uploadID`s). `t.Cleanup` runs `FLUSHDB`.

### 3.2 AppendPart fail-closed alignment (Phase A cont.)

File: `internal/api/upload_part_copy.go`.

Replace lines 806-813 (warn-and-continue) with the equivalent of
`handlers.go:2730-2757`: log error, record
`S3Error("AppendMPUPartState", ..., "StateUnavailable")`, emit
`audit.EventTypeMPUValkeyUnavail`, return `errStateUnavailable` to
the caller (new sentinel) and translate at the `handleUploadPartCopy`
call site into `HTTPStatus: 503 ServiceUnavailable`.

Caveat: the backend part has already been written when this fires.
This matches the existing UploadPart behaviour — client retries
(which the S3 contract encourages on 503) produce a duplicate backend
part that `CompleteMultipartUpload` discards via ETag-set
reconciliation.

Unit test (co-located in `upload_part_copy_test.go`):
`TestUploadPartCopy_MPU_AppendPartFailure_Returns503` using a
`failOnAppendStateStore` injector mirroring the one in
`encrypted_mpu_handler_test.go`.

### 3.3 The 10 UploadPartCopy integration tests (Phase B)

New file: `test/upload_part_copy_integration_test.go`, build-tagged
`//go:build integration`.

Shared fixture (`setupUploadPartCopyFixture(t)`):

- Start shared MinIO (`StartMinIOServer(t)`).
- Create two buckets with a unique prefix:
  `upc-src-<timestamp>`, `upc-dst-<timestamp>`.
- Start a gateway via `StartGateway(t, cfg)`.
- Seed three source objects via the gateway's PUT path (to guarantee
  metadata parity with production):
  - `plaintext-src.bin` (bucket `upc-src-*`, policy: no encryption) —
    seeded against a second gateway instance with a "no-encryption"
    policy, **or** via the raw backend client (which is lazier but
    bypasses the gateway's encryption path entirely; pick the former
    for byte-parity with the classifier).
  - `chunked-src.bin` (bucket `upc-src-*`, default chunked engine,
    `crypto.DefaultChunkSize = 64 KiB`).
  - `legacy-src.bin` (legacy-AEAD; seeded via direct
    `engine.Encrypt(...)` call + `PutObject`).

Per-test AWS SDK client:
```go
cl := s3.NewFromConfig(aws.Config{
    Region:       "us-east-1",
    Credentials:  staticCreds("test-ak","test-sk"),
    BaseEndpoint: aws.String(gateway.URL()),
    UsePathStyle: true,
})
```

Each test owns its own destination upload. `t.Cleanup` calls
`AbortMultipartUpload` on the destination upload ID if
`Complete` did not run.

#### Test-by-test notes

**`TestUploadPartCopy_Chunked`:** `CreateMultipartUpload` on dst;
`UploadPartCopy` of full chunked source; `CompleteMultipartUpload`;
`GetObject` on dst; assert bytes equal source plaintext.

**`TestUploadPartCopy_Chunked_WithRange`:** three sub-tests with
`x-amz-copy-source-range: bytes=<start>-<end>` chosen so that:
- `start`, `end` both mid-chunk (not aligned to 64 KiB)
- `start` on chunk boundary, `end` in next chunk
- range spans 3+ chunks
Assert downloaded object matches `srcPlain[start:end+1]`.

**`TestUploadPartCopy_Legacy`:** full-object copy of a legacy source.
Assert `gateway_upload_part_copy_legacy_fallback_total` metric
incremented by reading the Prometheus HTTP endpoint on the gateway's
`/metrics` route.

**`TestUploadPartCopy_Plaintext`:** plaintext source to plaintext dst.
Assert no encrypt/decrypt via Prometheus counter delta check on
`gateway_crypto_aead_operations_total` (or equivalent crypto
activity metric — inspect `internal/metrics/metrics.go` at
implementation time and pick the narrowest).

**`TestUploadPartCopy_LargeSource_MustUseRange`:** the V0.6-S3-3
issue at `docs/issues/v0.6-issues.md:646-652` prescribes a 6 MiB
source with a `HeadObject` Content-Length override reporting > 5 GiB.
Implementation: create the source via the gateway, then inject the
override either by (a) MinIO `mc` metadata-set, or (b) a custom
gateway test middleware. Choose (b): add a test-only
`reportContentLengthFunc(bucket, key) int64` hook on the test
`Client` wrapper that, when set, overrides the `ContentLength` in the
`HeadObject` return. Wire via a new variadic `WithHeadObjectOverride`
option on `TestGateway`.

First assertion: `UploadPartCopy` with no range → HTTP 400
`InvalidRequest` (from
`internal/api/upload_part_copy.go:233-244`).

Second assertion: `UploadPartCopy` with
`x-amz-copy-source-range: bytes=0-<6MiB-1>` → HTTP 200.

**`TestUploadPartCopy_CrossBucket`:** source bucket with one engine,
destination bucket with a different engine (different password). Use
two `PolicyConfig` entries via a single `PolicyManager`:

```yaml
- id: upc-src-policy
  buckets: ["upc-src-*"]
  encryption:
    password: "source-password-32-chars-..."
- id: upc-dst-policy
  buckets: ["upc-dst-*"]
  encryption:
    password: "dest-password-32-chars-..."
```

The gateway's per-bucket-engine resolution at
`Handler.getEncryptionEngine(bucket)` must yield the right engine for
the decrypt and re-encrypt halves. Assert byte-for-byte round trip.

**`TestUploadPartCopy_AbortMidway`:** create MPU, issue 3
`UploadPartCopy`s, then `AbortMultipartUpload`. Assertions:
- Subsequent `CompleteMultipartUpload` returns 404
  `NoSuchUpload`.
- `ListObjects` on dst bucket returns zero objects under the target
  key prefix.
- Audit events (captured via a test `audit.Logger` fake passed via
  `WithAuditLogger`) include `mpu.abort`.

**`TestUploadPartCopy_MixedWithUploadPart`:** interleave 2 × regular
`UploadPart` (with a random 5 MiB body) and 2 × `UploadPartCopy`
across part numbers 1–4. `CompleteMultipartUpload` with all four;
download; assert byte-for-byte match against the expected
concatenation.

**`TestUploadPartCopy_CrossBucket_ReadDenied_Integration`:** needs two
MinIO credential sets. Implementation:
1. `cfg.Backend.UseClientCredentials = true`.
2. Seed source object with root MinIO credentials.
3. Create a second MinIO user `alice/alice-secret` via `mc admin user
   add`; grant `s3:PutObject,s3:CreateMultipartUpload` on the
   destination bucket only.
4. Issue `UploadPartCopy` with SigV4 creds for `alice`. Expected:
   the gateway issues `HeadObject` on the source bucket with
   `alice` creds, MinIO returns `AccessDenied`, gateway translates
   to HTTP 403.
5. Assert zero destination parts exist (ListParts returns empty).

If `mc` is not available, `t.Skipf("mc CLI not found; skipping
multi-credential test")`.

**`TestUploadPartCopy_PlaintextSource_EncryptedDestBucket_Refused_Integration`:**
Load a `PolicyManager` where dst bucket has
`require_encryption: true`. Seed a plaintext source. Issue
`UploadPartCopy`. Assert HTTP 500 `InternalError`, audit event
`copy_part_refused`, zero destination parts.

### 3.4 The 3 `TestUploadPartCopy_MPU_*` tests (Phase C)

File: `test/upload_part_copy_mpu_integration_test.go`, build-tagged
`//go:build integration`.

Shared fixture: as above, **plus**
- `kmMem = crypto.NewPasswordKeyManager("integration-test-password-32ch...")`
- `store = test.NewTestMPUStateStore(t)`
- `gateway = test.StartGateway(t, cfg, test.WithKeyManager(kmMem),
   test.WithMPUStateStore(store),
   test.WithPolicyManager(encryptedMPUPolicy(t, dstBucket)))`

where `encryptedMPUPolicy` writes:

```yaml
- id: upc-mpu-encrypted
  buckets: ["upc-dst-*"]
  encrypt_multipart_uploads: true
  encryption:
    password: "integration-test-password-32ch..."
```

**`TestUploadPartCopy_MPU_PlaintextSource_EncryptedDest`:** full-object
copy. Assertions:
1. `CompleteMultipartUpload` returns 200.
2. Gateway `GetObject` returns bytes equal source plaintext.
3. Raw-backend `GetObject` returns bytes that do **not** contain the
   first 1 KiB of source plaintext (at-rest ciphertext smoke).
4. Valkey `mpu:*` key for the upload ID is deleted post-Complete.
5. Audit events: `mpu.create`, `mpu.part` (with part-count), `mpu.complete`.

**`TestUploadPartCopy_MPU_ChunkedSource_EncryptedDest_WithRange`:**
Seed a chunked source via the gateway. Create encrypted-MPU upload
on dst. `UploadPartCopy` with a mid-chunk `x-amz-copy-source-range`.
Assert downloaded object matches `srcPlain[range]`. Assert raw-backend
bytes ≠ plaintext slice. Assert Valkey `PartRecord.PlainLen` matches
`range-size`.

**`TestUploadPartCopy_MPU_LegacySource_EncryptedDest`:** seed a legacy
source via `engine.Encrypt(...) + PutObject`. Copy into the
encrypted-MPU dst. Assert round-trip + at-rest ciphertext. Assert
the copy path hits the legacy cap check (verify via
`gateway_upload_part_copy_legacy_fallback_total` metric).

### 3.5 The 4 end-to-end encrypted-MPU integration tests (Phase D)

File: `test/encrypted_mpu_integration_test.go` (new; the existing
`test/encrypted_mpu_test.go` is env-gated and uses external gateway
— this plan leaves it in place for external-gateway regression and
adds a proper in-process CI file alongside).

Shared fixture: same as §3.4 (PasswordKeyManager + Valkey
StateStore + encrypted-MPU policy).

**`TestEncryptedMPU_PasswordKeyManager_SmallObject`:** 16 MiB /
2 × 8 MiB parts. Create → UploadPart × 2 → Complete → GetObject.
Assert byte-for-byte. Assert raw-backend bytes ≠ plaintext. Assert
Valkey key deleted post-Complete.

**`TestEncryptedMPU_PasswordKeyManager_Ranged_GET`:** 64 MiB /
8 × 8 MiB parts. Complete. Issue three ranged GETs:
- mid-chunk (offset 10 KiB, length 50 KiB within part 1)
- cross-chunk (offset 60 KiB, length 20 KiB spanning 64 KiB boundary)
- cross-part (offset ~7.9 MiB, length 500 KiB spanning part 1 / part 2)

Assert returned bytes equal the corresponding plaintext slice.

**`TestEncryptedMPU_PasswordKeyManager_AtRestCiphertext`:** 8 MiB /
1 part. Complete. Use `NewRawBackendS3Client` to fetch the raw
backend bytes. Assert:
- `len(raw) > len(plaintext)` (AEAD tags + chunk overhead)
- No 1 KiB-aligned substring of plaintext appears in raw.
- The MPU manifest object (key ending `.mpu-manifest` or the
  metadata header `x-amz-meta-encryption-mpu`) is present.

**`TestEncryptedMPU_PasswordKeyManager_AbortDeletesState`:** Create →
UploadPart × 1 → Abort. Assert:
- Valkey `Get` returns `ErrUploadNotFound`.
- Backend `ListParts` returns empty.
- Audit events: `mpu.create`, `mpu.part`, `mpu.abort`.

### 3.6 Cosmian KMS encrypted-MPU test (Phase E)

File: `test/cosmian_kms_integration_test.go` — append one test.

**`TestCosmianKMS_EncryptedMPU_RoundTrip`:**
1. Reuse the file's Cosmian docker-compose fixture (`setupCosmianEnv`

2. Construct a `cosmianKeyManager` via the same registry path the

## 5. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| MinIO fixture is shared across all integration tests (`sync.Once`); parallel test runs could race on bucket creation or upload IDs. | Each test names buckets with `<test-name>-<unix-nanos>`; each test's `uploadID` is produced by MinIO so collisions are impossible. |
| Valkey DB-isolation scheme (hash(t.Name()) → DB 0..15) collides; one test flushes another's state. | 16 DBs for ~17 tests means 2 share a DB. Mitigation: additionally namespace `uploadKey` with a per-test prefix, or scale to `redis.Options.DB = (hash % 15) + 1` leaving DB 0 reserved. Chosen: rely on distinct `uploadID` values (UUID → hash → unique `mpu:` key), not DB isolation. `t.Cleanup` does `DEL` on known `mpu:*` keys, not `FLUSHDB`. |
| `mc admin user add` unavailable in CI. | Test 9 (`..._ReadDenied_Integration`) skips cleanly if `mc` binary absent. Existing `hasDocker()` pattern in `test/minio.go` extended. CI runs Docker + `mc` via the same docker-compose. |
| `HeadObject` size-override hook leaks into non-test builds if plumbed through the production `Handler`. | Put the override on the test `s3.Client` wrapper only (a new `testS3ClientOverride` type in `test/`), never in `internal/s3/`. |
| Cosmian KMS integration server flakiness on CI. | Test 18 is gated on the existing `cosmianAvailable(t)` helper. Skip if unreachable. |
| `AppendPart` fail-closed alignment is technically a behavioural change for an in-flight release and could surprise external operators who depended on the warn-and-continue. | Zero known external depends. Audit the CHANGELOG note as **Fixed**, not **Added**; flag as a security-critical fix with reference to `internal/api/handlers.go:2730-2757` as the authoritative precedent. |
| Raw-backend at-rest ciphertext test reports false negative if plaintext happens to contain a 1 KiB run of zeros or other low-entropy pattern present in AES-GCM ciphertext at the same position. | Use a random 8 MiB plaintext (`io.CopyN(dst, rand.Reader, 8*1024*1024)`); assert "first 1 KiB of plaintext does not appear anywhere in ciphertext" — probability of coincidence for random data is cryptographically negligible. |
| Gateway audit-logger fake must not leak across tests when the global `audit.Logger` is package-level. | `test.WithAuditLogger(fake)` wires via `NewHandlerWithFeatures` arg 7; the handler holds it by pointer. Each test instantiates its own handler via a fresh `StartGateway`. |
