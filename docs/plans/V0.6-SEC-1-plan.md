# V0.6-SEC-1 — Pluggable KeyManager Interface — Implementation Plan

Status: Draft (fresh plan, supersedes any prior proposal)
Owner: Crypto
Priority: P0
Labels: `milestone:v0.6`, `area:kms`

> This plan is independent and does not reuse `docs/V0.6-SEC-1-IMPLEMENTATION.md`.
> It focuses on producing a stable, HSM-friendly `KeyManager` abstraction and
> decoupling `internal/crypto/engine.go` from any concrete KMS integration.

---

## 1. Context & Current State

The codebase already contains a `KeyManager` interface at
`internal/crypto/keymanager.go:20` and a Cosmian KMIP implementation at
`internal/crypto/keymanager_cosmian.go:48`. The engine
(`internal/crypto/engine.go:88`) holds a `kmsManager KeyManager` field set via
`SetKeyManager` (`internal/crypto/engine.go:205`). However the current state has
several gaps relative to V0.6-SEC-1 goals:

- The interface is present but has **no formal conformance test suite** that
  third-party/HSM adapters can import and run.
- There is **no in-memory adapter** suitable for unit tests outside the `crypto`
  package, benchmarks, and local development (tests currently depend on a full
  KMIP mock server).
- There is **no HSM placeholder adapter** (e.g. PKCS#11 skeleton) documenting
  the integration contract for future HSM work.
- `SetKeyManager` mutates the engine after construction — an anti-pattern that
  couples lifecycle and increases the risk of a racy "half-configured" engine.
  The refactor should make the KeyManager a **construction-time dependency**
  (dependency injection) consistent with Go idioms.
- Invariants (context propagation, zeroization, idempotency of `Close`, nil
  safety) are not documented on the interface.
- The engine still calls `context.Background()` in three hot paths
  (`engine.go:812`, `engine.go:938`, `engine.go:1118`, `engine.go:1280`),
  bypassing caller-provided deadlines/cancellation — this must be fixed so the
  interface can be honoured by HSM adapters where network I/O is the norm.

The objective of V0.6-SEC-1 is therefore to **harden, document, and
generalise** the existing interface so that additional adapters (HSM, AWS KMS,
Vault Transit, GCP KMS, etc.) can be implemented without touching the engine.

## 2. Design Goals & Non-Goals

### Goals
1. A **stable, well-documented** `KeyManager` interface with explicit
   invariants (nil handling, zeroization, context deadlines, concurrency,
   Close idempotency).
2. The `engine` depends **only on the interface** — no type assertions,
   no provider-specific branches, no `SetKeyManager` post-construction mutation.
3. A reusable **interface conformance test suite** callable from every adapter
   package (`ConformanceSuite(t, factory)`).
4. Ship three adapters:
   - **`InMemoryKeyManager`** — production-grade, used for tests and
     single-password-equivalent deployments without external KMS.
   - **Cosmian KMIP** — existing implementation, re-plugged into the new
     constructor/registry.
   - **HSM placeholder (PKCS#11-shaped)** — stub with explicit `ErrNotImplemented`
     returns plus documentation of required build tags / cgo dependencies.
5. A small **adapter registry** (`Register(name, factory)`) to enable
   configuration-driven selection without a `switch` statement in
   `internal/api/crypto_factory.go`.
6. Structured **errors** (`ErrKeyNotFound`, `ErrUnwrapFailed`,
   `ErrProviderUnavailable`) usable with `errors.Is`.

### Non-Goals
- Implementing real AWS KMS / Vault Transit (deferred to v1.0).
- Implementing a functional PKCS#11 adapter (only the skeleton + docs).
- FIPS build profile (tracked separately as V0.6-SEC-2).
- Admin API for rotation (V0.6-CFG-1).

## 3. Target Interface (proposed)

File: `internal/crypto/keymanager.go`

```go
// KeyManager abstracts an external Key Management System that wraps/unwraps
// per-object Data Encryption Keys (DEKs).
//
// Invariants (MUST hold for every implementation):
//   1. All methods are safe for concurrent use by multiple goroutines.
//   2. ctx is honoured: implementations MUST use ctx for any blocking I/O
//      and return ctx.Err() wrapped when cancellation occurs.
//   3. Plaintext DEKs returned by UnwrapKey MUST be owned by the caller; the
//      caller is responsible for zeroization. Implementations MUST NOT retain
//      a reference to the returned slice. Implementations SHOULD zeroize any
//      internal copies before returning.
//   4. WrapKey MUST NOT log or export plaintext input.
//   5. Close is idempotent; subsequent calls return nil. After Close, all
//      other methods MUST return ErrProviderUnavailable.
//   6. A nil KeyManager is never valid; callers must check.
type KeyManager interface {
    Provider() string
    WrapKey(ctx context.Context, plaintext []byte, metadata map[string]string) (*KeyEnvelope, error)
    UnwrapKey(ctx context.Context, envelope *KeyEnvelope, metadata map[string]string) ([]byte, error)
    ActiveKeyVersion(ctx context.Context) (int, error)
    HealthCheck(ctx context.Context) error
    Close(ctx context.Context) error
}

// Sentinel errors for use with errors.Is.
var (
    ErrProviderUnavailable = errors.New("keymanager: provider unavailable")
    ErrKeyNotFound         = errors.New("keymanager: key not found")
    ErrUnwrapFailed        = errors.New("keymanager: unwrap failed")
    ErrInvalidEnvelope     = errors.New("keymanager: invalid envelope")
)
```

Rationale:
- Adding sentinel errors enables callers (engine, admin API, readiness probes)
  to react semantically instead of string matching.
- The concurrency + zeroization invariants match guidance in *Computer and
  Information Security Handbook (4th ed., 2024)* on key-lifecycle hygiene in
  software KMS clients [1].
- The interface remains small (6 methods) — consistent with "accept
  interfaces, return structs" and the small-interface guidance in
  *Hands-On Software Engineering with Golang* (Anagnostopoulos, 2020) [2].

## 4. Work Breakdown

### Phase A — Interface hardening (no behaviour change)
- A1. Add sentinel errors + invariant doc comments to `keymanager.go`.
- A2. Extend `KeyEnvelope` with `CreatedAt time.Time` (optional, zero-value
      safe) so HSM adapters can record wrap times; back-compat preserved.
- A3. Add `context.Context` to all engine→KeyManager call sites that still use
      `context.Background()` (`engine.go:812,938,1118,1280`).
- A4. Update the existing Cosmian implementation to wrap its internal errors
      with the new sentinels via `fmt.Errorf("...: %w", ErrUnwrapFailed)`.

### Phase B — Engine decoupling
- B1. Introduce an **EngineOption** pattern (functional options) so that
      `NewEngineWith…` constructors accept `WithKeyManager(km KeyManager)`.
      This replaces the out-of-band `SetKeyManager` mutator for new callers;
      the old function remains for one release with a `// Deprecated:` notice.
      The functional-options pattern is the idiomatic Go approach recommended
      by *Go: Design Patterns for Real-World Projects* (Vivien et al., 2017) [3].
- B2. Remove the `if e, ok := enc.(*engine); ok { … }` type assertions in
      `SetKeyManager` / `SetKeyResolver` by making the engine expose a
      narrower configuration interface.
- B3. Collapse the repeated KMS branches in `Encrypt` / `encryptChunked` /
      `encryptChunkedWithMetadataFallback` / `Decrypt` / `decryptChunked` /
      `DecryptRange` into two helpers:
      `wrapOrDeriveKey(ctx, alg, metadata) ([]byte, *KeyEnvelope, error)` and
      `unwrapOrDeriveKey(ctx, alg, metadata) ([]byte, error)`.
      This reduces six copies of the branch to a single well-tested unit and
      is the direct application of the adapter-seam extraction pattern from
      *Hands-On Software Architecture with Golang* (Raiturkar, 2018) [4].

### Phase C — In-memory adapter
- C1. New file `internal/crypto/keymanager_memory.go`:
      `NewInMemoryKeyManager(masterKey []byte, opts …MemoryOption) (KeyManager, error)`.
      Implements AES-256 key-wrap (RFC 3394 / `crypto/aes` + NIST KW) for
      parity with what Cosmian does on the wire.
      Supports multiple historical versions for rotation tests.
      Zeroizes its internal master key copies on `Close`.
- C2. Register under the name `"memory"` in the adapter registry.
- C3. Rewrite `internal/crypto/engine_test.go` helpers that currently rely on
      the KMIP mock to use the in-memory adapter for the happy paths
      (faster, no TLS, no goroutines).

### Phase D — Adapter registry
- D1. New file `internal/crypto/keymanager_registry.go` with:
      - `type Factory func(ctx context.Context, cfg map[string]any) (KeyManager, error)`
      - `Register(name string, f Factory)` (panics on duplicate, matches the
        `database/sql` convention).
      - `Open(ctx, name, cfg)` returning `KeyManager` or
        `ErrProviderUnavailable` if unknown.
- D2. Move the `switch provider` in `internal/api/crypto_factory.go:22` to
      an `init()` that registers `"cosmian"`, `"kmip"`, `"memory"`, and `"hsm"`
      (stub). `BuildKeyManager` becomes a thin shim over `Open`.
- D3. Document in `docs/KMS_COMPATIBILITY.md` how downstream projects add a
      third-party adapter by calling `crypto.Register` from their own package.

### Phase E — HSM placeholder adapter
- E1. New file `internal/crypto/keymanager_hsm.go` gated by `//go:build hsm`
      build tag. Default build uses
      `internal/crypto/keymanager_hsm_stub.go` returning
      `ErrProviderUnavailable` on every method with a clear message pointing
      to documentation.
- E2. Document the PKCS#11 contract (slot / token / pin-source /
      wrapping-mechanism) in a new ADR `docs/adr/0004-hsm-adapter-contract.md`.
      Align with guidance in *Computer and Information Security Handbook*
      chapters on HSM integration [1].
- E3. Register as `"hsm"` in the adapter registry.

### Phase F — Conformance test kit
- F1. New file `internal/crypto/keymanager_conformance.go` (non-`_test.go` so
      external adapter packages can import it):
      ```go
      func ConformanceSuite(t *testing.T, newFactory func(t *testing.T) KeyManager)
      ```
      Runs:
      - Wrap → Unwrap round-trip with random DEK sizes (16, 24, 32 bytes).
      - `ActiveKeyVersion` monotonicity.
      - `HealthCheck` returns nil on a live manager.
      - `Close` is idempotent and post-Close calls yield `ErrProviderUnavailable`.
      - Context cancellation stops Wrap/Unwrap within a grace window.
      - Concurrent Wrap/Unwrap on 64 goroutines produces no races
        (verified under `-race`).
- F2. Wire the suite into `keymanager_test.go` (Cosmian),
      `keymanager_memory_test.go` (in-memory), and a build-tagged
      `keymanager_hsm_test.go` (HSM — expected to skip when tag absent).
- F3. This testing-toolkit pattern follows
      *Test-Driven Development in Go* (Simion, 2023) guidance for shared
      contract tests across interface implementations [5].

### Phase G — Config & migration
- G1. `internal/config/config.go` — extend `KeyManagerConfig` to accept
      `provider: memory` and the associated master-key source
      (`env:` / `file:` / `vault:` prefixed secret reference, existing
      pattern).
- G2. Preserve backward compatibility: `provider: cosmian` or `kmip` continues
      to work; missing provider defaults to `memory` only when
      `Encryption.KeyManager.Enabled` is false (no behaviour change for
      existing deployments).
- G3. Update `config.yaml.example` with an annotated `memory` block.

## 6. Test Strategy

| Layer | Tests |
|---|---|
| Unit — interface | `ConformanceSuite` exercised by all three adapters under `-race`. |
| Unit — engine | Restructured `engine_test.go` uses in-memory adapter to remove the TLS/KMIP dependency from hot-path tests. Existing KMIP mock retained for one dedicated integration test. |
| Unit — registry | `Open` returns `ErrProviderUnavailable` for unknown names; `Register` panics on duplicate; `Factory` receives untouched cfg map. |
| Fuzz | Re-enable `internal/crypto/fuzz_test.go` seeds against the in-memory adapter for envelope edge cases (truncated / extended / random ciphertext). |
| Integration | `test/integration` — existing Cosmian container test runs conformance suite against a real KMIP endpoint. |
| Bench | `BenchmarkEncryptWithMemoryKM` to provide a CPU-only baseline for future KMS latency comparisons (feeds V0.6-QA-1). |

Commands to verify locally:
```
go test ./internal/crypto/... -race -count=1
go test ./internal/api/... -race
go test ./... -tags=hsm ./internal/crypto/...   # skips, should compile
go vet ./...
golangci-lint run
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Functional-options change breaks external callers of `NewEngineWith…`. | Keep the existing constructors; add new `NewEngine(opts ...Option)` alongside. Mark old ones `// Deprecated:` with a migration example. |
| `SetKeyManager` removal breaks `internal/api/handlers.go:589`. | Keep `SetKeyManager` as a thin wrapper but emit a deprecation log; migrate handlers to pass the KM through the engine constructor in the same PR. |
| Registry side-effects in `init()` can cause hidden coupling. | Register only the pure-Go adapters (memory, stub HSM) in `crypto` package init. `cosmian` registered from `internal/api/crypto_factory.go` init to avoid pulling the KMIP dependency into `crypto` package tests. |
| Conformance suite false-negatives on slow CI. | All timeouts configurable via `ConformanceOptions`; default 2 s wrap/unwrap, 10 s health. |
| Context propagation refactor touches six call sites; risk of incorrect ctx reuse across goroutines. | Add a lint rule (`contextcheck`) to CI; add a unit test that intentionally cancels the caller context and asserts propagation. |
