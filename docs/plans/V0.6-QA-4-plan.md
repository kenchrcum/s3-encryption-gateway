Root cause traced to `test/minio.go:96-168`

Key rules:

1. **A test lives in exactly one tier.** If a test needs Docker, it
   is tier 2. If a test spins up multiple containers and runs for
   minutes, it is tier 3. No exceptions.
2. **`go test ./...` is tier 1 only.** By construction (build tag
   gating), tier 2/3 tests cannot compile under the default build
   and so cannot be run accidentally.
3. **Docker unavailability is a `t.Skip`, not an error.** Tier 2
   cleanly skips when Docker is not present; the skip message names
   the missing fixture.
4. **Flaky tier 1 is a P0 bug.** Tier 1 is the fastest feedback
   loop; any flake in tier 1 is prioritised over feature work.

### 2.2 The `Provider` interface

```go
// test/provider/provider.go
package provider

import (
    "context"
    "testing"

    "github.com/kenneth/s3-encryption-gateway/internal/config"
)

// Provider is the contract every S3-compatible backend-under-test
// satisfies. Implementations are registered via init() and discovered
// at test start time.
type Provider interface {
    // Name returns a stable identifier used in subtest names and
    // skip messages. Must not contain characters that break
    // `go test -run`.
    Name() string

    // Start brings the backend up and returns a configured client
    // and a unique bucket name. The bucket is already created.
    // t.Cleanup is registered internally; callers do not manage
    // teardown.
    Start(ctx context.Context, t *testing.T) Instance

    // Capabilities declares which features this backend supports.
    // Tests consult this bitmap and call t.Skipf when the backend
    // does not support the feature being tested.
    Capabilities() Capabilities

    // CleanupPolicy declares whether the conformance suite should
    // delete objects it creates. Minimum-storage-duration backends
    // (Wasabi) declare SkipDelete to avoid billing churn; free-delete
    // backends declare Delete.
    CleanupPolicy() CleanupPolicy

    // BackendConfig returns the config stanza the gateway-under-test
    // should load to talk to this backend.
    BackendConfig(inst Instance) config.BackendConfig
}

type Instance struct {
    Endpoint     string
    Region       string
    AccessKey    string
    SecretKey    string
    Bucket       string          // Already created, empty
    // Raw exposes provider-specific handles for edge-case tests
    // (e.g. MinIO admin client for at-rest inspection). Most tests
    // do NOT use this.
    Raw          interface{}
}

type Capabilities uint64

const (
    CapObjectLock Capabilities = 1 << iota
    CapObjectTagging
    CapMultipartUpload
    CapMultipartCopy
    CapVersioning
    CapServerSideEncryption
    CapPresignedURL
    CapConditionalWrites   // If-None-Match / If-Match
    CapBatchDelete         // DeleteObjects XML batch
    CapKMSIntegration      // Cosmian KMS integration works
    // ...
)

type CleanupPolicy uint8

const (
    CleanupPolicyDelete     CleanupPolicy = iota // Delete test objects on teardown
    CleanupPolicySkipDelete                      // Leave for provider lifecycle (cost-sensitive)
)

// Registry ŌĆö tests iterate this.
var registered []Provider

func Register(p Provider)   { registered = append(registered, p) }
func All() []Provider       { return append([]Provider(nil), registered...) }
```

Provider implementations use Testcontainers-Go (see
*Modern REST API Development in Go* ch. 10). Example (abbreviated):

```go
// test/provider/minio.go
func init() {

        provider.Register(&minioProvider{})
    }
}

type minioProvider struct{}

func (p *minioProvider) Name() string { return "minio" }

func (p *minioProvider) Capabilities() provider.Capabilities {
    return provider.CapMultipartUpload |
        provider.CapMultipartCopy |
        provider.CapObjectTagging |
        provider.CapServerSideEncryption |
        provider.CapPresignedURL |
        provider.CapConditionalWrites
    // NOT: CapObjectLock, CapVersioning (MinIO AGPL as of the
    // pinned tag supports both but we want to prove the skip path
    // works, so we run ObjectLock only against Garage+RustFS.
    // Flip when conformance is green across all three.)
}

func (p *minioProvider) Start(ctx context.Context, t *testing.T) provider.Instance {
    t.Helper()
    req := testcontainers.ContainerRequest{
        Image:        "minio/minio:RELEASE.2024-11-07T00-52-20Z",

        Cmd:          []string{"server", "/data"},
        Env: map[string]string{
            "MINIO_ROOT_USER":     "minioadmin",
            "MINIO_ROOT_PASSWORD": "minioadmin",
        },
        WaitingFor: wait.ForHTTP("/minio/health/live").WithPort("9000"),
    }
    c, err := testcontainers.GenericContainer(ctx,
        testcontainers.GenericContainerRequest{
            ContainerRequest: req, Started: true,
        })
    require.NoError(t, err)
    t.Cleanup(func() { _ = c.Terminate(context.Background()) })

    host, _ := c.Host(ctx)
    port, _ := c.MappedPort(ctx, "9000")
    endpoint := fmt.Sprintf("http://%s:%s", host, port.Port())

    inst := provider.Instance{
        Endpoint:  endpoint,
        Region:    "us-east-1",
        AccessKey: "minioadmin",
        SecretKey: "minioadmin",
        Bucket:    fmt.Sprintf("conf-%s-%d", p.Name(), time.Now().UnixNano()),
    }
    createBucket(t, inst)   // <-- THE FIX: always create the bucket
    return inst
}
```

Garage and RustFS implementations follow the same shape. All three are
Testcontainers-backed; no `exec.Command`, no `pkill`, no pre-running
docker-compose.

### 2.3 The `ConformanceSuite`

```go
// test/conformance/suite.go
//go:build conformance

package conformance

import (
    "testing"

    "github.com/kenneth/s3-encryption-gateway/test/provider"
)

// RunAll runs the conformance suite against every registered
// provider. Each provider is its own top-level t.Run, each test
// function is a nested t.Run. A failure in one provider never
// aborts the others.
func TestConformance(t *testing.T) {
    for _, p := range provider.All() {
        p := p
        t.Run(p.Name(), func(t *testing.T) {
            t.Parallel()  // different providers run concurrently
            ctx := t.Context()
            inst := p.Start(ctx, t)

            cases := []struct {
                name string
                cap  provider.Capabilities
                fn   func(*testing.T, provider.Instance)
            }{
                {"PutGet",                 0,                            testPutGet},
                {"HeadObject",             0,                            testHead},
                {"ListObjects",            0,                            testList},
                {"DeleteObject",           0,                            testDelete},
                {"DeleteObjects",          0,                            testDeleteMulti},
                {"RangedRead",             0,                            testRangedRead},
                {"CopyObject",             0,                            testCopyObject},
                {"Multipart_Basic",        provider.CapMultipartUpload,  testMultipartBasic},
                {"Multipart_Abort",        provider.CapMultipartUpload,  testMultipartAbort},
                {"Multipart_ListParts",    provider.CapMultipartUpload,  testMultipartListParts},
                {"UploadPartCopy_Full",    provider.CapMultipartCopy,    testUPC_Full},
                {"UploadPartCopy_Range",   provider.CapMultipartCopy,    testUPC_Range},
                {"Chunked_RoundTrip",      0,                            testChunkedRoundTrip},
                {"Chunked_RangedRead",     0,                            testChunkedRanged},
                {"Legacy_RoundTrip",       0,                            testLegacyRoundTrip},
                {"Tagging",                provider.CapObjectTagging,    testTagging},
                {"Presigned",              provider.CapPresignedURL,     testPresigned},
                {"Rotation_DualRead",      0,                            testRotationDualRead},
                {"EncryptedMPU_PutGet",    provider.CapMultipartUpload,  testEncryptedMPU},
                {"EncryptedMPU_AtRest",    provider.CapMultipartUpload,  testEncryptedMPU_AtRest},
                {"EncryptedMPU_Ranged",    provider.CapMultipartUpload,  testEncryptedMPU_Ranged},
                {"EncryptedMPU_Tamper",    provider.CapMultipartUpload,  testEncryptedMPU_Tamper},
                {"UPC_MPU_Plaintext",      provider.CapMultipartCopy,    testUPC_MPU_Plaintext},
                {"UPC_MPU_Chunked",        provider.CapMultipartCopy,    testUPC_MPU_Chunked},
                {"UPC_MPU_Legacy",         provider.CapMultipartCopy,    testUPC_MPU_Legacy},
                {"ObjectLock_Retention",   provider.CapObjectLock,       testObjectLockRetention},
                {"ObjectLock_LegalHold",   provider.CapObjectLock,       testObjectLockLegalHold},
                {"ObjectLock_BypassRefused", provider.CapObjectLock,     testObjectLockBypass},
                // ~40 total
            }

            for _, tc := range cases {
                tc := tc
                t.Run(tc.name, func(t *testing.T) {
                    if tc.cap != 0 && p.Capabilities()&tc.cap == 0 {
                        t.Skipf("%s does not advertise %s",
                            p.Name(), tc.cap)
                    }
                    tc.fn(t, inst)
                })
            }
        })
    }
}
```

Each `testXxx` takes `(t *testing.T, inst provider.Instance)` and is
provider-agnostic. There are **zero** `if providerName == "minio"`
branches inside test bodies ŌĆö capability bits handle the
differentiation. This is the pattern from *Contract Testing in Action*
ch. 2: one contract, many implementations, one suite asserting the
contract.

### 2.4 Gateway harness

One harness function replaces the four today:

```go
// test/harness/gateway.go
// Spawns an in-process gateway bound to the provider under test.
// Returns an HTTP client + addr. Cleanup is registered via t.Cleanup.
func StartGateway(t *testing.T, inst provider.Instance, opts ...Option) *Gateway
```

All gateway-config knobs (encryption mode, policy YAML, admin API,
Valkey stanza, rotation state, client-credentials mode) are Options
functions. Tests compose only what they need.

### 2.5 The headline security assertion

The `testEncryptedMPU_AtRest` test is the headline security smoke
test. Pseudo-code:

```go
func testEncryptedMPU_AtRest(t *testing.T, inst provider.Instance) {
    gw := harness.StartGateway(t, inst,
        harness.WithEncryptedMPU(true),
        harness.WithValkey(startMiniredis(t)))

    plaintext := bytes.Repeat([]byte("SECRET"), 1<<20) // 6 MiB ŌåÆ auto-multipart
    uploadViaAWSCLI(t, gw, "key1", plaintext)

    // Read the raw bytes directly from the backend, bypassing the gateway.
    rawClient := newRawS3Client(inst)
    raw := rawClient.GetObject(inst.Bucket, "key1")

    require.NotContains(t, raw, []byte("SECRET"),
        "at-rest ciphertext must not contain plaintext markers ŌĆö "+
            "security goal not met for provider %s", inst.Name())
}
```

This runs against MinIO, Garage, and RustFS automatically. If any
provider ever leaks plaintext through an edge-case path, CI goes red.

### 2.6 External-provider plug-in architecture

New public S3 vendors plug in via a single file. The worked shape:

```go
// test/provider/external.go
//
// One file per external vendor. Registration is conditional on
// the vendor's credentials being present in the environment, so
// CI can run this package unmodified and the matrix auto-expands.

package provider

import (
    "context"
    "os"
    "testing"
)

// externalProvider is a shared implementation for credential-
// based S3 vendors. Per-vendor files (aws.go, wasabi.go,
// b2.go, hetzner.go, digitalocean.go, r2.go, ...) each call
// Register(&externalProvider{...}) in init() with the right
// endpoint/capabilities/cleanup bits.
type externalProvider struct {
    name      string
    endpoint  string        // "" = use SDK default for region
    region    string
    keyEnv    string
    secretEnv string
    bucketEnv string
    caps      Capabilities
    cleanup   CleanupPolicy
}

func (p *externalProvider) Name() string              { return p.name }
func (p *externalProvider) Capabilities() Capabilities { return p.caps }
func (p *externalProvider) CleanupPolicy() CleanupPolicy { return p.cleanup }

func (p *externalProvider) Start(ctx context.Context, t *testing.T) Instance {
    t.Helper()
    ak := os.Getenv(p.keyEnv)
    sk := os.Getenv(p.secretEnv)
    bk := os.Getenv(p.bucketEnv)
    if ak == "" || sk == "" || bk == "" {
        t.Skipf("%s credentials not set (%s, %s, %s)",
            p.name, p.keyEnv, p.secretEnv, p.bucketEnv)
    }
    // Build a unique prefix for this run so parallel CI jobs
    // against the same real bucket do not collide. We DO NOT
    // create a new bucket on external providers (many require
    // regional setup, payment authorisation, or verification).
    inst := Instance{
        Endpoint:  p.endpoint,
        Region:    p.region,
        AccessKey: ak,
        SecretKey: sk,
        Bucket:    bk,
    }
    // Register per-test key-prefix cleanup via t.Cleanup
    // (subject to CleanupPolicy).
    t.Cleanup(func() { cleanupPrefix(inst, p.cleanup, t.Name()) })
    return inst
}
```

Per-vendor registration files:

```go
// test/provider/wasabi.go ŌĆö same shape, CleanupPolicySkipDelete
// test/provider/aws.go    ŌĆö same shape, CleanupPolicyDelete
// test/provider/hetzner.go ŌĆö same shape, CleanupPolicyDelete
```

**Adding a new public vendor (the recipe, as it will appear in
`docs/TESTING.md`):**

1. Create `test/provider/<vendor>.go`.
2. Register an `externalProvider` with the vendor's endpoint,
   region, env-var names, capability bitmap, and cleanup policy.
3. Run `make test-conformance-external` with the vendor's
   credentials set.
4. If any conformance test fails because the vendor does not
   support a feature, **narrow the capability bitmap**, do not
   add a branch inside the test.
5. If any test fails for a behavioural quirk (e.g. different
   error code shape), open a ticket ŌĆö the gateway may need a
   compatibility fix.
6. Submit PR. The CI matrix picks up the new provider
   automatically when credentials are supplied as repo secrets.

### 2.8 Why Docker-only (and why Testcontainers, not
`docker-compose`)

The current state has four mutually-inconsistent deployment
paths, documented in ┬¦1.2. This produces three distinct failure
modes seen in the comprehensive log and in day-to-day development:

1. **"I forgot to `docker-compose up`"** ŌĆö test skips or fails
   connecting to MinIO/Valkey on a well-known port. Hides real
   regressions under noise.
2. **"Garage binary isn't on PATH on CI"** ŌĆö conditional skip
   that makes CI green despite losing an entire provider's
   coverage. Currently the default situation.
3. **"Previous run left orphan containers/processes"** ŌĆö
   `pkill garage` on `test/garage.go:43-45` is the tell that
   this has happened often enough to warrant a safety net. The
   safety net itself races the next test's start.

Testcontainers-Go solves all three mechanically:

1. **No prerequisite.** `Start(ctx, t)` spins the container up
   when the test wants it. Forgetting becomes impossible.
2. **One deployment model for every dependency.** MinIO, Garage,
   RustFS, Valkey, Cosmian KMS all launch the same way, through
   the same client library, with the same wait-for-ready
   semantics, and the same teardown guarantees.
3. **Ryuk reaper.** On first Testcontainers call, a sidecar
   container (`testcontainers/ryuk`) is started with a heartbeat
   TCP connection to the test process. When the test process
   disappears (clean exit, panic, SIGKILL, `go test -timeout`
   expiry), Ryuk's heartbeat read hits EOF and Ryuk kills every
   container carrying the `org.testcontainers.session-id` label
   for that session. Orphan containers become impossible unless
   the Docker daemon itself dies ŌĆö at which point nothing works
   anyway.

Why **not** keep `docker-compose` alongside Testcontainers:

- Two systems = two mental models. Developers write the wrong
  kind of fixture because "the other one is working."
- `docker-compose` uses fixed host ports
  (9000, 3900, 6379, 5696, 9998). Concurrent test runs on the
  same host collide. This is observable today when two
  developers on a shared CI runner both run `make test`.
- `docker-compose` teardown requires an explicit
  `docker-compose down`. Forgetting is common; the cleanup-
  leak class of bug is endemic.
- `docker-compose.yaml` has to be kept in sync with Go test code
  by hand. Drift produces "tests expect valkey 7.2.5 but compose
  says valkey 7.2.4" bugs that only surface weeks later.

The sole surviving role of Compose in the project: the
`docker/dev-compose.yml` kept for developers who want a
long-running local environment for manual smoke/REPL sessions.
**No test reads it.** A linter (`scripts/test-isolation.sh`)
greps for the string `docker-compose` under `test/` and fails CI
if any match appears.

## 6. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Testcontainers-Go startup latency dominates tier-2 runtime | Each provider caches its container image; parallel `t.Parallel()` per provider; total wall-clock stays < 15 min per the budget. |
| Pinning MinIO to an old AGPL tag rots | Quarterly review checkpoint; if a newer AGPL release is still usable, bump it; if not, drop MinIO to "compatibility only" and make Garage the reference. |
| RustFS upstream instability | Pin a specific tag; do not use `:latest`; re-evaluate quarterly. |
| Capability-bitmap drift from reality | `testProviderCapabilities` in the conformance self-test asserts the bitmap matches the backend's actual behaviour on a probe call; regression-guarded. |
| Mass delete + rewrite blows a merge window | Phased rollout (Phases 2-3 land incrementally; old and new coexist until Phase 5). Every PR is independently green. |
| External-provider tests flake on real AWS / Wasabi / B2 | External tests are nightly-only and their failures do not block merges; flakes surface in a dashboard, not a red PR. Per-vendor retry budget documented in `docs/TESTING.md`. |
| Goroutine / TCP leaks in the new harness (like the chaos test leak) | `goleak.VerifyNone` in `TestMain` of every conformance test; `netstat`-based assertion in the harness teardown. |
| Test wall-clock inflation over time | Enforce per-tier budget in CI: tier 1 > 60 s ŌåÆ fail; tier 2 > 15 min ŌåÆ fail. Budget gates live in the CI workflow, not in soft guidance. |
| Developer regresses to `docker-compose` habits | `scripts/test-isolation.sh` is a required PR check; any re-introduction of `docker-compose` / `exec.Command` / hard-coded ports fails the build mechanically. |
| Cosmian KMS container image tag churn | Pin a specific tag from <https://github.com/Cosmian/kms/pkgs/container/kms>; Dependabot / Renovate auto-opens a PR on new releases; the PR is a forcing function for re-running the KMS conformance suite against the new image before merge. |
| B2 cost creep from load tests | `test-load-external` is not on the PR path; runs nightly or on-demand only. Load-test profile sized to stay within B2's 1 GB/day free download quota where possible; bigger runs are explicit release-gate work. |
| New public-provider plug-in falsely advertises capabilities | `matrix_selftest.go` does not protect against this, but the capability-bit-probe `testProviderCapabilities` runs in tier 2 against every registered provider and asserts the declared bitmap matches probed behaviour on HeadBucket / PutObject with Object-Lock headers / etc. Divergence fails CI. |

## 9. Open Questions (for review before starting)

1. **MinIO pinning policy** ŌĆö do we pin to the last pre-console-strip
   tag (`RELEASE.2024-11-07T00-52-20Z`) or follow the latest AGPL
   release? Recommendation: pin, re-evaluate quarterly.
2. **Garage as the reference provider** ŌĆö if MinIO drops features we
   depend on, is Garage the new reference? Recommendation: yes,
   with RustFS as the tie-breaker on new-feature support.
3. **External-provider credentials in CI** ŌĆö which org secrets are
   we willing to burn budget on? Recommendation: AWS (US-East-1) +
   Wasabi (free-tier-friendly), skip Hetzner and B2 in CI until
   funding decision.
4. **Conformance-suite parallelism** ŌĆö `t.Parallel()` per provider
   is safe; `t.Parallel()` per subtest within a provider runs many
   bucket operations against a single backend instance
   simultaneously. Recommendation: start serial-per-provider,
   promote to parallel once the suite is stable.
5. **Build tag vs. separate module** ŌĆö should
   `test/conformance/` be a separate Go module so it can use
   Testcontainers without polluting the main `go.sum`?
   Recommendation: separate module under
   `test/conformance/go.mod`; keeps main binary deps minimal.
