## 1. Context & Current State

### 1.1 What the chart supports today

`helm/s3-encryption-gateway/` (chart version 0.5.10) ships with:

| Surface | Location | Notes |
|---|---|---|
| Single `Deployment` | `templates/deployment.yaml:1-321` | `replicaCount` from values; one `app.kubernetes.io/instance` label per release. |
| Single `Service` (ClusterIP) | `templates/service.yaml` | Selects all pods via `selectorLabels`; no colour/track label anywhere. |
| Optional `Ingress` | `templates/ingress.yaml` | Plain `networking.k8s.io/v1` Ingress; single backend; no weighted routing; controller-agnostic (consumed by Traefik, Contour, HAProxy Ingress, and — historically — ingress-nginx). |
| `HorizontalPodAutoscaler` | `templates/hpa.yaml` | Single target; not track-aware. |
| `PodDisruptionBudget` | `templates/poddisruptionbudget.yaml` | Single target; needs per-track PDBs in a blue/green topology. |
| `ServiceMonitor` / `PodMonitor` | Two template files | Track-agnostic; will need track labels for per-track dashboards. |
| `NetworkPolicy` | `templates/networkpolicy.yaml` | Selects via `selectorLabels`; track-agnostic. |
| `serviceaccount.yaml`, `issuer.yaml`, `certificate.yaml` | Templates | Singletons; no per-track duplication needed. |
| Valkey subchart | `Chart.yaml:18-28` | One `valkey.enabled` bool; subchart is deployed per release. |

The chart is built for **single-tenant, single-release** deployments.
Running blue/green today would require two parallel chart releases
(`helm install gw-blue`, `helm install gw-green`) plus an operator-
maintained **out-of-chart** `Service` that does the selector flip.
That path works but is undocumented, has three footguns (shared
Valkey, active-key-version parity, readiness semantics), and is the
first thing a new operator asks "is there an example?" for.

### 1.2 What "blue/green" means for this gateway specifically

Canonical blue/green (Fowler / Humble / Farley) flips **100 % of
traffic** from the old version to the new at a single point in time.
For a stateless HTTP service that is straightforward: two Deployments
sharing one Service, operator edits the `Service.spec.selector` to
flip from `track: blue` to `track: green`. Ibryam & Huss,
*Kubernetes Patterns, 2nd Edition* (O'Reilly, 2023) Ch. 3 "Declarative
Deployment" describe exactly this pattern and it is the pattern the
CNCF `Deployment` primitive was designed for (the `Deployment` object
does **rolling**; blue/green is a layer above).

For the **S3 encryption gateway** that naïve story has three
non-obvious failure modes that this plan must solve before we
publish an example an operator would actually copy:

1. **Active-key-version parity.** Both colours must wrap DEKs
   against the same active `key_version` and must be able to unwrap
   the N most recent versions (`dualReadWindow`). If "green" ships
   with a newer `KeyManager` config that advances the active key
   *before* blue is drained, blue continues to PUT objects wrapped
   against the now-older version — legal on read via the dual-read
   window, but the operator must understand the ordering. The blue/
   green recipe must sequence this: **rotation happens on green
   *after* the flip, never before**. Cross-reference ADR 0007
   "Admin API & Key Rotation" and the footgun note at
   `V0.6-CFG-1-plan.md:523`.
2. **Encrypted-MPU state store sharing.** A multipart upload started
   on blue produces Valkey state keyed `mpu:<sha256(uploadId)>`
   (ADR 0009 §Design). If the traffic flip happens mid-upload and
   the next `UploadPart` lands on green, green **must** see the same
   Valkey state or the upload aborts with `NoSuchUpload`. Therefore
   **blue and green must share a single Valkey instance** — the
   chart's per-release Valkey subchart is *wrong* for this topology.
   The blue/green recipe documents an **external Valkey** pattern
   (both releases set `valkey.enabled: false` and point
   `config.multipartState.valkey.addr` at a shared cluster-scoped
   Valkey). This is the most important correctness item in the whole
   plan.
3. **Readiness + connection-draining semantics.** The `Service`
   selector flip is atomic from the control plane's point of view
   but kube-proxy propagates iptables / IPVS updates asynchronously
   with a typical tail of 5-30 s per node (Burns et al., *Kubernetes
   Best Practices, 2nd Edition* (O'Reilly, 2023), Ch. 4 "Services";
   Luksa, *Kubernetes in Action, 2nd Edition* (Manning, 2026), Ch. 11
   "Services"). In-flight S3 requests on blue (long CopyObject, 5 GiB
   multipart UploadPart, range GET of a 100 MiB object) need
   `terminationGracePeriodSeconds` ≥ the 99th-percentile request
   duration plus a pre-stop `sleep` equal to the kube-proxy
   propagation tail. Without this, the flip *visibly* drops
   connections.

### 1.3 What "canary" means for this gateway specifically

Canary (Humble / Farley, *Continuous Delivery*; Beyer et al., *The
SRE Workbook* Ch. 16 "Canarying Releases") shifts a **fraction** of
traffic from the old version to the new for a soak window, then
either rolls forward (traffic → 100 % new) or rolls back (traffic →
0 % new).

**Ingress controller choice — why Traefik, not NGINX Ingress.**
The ingress-nginx project has been in maintenance-only mode with
declining commit velocity and unresolved CVE backlog; the Kubernetes
SIG-Network path forward is Gateway API, and Traefik is the
actively-maintained L7 router that implements both its own CRDs
(`IngressRoute` / `TraefikService`) *and* Gateway API conformance.
Blessing ingress-nginx as the reference for a v0.6 release we expect
to live for 12+ months is a footgun; we avoid it. This plan ships
Traefik as the primary recipe and Gateway API `HTTPRoute` as the
portable appendix. Operators on other controllers (Contour,
Emissary, HAProxy Ingress, etc.) adapt the Traefik pattern via
Gateway API, which every serious controller now implements.

The three viable mechanisms for this gateway are:

| Mechanism | Where the split happens | Suitability |
|---|---|---|
| **Traefik `IngressRoute` + `TraefikService` (kind: Weighted)** | Traefik controller, L7, native weighted-round-robin across N Kubernetes Services declared inside a single `TraefikService` CRD. Sticky sessions supported via the `sticky` field. | **Recommended default.** Actively maintained, first-class CRD support, zero annotation hacks. Documented in Sharma & Mathur, *Traefik API Gateway for Microservices* (Apress, 2020) Ch. 7 and in Traefik's current docs. Both blue/green (weight 100/0) and canary (weight 95/5) use the same primitive. |
| **Gateway API `HTTPRoute` with `backendRefs` weights** | Kubernetes-standard L7 API; Traefik is a conformant implementation (and so are Envoy Gateway, Cilium, Istio). Weights on `backendRefs`. | **Portable appendix.** Ships as an equivalent recipe for operators committed to Gateway API or moving cross-controller. Slightly less mature across controllers but the future-proof path. Documented in Muschko, *CKA Study Guide, 2nd Edition* (O'Reilly, 2026) and Sayfan, *Mastering Kubernetes, 4th Edition* (Packt, 2023). |
| **Service mesh (Istio / Linkerd) `VirtualService` / `ServiceProfile`** | Sidecar-injected, mTLS-capable, subset-based routing | Overkill for most deployments; filed as follow-up in §8 (`V0.6-OPS-1-followup-mesh`). |
| **Argo Rollouts / Flagger** | Custom controller owns the traffic split, consumes Prometheus metrics for automated promotion | Powerful but introduces a hard dependency on a progressive-delivery operator. Documented as an **optional** overlay, not the primary recipe. |

For canary, the three stateful footguns from §1.2 apply **plus** a
fourth: during canary, any single S3 client session can bounce
between stable and canary for consecutive requests. The gateway's
wrap/unwrap paths are idempotent-by-design (DEK is derived from the
manifest on read, per ADR 0009), so this is safe for **reads**. For
an in-flight **multipart upload** where `UploadPart` requests may
bounce, shared Valkey again makes it work. **This is a hard
requirement**, not an optional hardening — the recipe fails closed
without it. Traefik's `sticky` cookie on the `TraefikService` can
additionally pin a given client to one backend for the duration of
an MPU to reduce Valkey round-trips, but correctness does **not**
depend on it.

## 2. Design Goals & Non-Goals

### Goals

1. Ship **two** copy-paste-runnable topologies that real operators
   can deploy in under 15 minutes:
   - **Blue/green overlay** using a single chart with colour-aware
     values (`values.blue.yaml`, `values.green.yaml`), a shared
     external `Service` and shared external Valkey, and a documented
     `kubectl patch` cutover command.
   - **Canary overlay** using two chart releases behind a single
     Traefik `IngressRoute` + weighted `TraefikService`, shared
     external Valkey, and a documented promote / rollback procedure.
     A Gateway API `HTTPRoute` equivalent ships as a documentation
     appendix for cross-controller portability.
2. **Fail-closed defaults.** The chart's default single-release
   behaviour is unchanged (`track` label defaults to empty;
   `ingress.traefik.enabled` defaults to `false`;
   `ingress.traefik.weighted.enabled` defaults to `false`;
   `valkey.enabled` defaults to `false` which is already the
   current state). Nobody who does `helm upgrade --reuse-values`
   inherits blue/green semantics by surprise.
3. **Correctness first, convenience second.** The stateful
   invariants from §1.2 / §1.3 are encoded as:
   - Helm template-time `required` / `fail` guards that refuse to
     render blue/green or canary values without a shared external
     Valkey address.
   - A precondition checklist in `docs/OPS_DEPLOYMENT.md` that
     operators must tick before running the cutover.
   - Reference `preStop` hooks + `terminationGracePeriodSeconds`
     baked into the blue/green example values.
4. **Operator runbook.** One new file, `docs/OPS_DEPLOYMENT.md`,
   that is the single source of truth. Replaces any ad-hoc
   deployment notes in the chart README (which becomes a pointer).
5. **CI coverage.** `helm template` renders all four scenarios
   (default, blue, green, canary) in the existing `helm-test.yml`
   workflow. A new `scripts/test-progressive-delivery.sh` drives
   the rendering checks locally and in CI.
6. **Zero chart schema divergence with OPS-2.** This plan's new
   values (`track`, `ingress.traefik.*`) are proposed here and
   cross-linked to the OPS-2 schema plan so both land together.

### Non-Goals

- **Service mesh (Istio / Linkerd) integration.** Out of scope for
  v0.6. Filed as *V0.6-OPS-1-followup-mesh* (§8).
- **ingress-nginx recipes.** Explicitly dropped per the reasoning
  in §1.3. The chart's existing `templates/ingress.yaml` (plain
  `networking.k8s.io/v1` Ingress) continues to render — Traefik
  consumes it too — but neither this plan nor the runbook will
  bless ingress-nginx-specific annotations (`canary-weight` and
  friends).
- **Gateway API as a first-class chart template.** The portable
  `HTTPRoute` recipe ships as a *documentation* appendix only in
  v0.6. Promotion to `templates/httproute.yaml` is filed as
  *V0.6-OPS-1-followup-gateway-api* for v0.7.
- **Argo Rollouts / Flagger as the primary recipe.** They are
  documented as an **optional** overlay with a one-page recipe;
  not the default, because the chart must stay dependency-light.
- **Automated metric-driven canary promotion.** The recipe hands
  operators a Prometheus query to run manually (error-rate + p99
  latency on the canary service) and tells them when to promote.
  Automated promotion is Flagger's job and belongs in the optional
  overlay.
- **Multi-region / multi-cluster.** Every example here is
  single-cluster. Multi-region blue/green (flip a DNS record) is
  out of scope; the patterns here compose cleanly if an operator
  puts a global load balancer in front.
- **KMS / HSM adapter changes.** Rotation is consumed as-is (via
  the CFG-1 admin API). We add *sequencing guidance* to the
  runbook; we do not change any crypto code.
- **Helm chart versioning / semver split** (e.g. shipping a
  separate `s3-encryption-gateway-bluegreen` chart). Keep one
  chart; operators select a topology via values.

## 3. Architecture

### 3.1 Topology A — Blue/Green (shared Service, selector flip)

```
                           ┌────────────────────────────┐
                           │  gateway  Service          │
                           │  (out-of-chart, operator-  │
                           │   owned; selector:         │
                           │   track=blue ← flip to     │
                           │   track=green)             │
                           └───────────┬────────────────┘
                                       │
                  ┌────────────────────┴────────────────────┐
                  │                                         │
     ┌────────────▼─────────────┐            ┌──────────────▼───────────┐
     │ Release: gw-blue         │            │ Release: gw-green        │
     │ Helm chart v0.6.x        │            │ Helm chart v0.6.x        │
     │ values.blue.yaml         │            │ values.green.yaml        │
     │ label: track=blue        │            │ label: track=green       │
     │ valkey.enabled=false     │            │ valkey.enabled=false     │
     │ ────────────┬────────── │            │ ────────────┬───────────  │
     └─────────────┼─────────── ┘            └─────────────┼────────────┘
                   │                                       │
                   └─────────────────┬─────────────────────┘
                                     │
                       ┌─────────────▼─────────────┐
                       │ External Valkey           │
                       │ (cluster-scoped, shared)  │
                       │ MPU state store           │
                       └───────────────────────────┘
```

Key contracts:
- Both releases set `track: blue` or `track: green` on the pod
  selector labels via a new chart value.
- Both releases set `service.enabled: false`; the operator owns the
  `Service` outside the chart (its YAML is shipped as an example in
  `docs/examples/bluegreen/service.yaml`).
- Both releases set `valkey.enabled: false` and
  `config.multipartState.valkey.addr` to a shared external Valkey.
- Both releases MUST use the same `ENCRYPTION_PASSWORD` / same
  active KMS `key_version` so objects wrapped by one can be
  unwrapped by the other. The runbook makes this explicit and
  provides a verification script.
- Each release has its own HPA, PDB, ServiceMonitor/PodMonitor.

Cutover command (documented in runbook):
```bash
kubectl patch service gateway -n <ns> \
    -p '{"spec":{"selector":{"app.kubernetes.io/name":"s3-encryption-gateway","track":"green"}}}'
```

Rollback is the same command with `track: blue`. Under 1 s at the
API-server; 5-30 s to propagate to kube-proxy tables on worker
nodes.

### 3.2 Topology B — Canary (Traefik IngressRoute + weighted TraefikService)

```
                ┌────────────────────────────────────────┐
                │ IngressRoute (Traefik CRD)             │
                │ host: s3.example.com                   │
                │ routes:                                │
                │   match: Host(`s3.example.com`)        │
                │   services:                            │
                │     - name: gateway-weighted           │
                │       kind: TraefikService             │
                └────────────────┬───────────────────────┘
                                 │
                ┌────────────────▼───────────────────────┐
                │ TraefikService (kind: Weighted)        │
                │  weighted:                             │
                │   services:                            │
                │    - name: gw-stable   weight: 95      │
                │    - name: gw-canary   weight:  5      │
                │   sticky:                              │
                │    cookie: { name: gw-pin, httpOnly }  │
                └────────┬────────────────────┬──────────┘
                         │                    │
             ┌───────────▼────────┐   ┌───────▼───────────┐
             │ Release: gw-stable │   │ Release: gw-canary│
             │ chart v0.5.x       │   │ chart v0.6.x-rc   │
             │ Service: gw-stable │   │ Service: gw-canary│
             │ ingress.enabled=   │   │ ingress.enabled=  │
             │   false            │   │   false           │
             │ service.enabled=   │   │ service.enabled=  │
             │   true             │   │   true            │
             └─────────┬──────────┘   └─────────┬─────────┘
                       │                        │
                       └───────────┬────────────┘
                                   │
                    ┌──────────────▼──────────────┐
                    │ External Valkey             │
                    │ (shared, cluster-scoped)    │
                    └─────────────────────────────┘
```

Key contracts:

- Two chart releases, each owning its own `Service`
  (`gw-stable`, `gw-canary`). Neither owns the `IngressRoute` —
  the operator owns a single `IngressRoute` + `TraefikService`
  pair outside the chart, shipped as raw manifests under
  `docs/examples/canary/`.
- The `TraefikService` of kind `Weighted` references both
  Services with explicit integer weights. Traefik does native
  weighted-round-robin at the L7 layer with no annotation magic.
- `sticky.cookie` (optional, **recommended for this gateway**)
  pins a given S3 client session to one backend. S3 SDKs keep
  TCP connections alive for minutes, so the cookie effectively
  pins an in-flight multipart upload to one colour without
  depending on Valkey for cross-replica state reads. Correctness
  still requires shared Valkey, but stickiness reduces state-store
  round-trips and softens the canary UX.
- Shared external Valkey, same as Topology A.
- Same active-key-version / dual-read-window invariant as
  Topology A.
- Promote = `kubectl patch traefikservice gateway-weighted -p
  '{"spec":{"weighted":{"services":[...weight:100...,
  ...weight:0...]}}}'`. Progressive: 5 → 25 → 50 → 100.
- Rollback = weight → 0 on the canary service; `helm uninstall
  gw-canary` once state has drained.

The chart contribution here is:
1. New template `templates/ingressroute.yaml`, gated by
   `ingress.traefik.enabled`, that renders a **single-backend**
   IngressRoute pointing at the release's own Service. This is the
   drop-in replacement for the standard Ingress for operators who
   are already on Traefik and want per-release routing (not a
   weighted split). Used for the "single release on Traefik" path.
2. New template `templates/traefikservice.yaml`, gated by
   `ingress.traefik.weighted.enabled`, that renders a
   `kind: Weighted` TraefikService referencing both a stable and
   canary backend by Service name. Only **one** release in a
   canary topology owns this (by convention, the operator deploys
   it alongside the stable release).
3. Raw manifests under `docs/examples/canary/` for operators who
   want the canary topology without the chart owning the weighted
   service — this is the operator-owned pattern that mirrors the
   blue/green shared-Service pattern.

### 3.3 Topology C — Canary via Gateway API (portable appendix)

For operators standardised on Gateway API, an equivalent recipe
under `docs/examples/gateway-api/` ships a single `HTTPRoute`:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: gateway
spec:
  parentRefs:
    - name: public-gateway
  hostnames: ["s3.example.com"]
  rules:
    - backendRefs:
        - name: gw-stable
          port: 80
          weight: 95
        - name: gw-canary
          port: 80
          weight: 5
```

Same stateful invariants apply (shared Valkey, same active key).
This is a documentation appendix only; the chart does not template
`HTTPRoute` in v0.6 because Gateway API `HTTPRoute` v1 reached GA
late in 2024 and controller-coverage of the weighted-backend
behaviour is still uneven. Filed as follow-up
`V0.6-OPS-1-followup-gateway-api` for v0.7 promotion to a first-class
chart template once the cross-controller story stabilises.

### 3.4 Stateful invariants (reproduced for reviewer convenience)

| Invariant | Enforcement |
|---|---|
| Both colours/tracks wrap DEKs against the same active key version | Runbook precondition; verification script `scripts/verify-key-parity.sh` (renders each release's configmap, diffs the `KEY_MANAGER_PROVIDER` / `COSMIAN_KMS_KEYS` envs) |
| MPU state is shared across colours/tracks | Chart template-time guard: if `track` is set AND `valkey.enabled: true`, fail rendering with a clear message pointing to the external-Valkey pattern |

### 4.2 Selector label wiring

```yaml
{{- define "s3-encryption-gateway.trackLabel" -}}
{{- if .Values.track -}}
track: {{ .Values.track | quote }}
{{- end -}}
{{- end }}
```

And `selectorLabels` is extended **non-breakingly** — the `track`

single-release deployments see no change in their selectors.

### 4.3 Template-time guards

In `templates/deployment.yaml` (or a new `templates/_validate.tpl`
helper), a `fail` guard:

```yaml
{{- if and .Values.track .Values.valkey.enabled -}}
{{- fail (printf "When `track=%q` is set (blue/green or canary topology), `valkey.enabled` must be `false` and `config.multipartState.valkey.addr` must point to a shared external Valkey. See docs/OPS_DEPLOYMENT.md §Stateful Invariants." .Values.track) -}}
{{- end -}}
```

A second guard validates that if `track` is set, either
`config.multipartState.valkey.addr.value` or
`config.multipartState.valkey.addr.valueFrom` is non-empty. **Unless**
the operator has explicitly disabled encrypted MPU
(`encryptMultipartUploads=false` on all policies), which is a future
value that OPS-2 / policy templates may expose.

A third guard enforces mutual exclusivity between the two ingress
paths:

```yaml
{{- if and .Values.ingress.enabled .Values.ingress.traefik.enabled -}}
{{- fail "Cannot enable both `ingress.enabled` (standard Ingress) and `ingress.traefik.enabled` (Traefik IngressRoute) on the same release. Choose one. See docs/OPS_DEPLOYMENT.md §3." -}}
{{- end -}}
```

A fourth guard enforces that `ingress.traefik.weighted.enabled`
only turns on when `ingress.traefik.enabled` is also true, and
that the `services` list has at least two entries with weights
summing to 100 (± 0):

```yaml
{{- if .Values.ingress.traefik.weighted.enabled -}}
{{- if not .Values.ingress.traefik.enabled -}}
{{- fail "`ingress.traefik.weighted.enabled` requires `ingress.traefik.enabled: true`." -}}
{{- end -}}
{{- $sum := 0 -}}
{{- range .Values.ingress.traefik.weighted.services -}}
{{- $sum = add $sum (int .weight) -}}
{{- end -}}
{{- if ne $sum 100 -}}
{{- fail (printf "ingress.traefik.weighted.services weights must sum to 100; got %d" $sum) -}}
{{- end -}}
{{- end -}}
```

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Operators naïvely copy the blue/green example but forget to set up external Valkey, causing MPU aborts on cutover. | Template-time `fail` guard refuses to render; runbook precondition checklist; verification script. |
| Active-key-version drift between colours causes 500s on green reads. | Runbook explicit sequencing ("rotate on green *after* the flip"); verification script diffs the two rendered configmaps. |
| Traefik CRD API-group migration (`traefik.containo.us` → `traefik.io`) on older clusters causes `IngressRoute` rendering to be rejected. | Chart emits `traefik.io/v1alpha1` (v3+). Runbook lists Traefik ≥ v3.0 as a precondition. Pre-v3 operators either upgrade Traefik first or stay on the standard Ingress path. |
| `TraefikService` weighted routing silently accepts weights that don't sum to 100 (Traefik normalises). | Template-time guard rejects non-100 weight sums at render time (§4.3 fourth guard). |
| `ServiceMonitor` per-track relabel rule breaks existing dashboards if operators upgrade without noticing. | Chart value defaults to empty `track`, so behaviour is unchanged for single-release deployments. Release notes call out the new label only for blue/green users. |
| The example Service manifest under `docs/examples/bluegreen/` drifts from the chart's selectorLabels. | Golden-file test compares the chart's rendered selector against the raw manifest's selector; fails if they diverge. |
| Operators run blue/green with `replicaCount: 1` and hit unavailability during the kube-proxy propagation tail. | Runbook recommends `replicaCount ≥ 2` per colour, PDB with `minAvailable: 1`, and an explicit analysis window between readiness and cutover. |
| FIPS build mismatch between colours (one on `-tags=fips`, the other not) causes decrypt failures. | Runbook precondition; verification script asserts both releases' images match on the FIPS tag. |
| Helm schema (OPS-2) lands before OPS-1 and rejects our new values. | OPS-2 plan is pre-coordinated — schema additions for `track`, `ingress.traefik.*`, `lifecycle`, `terminationGracePeriodSeconds` are listed as OPS-2 Phase-D inputs in `docs/plans/V0.6-OPS-2-plan.md` (to be written when OPS-2 starts). |
| Operator on ingress-nginx expects the chart to still generate canary annotations. | Explicitly not supported. Migration note (runbook §12) directs them to Traefik or the Gateway API appendix. The chart's standard `ingress.enabled` path remains unchanged — ingress-nginx still *consumes* the plain Ingress, we just don't bless progressive-delivery annotations for it. |

## Appendix A — Worked render examples

### A.1 `values-blue.yaml` (excerpt)

```yaml
# helm/s3-encryption-gateway/examples/values-blue.yaml
#
# Blue side of a blue/green deployment.
# See docs/OPS_DEPLOYMENT.md §3 for the full procedure.
#
# PRECONDITIONS (enforced by template guards + runbook checklist):
#   1. Shared external Valkey cluster reachable at VALKEY_ADDR.
#   2. Same ENCRYPTION_PASSWORD / same active KMS key_version as
#      the green release.
#   3. The shared Service (docs/examples/bluegreen/service.yaml)
#      is deployed separately and points at track=blue by default.

replicaCount: 3

# Label this release's pods with track=blue so the shared Service
# can select them.
track: blue

# The chart must NOT own the Service — the operator owns it so
# the selector can be patched to flip traffic.
service:
  enabled: false

# Shared external Valkey (disable the subchart, set the addr).
valkey:
  enabled: false

config:
  # ... backend, encryption, policies as usual ...
  multipartState:
    valkey:
      addr:
        value: "valkey-shared.mpu-state.svc.cluster.local:6379"

# Pod lifecycle — critical for safe cutover.
terminationGracePeriodSeconds: 60
lifecycle:
  preStop:
    exec:
      command: ["sh", "-c", "sleep 10"]

# Separate HPA / PDB / NetworkPolicy / ServiceMonitor per colour.
podDisruptionBudget:
  enabled: true
  minAvailable: 2

serviceMonitor:
  enabled: true
  labels:
    track: blue
```

### A.2c `promote.sh` (weight-shift helper)

```bash
# docs/examples/canary/promote.sh
#!/usr/bin/env bash
set -euo pipefail

STEP="${1:-}"
NS="${NAMESPACE:-s3-gateway}"
TS="${TRAEFIKSERVICE:-gateway-weighted}"

case "$STEP" in
  --step) CANARY_WEIGHT="$2" ;;
  --rollback) CANARY_WEIGHT=0 ;;
  5|25|50|100) CANARY_WEIGHT="$STEP" ;;
  *) echo "Usage: $0 {5|25|50|100|--rollback|--step N}" >&2; exit 2 ;;
esac
STABLE_WEIGHT=$((100 - CANARY_WEIGHT))

kubectl -n "$NS" patch traefikservice "$TS" --type=merge \
  -p "{\"spec\":{\"weighted\":{\"services\":[
        {\"name\":\"gw-stable\",\"port\":80,\"weight\":${STABLE_WEIGHT}},
        {\"name\":\"gw-canary\",\"port\":80,\"weight\":${CANARY_WEIGHT}}
      ]}}}"

echo "Canary weight → ${CANARY_WEIGHT}%"
```

### A.4 Cutover command (documented in runbook and `cutover.sh`)

```bash
# docs/examples/bluegreen/cutover.sh
#!/usr/bin/env bash
set -euo pipefail

NEW_TRACK="${1:-green}"  # pass "blue" to roll back
NAMESPACE="${NAMESPACE:-s3-gateway}"
SERVICE_NAME="${SERVICE_NAME:-gateway}"

# Precondition: both tracks are READY.
kubectl -n "$NAMESPACE" wait --for=condition=available \
    deployment/gw-"$NEW_TRACK" --timeout=120s

# Flip the selector.
kubectl -n "$NAMESPACE" patch service "$SERVICE_NAME" \
    --type=strategic \
    -p "{\"spec\":{\"selector\":{\"app.kubernetes.io/name\":\"s3-encryption-gateway\",\"track\":\"${NEW_TRACK}\"}}}"

echo "Service $SERVICE_NAME now points at track=$NEW_TRACK."
echo "Observe error rate and latency for the analysis window."
echo "Rollback: $0 $( [ "$NEW_TRACK" = green ] && echo blue || echo green )"
```
