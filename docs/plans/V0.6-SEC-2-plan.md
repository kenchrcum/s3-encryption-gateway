# V0.6-SEC-2 — FIPS-Compliant Crypto Profile (Optional) — Implementation Plan

Status: Draft
Owner: Crypto
Priority: P2
Labels: `milestone:v0.6`, `area:kms`, `area:security`, `area:build`

> This plan defines how `s3-encryption-gateway` will offer an **optional**,
> opt-in build profile that restricts cryptography to FIPS 140-3 approved
> algorithms and uses a FIPS-validated module under the hood. It is the direct
> follow-up to V0.6-SEC-1 (Pluggable KeyManager) and mirrors the existing
> `hsm` / `!hsm` build-tag split used in `internal/crypto/keymanager_hsm*.go`.

---

## 1. Context & Current State

### 1.1 What the gateway uses today

Project-wide crypto inventory (see exploration report):

| Primitive                    | Package                                        | FIPS 140-3 status                                     |
| ---------------------------- | ---------------------------------------------- | ----------------------------------------------------- |
| AES-256-GCM (AEAD, default)  | `crypto/aes` + `crypto/cipher` (stdlib)        | **Approved** (SP 800-38D)                             |
| ChaCha20-Poly1305 (AEAD)     | `golang.org/x/crypto/chacha20poly1305`         | **Not approved** (no SP/FIPS validation)              |
| PBKDF2-HMAC-SHA256 (KDF)     | `golang.org/x/crypto/pbkdf2` (to be migrated)  | Algorithm approved (SP 800-132); package not in module boundary |
| HMAC-SHA256 (request auth)   | `crypto/hmac`, `crypto/sha256`                 | **Approved**                                          |
| AES key-wrap (memory KM)     | `crypto/aes` (AES-KW via `cipher`)             | **Approved** (SP 800-38F)                             |
| SHA-256                      | `crypto/sha256`                                | **Approved**                                          |
| MD5 (S3 ETag parity only)    | `crypto/md5`                                   | Not a security primitive — **allowed** under FIPS policy carve-out |
| TLS                          | `crypto/tls` + `crypto/x509`                   | Approved cipher suites only when module is in FIPS mode |
| `crypto/rand`                | stdlib                                         | Approved DRBG when module is in FIPS mode             |

"Designing to FIPS 140: A Guide for Engineers and Programmers" (Johnston & Fant,
Apress 2024) stresses that FIPS compliance is a property of **the module, not
the algorithm**: a program that calls AES-256-GCM is only FIPS-compliant if the
AES implementation resides inside a **validated cryptographic module** whose
boundary is well-defined and whose power-on self-tests (KATs) pass before any
approved service is used [1].

### 1.2 Constraints that bound the design

1. **Go 1.25+ ships a native FIPS 140-3 module** (`GOFIPS140`). It is
   pure-Go, works with `CGO_ENABLED=0`, and — unlike `GOEXPERIMENT=boringcrypto`
   — does not require glibc. It is also undergoing CMVP validation. This is
   the right base for the gateway (go.mod already targets `go 1.25.0`).
2. When FIPS mode is active, `crypto/tls`, `crypto/rand`, `crypto/aes`,
   `crypto/cipher`, `crypto/sha256`, `crypto/hmac` are served from the module;
   **non-approved algorithms invoked through stdlib panic or return errors**.
3. **ChaCha20-Poly1305 is not on the FIPS-approved list** and must be
   unavailable in a FIPS build. Serious Cryptography, 2nd ed. (Aumasson, No
   Starch 2024) is explicit that ChaCha20-Poly1305, despite being a strong
   modern AEAD, is **not FIPS-approved** and therefore cannot be used in a
   regulated deployment even when otherwise preferable [2].
4. `golang.org/x/crypto/pbkdf2` is outside the FIPS module boundary. Go 1.24
   introduced a **stdlib** `crypto/pbkdf2` with a redesigned API (generic,
   error-returning, different argument order) that is inside the FIPS module
   boundary when FIPS mode is active. Because we already target `go 1.25.0`,
   we migrate **all** callers to `crypto/pbkdf2` unconditionally — the
   package ships with every supported Go toolchain and is the Go team's
   recommended replacement. This eliminates one external crypto dependency
   and the need for a build-tag shim.
5. FIPS 140-3 §7.10 mandates **power-on self-tests** (KATs) and a
   **conditional self-test** for the DRBG. The Go FIPS module runs these
   automatically at `init()` when the program is launched with
   `GOFIPS140=on`; the gateway must nevertheless **assert FIPS mode on
   startup** and fail closed if the environment says "fips" but the runtime
   reports otherwise.
6. MD5 is used exclusively for S3 ETag compatibility (`engine.go:8`). FIPS
   permits non-approved hashes for non-security uses (e.g. integrity tags not
   used for security decisions). We must document the carve-out.

### 1.3 What this plan is NOT

- Not a CMVP submission. We rely on Go's upstream module validation and
  document which algorithms the gateway exercises.
- Not a FedRAMP/DoD accreditation. Those are deployment concerns.
- Not a re-implementation of cryptography. All primitives remain stdlib.

## 2. Design Goals & Non-Goals

### Goals
1. A **`fips` build tag** that, when set, produces a binary which:
   - Refuses to register ChaCha20-Poly1305 in `algorithms.go`.
   - Verifies at startup that `crypto/fips140.Enabled()` returns `true`; exits
     non-zero otherwise with a clear log line.
   - Emits a `gateway_fips_mode` Prometheus gauge and a structured log banner.
1a. An **unconditional migration** from `golang.org/x/crypto/pbkdf2` to the
   stdlib `crypto/pbkdf2` (Go 1.24+). This is independent of the FIPS tag but
   is a prerequisite: it puts the KDF inside the FIPS module boundary when
   the binary runs in FIPS mode, and removes one external dependency from
   every build.
2. A **default (non-FIPS) build** that behaves exactly as today — zero
   behavioural drift for users who do not opt in.
3. A **second Dockerfile** (`Dockerfile.fips`) producing an image tagged
   `ghcr.io/.../s3-encryption-gateway:<ver>-fips` built with
   `-tags=fips` and `GOFIPS140=v1.0.0` baked into the runtime env.
4. **Documentation** (`docs/FIPS.md`) enumerating: approved algorithms,
   build instructions, runtime environment (`GOFIPS140`, `GODEBUG=fips140=on`),
   operational guarantees, limitations (no ChaCha20, no compress+encrypt with
   non-approved hashes), and evidence for auditors (algorithm inventory + CMVP
   certificate reference).
5. **Conformance test** that runs the existing `algorithms_test.go` under
   both build tags and asserts:
   - With `-tags=fips`: registering ChaCha20-Poly1305 returns
     `ErrAlgorithmNotApproved`; registering AES-256-GCM succeeds; a
     round-trip encrypt/decrypt works; `crypto/fips140.Enabled()` is true.
   - Without `-tags=fips`: both algorithms register; behaviour unchanged.

### Non-Goals
- Replacing AES-256-GCM with AES-256-GCM-SIV or similar nonce-misuse
  resistant schemes (out of scope — would change on-disk envelope).
- Adding new key derivation algorithms (Argon2id, scrypt) — both are
  unavailable under FIPS.
- Swapping the KeyManager adapter set. The `memory`, `cosmian`, and `hsm`
  adapters continue to work; FIPS only constrains the **in-process** crypto.
- TLS hardening beyond what Go's FIPS mode enforces (separate v0.7 work).

## 3. Architecture & File Layout

### 3.1 Build-tag split (mirrors the existing `hsm` pattern)

```
internal/crypto/
├── algorithms.go              # unchanged: public API + registry
├── algorithms_default.go      # //go:build !fips   — registers AES-GCM + ChaCha20
├── algorithms_fips.go         # //go:build  fips   — registers AES-GCM only; ChaCha20 returns ErrAlgorithmNotApproved
├── fips_profile.go            # //go:build  fips   — startup assertion: panics if !fips140.Enabled()
└── fips_profile_stub.go       # //go:build !fips   — no-op; exports FIPSEnabled() = false
```

`engine.go` is migrated to import `crypto/pbkdf2` directly in **all** builds —
no build-tag shim is needed because the stdlib package is available in every
supported Go toolchain (we already target `go 1.25.0`). The build-tag split
is reserved strictly for primitives that genuinely differ between the two
profiles (algorithm registry, startup assertion).

### 3.2 New public surface

```go
// internal/crypto/fips_profile.go   (build tag: fips)
package crypto

import "crypto/fips140"

// FIPSEnabled reports whether the binary was built and is running with the
// FIPS 140-3 profile active. It is always false in non-fips builds.
func FIPSEnabled() bool { return fips140.Enabled() }

// AssertFIPS returns an error if the binary was built with -tags=fips but
// the runtime FIPS module is not active. Called from main.go at startup.
func AssertFIPS() error { /* ... */ }
```

```go
// internal/crypto/algorithms.go  (shared)
// New sentinel, mentioned in errors returned by createAEADCipher in fips builds.
var ErrAlgorithmNotApproved = errors.New("crypto: algorithm not FIPS-approved in this build")
```

### 3.3 Startup wiring (`cmd/server/main.go`)

Right after `logger.Info("Starting S3 Encryption Gateway")` (line ~314):

```go
if err := crypto.AssertFIPS(); err != nil {
    logger.Error("FIPS profile assertion failed", "error", err)
    os.Exit(1)
}
logger.Info("crypto profile", "fips", crypto.FIPSEnabled())
metrics.SetFIPSMode(crypto.FIPSEnabled())   // new gauge
```

`AssertFIPS` succeeds trivially in non-fips builds and fails loudly if the
operator set `GOFIPS140=off` on a fips-tagged binary.

### 3.4 Algorithm registry changes

`algorithms.go` currently exposes `DefaultAlgorithmConfig()` returning an
entry for each AEAD. Under `-tags=fips`:

- `ChaCha20Poly1305` entry is omitted from the default config.
- `createAEADCipher("ChaCha20-Poly1305", …)` returns `ErrAlgorithmNotApproved`.
- A new flag on `AlgorithmEntry` (`FIPSApproved bool`) lets the registry
  filter out non-approved algorithms even if user configuration requests them.

This is the defence-in-depth recommended by Johnston & Fant: reject
non-approved primitives at the API boundary, not only via build exclusion,
so that misconfiguration cannot silently fall back [1].

## 4. Implementation Steps

Numbered so they map cleanly onto commits/PRs.

1. **Migrate PBKDF2 to stdlib (prerequisite, no build tag).** Replace the
   import at `engine.go:22` with `crypto/pbkdf2` and rewrite both call sites
   (`engine.go:222`, `engine.go:686`) for the new API:
   - argument order: `Key(h, password, salt, iter, keyLen)` (hash first)
   - `password` is now `string` (drop the `[]byte(...)` conversion on
     `e.password`; use `string(altPass)` at the second site)
   - returns `(key []byte, err error)` — handle the error. Because
     `iter=100000` and `keyLen` are compile-time constants, the error path
     is statically unreachable; return a wrapped error anyway so a future
     refactor cannot introduce a silent failure.
   Remove `golang.org/x/crypto/pbkdf2` from `go.mod` if no other consumer
   remains (ChaCha20-Poly1305 still pulls in `golang.org/x/crypto`, so the
   module stays; only the `pbkdf2` sub-package goes). Add a KAT regression
   test that verifies the migrated code produces byte-identical output to
   a pre-recorded vector (protects against future drift).
2. **Introduce `FIPSApproved` metadata** on `AlgorithmEntry` in
   `algorithms.go`. No behavioural change yet.
3. **Split algorithm registration** into `algorithms_default.go`
   (`!fips`) and `algorithms_fips.go` (`fips`). The fips variant omits
   ChaCha20-Poly1305 and overrides `createAEADCipher` to reject it.
4. **Add `ErrAlgorithmNotApproved`** to `algorithms.go` (shared, both tags).
5. **Create `fips_profile.go` / `fips_profile_stub.go`** with `FIPSEnabled`
   and `AssertFIPS`. The fips variant wraps `crypto/fips140.Enabled()`.
6. **Wire startup** in `cmd/server/main.go` (call `AssertFIPS`, log banner,
   metric).
7. **Add Prometheus gauge** `gateway_fips_mode` (0/1) in
   `internal/metrics/metrics.go` with a `SetFIPSMode(bool)` helper.
8. **Add `Dockerfile.fips`**:
   - Base: `golang:1.25` (Debian, *not* Alpine — musl lacks some stdlib
     behaviour the module expects; matches Go upstream's guidance).
   - Build: `GOFIPS140=v1.0.0 CGO_ENABLED=0 go build -tags=fips -ldflags=... ./cmd/server`.
   - Runtime: `gcr.io/distroless/static-debian12:nonroot`.
   - Env: `GOFIPS140=v1.0.0`, `GODEBUG=fips140=on`.
9. **CI matrix**: add a `fips` job that runs `go build -tags=fips ./...`,
   `go vet -tags=fips ./...`, and the algorithm registry tests with
   `GOFIPS140=v1.0.0` set.
10. **Unit & conformance tests** (see §5).
11. **Docs**: `docs/FIPS.md` (audience: operators & auditors). Cross-link
    from `README.md`, `docs/ROADMAP.md`, and `docs/issues/v0.6-issues.md`.
12. **Helm**: add `values.fips.yaml` overlay that pins `image.repository`
    suffix `-fips` and sets `env.GOFIPS140=v1.0.0`. No chart template changes
    beyond env passthrough.

Each step is independently reviewable; steps 1-7 touch Go code, 8-12 are
infra/docs. Step 1 (PBKDF2 migration) is **independent of the FIPS tag** and
can land as its own PR ahead of the others.

## 5. Tests

### 5.1 PBKDF2 migration (both builds)

- `internal/crypto/pbkdf2_kat_test.go` (no build tag): deterministic
  known-answer test using a fixed password / salt / iter / keyLen; asserts
  byte-identical output to a hard-coded expected vector. Guards against any
  future accidental algorithm drift when the stdlib package evolves.
- Existing engine round-trip tests must pass unchanged under both build tags.

### 5.2 Unit (tag-gated)

- `internal/crypto/algorithms_fips_test.go` (`//go:build fips`):
  - `TestChaCha20Rejected`: `createAEADCipher("ChaCha20-Poly1305", key)` →
    `ErrAlgorithmNotApproved`.
  - `TestAESGCMApproved`: round-trip encrypt/decrypt succeeds.
  - `TestFIPSRuntimeEnabled`: `crypto/fips140.Enabled()` is true (depends on
    `GOFIPS140` env at test time; the CI job sets it).

- `internal/crypto/algorithms_default_test.go` (`//go:build !fips`):
  - `TestChaCha20Registered`: both AEADs register; behaviour unchanged.
  - `TestFIPSDisabled`: `FIPSEnabled()` returns false.

### 5.3 Startup assertion

- `cmd/server/main_fips_test.go` (`//go:build fips`): exec the binary with
  `GOFIPS140=off` and assert exit code 1 plus the expected log line.

### 5.4 Integration

- Extend the existing `KeyManager` `ConformanceSuite` to run under `-tags=fips`
  against the in-memory adapter (AES-KW only); this proves the wrap/unwrap
  path has no incidental non-approved primitives.
- Large-object streaming test (`test/integration_test.go`) runs in both
  matrix cells.

### 5.5 Mutation / race

- `go test -race -tags=fips ./internal/crypto/...` in CI.

## 7. Risks & Mitigations

| Risk | Mitigation |
| ---- | ---------- |
| Go FIPS module CMVP status changes between releases | Pin `GOFIPS140=v1.0.0` explicitly; revisit annually. |
| Operators who depended on ChaCha20-Poly1305 adopt FIPS image by mistake | Startup log + config-validator error make it impossible to run silently. |
| Alpine base incompatibility | FIPS image uses Debian/distroless; default image stays on Alpine. |
| `golang.org/x/crypto/pbkdf2` and `crypto/pbkdf2` have different APIs (argument order, `string` password, error return, generic signature) | Migrate both call sites in a dedicated PR; KAT regression test verifies byte-identical output against a pre-recorded vector; call sites explicitly handle the new error return even though it is statically unreachable. |
| Build-tag drift between local dev and CI | Makefile target `make build-fips` and `make test-fips` used in CI verbatim. |
