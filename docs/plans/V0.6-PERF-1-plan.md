## 1. Context & Current State

### 1.1 What the hot paths currently do

Every audited path materialises one or more full-object byte slices in
heap before forwarding them to the next stage. The concrete sites and
sizes are:

| File:line | Handler | Current behaviour | Worst-case heap per request |
|---|---|---|---|
| `internal/api/handlers.go:1111` | `handleGetObject` (non-optimised ranged) | `io.ReadAll(decryptedReader)` into `decryptedData` then range-slice | 1× plaintext |
| `internal/api/handlers.go:1198` | `handleGetObject` (optimised range) | `io.ReadAll(decryptedReader)` into `outputData` before `w.Write` | 1× range payload (small, but avoidable) |
| `internal/api/handlers.go:2672` | `handleUploadPart` (encrypted MPU) | `io.ReadAll(encReader)` to satisfy SDK seekable-body contract | 1× ciphertext part (≤ 5 GiB) |
| `internal/api/handlers.go:2718` | `handleUploadPart` (plaintext MPU) | `io.ReadAll(r.Body)` | 1× part body |
| `internal/api/handlers.go:3783` | `handleCopyObject` | `io.ReadAll(decryptedReader)` → `decryptedData` | 1× plaintext |
| `internal/api/handlers.go:3828` | `handleCopyObject` | `dstEngine.Encrypt(bytes.NewReader(decryptedData), ...)` + `io.ReadAll(encryptedReader)` → `encryptedData` | +1× ciphertext (≈ plaintext + 2 KiB/chunk tags) |
| `internal/api/upload_part_copy.go:557,587` | `uploadPartCopyLegacy` | `io.ReadAll(LimitReader(…, cap+1))` | 1× cap (256 MiB default) — **intentional** (single AEAD cannot stream) |
| `internal/api/upload_part_copy.go:730,771,790,816` | `uploadPartCopyReencryptMPU` | Three-class: `ReadAll` source plaintext → `bytes.NewReader` → `encryptMPUPart` → `ReadAll` ciphertext | 2× part payload |
| `internal/crypto/mpu_encrypter.go:39` | `NewMPUPartEncryptReader` | `io.ReadAll(body)` into `plaintext`, then loop emits chunks into a `bytes.Buffer` | 1× plaintext + 1× ciphertext (same buffer) |
| `internal/crypto/compression.go:128,195` | `Compress`/`Decompress` (legacy non-chunked only) | `io.ReadAll` then gzip Write/Read in a `bytes.Buffer` | 1× source + 1× (de)compressed |
| `internal/crypto/engine.go:335` | `engine.Encrypt` legacy (non-chunked) branch | `io.ReadAll(reader)` into `plaintext` — used for ETag + compression decisions | 1× plaintext |

Chunked encryption (`internal/crypto/chunked.go`) already streams in
both directions with a pooled 64 KiB chunk buffer and is the reference
implementation to mimic. `mpuDecryptReader` (`mpu_encrypter.go:99-201`)
also streams correctly and is the model for the new MPU encrypt reader.

### 1.2 Why this has not been fixed yet

Three reasons, each of which this plan addresses:

1. **AWS SDK seekable-body requirement for `UploadPart`.** SDK V2's
   HTTP signer re-reads the body for SigV4 payload hashing and for
   retries. The naive fix (`r.Body` passed directly) breaks SigV4
   when the backend uses plaintext-HTTP (Ceph RGW, Hetzner, MinIO in
   some modes). Current code works around this by buffering. A
   proper fix requires either (a) a pooled seekable wrapper bounded
   by `MaxPartBuffer`, or (b) SDK retry-loop short-circuiting via
   `aws.IsReaderSeekable(false)` + `ContentLength` set. See §4.4.

2. **ETag coupling in legacy engine encrypt.** `engine.go:350`
   computes `originalETag = computeETag(plaintext)` for the
   pre-compression plaintext. A streaming rewrite must thread an MD5
   hasher through the pipe so the ETag is computed on the fly. See
   §4.6.

3. **Deterministic per-chunk IV derivation in MPU encrypt.**
   `DeriveMultipartIV` (`mpu_iv.go:40`) is cheap (HKDF-Expand over
   fixed-size info) and depends only on `(DEK, uploadIDHash,
   ivPrefix, partNumber, chunkIndex)` — nothing about the plaintext.
   So the encrypt reader can emit chunks as plaintext arrives;
   retries yield byte-identical ciphertext because IVs are
   deterministic. The only subtlety is the final chunk, which may be
   short — the streaming reader must flush any remainder at source
   EOF. See §4.3.

### 1.3 Known non-goals / intentionally buffered paths

- `uploadPartCopyLegacy` (`upload_part_copy.go:557`). Legacy format
  is a single AEAD envelope; authenticity requires the entire
  ciphertext before any plaintext may be emitted (this is RFC 5116
  "atomic authenticated encryption", reinforced in *Serious
  Cryptography, 2nd Ed.* §8.4 and *API Security in Action* §6.4.2 on
  the AEAD commit-before-release rule). We document this explicitly
  and leave the ≤ 256 MiB cap in place.
- `engine.go` legacy (non-chunked) Encrypt/Decrypt. Same argument.
  Streaming is only meaningful in chunked mode, which is already
  streaming. The fix here is narrower: eliminate the
  `bytes.NewReader(compressedData)` double-buffer by piping the
  gzip/zstd stream directly into the AEAD writer boundary (which
  still finalises atomically for legacy, but without the
  double-allocation).

### 1.4 Prior art inside the codebase

- `internal/crypto/chunked.go` — streaming chunked encrypt/decrypt with
  `BufferPool` integration. Reference for the new MPU streaming
  encrypt reader.
- `internal/crypto/mpu_encrypter.go:99` — `mpuDecryptReader`. Already
  streams, proves the pattern works end-to-end.
- `internal/crypto/buffer_pool.go` — pooled byte slices at four size
  classes including `Get64K` (the natural `io.CopyBuffer` buffer).
  Already instrumented with `RecordBufferPoolHit`/`Miss` metrics.
- `encBufPool` (`mpu_encrypter.go:124`) — `sync.Pool` of 65 552-byte
  slices. This is the model for the new `ciphertextChunkPool` used
  by the streaming MPU encrypt reader.

### 1.5 External reference material

Cross-referenced during context gathering:

- *Efficient Go* (Plotka, O'Reilly 2022) — chapters on allocation
  profiling (`-benchmem`, `benchstat`), `sync.Pool` usage patterns,
  and `io.CopyBuffer` semantics. Informs Phase A tooling.
- *Learning Go, 2nd Ed.* (Bodner, 2024) ch. "io.Reader and io.Writer"

- *Serious Cryptography, 2nd Ed.* (Aumasson, 2024) ch. 8 — AEAD

## 2. Goals & Non-Goals

### Goals

1. **Bounded heap per request** on GET, PUT-via-CopyObject,
   UploadPart (encrypted and plaintext MPU), and UploadPartCopy
   with a chunked source. Peak in-flight buffered bytes per request
   ≤ `2 × crypto.DefaultChunkSize + 64 KiB ≈ 192 KiB`, independent
   of object/part size.
2. **Measurable throughput improvement** on the 100 MiB benchmark
   fixture: ≥ 50 % reduction in allocated bytes/op, ≥ 30 %
   wall-time reduction, under `-benchmem -benchtime=10s` measured
   via `benchstat`.
3. **AEAD authenticity unchanged.** All existing crypto tests,
   including `-race`, `-tags=fips`, `-tags=hsm`, pass unmodified.
   Tamper-injection tests continue to detect chunk-level corruption
   at the same point in the stream as before.
4. **Deterministic ciphertext for retries.** The streaming MPU
   encrypt reader MUST produce byte-identical output when the same
   source is re-encrypted with the same `(DEK, uploadID, partNumber,
   ivPrefix)` — the SDK retry contract depends on this.
5. **No API / on-disk format change.** Chunked envelope layout,
   manifest schema, and the per-part ciphertext byte layout are
   preserved.

### Non-Goals

- Rewriting legacy (non-chunked) AEAD to streaming. Atomic
  authenticity of a single-nonce AEAD is incompatible with emitting
  plaintext before the tag is verified; the cap stays.
- New compression algorithms. Phase E only converts the existing
  gzip path to streaming; zstd remains where it is (already a
  streaming implementation in the chunked path, buffered in the
  legacy path, per §1.3 it stays buffered).
- Request-level backpressure policy changes. `io.Pipe` gives
  natural backpressure; no new config knobs for flow-control.
- Any change to `handleCopyObject`'s metadata selection rules
  (filter keys, Content-Type inheritance). Pure refactor.

### Phase B — `handleGetObject` streaming

**Touch:** `internal/api/handlers.go:1107-1287`.

Two `io.ReadAll` sites:

1. `:1111` — non-optimised range buffer. Replace with a `bytes.Buffer`
   limited by `plaintextEnd+1` (already known) filled via
   `io.CopyN(buf, decryptedReader, plaintextEnd+1)`, then
   `applyRangeRequest(buf.Bytes(), ...)`. Legacy AEAD path remains
   bounded by `MaxLegacyCopySourceBytes` (enforced already upstream).
2. `:1198` — optimised range path. Replace with
   `io.CopyBuffer(w, decryptedReader, pool.Get64K())`. Set
   `Content-Length` from `plaintextEnd - plaintextStart + 1` (already
   computed). Remove the intermediate `outputData` slice entirely.

Also: the MPU decrypt branch (`:971-1056`) already streams — no
change, but verify the `firstChunk` tamper-gate still works.

**Tests:** `TestHandleGetObject_Streaming_BoundedHeap` (Phase G
migrates this to a heap assertion); existing range and chunked
suites pass unchanged.

### Phase C — `handleCopyObject` streaming pipeline

**Touch:** `internal/api/handlers.go:3708-3909`.

Replace the `ReadAll → Encrypt → ReadAll` pair at `:3783,:3843` with
a single `io.Pipe` pipeline:

```
srcReader → srcEngine.Decrypt → [plaintext pipe] → dstEngine.Encrypt → [ciphertext pipe] → s3Client.PutObject
```

Implementation notes:

- Use `pr, pw := io.Pipe()`. Spawn a goroutine:
  ```go
  go func() {
      _, err := io.Copy(encWriter, decryptedReader) // or an
                                                      // io.Copy wrapping
                                                      // the encrypt writer
      pw.CloseWithError(err)
  }()
  ```
  Then pass `pr` to `s3Client.PutObject` with a precomputed
  `ContentLength`.
- `dstEngine.Encrypt` currently returns an `io.Reader`. For
  chunked mode that's already streaming; wrap it as the
  `PipeReader` source directly (no outer goroutine needed).
  Legacy mode continues to buffer inside `engine.Encrypt`, which is
  acceptable because legacy requires full-object access anyway — but
  the `bytes.NewReader(encryptedData)` wrapper at the caller side
  disappears.
- **ContentLength computation.** For chunked destinations, length
  is `plainLen + ceil(plainLen/ChunkSize) × tagSize + manifestSize`;
  the formula is already derived in
  `handleCreateMultipartUpload` and in the chunked encrypt path.
  Extract into a reusable helper `crypto.CiphertextLenForPlaintext(
  plainLen int64, chunkSize int, algo string) int64`.
- **Error propagation.** If `dstEngine.Encrypt` errors partway
  through, `pw.CloseWithError` causes `PutObject` to fail with that
  error; we translate via `TranslateError` as today. There is no
  partial-object write because S3 `PutObject` is atomic (either the
  upload completes and the object appears at the new version, or
  it doesn't).
- **Content-Length mismatch defence.** Wrap `pr` in a
  `countingReader` that, on EOF, compares bytes-read vs declared
  length and returns an error if they disagree. This is cheap
  insurance against a future refactor breaking the length formula.
- `handleCopyObject` currently lacks a legacy-source cap. Before
  landing this phase, honour `MaxLegacyCopySourceBytes` on the
  source side (legacy source → bounded buffer; chunked source →
  streaming). Mirrors the UploadPartCopy enforcement at
  `upload_part_copy.go:557`. This is a safety tighten that we
  should have done in S3-1 but missed.

**Tests:**
- `TestHandleCopyObject_Chunked_Streams_Bounded` — 100 MiB
  chunked→chunked copy, heap-bound assertion via the tracking
  reader.
- `TestHandleCopyObject_Legacy_CapEnforced` — legacy source
  exceeding cap returns 413 (or the existing `errLegacySourceTooLarge`
  equivalent) before decrypt begins.
- `TestHandleCopyObject_ContentLengthMismatch_FailsClosed` — inject
  a synthetic engine that writes N+1 bytes; confirm 5xx.
- Existing `TestHandleCopyObject_*` suite passes unchanged.

### Phase D — `handleUploadPart` seekable-body wrapper

**Touch:** `internal/api/handlers.go:2600-2734`,
`internal/s3/seekable_body.go` (new),
`internal/config/config.go` (new `MaxPartBuffer` knob).

Two sites:

1. **Encrypted MPU path (`:2672`).** The current `ReadAll` of the
   encrypt reader exists solely to satisfy the SDK's SigV4 body-hash
   retry. Two options:

   - **Option D-1 (preferred).** Bypass the SigV4 seekable
     requirement by setting
     `input.ChecksumAlgorithm = types.ChecksumAlgorithmSha256` (or
     SHA1, for older backends) and a precomputed
     `input.ChecksumSHA256 = <hex>`. The SDK then trusts the
     precomputed digest and streams the body without seeking. Compute
     the digest incrementally through a `hashReader` wrapped around
     the encrypt reader's pipe. *Caveat:* not every backend accepts
     trailing checksums; add a capability flag in
     `Backend.SupportsStreamingChecksums` and fall back to D-2.
   - **Option D-2 (fallback).** Introduce a pooled bounded-buffer
     seekable wrapper in `internal/s3/seekable_body.go`. Cap at
     `Server.MaxPartBuffer` (default 64 MiB). Parts above the cap
     are rejected with a specific S3 error
     (`EntityTooLarge`, HTTP 413) referencing the config knob.
     Pool the backing slice via a new size class on `BufferPool`
     (8 MiB and 64 MiB classes), so the hot path doesn't allocate.

   Phase D ships D-2 unconditionally; D-1 is tracked as a follow-up
   to avoid blocking the rest of PERF-1 on SDK capability work.

2. **Plaintext MPU path (`:2718`).** Same D-2 treatment. Cap at
   `MaxPartBuffer`; larger parts fall through to a streaming
   direct-forward if the backend client supports it (today the
   SDK path does, via `ContentLength` + non-seekable reader with
   retries disabled for oversize parts — worth a fresh test
   matrix, see Acceptance).

**Fail-closed interactions:**
- The encrypted path must still call `AppendPart` after
  `UploadPart`. The seekable wrapper does not change this; the
  503-on-Valkey-failure behaviour at `:2755-2780` is untouched.
- The length-mismatch assertion at `:2690-2708` is redundant after
  D-2 (the wrapper guarantees consistent length). Keep it; it's a
  belt-and-braces check and costs nothing.

**Config:**

```yaml
server:
  max_part_buffer: 67108864   # 64 MiB, default
```

Added to `ServerConfig` with `yaml:"max_part_buffer"` and
`env:"SERVER_MAX_PART_BUFFER"`. Default constant
`DefaultMaxPartBuffer = 64 * 1024 * 1024` in
`internal/config/config.go`.

**Tests:**
- `TestHandleUploadPart_Encrypted_Streaming_BoundedHeap` — the
  pooled wrapper never allocates more than `MaxPartBuffer` bytes.
- `TestHandleUploadPart_OversizePart_Refused` — part > cap returns
  413 with a clear message and no partial backend write.
- `TestSeekableBody_ReuseAcrossRetries` — unit test of the wrapper:
  simulate two `Seek(0, SeekStart)` + `ReadAll` passes, assert
  byte-identical output on both.
- Concurrent upload: 10 parallel 5 MiB parts; assert
  `BufferPoolMetrics` shows ≥ 90 % hit rate on the new size class.

### Phase E — Compression streaming

**Touch:** `internal/crypto/compression.go:121-219`.

Current `Compress` does `ReadAll → gzip.Writer → bytes.NewReader`.
Current `Decompress` does `ReadAll → gzip.NewReader → ReadAll`.

Convert both to streaming:

```go
// Compress
pr, pw := io.Pipe()
go func() {
    gw, _ := gzip.NewWriterLevel(pw, c.level)
    _, cpErr := io.Copy(gw, reader)
    closeErr := gw.Close()
    pw.CloseWithError(firstNonNil(cpErr, closeErr))
}()
return pr, metadata, nil
```

```go
// Decompress
gr, err := gzip.NewReader(reader)
if err != nil { return nil, err }
// caller-owned close — document the ownership change
return &readCloserWrapper{gr}, nil
```

Important constraint: `Compress` currently refuses to use
compression if the output is not smaller than the input ("Only use
compression if it actually saves space", `compression.go:164-168`).
This decision needs the final size, which streaming precludes. Two
options:

- **E-1 (recommended).** Drop the post-hoc size check. Trust the
  `ShouldCompress(size, contentType)` pre-filter (size + content
  type), which is the industry-standard approach and matches how
  nginx, Envoy, and the AWS SDK itself decide compression. Document
  in the ADR addendum.
- **E-2.** Peek the first ~8 KiB, compute a compressed:uncompressed
  ratio, and decide. Complex and fragile. Skip.

ADR 0006 (and a new addendum) should be updated to record this
behaviour change: "compression decision is pre-filter only; small
incompressible payloads may grow by < 1 % (gzip overhead) but are
no longer rejected post-facto."

**Tests:**
- `TestCompress_Streaming_Roundtrip_BoundedHeap` — 100 MiB text
  payload, assert peak buffered bytes ≤ 256 KiB.
- `TestCompress_IncompressiblePayload_GrowsSlightly` — random
  bytes, confirm output size ≈ input size + ~ 20 bytes, and that
  the metadata marker is written (so Decompress knows to gunzip).
- Existing round-trip tests pass unchanged.

### Phase F — Engine compression-branch refactor

**Touch:** `internal/crypto/engine.go:330-360` and the sibling
`:722,:874,:997,:1087,:1380,:1500,:1608,:1701` sites identified in
the audit.

Narrow scope: only the `bytes.NewReader(compressedData)` /
`bytes.NewReader(plaintext)` wrappers used to re-read compressed
output into the AEAD writer. Replace with direct piping of the
streaming compression reader into the AEAD boundary. Legacy
non-chunked AEAD still buffers (§1.3); the change is just removing
one of the two copies, not both.

**Invariants to preserve:**
- `x-amz-meta-encryption-original-size` metadata emission
  (`engine.go:350-ish`). Currently computed from `len(plaintext)`
  before compression. Streaming: thread a `sizeCountingReader`
  around the source before the compression pipe; read the final
  count after EOF. Must be emitted in the metadata map returned
  from `Encrypt`, so compute it by the time the AEAD writer
  finalises (i.e. synchronously before the ciphertext returns to
  the caller).
- ETag computation (`computeETag(plaintext)`). Thread an `md5.New`
  through the same `sizeCountingReader` — it becomes a
  `sizeAndHashReader`. Return the ETag in metadata as today.

**Streaming concerns / risks:**
- The `sizeAndHashReader` must read the source exactly once; both
  compression and ETag must tee from the same read. Use `io.TeeReader`
  or a fused reader that forwards to both sinks. The pattern is
  standard; *Efficient Go* §10.5 discusses tee pipelines.
- Legacy `Decrypt` returns plaintext after fully authenticating;
  streaming the decompression *after* AEAD verification is safe
  and is what chunked mode already does.

**Tests:**
- `TestEngine_LegacyEncrypt_CompressionStreams` — heap-bound
  assertion.
- `TestEngine_LegacyEncrypt_OriginalSizeMetadata` — explicit assert
  on `x-amz-meta-encryption-original-size` value after streaming
  refactor.
- `TestEngine_LegacyEncrypt_ETag` — explicit ETag-on-plaintext
  assertion.
- Full crypto round-trip suite (both compressed and
  non-compressed) passes unchanged.

### Phase G — Streaming MPU encrypt reader + heap-bound assertions

**Touch:** `internal/crypto/mpu_encrypter.go:21-89`,
`internal/api/handlers.go` (no change — just the encrypt reader is
swapped), new heap-assertion tests.

**G-1. Streaming `NewMPUPartEncryptReader`.** Replace the
`ReadAll` at `:39` with a streaming reader:

```go
type mpuEncryptReader struct {
    src       io.Reader
    gcm       cipher.AEAD
    dek       []byte
    hash      [32]byte
    ivPrefix  [12]byte
    part      uint32
    chunkSize int

    plainBuf  []byte     // reusable, pooled, cap=chunkSize
    cipherBuf bytes.Buffer // output buffer for current chunk
    chunkIdx  uint32
    srcDone   bool
    outPos    int
}
```

- On each `Read(p)`:
  1. If `cipherBuf` has unread bytes, serve from it.
  2. Otherwise, if `srcDone`, return `io.EOF`.
  3. Otherwise, `io.ReadFull(src, plainBuf[:chunkSize])` — partial
     read at EOF means a short final chunk; set `srcDone`.
  4. Derive the IV, `gcm.Seal(nil, iv, plainBuf[:n], nil)` into
     `cipherBuf`.
  5. Increment `chunkIdx`; goto 1.
- Pool the `plainBuf` (chunkSize bytes) via a new
  `plainChunkPool`; reuse `encBufPool` for cipher output.
- Return a computed ciphertext length upfront.
  `encLen = plainLen + ceil(plainLen/chunkSize) × tagSize`.
  Caller still uses this for `ContentLength`. *Caveat:* if
  `plainLen` is unknown (not provided), we cannot return the length
  without reading the source first, which defeats the purpose.
  Fallback: when `plainLen == 0 && body != nil`, fall back to the
  legacy buffered behaviour (documented via a metric). All current
  callers supply `plainLen` per the encrypted-MPU fast path at
  `handlers.go:2644-2650`.

**Determinism invariant:** the per-chunk IV depends only on
`(DEK, uploadIDHash, ivPrefix, partNumber, chunkIndex)`. A
re-execution of the streaming reader over the same source produces
identical IVs and therefore identical ciphertext. This is the SDK
retry safety property and must be asserted by a new test
`TestMPUEncryptReader_DeterministicCiphertext_AcrossRetries`.

**Interaction with Phase D.** The seekable-body wrapper (Phase D)
now wraps a *streaming* ciphertext source. Because the wrapper
re-reads the underlying buffer on Seek, and the buffer content is
the already-sealed ciphertext, determinism is trivially satisfied
for the SDK. Phase G's determinism property matters primarily for
the D-1 (streaming SHA-256 checksum) option if we ever ship it.

**G-2. Heap-bound assertions.** Add tests (one per changed
handler) that wrap every `io.Reader` in the hot path with the
`trackingReader` from Phase A, run a 64 MiB or 100 MiB payload
through, and assert peak simultaneous buffered bytes ≤
`2 × crypto.DefaultChunkSize + MaxPartBuffer + 64 KiB` (chunked
paths) or ≤ `MaxPartBuffer + 64 KiB` (MPU upload). A CI
regression here catches any future `ReadAll` re-introduction.

**Tests:**
- `TestMPUEncryptReader_Streaming_BoundedHeap` — peak buffer ≤
  `chunkSize + tagSize + pooled cipher buf`.
- `TestMPUEncryptReader_DeterministicCiphertext_AcrossRetries` —
  re-seal, byte-identical.
- `TestHandleGetObject_Streaming_BoundedHeap`.
- `TestHandleCopyObject_Streaming_BoundedHeap`.
- `TestHandleUploadPart_Streaming_BoundedHeap`.
- `TestHandleUploadPartCopy_ReencryptMPU_Streaming_BoundedHeap`
  (this one targets the `uploadPartCopyReencryptMPU` ReadAlls at
  `upload_part_copy.go:730,771,790,816`; if wholesale streaming is
  impractical there due to the three-class switch, at minimum we
  eliminate the second buffer at `:816` by streaming the encrypt
  reader into UploadPart directly).

## 4. Technical Deep-Dives

### 4.1 `io.Pipe` error propagation

The standard footgun: a goroutine that calls `pw.Close()` (not
`CloseWithError(err)`) on the error path causes the reader side to
see `io.EOF` instead of the real error, producing a short-write that
is silently accepted by `PutObject`. Every pipe in this plan uses
the pattern:

```go
go func() {
    defer pw.Close() // only runs on success path
    if _, err := io.Copy(pw, src); err != nil {
        pw.CloseWithError(err)
        return
    }
}()
```

or equivalently an explicit `pw.CloseWithError(io.Copy(pw, src))`.

### 4.2 Cancellation

The request context must abort the pipe promptly on client
disconnect. Wrap the goroutine's source in a `ctxReader` that
returns `ctx.Err()` on every `Read` once the context is cancelled.
Without this, a 10 GiB copy whose client hung up at byte 1 will
continue to decrypt + encrypt + upload for minutes.

Existing code already plumbs `r.Context()` into `s3Client.GetObject`
and `s3Client.PutObject`, but the goroutine in between is blind.
Fix by wrapping `decryptedReader` once at entry.

### 4.3 MPU streaming encrypt: final-chunk semantics

Current `NewMPUPartEncryptReader` iterates over a known-size
plaintext and knows when `end == len(plaintext)`. Streaming loses
that certainty. Two paths, and we need to pick one:

- **Path A (length-known):** caller passes `plainLen`; reader does
  `io.ReadFull` for every full-size chunk and a final `io.ReadFull`
  with `buf[:plainLen % chunkSize]` if the remainder is non-zero.
  `ReadFull` distinguishes EOF from partial read; short read is a
  hard error (truncation attempt). This is the default path and
  matches how the plaintext MPU upload handler already operates
  (`handlers.go:2644-2650` sets `plainLen`).
- **Path B (length-unknown):** reader does `io.ReadAtLeast(src,
  buf, 1)`; a return with `n < chunkSize` and no error is the
  signal for "final chunk, finalise". This is rarely needed — only
  if a future caller passes a `chunked-transfer-encoding` body
  without `x-amz-decoded-content-length`. Implement but treat as
  the slow path; bench-gate in Phase A.

### 4.4 SDK seekable body: the actual contract

From the aws-sdk-go-v2 `service/s3` source
(`PutObjectInput.Body`): the body is wrapped by the signer, which
needs either
  (a) a `io.ReadSeeker` for single-shot SigV4 body-hash, or
  (b) a pre-computed `ChecksumSHA256` and `ChecksumAlgorithm` to
  skip hashing, or
  (c) streaming payload hash mode (`x-amz-content-sha256:
  STREAMING-AWS4-HMAC-SHA256-PAYLOAD`), which the SDK enables
  automatically when the body is non-seekable and
  `ContentLength` is set, iff the backend supports it.

Current code uses (a) by buffering. Phase D ships (a) with a
pooled bounded buffer (so the *allocation* is amortised, even if
the *copy* is not). Phase D-1 (follow-up) would ship (b) or (c).
The correctness risk is entirely on (c): some backends (older Ceph
RGW, some S3-alike appliances) reject
`STREAMING-AWS4-HMAC-SHA256-PAYLOAD`. We gate it behind a backend
capability flag as in §4.5 of V0.6-S3-1.

### 4.5 Counting-reader / length-assertion helper

Introduce `internal/util/counting.go`:

```go
type CountingReader struct { R io.Reader; N int64 }
func (c *CountingReader) Read(p []byte) (int, error) {
    n, err := c.R.Read(p)
    c.N += int64(n)
    return n, err
}

type LengthAssertingReader struct {
    R        io.Reader
    Expected int64
    seen     int64
}
func (l *LengthAssertingReader) Read(p []byte) (int, error) {
    n, err := l.R.Read(p)
    l.seen += int64(n)
    if err == io.EOF && l.seen != l.Expected {
        return n, fmt.Errorf("length mismatch: expected %d, got %d", l.Expected, l.seen)
    }
    return n, err
}
```

Used in Phase C for ContentLength assertion and in Phase D for
part-length assertion.

### 4.6 ETag preservation through streaming

`engine.Encrypt` records `originalETag` for the unmodified plaintext.
In streaming form:

```go
h := md5.New()
sz := &CountingReader{R: plaintextSource}
tee := io.TeeReader(sz, h)
compressed, cmeta, _ := c.Compress(tee, contentType, hintedSize)
// pipe compressed → AEAD writer
// after EOF:
originalETag := hex.EncodeToString(h.Sum(nil))
originalSize := sz.N
```

MD5 is used for ETag only — not for security — so FIPS profiles
continue to compile (`crypto/md5` is available in FIPS mode per
go-1.21 FIPS 140 support). Confirm in Phase F tests under
`-tags=fips`.

## 7. Risks & Mitigations

| Risk | Likelihood | Severity | Mitigation |
|---|---|---|---|
| `io.Pipe` goroutine leaks on error | M | H | Use `CloseWithError` exclusively; add `-race` test that cancels context mid-copy and asserts goroutine count returns to baseline. |
| SDK signer rejects non-seekable body on a backend we don't test against | M | H | Phase D ships pooled-seekable (option D-2) by default; streaming SigV4 (D-1) is a follow-up gated by `Backend.SupportsStreamingChecksums`. |
| Post-hoc compression size check removal degrades small-payload compression ratio | L | L | ADR addendum; pre-filter already excludes `image/*`, `video/*`, etc; gzip overhead on random bytes ≈ 20 bytes; benign. |
| Streaming MPU encrypt produces non-deterministic ciphertext under source re-reads | L | H | IV derivation is pure function of index, not content; asserted by `TestMPUEncryptReader_DeterministicCiphertext_AcrossRetries`. |
| ETag on legacy Encrypt drifts if streaming ordering changes | L | M | `TeeReader` into `md5.New` happens before any transform; test asserts bit-identical ETag vs pre-change. |
| Content-Length miscalculation for chunked destination | L | H | `CiphertextLenForPlaintext` helper with unit tests per algorithm + chunk-size; `LengthAssertingReader` fails closed at the backend boundary. |
| Bench fixtures too small to show the promised ≥ 50 % allocation reduction | L | M | Phase A commits a 100 MiB fixture; if the reduction is not seen there, the claim is retracted in the PR body and thresholds are re-stated rather than the phase being declared "done". |
| `MaxPartBuffer` default too low for real-world 64-MiB+ parts | M | M | Default 64 MiB matches the typical "large part" upper bound; document the override; alert operators via a `mpu.part_oversize_rejected` metric. |
