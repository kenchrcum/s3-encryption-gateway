# V0.6-S3-1 — Multipart Copy (`UploadPartCopy`) — Implementation Plan

Status: Draft
Owner: API
Priority: P1
Labels: `milestone:v0.6`, `area:s3`

> Scope clarified relative to the one-line tracker entry at
> `docs/issues/v0.6-issues.md:111`. The task is expanded to cover dispatch
> correctness, source-format awareness (chunked / legacy / plaintext), a
> backend-native fast path, streaming behaviour, response contract
> (`CopyPartResult` ETag), cross-key-space handling, and the currently-broken
> `ProxyClient.CopyObject` stub. Encrypted multipart **destinations** are
> explicitly out of scope — see §2 Non-Goals.

---

## 1. Context & Current State

The gateway currently has a silent-failure trap on the S3 `UploadPartCopy`
operation and several related gaps.

- **UploadPartCopy is unrouted.** The route table at
  `internal/api/handlers.go:105` maps `PUT ?partNumber=N&uploadId=X` to
  `handleUploadPart` regardless of whether the request carries an
  `x-amz-copy-source` header. The handler at
  `internal/api/handlers.go:2116` ignores that header entirely, treats the
  body as part data (`io.ReadAll(r.Body)` at `:2236`) and uploads a
  zero-byte part to the backend. No S3 error is returned. Clients that issue
  a valid `UploadPartCopy` today observe silently-corrupted multipart
  uploads.
- **Multipart parts are plaintext by design.** The comment at
  `internal/api/handlers.go:2169-2173` and the decision in
  `docs/adr/0002-multipart-upload-interoperability.md` (§Alternative 1,
  rejected) fix multipart parts as plaintext to avoid the "concatenated
  independent AEAD streams" decryption problem. This plan does **not**
  revisit that decision.
- **Regular `CopyObject` is a full decrypt / re-encrypt cycle.** The handler
  at `internal/api/handlers.go:2536-2743` fetches the source via
  `s3Client.GetObject` (`:2584`), decrypts (`:2601-2611`),
  `io.ReadAll`s plaintext into memory (`:2626`), re-encrypts with a fresh
  salt/nonce/envelope (`:2658-2668, :2683`), and uploads via `PutObject`
  (`:2706`). The destination ETag is fetched with a follow-up `HeadObject`
  (`:2722-2723`). Two full-object buffers in memory; no streaming.
- **Backend-native copy exists but is unused.**
  `internal/s3/client.go:660-695` implements `CopyObject` via the AWS SDK
  `CopyObjectInput`. The gateway never calls this path because every copy is
  mediated for encryption. No `UploadPartCopy` wrapper exists on the backend
  interface at all — see the `Client` interface at
  `internal/s3/client.go:33-37, 39-41` where UploadPartCopy is absent.
- **`ProxyClient.CopyObject` is a stub.**
  `internal/s3/proxy_client.go:144-147` returns
  `"ProxyClient.CopyObject not implemented"`. Copy is therefore entirely
  broken in proxy/passthrough deployments today; this plan fixes that as a
  cross-cutting concern.
- **Chunked-source range-decrypt infrastructure exists and is reusable.**
  `internal/crypto/range_decrypt.go`, `internal/crypto/range_optimization.go`,
  and the call sites at `internal/api/handlers.go:708-824` expose
  `ParseHTTPRangeHeader`, `CalculateEncryptedRangeForPlaintextRange`, and
  `DecryptRange`. These are precisely the primitives needed to honour
  `x-amz-copy-source-range` efficiently against chunked sources.
- **Legacy single-AEAD sources (non-chunked) are monolithic.** The
  `engine.Encrypt` non-chunked branch emits one nonce/tag over the whole
  object (see `internal/crypto/engine.go:282+`). Any range against such a
  source must decrypt the entire object into memory — correct, but costly.
- **ETag conventions vary by path.** Single-part encrypted PUT stores the
  plaintext MD5 in `x-amz-meta-encryption-original-etag`
  (`internal/crypto/engine.go:322`) and restores it on HEAD/List
  (`handlers.go:1449-1452, :1615-1617`). Multipart parts pass the backend
  ETag straight through (`handlers.go:2270`).
- **Bucket-validation middleware** at
  `internal/middleware/bucket_validation.go:63-65` already inspects
  `x-amz-copy-source` for CopyObject; the new route must keep triggering it.

The S3 reference behaviour for `UploadPartCopy` (request / response /
error surface) is documented by AWS at [6] and the SDK V2 input struct at
[7]; this plan follows those contracts.

## 2. Design Goals & Non-Goals

### Goals
1. **Correct dispatch** — an HTTP request carrying `x-amz-copy-source` to a
   multipart part URL routes to a dedicated `handleUploadPartCopy` handler;
   malformed or unsupported inputs return proper S3 error codes rather than
   silently producing empty parts.
2. **Encrypted-source correctness** — copying from an encrypted source
   (chunked or legacy) produces a part whose bytes equal the caller-visible
   plaintext bytes of the requested source range.
3. **Source-format-aware performance** —
   - chunked source + range: decrypt only the chunks needed;
   - plaintext source + no re-encrypt required: use a backend-native
     `UploadPartCopy` fast path (zero bytes through the gateway);
   - legacy source + range: fall back to full decrypt (documented cost).
4. **Response contract** — emit a compliant `CopyPartResult` XML body with
   the destination-part ETag and `LastModified`, matching [6].
5. **Cross-key-space correctness** — source and destination may resolve to
   different engines / KEKs; reuse the per-object engine resolution already
   used by `handleCopyObject`.
6. **Streaming, bounded memory** — the mediated path MUST NOT double-buffer
   full objects in memory; uses `io.Pipe` / `io.CopyBuffer` and, for the
   legacy fallback, a disclosed upper-bound on buffered bytes.
7. **Proxy-mode parity** — `ProxyClient.CopyObject` and the new
   `UploadPartCopy` path both work or both return a proper S3 error; no
   silent failures.
8. **Observability** — metrics and structured logs to make the operation
   debuggable in production.

### Non-Goals
- **Encrypted multipart destinations.** Parts remain plaintext, consistent
  with ADR 0002. Any work toward encrypted MPU is a separate proposal and
  would need a new per-upload envelope scheme (likely chunked-MPU with a
  finalization manifest on `CompleteMultipartUpload`).
- **Client-side encryption changes.**
- **ACL / Object Lock header pass-through.** Tracked separately under
  V0.6-S3-2.
- **Zero-copy crypto optimisations in the general case** (V0.6-PERF-1). This
  plan guarantees bounded memory for UploadPartCopy, not zero-copy
  everywhere.
- **Replacing `handleCopyObject`'s double-buffering.** A drive-by clean-up
  is welcome but not required; the success criterion for this task is the
  new handler.

## 3. Target Request / Response Semantics

### 3.1 Routing predicate

Inside `handleUploadPart` (`internal/api/handlers.go:2116`), as the first
substantive check after parameter validation, branch on
`r.Header.Get("x-amz-copy-source") != ""` and delegate to
`handleUploadPartCopy`. This mirrors the single-object
`x-amz-copy-source` branch at `handlers.go:1074-1080`.

### 3.2 Source resolution matrix

For each incoming request, the handler classifies the source:

| Source class | Encryption metadata (on HEAD) | Strategy |
|---|---|---|
| Chunked-encrypted | `x-amz-meta-encryption-chunked=true` | Mediated: translate plaintext range → encrypted byte range via `crypto.CalculateEncryptedRangeForPlaintextRange`, GET range, `DecryptRange`, stream to `UploadPart`. |
| Legacy single-AEAD | `x-amz-meta-encrypted=true` without chunked flag | Mediated (slow): GET full, decrypt full, slice plaintext by source range, stream to `UploadPart`. Bounded memory = whole-object size; emit `object_size` in structured log. |
| Plaintext passthrough | No encryption metadata | Fast path: call backend-native `UploadPartCopy` (new wrapper) and forward the backend's `CopyPartResult` verbatim. Zero bytes through the gateway. |

The classification is done by a new helper `classifyCopySource(ctx,
s3Client, bucket, key, versionID)` that issues a single `HeadObject` — the
same pattern already used by the regular copy handler.

### 3.3 `x-amz-copy-source-range` handling

Supported format per [6]: `bytes=first-last` with both ends specified and
`first <= last`.

- Parsed via `crypto.ParseHTTPRangeHeader` (reuses `handlers.go:749`).
- For chunked sources, translated to an encrypted byte range via
  `crypto.CalculateEncryptedRangeForPlaintextRange`.
- Missing header: whole source object (still subject to MPU part-size
  rules, §3.4).
- Malformed: respond `400 InvalidArgument` per S3 error contract [8].
- Out-of-range (`first >= plaintextSize`): respond `416
  InvalidRange`.

### 3.4 MPU part-size rules

S3 enforces (see [6]):

- Non-final parts: `5 MiB ≤ size ≤ 5 GiB`.
- Any single UploadPartCopy source range: `≤ 5 GiB`.
- If source object is `> 5 GiB` **and** no range is given: `400
  InvalidRequest` (AWS behaviour: "The specified copy source is larger than
  the maximum allowable size for a copy source: 5368709120").

The handler enforces the 5 GiB per-call cap locally. The 5 MiB minimum is
**not** enforced locally — AWS applies it only at
`CompleteMultipartUpload`, so we must let short parts through.

### 3.5 Response body

```
<CopyPartResult xmlns="http://s3.amazonaws.com/doc/2006-03-01/">
  <ETag>"...."</ETag>
  <LastModified>2026-04-17T10:00:00.000Z</LastModified>
</CopyPartResult>
```

- `ETag`: backend's raw `UploadPart` ETag (mediated path) or backend's
  `UploadPartCopy` ETag (fast path). Consistent with the existing multipart
  ETag contract at `handlers.go:2270`. Parts remain plaintext, so the
  backend ETag is meaningful to clients that later reconstruct the object
  ETag.
- `LastModified`: returned by the backend on the fast path; set to the
  gateway's current time on the mediated path (AWS does the same — the
  value reflects when the *part* was written, not the source).

### 3.6 Error surface

| Condition | HTTP | Code |
|---|---|---|
| Missing `uploadId` / `partNumber` | 400 | `InvalidRequest` |
| Malformed `x-amz-copy-source` | 400 | `InvalidArgument` |
| Malformed `x-amz-copy-source-range` | 400 | `InvalidArgument` |
| Source not found | 404 | `NoSuchKey` |
| Source bucket not found | 404 | `NoSuchBucket` |
| Source > 5 GiB with no range | 400 | `InvalidRequest` |
| Source range start ≥ object size | 416 | `InvalidRange` |
| Multipart uploads disabled (`Server.DisableMultipartUploads`) | 501 | `NotImplemented` (reuse path at `handlers.go:2134-2143`) |
| Proxy mode, mediated needed | 501 | `NotImplemented` (until `ProxyClient` support lands) |

## 4. Work Breakdown

### Phase A — Backend client surface
- A1. Extend the `Client` interface at `internal/s3/client.go:33-41` with:
  ```go
  UploadPartCopy(
      ctx context.Context,
      dstBucket, dstKey, uploadID string,
      partNumber int32,
      srcBucket, srcKey string, srcVersionID *string,
      srcRange *CopyPartRange, // nil = whole source
  ) (*CopyPartResult, error)
  ```
  with supporting types `CopyPartRange{First, Last int64}` and
  `CopyPartResult{ETag string; LastModified time.Time}`.
- A2. Implement on `s3Client` via `s3.UploadPartCopyInput` (see SDK V2 [7]).
  `CopySource` construction mirrors `CopyObject` at
  `internal/s3/client.go:662-664`. When `srcRange != nil`, set
  `CopySourceRange = aws.String(fmt.Sprintf("bytes=%d-%d", first, last))`.
- A3. Implement on `ProxyClient` (`internal/s3/proxy_client.go`): forward
  the `PUT ?partNumber=...&uploadId=...` with `x-amz-copy-source` and
  optional `x-amz-copy-source-range` to the upstream, parse
  `CopyPartResult` from the response. This also exercises the proxy HTTP
  path needed by A4.
- A4. **Fix `ProxyClient.CopyObject`** (`proxy_client.go:145`) in the same
  PR — forward the `PUT` with `x-amz-copy-source` and parse
  `CopyObjectResult`. Symmetrical to A3, reuses helpers.

### Phase B — Handler wiring
- B1. In `handleUploadPart` (`handlers.go:2116`), insert the dispatch branch
  before the multipart-disabled check so that disabled-multipart still
  yields the correct error:
  ```go
  if r.Header.Get("x-amz-copy-source") != "" {
      h.handleUploadPartCopy(w, r)
      return
  }
  ```
- B2. New function `handleUploadPartCopy` (likely in a new file
  `internal/api/upload_part_copy.go` to keep `handlers.go` in check; it's
  already 2.8 k LOC). Responsibilities:
  - Parse & validate `bucket`, `key`, `uploadID`, `partNumber`.
  - Re-use existing config gate (`Server.DisableMultipartUploads`).
  - Parse `x-amz-copy-source` (supports `/bucket/key`, `bucket/key`, and
    `?versionId=...` suffix). Reuse the helper already present for
    `handleCopyObject` (`handlers.go:2536+`) — extract to
    `parseCopySource(header string) (srcBucket, srcKey, srcVersionID,
    error)`.
  - Parse optional `x-amz-copy-source-range`.
  - Classify source (§3.2) via a single `HeadObject`.
  - Execute the chosen strategy (§3.3, §3.4).
  - Emit `CopyPartResult` XML (§3.5).

### Phase C — Mediated strategies
- C1. Chunked path: build a `crypto.ByteRange` from the plaintext source
  range; call `CalculateEncryptedRangeForPlaintextRange` to get the
  backend byte range; issue `s3Client.GetObject` with `Range`; invoke
  `engine.DecryptRange` to produce a plaintext `io.Reader` bounded by the
  requested slice; pipe into `s3Client.UploadPart`.
  **Bounded memory**: one chunk (default 64 KiB, max 1 MiB per
  `internal/crypto/chunked.go:14-23`) plus a transfer buffer. No
  `io.ReadAll`.
- C2. Legacy path: `s3Client.GetObject` (full), `engine.Decrypt`, slice
  resulting plaintext by source range, pipe to `s3Client.UploadPart`. Log
  a `WARN` with `source_size_bytes` so operators can spot the slow path in
  dashboards. Emit a counter `gateway_upload_part_copy_legacy_fallback_total`.
- C3. Plaintext fast path: `s3Client.UploadPartCopy` (new from A1),
  forward `CopyPartResult` as-is.

### Phase D — Metrics, logs, audit
- D1. Metrics (Prometheus, via existing `internal/metrics`):
  - `gateway_upload_part_copy_total{source_mode="chunked|legacy|plaintext",result="ok|error"}`
  - `gateway_upload_part_copy_bytes_total{source_mode=…}`
  - `gateway_upload_part_copy_duration_seconds{source_mode=…}` histogram
  - `gateway_upload_part_copy_legacy_fallback_total` counter
- D2. Structured log at INFO: `{src_bucket, src_key, src_version,
  src_range, src_mode, dst_bucket, dst_key, upload_id, part_number,
  dst_etag, duration_ms}`.
- D3. Audit event hook (`internal/audit`, same pattern as `handleCopyObject`
  at the end of its handler).

### Phase E — Documentation
- E1. Update `docs/S3_API_IMPLEMENTATION.md` multipart section (currently
  documents multipart as plaintext-only at `:41-46`) to include
  UploadPartCopy semantics and the source-class matrix.
- E2. Add ADR `docs/adr/0006-upload-part-copy.md` recording:
  - Decision to keep destinations plaintext (upholds ADR 0002).
  - Source-format dispatch matrix.
  - Cross-key-space behaviour.
- E3. Append a short "Known limitations" note to ADR 0002's "Future
  Considerations" section (`0002-…:323-325`) pointing to ADR 0006.

### Phase F — Tests (detailed in §6)
- Parallel to each phase; merge with the production change.

## 7. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| UploadPartCopy crosses bucket/key-space boundaries implicitly (client presents one URL, data is pulled from another). A caller authorised to write the destination might not be authorised to read the source. | Explicit **source-bucket read authorisation** check using the same policy path that guards `GetObject`, independent of destination write authorisation. Promoted to a DoD checkbox (§8). Rationale: *Building Secure and Reliable Systems* Ch. 5 §"Classifying Access Based on Risk" and §"Advanced Authorization Controls" [9]. |
| Adding dispatch branch regresses `handleUploadPart` hot path. | Single header check (`r.Header.Get` is O(1)); guarded by an existing parameter-validation block; covered by the existing UploadPart test suite. |
| Legacy source + range → full-object buffer causes OOM on large objects. | Hard cap via config `Server.MaxLegacyCopySourceBytes`; **default 256 MiB** (opt-out safety posture: operators must explicitly raise the limit and size pod memory accordingly — `max_concurrent_copies × cap` headroom required). Exceeding the cap returns `InvalidRequest` with a clear message pointing operators at the chunked-encryption migration path. Rationale: *Building Secure and Reliable Systems* (Adkins et al., 2020) Ch. 8 §"Differentiate Costs of Failures" and Ch. 10 §"Self-Inflicted Attacks" argue for defaulting dangerous code paths to a conservative limit that operators opt *out* of rather than opt *in* to restricting [9]. |
| Backend-native `UploadPartCopy` on the plaintext fast path can succeed even if the gateway has the destination bucket marked encrypted (config drift). | **Hard refusal**, not best-effort: if the destination bucket config mandates encryption but the classified source is plaintext, the handler MUST refuse with `500 InternalError` and emit an audit event. This is promoted from "mitigation" to a DoD checkbox (§8) so it cannot be quietly dropped during implementation. Classification uses the source object's actual metadata (not its bucket config), consistent with *Real-World Cryptography* (Wong, 2021) Ch. 16 §"Cryptography is not an island" [10] on trust-on-first-read hazards. |
| `x-amz-copy-source` URL-encoding edge cases (slashes, unicode). | Share the parser with `handleCopyObject`; add a property-based test; reference AWS docs [6] for canonical encoding rules. |
| ETag mismatch between backend-native copy and mediated copy surprises clients that checksum parts client-side. | Documented in ADR 0006 and in `S3_API_IMPLEMENTATION.md`; both paths return the backend ETag of the resulting part, which is what clients actually observe on `ListParts`. |
| Cross-key-space copy where destination engine cannot be resolved (mis-configuration). | Reuse `handleCopyObject`'s engine-resolution error path (returns `500 InternalError`); covered by unit test. |
| ProxyClient.CopyObject fix expands scope. | It is a small, well-contained HTTP-forwarding implementation that unblocks every copy test in proxy mode; bundling it avoids shipping a half-fixed copy story. |
| `CompleteMultipartUpload` currently has no notion of "mixed part origins" (some uploaded, some copied). | No change needed — S3 semantics make this transparent; confirmed by integration test `TestUploadPartCopy_MixedWithUploadPart`. |
| Concurrent `UploadPartCopy` against the same `uploadId` & `partNumber`. | S3 semantics: last write wins (AWS contract). Add a note to the ADR; no local serialization. |

## 8. Definition of Done

Per the tracker entry `docs/issues/v0.6-issues.md:99-118`:

- [ ] `UploadPartCopy` routed to a dedicated handler; never falls through to
      `handleUploadPart`'s body-read path.
- [ ] All three source classes (chunked, legacy, plaintext) supported with
      correctness tests and bounded-memory assertion for the chunked path.
- [ ] Backend-native `UploadPartCopy` wrapper added to the `Client`
      interface and implemented on both `s3Client` and `ProxyClient`.
- [ ] `ProxyClient.CopyObject` (`proxy_client.go:145`) implemented.
- [ ] `x-amz-copy-source-range` honoured; malformed ranges produce correct
      S3 error codes.
- [ ] **Source-bucket read authorisation** checked on every UploadPartCopy
      via the same policy path as `GetObject`; denial produces `403
      AccessDenied` independently of destination write authorisation.
      Integration test `TestUploadPartCopy_CrossBucket_ReadDenied` covers
      the case. (Per *Building Secure and Reliable Systems* Ch. 5 [9].)
- [ ] **Destination-policy / source-mode mismatch hard-refusal**: if the
      destination bucket config mandates encryption and the classified
      source is plaintext, the handler returns `500 InternalError` and
      emits an audit event. Integration test
      `TestUploadPartCopy_PlaintextSource_EncryptedDestBucket_Refused`
      covers the case.
- [ ] Response XML validated against the SDK V2 `UploadPartCopy` response
      shape [7].
- [ ] Metrics (§D1) and structured logs (§D2) emitted; Prometheus scrape
      test passes.
- [ ] Audit event emitted per copy, matching `handleCopyObject`'s shape.
- [ ] Integration tests against MinIO pass (all six test names in §6).
- [ ] Benchmarks recorded in the v0.6 performance baseline artifact
      (feeds V0.6-QA-1).
- [ ] ADR 0006 merged; ADR 0002 cross-referenced;
      `docs/S3_API_IMPLEMENTATION.md` updated.
- [ ] CHANGELOG entry under **v0.6 – Unreleased** describing
      UploadPartCopy support and the plaintext-destination caveat.
- [ ] Coverage for new code ≥ 85 %, contributing to the v0.6 ≥ 80 %
      milestone gate.

## 10. Follow-ups / Out-of-scope items surfaced during planning

These are explicitly **not** part of V0.6-S3-1 but are worth filing as
separate tickets at merge time:

1. **Encrypted multipart destinations** — new ADR, likely chunked-MPU with
   a manifest finalised on `CompleteMultipartUpload`. Deep change.
2. **Streaming rewrite of `handleCopyObject`** — remove the two
   `io.ReadAll` calls at `handlers.go:2626, :2683`. Aligns with
   V0.6-PERF-1.
3. **Backend-native CopyObject when source is plaintext** — same
   optimisation as the UploadPartCopy fast path, applied to regular copy.
4. **Conditional copy headers** (`x-amz-copy-source-if-match`, etc.) —
   forward on the fast path; implement `HeadObject`-based check on the
   mediated path. Small, self-contained follow-up.
