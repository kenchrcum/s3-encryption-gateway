## Critical / High Severity Findings

### [DAF-01] SigV4 Header Auth Missing Clock-Skew / Replay Window
- **Severity:** HIGH
- **CVSS Estimate:** 6.5 (Medium-High)
- **Files:** `internal/api/auth.go` (lines ~64-91, ~127-146)
- **Description:** `ValidateSignatureV4` validates expiry for **presigned URLs** (line ~138), but for **header-based Signature V4 requests** (the `else` branch starting line ~64), the `X-Amz-Date` timestamp is extracted (line ~87) but **never validated against server time**. There is no clock-skew check, no maximum age enforcement, and no anti-replay nonce.
- **Impact:** An attacker who captures a valid signed request (e.g., via packet capture, proxy logs, or browser dev tools) can replay it indefinitely. This enables repeated destructive operations (DELETE, PUT overwrite) or data exfiltration (captured GET signatures) without time bound.
- **Reproduction:** Capture a valid `Authorization: AWS4-HMAC-SHA256 ...` header from any authenticated request. Replay the same request hours or days later; it will be accepted.
- **Remediation:**
  1. Parse `X-Amz-Date` for all SigV4 requests (not just presigned).
  2. Reject requests where the timestamp is outside a configurable skew window (default 15 minutes, matching AWS specification).
  3. Consider adding a short-lived nonce cache for header-auth requests to prevent near-instant replay within the skew window.
  4. Log rejected replays at WARN level for security monitoring.
- **References:** O'Reilly "API Security in Action" (Madden) — Chapter 4: Request Authentication & Replay Protection; AWS SigV4 documentation — "Handling clock skew".

### [DAF-02] Key Padding Creates Catastrophically Weak Keys
- **Severity:** HIGH
- **CVSS Estimate:** 7.5 (High)
- **Files:** `internal/crypto/engine.go` (multiple locations: encrypt ~429, decrypt ~648, chunked encrypt ~873, chunked decrypt ~1002, range decrypt ~1180)
- **Description:** When a derived or KMS-unwrapped key is shorter than the expected `keySize`, the engine "pads" it by copying the start of the key over the remainder:
  ```go
  if len(key) < keySize {
      copy(adjustedKey[len(key):], key[:keySize-len(key)])
  }
  ```
  A 16-byte key padded to 32 bytes becomes `key[:16] + key[:16]` — a key with full 16-byte block repetition. This is cryptographically catastrophic.
- **Impact:** If a misconfigured KMS or key resolver returns an undersized key, encryption becomes trivially weak without any operator-visible failure. The ciphertext could be broken with standard cryptanalysis.
- **Reproduction:** Inject a short key (e.g., 16 bytes) into the decrypt path via a mock KMS returning 16 bytes; the padded 32-byte key exhibits perfect periodicity.
- **Remediation:**
  1. **Remove the padding fallback entirely.** If `len(key) != keySize`, return a hard error: `fmt.Errorf("key size mismatch: got %d, want %d", len(key), keySize)`.
  2. Keys must be exactly the required size. Padding should never be a silent operation.
  3. Document this as a breaking change for any deployment that inadvertently relied on the padding (highly unlikely, as it indicates a broken KMS).
- **References:** O'Reilly "Cryptography Engineering" (Ferguson, Schneier, Kohno) — Key Management: never derive or pad keys in ways that reduce entropy; "Security with Go" (Leon) — Chapter 6: AES key handling.

### [DAF-03] Unbounded Goroutine Spawning in Audit BatchSink
- **Severity:** HIGH
- **CVSS Estimate:** 6.2 (Medium)
- **Files:** `internal/audit/sink.go` (lines ~72-86)
- **Description:** `BatchSink.WriteEvent` spawns a new goroutine (`go s.writeWithRetry(events)`) every time the buffer reaches `bufferSize`. Under sustained audit traffic or a slow HTTP sink, goroutines accumulate without bound because there is no semaphore, worker pool, or backpressure mechanism.
- **Impact:** An attacker can generate high-volume S3 requests to exhaust memory and file descriptors, causing a gateway-wide denial of service. The gateway process will OOM or hit the OS goroutine limit.
- **Reproduction:** Send sustained S3 request traffic with audit logging enabled and a slow HTTP sink endpoint. Observe `go runtime.NumGoroutine()` climbing without bound.
- **Remediation:**
  1. Replace the fire-and-forget goroutine with a **bounded worker pool** (e.g., a channel + fixed number of goroutines, or `golang.org/x/sync/semaphore`).
  2. Alternatively, use the existing `run()` ticker loop exclusively for flushes and remove the async goroutine path entirely. The ticker already handles periodic flushing.
  3. If async flushes are required for latency, bound concurrency with a `sync/semaphore` or buffered channel (e.g., max 4 concurrent flushes).
  4. Add a metric `audit_flush_backpressure_total` when the worker pool is saturated.
- **References:** O'Reilly "Building Secure and Reliable Systems" (Adkins, Beyer et al.) — Chapter 8: Resource exhaustion and backpressure; Go Concurrency Patterns.

### [DAF-04] Chunked Encrypt Path Reads Entire Plaintext into Memory
- **Severity:** HIGH
- **CVSS Estimate:** 6.2 (Medium)
- **Files:** `internal/crypto/engine.go` (`encryptChunked`, line ~788; `encryptChunkedWithMetadataFallback`, line ~956)
- **Description:** Both `encryptChunked` and `encryptChunkedWithMetadataFallback` call `io.ReadAll(reader)` on the entire source before starting the chunked encryption pipeline. The so-called "chunked" reader then processes the already-buffered plaintext.
- **Impact:** For a 5 GB upload, the gateway allocates 5 GB of heap before encryption begins. This creates a trivial remote memory-exhaustion DoS — an attacker simply uploads large objects.
- **Reproduction:** Upload a 2 GB object with chunked mode enabled. Monitor heap allocation via pprof; observe a single 2 GB allocation in `encryptChunked`.
- **Remediation:**
  1. Pass the raw `io.Reader` directly into `newChunkedEncryptReader` without the `io.ReadAll` step.
  2. The chunked reader (`internal/crypto/chunked.go`) already handles streaming correctly; the full-buffer step in `encryptChunked` is vestigial.
  3. For the metadata-fallback path, stream the chunked output into a pipe or use a bounded buffer to build the fallback payload without full materialization.
- **References:** O'Reilly "Systems Performance" (Gregg) — Memory-bound vs. streaming architectures; Go best practices for large file handling.

### [DAF-05] KMS Key Length Not Validated in Chunked Decrypt Paths
- **Severity:** HIGH
- **CVSS Estimate:** 7.5 (High)
- **Files:** `internal/crypto/engine.go` (`decryptChunked`, lines ~1175-1201; `DecryptRange`, lines ~1348-1374)
- **Description:** The legacy `Decrypt()` function validates KMS-unwrapped key size and rejects wrong-sized keys (line ~640). However, `decryptChunked()` and `DecryptRange()` **omit this validation** in the KMS branch and fall through to the post-branch `adjustedKey` padding logic (DAF-02). A KMS returning an unexpectedly short key is silently padded instead of rejected.
- **Impact:** Same as DAF-02 — silently weakened encryption when a KMS misbehaves, but in the decrypt path where the operator may not notice.
- **Remediation:**
  1. Add explicit key length validation immediately after KMS `UnwrapKey` in all three decrypt functions.
  2. Remove the `adjustedKey` padding logic entirely (see DAF-02).
- **References:** Same as DAF-02.

## Medium Severity Findings

### [DAF-06] Tracing Middleware Blindly Trusts X-Forwarded-For / X-Real-IP
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/middleware/tracing.go` (lines ~138-152)
- **Description:** `getRemoteAddr` in the tracing middleware unconditionally uses `X-Real-IP` and the **leftmost** IP in `X-Forwarded-For` without consulting the `TrustedProxies` configuration. While `V1.0-SEC-6` hardened `getClientIP` and `getClientKey`, the tracing path was **missed entirely**.
- **Impact:** Attackers can spoof client IPs in distributed traces, polluting observability data, evading trace-based security analytics, and causing operators to misattribute traffic.
- **Remediation:** Refactor `getRemoteAddr` to accept an `*util.IPExtractor` (the one already configured with `TrustedProxies`). If no extractor is configured, fall back to `RemoteAddr`.
- **References:** O'Reilly "Defending APIs" (Domoney) — Chapter 3: Source IP validation and proxy trust.

### [DAF-07] OpenTelemetry Spans Leak Presigned URL Signatures
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/middleware/tracing.go` (lines ~32-34, ~50-56)
- **Description:** The tracing middleware adds `semconv.HTTPURL(r.URL.String())` to every span. Even when `redactSensitive=true`, this attribute still contains the **full unredacted query string**, including presigned URL signatures (`X-Amz-Signature`), temporary credentials, and tokens. A separate `http.query` attribute exists but does not protect `HTTPURL`.
- **Impact:** Trace backends (Jaeger, OTLP) often have different retention/access controls than application logs. Sensitive tokens can be persisted in trace storage and exposed to operators or compromised trace pipelines.
- **Remediation:**
  1. Strip or redact the query string from `HTTPURL` regardless of the `redactSensitive` flag. Use `r.URL.Path` + a redacted query summary.
  2. Parse the URL and only include specific known-safe query parameters (e.g., `versionId`) if needed.
- **References:** O'Reilly "Building Secure and Reliable Systems" — Chapter 11: Observability and data classification; "Secure APIs" (Haro) — Logging and tracing hygiene.

### [DAF-08] Debug Printf in S3 Client Leaks Metadata Previews
- **Severity:** MEDIUM (downgraded from HIGH due to requiring debug mode)
- **CVSS Estimate:** 3.7 (Low-Medium)
- **Files:** `internal/s3/client.go` (lines ~413-434, ~492-512)
- **Description:** Despite V1.0-SEC-3 (which removed `fmt.Printf` from `internal/crypto/engine.go`), the S3 client still contains `fmt.Printf` debug blocks gated by `debug.Enabled()` that emit previews (up to 30 characters) of encryption metadata values: salt, IV, wrapped key, and KMS key ID.
- **Impact:** If debug mode is accidentally enabled in production (accident, misconfiguration, or during an incident), these prints write cryptographic metadata to stdout/stderr, where it may be captured by container log aggregators.
- **Remediation:**
  1. Replace all `fmt.Printf` calls in `client.go` with `slog.Debug(...)` and emit **only lengths**, never value previews.
  2. Add a CI linter gate (`grep -n 'fmt.Printf.*debug.Enabled'`) to prevent regression.
- **References:** V1.0-SEC-3 implementation guide; O'Reilly "Crafting Secure Software" — Debug mode hygiene.

### [DAF-09] Password Loaded into Immutable String Before Engine Copy
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `cmd/server/main.go` (lines ~381, ~398)
- **Description:** The encryption password is loaded from config/file into a Go `string` (`encryptionPassword`), then passed to the engine constructor which copies it into `[]byte`. Go strings are immutable and cannot be securely wiped from memory. The original string remains in the heap until GC and may be interned or copied by the runtime.
- **Impact:** Even though V1.0-SEC-1 zeroized the engine's internal `[]byte`, the password still exists in at least one other location (the `activePassword` string) that lives for the lifetime of the process.
- **Remediation:**
  1. Read the password file directly into `[]byte` using `os.ReadFile`.
  2. Immediately copy it into a separately allocated buffer, then zero the original.
  3. Pass only `[]byte` to engine constructors. Do not assign it to a `string` variable.
  4. Consider using `memguard` or similar for sensitive buffer management.
- **References:** O'Reilly "Security with Go" (Leon) — Chapter 3: Memory safety and secure deletion; "Cryptography Engineering" (Ferguson et al.) — Key lifetime management.

### [DAF-10] Cached Policy Engines Never Closed — Passwords Linger
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/api/handlers.go` (lines ~44, ~647-731)
- **Description:** `getEncryptionEngine` caches per-policy engines in `h.engineCache` (a `sync.Map`). These engines are never evicted, closed, or zeroized. If a policy is changed or removed, the old engine (and its password bytes) remains in memory indefinitely.
- **Impact:** This multiplies the password exposure from DAF-09: every unique policy ID leaves a permanent copy of the encryption password in heap memory. Old password material from rotated keys also persists.
- **Remediation:**
  1. Implement cache eviction (e.g., TTL of 1 hour, or LRU with max size).
  2. Call `engine.Close()` when an entry is evicted, which zeroizes the password bytes.
  3. Alternatively, do not cache engines that hold long-lived passwords; reconstruct them per-request or use a bounded cache with explicit cleanup.
- **References:** O'Reilly "Building Secure and Reliable Systems" — Secret lifecycle management; Go `sync.Map` patterns.

### [DAF-11] Admin Listener Lacks MaxHeaderBytes
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/admin/server.go` (lines ~89-95)
- **Description:** The admin `http.Server` does not set `MaxHeaderBytes`. The data-plane server sets it via `cfg.Server.MaxHeaderBytes` (default 1 MB), but the admin endpoint uses Go's default (1 MB) without configuration.
- **Impact:** An attacker with a valid bearer token (or exploiting a future auth bypass) can send requests with multi-megabyte headers to exhaust memory on the admin listener.
- **Remediation:** Set `MaxHeaderBytes` on the admin `http.Server`, either hardcoded to a reasonable limit (e.g., 64 KB) or aligned with the data-plane configuration.
- **References:** O'Reilly "Secure APIs" (Haro) — HTTP server hardening; Go `http.Server` documentation.

### [DAF-12] Admin Token File Re-read on Every Request
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/admin/server.go` (lines ~165-182)
- **Description:** `buildTokenSource` returns a closure that calls `os.ReadFile(path)` on **every authenticated request**. This performs a disk read + string allocation for every admin API call.
- **Impact:**
  1. **DoS amplification:** An attacker with a valid token can flood the admin API, causing excessive disk I/O.
  2. The runtime reader does not re-validate file permissions. If permissions are relaxed after startup, the server silently continues using the file.
- **Remediation:**
  1. Read the token once at startup, cache it in memory.
  2. Use `fsnotify` or a periodic refresh (e.g., every 30 seconds) if runtime rotation is required.
  3. Re-validate permissions on each refresh and emit a warning if they change.
- **References:** O'Reilly "API Security in Action" (Madden) — Token caching and rotation patterns.

### [DAF-13] TLS Configurations Allow Weak Default Cipher Suites
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.0 (Medium)
- **Files:** `internal/admin/server.go` (~line 116), `internal/api/crypto_factory.go` (~line 122)
- **Description:** Both the admin listener and the Cosmian KMS TLS config set `MinVersion: tls.VersionTLS12` but do not restrict `CipherSuites` or `CurvePreferences`. Go's default TLS 1.2 suite includes CBC-mode ciphers.
- **Impact:** A network attacker could potentially force a downgrade to a weaker TLS 1.2 cipher suite during the handshake.
- **Remediation:** Explicitly set `CipherSuites` to a modern list:
  ```go
  CipherSuites: []uint16{
      tls.TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384,
      tls.TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384,
      tls.TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305,
      tls.TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305,
      tls.TLS_AES_256_GCM_SHA384, // TLS 1.3 (mandatory, but explicit for clarity)
      tls.TLS_CHACHA20_POLY1305_SHA256,
  },
  CurvePreferences: []tls.CurveID{tls.X25519, tls.P256},
  ```
- **References:** O'Reilly "Network Security Assessment" (McNab) — TLS cipher suite hardening; Mozilla SSL Configuration Generator.

### [DAF-14] RecoveryMiddleware Innermost — Panics in Outer Middleware Crash Server
- **Severity:** MEDIUM
- **CVSS Estimate:** 5.3 (Medium)
- **Files:** `cmd/server/main.go` (lines ~607-609)
- **Description:** `RecoveryMiddleware` is the **first** middleware applied, making it the **innermost** wrapper. Panics in `LoggingMiddleware`, `SecurityHeadersMiddleware`, `TracingMiddleware`, `BucketValidationMiddleware`, or `RateLimitMiddleware` will **bypass recovery** and terminate the server goroutine.
- **Impact:** A panic in any outer middleware (e.g., a nil pointer during header processing) will crash the entire HTTP server, causing a denial of service for all concurrent requests.
- **Remediation:** Move `RecoveryMiddleware` to be the **outermost** middleware (applied last in the chain), so it wraps all other layers and catches panics from any middleware or handler.
- **References:** O'Reilly "Building Secure and Reliable Systems" — Fault isolation and recovery; Go middleware ordering best practices.

### [DAF-15] Multipart Upload Encryption Hardcodes AES256-GCM
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/api/handlers.go` (~line 2375), `internal/crypto/mpu_encrypter.go` (lines ~62-83)
- **Description:** `initMPUEncryptionState` hardcodes `Algorithm: "AES256GCM"`, and `NewMPUPartEncryptReader` always constructs an AES-GCM cipher. If the gateway is configured to use `ChaCha20-Poly1305`, MPU uploads ignore the policy.
- **Impact:** This violates the policy-driven algorithm selection and creates an architectural inconsistency. FIPS or compliance requirements mandating ChaCha20 are not honored for multipart uploads.
- **Remediation:** Thread the configured algorithm from `engine.preferredAlgorithm` through to `UploadState` and `NewMPUPartEncryptReader`. Support both AES-GCM and ChaCha20-Poly1305 in the MPU encrypt/decrypt paths.
- **References:** O'Reilly "Cryptography Engineering" — Algorithm agility and configuration consistency.

### [DAF-16] Audit FileSink Uses World-Readable Permissions (0644)
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.0 (Medium)
- **Files:** `internal/audit/sink.go` (~line 334)
- **Description:** `FileSink.WriteEvent` opens the audit log with `0644` permissions.
- **Impact:** Audit logs contain sensitive operational data (bucket names, object keys, client IPs, operation types). World-readable files violate the principle of least privilege and may expose data to compromised local users or sidecar containers.
- **Remediation:** Change the mode to `0600` and consider adding a config option for operators to override it.
- **References:** O'Reilly "Building Secure and Reliable Systems" — Logging permissions and data classification; CIS benchmarks.

### [DAF-17] `encryptChunkedWithMetadataFallback` Double-Buffers Plaintext + Ciphertext
- **Severity:** MEDIUM
- **CVSS Estimate:** 4.3 (Medium)
- **Files:** `internal/crypto/engine.go` (lines ~956-1124)
- **Description:** This function receives the entire plaintext as a `[]byte` argument, then at line ~1033 creates a chunked encrypt reader, and at line ~1036 fully buffers the ciphertext with `io.ReadAll(chunkedReader)`. This results in **both plaintext and ciphertext fully resident in memory simultaneously**, doubling peak memory usage.
- **Impact:** For large objects with metadata fallback enabled, the gateway requires roughly `2x object size` of heap memory, increasing the DoS surface.
- **Remediation:** Stream the chunked output through a pipe or use bounded buffering to avoid simultaneous full materialization of plaintext and ciphertext.
- **References:** Same as DAF-04.

## Low Severity Findings

### [DAF-18] Admin Token File Permission Check Follows Symlinks (TOCTOU)
- **Severity:** LOW
- **CVSS Estimate:** 3.1 (Low)
- **Files:** `internal/config/config.go` (~line 1345)
- **Description:** The admin token file permission check uses `os.Stat`, which follows symbolic links. A symlink pointing to a file with `0600` will pass the check, even if the symlink itself is attacker-controlled.
- **Impact:** A local attacker with the ability to create symlinks could potentially redirect the token read to a file they control after validation, or confuse operators about which file is actually being read.
- **Remediation:** Use `os.Lstat` for the permission check (to inspect the symlink itself) and require that the path not be a symlink, or validate that the final resolved file is within a trusted directory.
- **References:** O'Reilly "Secure APIs" (Haro) — TOCTOU and filesystem security; CWE-363.

## Project Security Standards Statement

### Strengths
1. **Formal Threat Modeling:** A comprehensive STRIDE-based threat model is documented in `docs/SECURITY_AUDIT.md`, demonstrating proactive security thinking.
2. **Crypto Hygiene:** The codebase uses `crypto/rand` exclusively (V1.0-SEC-7), implements constant-time comparisons (`subtle.ConstantTimeCompare`, `hmac.Equal`), and has made deliberate efforts toward memory zeroization (V1.0-SEC-1).
3. **Issue Tracking:** Security issues are tracked with severity ratings, implementation guides, and test criteria — a hallmark of mature security engineering.
4. **Fail-Closed Posture:** Examples include refusing to start if encrypted MPU is required but Valkey is unconfigured, and defaulting `TrustedProxies` to empty.

### Weaknesses (Pre-v1.0 Blockers)
1. **Incomplete Fix Scope:** V1.0-SEC-6 fixed XFF spoofing in two files but missed `tracing.go` (DAF-06). V1.0-SEC-3 removed debug prints from `engine.go` but left identical patterns in `s3/client.go` (DAF-08).
2. **Cryptographic Edge Cases:** The key-padding fallback (DAF-02) and missing KMS key length validation (DAF-05) create dangerous silent failures.
3. **DoS Vectors:** Unbounded goroutines (DAF-03) and full-buffering before chunked encryption (DAF-04, DAF-17) are exploitable resource-exhaustion bugs.
4. **Authentication Gaps:** Missing SigV4 replay protection (DAF-01) violates AWS best practices.

### O'Reilly Best Practices Alignment
| Best Practice | Alignment | Notes |
|---|---|---|
| Defense in Depth | Good | Multiple security layers present. Needs replay protection (DAF-01) and resource bounds (DAF-03, DAF-04). |
| Fail-Safe Defaults | Good | Insecure settings emit ERROR warnings. Key padding should fail closed (DAF-02). |
| Complete Mediation | Partial | Auth is mediated, but replay window missing (DAF-01). Recovery doesn't cover all layers (DAF-14). |
| Least Privilege | Partial | FileSink 0644 (DAF-16), token symlinks (DAF-18) need hardening. |
| Economy of Mechanism | Good | Minimal attack surface; remove remaining `fmt.Printf` debug blocks (DAF-08). |
| Secure Key Handling | Partial | Zeroization exists but originals persist as strings (DAF-09) and cache is unbounded (DAF-10). |
| Observability Without Exposure | Partial | Log redaction is good; trace spans leak data (DAF-07). |
| TLS Hardening | Partial | TLS 1.2 minimum set; ciphers should be explicit (DAF-13). |

## Recommended Action Plan

### Phase 1 — Critical (Block v1.0)
1. **DAF-02:** Remove key-padding fallback entirely; fail closed on key size mismatch.
2. **DAF-01:** Add SigV4 timestamp validation with configurable clock skew (default 15 min).
3. **DAF-05:** Add KMS key length validation in all decrypt paths (chunked, range).
4. **DAF-03:** Replace unbounded goroutines in `BatchSink` with bounded worker pool.
5. **DAF-04:** Eliminate `io.ReadAll(reader)` in `encryptChunked`; true streaming.

### Phase 2 — High Priority (v1.0 or v1.0.1)
6. **DAF-14:** Move `RecoveryMiddleware` to outermost position in middleware chain.
7. **DAF-09:** Load password as `[]byte` from file; avoid intermediate `string`.
8. **DAF-10:** Implement cache eviction with `engine.Close()` for policy engine cache.
9. **DAF-15:** Thread `preferredAlgorithm` through MPU encryption paths.
10. **DAF-08:** Remove remaining `fmt.Printf` debug blocks from `s3/client.go`.

### Phase 3 — Medium Priority (v1.0.x)
11. **DAF-06:** Align tracing `getRemoteAddr` with `TrustedProxies` logic.
12. **DAF-07:** Redact query strings from `HTTPURL` span attributes.
13. **DAF-11:** Set `MaxHeaderBytes` on admin server.
14. **DAF-12:** Cache admin token in memory with periodic refresh.
15. **DAF-13:** Explicitly configure TLS cipher suites.
16. **DAF-16:** Change audit FileSink permissions to `0600`.
17. **DAF-17:** Avoid double-buffering in metadata fallback encryption.
18. **DAF-18:** Use `os.Lstat` for token file permission checks.
