# Milestone v0.6 ‚Äî Issue Tracker and Implementation Guide

This document tracks all issues for v0.6 with implementation guides, tests, and acceptance criteria derived from `docs/ROADMAP.md`.

## Scope and Goals
- Security/Crypto: Pluggable KeyManager, optional FIPS-compliant crypto build profile
- S3 Compatibility: Multipart copy; Object Lock/Retention passthrough guidance
- Performance: Zero-copy streaming paths where feasible; retry policy customization
- Observability: pprof endpoints gated behind admin flag; profiling recipes
- Operations & Helm/K8s: Blue/green and canary examples; values validation/schema
- Config & UX: Admin API for safe key rotation (drain + cutover)
- Testing & Quality: Performance baselines; coverage gate >=80%; mutation testing on crypto-free critical code
- Docs: Migration guide for enabling KMS mode from single-password

## Milestone Acceptance Criteria
- Coverage gate (>=80%) and mutation testing on critical crypto-free code.
- Helm values validation (schema/JSONSchema) and default overlays.

## [V0.6-SEC-3] Encrypted Multipart Uploads (Per-Upload DEK + Finalization Manifest) ‚úÖ COMPLETE
- Owner: Crypto / API
- Priority: P0
- Status: **Implemented** ‚Äî 2026-04-20
- Labels: `milestone:v0.6`, `area:crypto`, `area:s3`, `area:security`
- Plan: `docs/plans/V0.6-SEC-3-plan.md`
- ADR: `docs/adr/0009-encrypted-multipart-uploads.md`
- Depends on: V0.6-SEC-1 (pluggable KeyManager ‚Äî complete),
  V0.6-S3-1 (UploadPartCopy dispatch ‚Äî complete),
  V0.6-CFG-1 (admin API for key rotation ‚Äî complete).

### Summary
Close the plaintext-at-rest hole for S3 multipart uploads documented in
ADR 0002 and flagged in `README.md:700`. Every part of every multipart
upload is currently stored **unencrypted** on the backend because
`internal/api/handlers.go:2211-2295` buffers and forwards the part body
verbatim. AWS SDK / CLI / boto3 auto-multipart kicks in at 5‚Äì16 MiB,
so any object ‚â• ~8 MiB under default client behaviour lands
unencrypted ‚Äî a headline defect for a gateway whose value proposition
is transparent encryption at rest.

Adopt the per-upload DEK + HKDF-derived per-part IV + finalization
manifest design (ADR 0009). State lives in **Valkey** (Redis-
protocol compatible, BSD-3 licensed) keyed
`mpu:<base64url(sha256(uploadId))>` with a 7-day TTL refreshed on
every `UploadPart`. Valkey is a required infrastructure dependency
when `EncryptMultipartUploads=true`; deployments that cannot run
Valkey use `Server.DisableMultipartUploads=true` (5 GiB single-PUT
ceiling, no state needed). Opt-in per bucket via
`PolicyConfig.EncryptMultipartUploads` in v0.6; default flip to
`true` in v0.7 after a soak.

### Motivation
1. **Plaintext at rest for the default client path.** AWS CLI / boto3
   / `aws-sdk-go-v2` / `minio-go` all auto-multipart ‚â• 5‚Äì16 MiB. A
   gateway that only encrypts single PUTs is silently leaking the
   majority of real-world data.
2. **The existing fail-closed option breaks defaults.**
   `DisableMultipartUploads=true` caps objects at 5 GiB and makes
   every mainstream client's default multipart flow fail. Not a
   real solution for production.
3. **Every primitive needed is already in-tree.** Chunked AEAD
   (`internal/crypto/chunked.go`), envelope encryption
   (`KeyManager.WrapKey/UnwrapKey`), fallback-pointer metadata
   (`engine.go:961+`), ranged decrypt
   (`engine.DecryptRange`, engine.go:1263) ‚Äî the gap is wiring, not
   crypto research.
4. **The rollout is safe.** Opt-in per-bucket, feature-flagged,
   byte-for-byte backward-compat when the flag is off. The
   `DisableMultipartUploads` fail-closed escape hatch is preserved
   unchanged for FIPS-restricted deployments.

### Implementation Guide (7 phases ‚Äî full detail in plan)
- [ ] **Phase A ‚Äî Crypto primitives.** `mpu_iv.go`
  (HKDF-SHA256 ‚Üí 12-byte IV), `mpu_manifest.go` (schema, round-trip,
  size budget), `Manifest` interface for polymorphism. KAT locked
  in tree.
- [ ] **Phase B ‚Äî Valkey state store.** `StateStore` interface,
  `ValkeyStateStore` using `github.com/redis/go-redis/v9`
  against Valkey 8.x. Unit tests via miniredis. TLS 1.3
  required in production; startup fails closed if
  `EncryptMultipartUploads=true` + plaintext Valkey +
  `insecure_allow_plaintext=false` (default). Orphan cleanup
  via native TTL ‚Äî no reaper daemon needed. `HealthCheck`
  surfaced on `/readyz`.
- [ ] **Phase C ‚Äî Handler wiring (happy path).** Thread DEK + state
  through Create / Upload / Complete / Abort.
  `EncryptMultipartUploads=false` default ‚Üí zero behaviour change.
- [ ] **Phase D ‚Äî GET + range decoding.** Extend `DecryptRange` +
  `CalculateEncryptedRangeForPlaintextRange` to consume the
  polymorphic manifest. Tamper-detection tests
  (reorder, truncate) are mandatory.
- [ ] **Phase E ‚Äî UploadPartCopy into encrypted MPU.** All three
  source classes (plaintext / chunked / legacy) re-encrypt under
  destination DEK. Plaintext fast path disabled for encrypted
  destinations.
- [ ] **Phase F ‚Äî Metrics, audit, admin.** Six Prometheus metrics,
  six audit events, two admin endpoints. Runbook addendum.
- [ ] **Phase G ‚Äî Integration matrix.** MinIO end-to-end + at-rest
  ciphertext verification; AWS CLI, boto3, `aws-sdk-go-v2`,
  `minio-go` √ó {1 MiB, 8 MiB, 100 MiB, 500 MiB}; docs updated;
  CHANGELOG entry.

### Security-critical properties (non-negotiable)
- [ ] **IV uniqueness** structurally guaranteed by
  `HKDF-Expand(DEK, salt=sha256(uploadId),
  info=iv_prefix || be32(part) || be32(chunk))`. No counter reuse
  possible across parts. Per AES-GCM nonce-uniqueness requirement
  (NIST SP 800-38D ¬ß8.2;
  *Serious Cryptography* 2nd ed. ¬ß9.3, 9.5).
- [ ] **Part reorder / truncate attacks** produce AEAD tag
  verification failures ‚Üí 500 InternalError + audit event; never
  silent mis-decryption.
- [ ] **Wrapped DEK in Valkey is ciphertext only.** Recovering
  plaintext still requires compromising the `KeyManager` (Cosmian
  KMIP / HSM). Valkey-read-only access discloses no plaintext.
  TLS 1.3 required on the wire.
- [ ] **Manifest written atomically with Complete** ‚Äî supplied as
  metadata to the backend `CompleteMultipartUpload` call, so no
  inter-call crash window between manifest persistence and object
  finalization.
- [ ] **Opt-in flag snapshot captured at Create** ‚Äî policy flips
  mid-upload do not affect in-flight MPUs; test asserts this.
- [ ] **FIPS-approved primitives only** ‚Äî AES-256-GCM +
  HKDF-SHA256 (ADR 0005 profile unchanged).

### Known Limitations (documented, not blockers)
- **Valkey is a required infrastructure dependency** when
  `EncryptMultipartUploads=true`. Deployments that cannot run
  Valkey use `DisableMultipartUploads=true` instead (5 GiB
  single-PUT ceiling).
- **Valkey AOF loss window ‚â§ 1 s** ‚Äî deterministic IV schedule
  makes client retries byte-identical; `Complete` validation
  against per-part `etag` catches and recovers from missing
  state rows.
- **Cryptographic rollback protection for Valkey state** is a
  v0.7 hardening (noted in ADR 0009 ¬ßSecurity Considerations and
  ¬ßFuture Considerations #5).
- **Default `EncryptMultipartUploads=false`** in v0.6; operators
  opt-in per bucket. v0.7 flips default after soak.
- **Minimum-storage-duration backends** (Wasabi, AWS Glacier /
  Standard-IA / One Zone-IA) are the reason backend-object state
  was rejected; Valkey avoids the cost. See ADR 0009
  ¬ßAlternatives E and ¬ßReference 13.

## [V0.6-PERF-1] Zero-Copy Streaming Where Feasible
- Owner: Perf
- Priority: P1
- Labels: `milestone:v0.6`, `area:perf`, `area:crypto`, `area:s3`

### Summary
Audit and eliminate full-object in-memory buffers on the hot data paths
(GET, PUT, CopyObject, UploadPart, UploadPartCopy). Replace with streaming
pipelines using `io.Pipe` / `io.CopyBuffer` while preserving AEAD
authenticity guarantees and existing chunked-envelope correctness.

### Motivation
Several handlers currently perform `io.ReadAll` over whole-object payloads
and then re-wrap the bytes in `bytes.NewReader` before emitting them to
the next stage. For large objects this materialises two or more
full-object copies in heap, dominates GC pressure, and caps practical
throughput well below what the crypto primitives sustain. The most
egregious paths were documented as follow-ups during V0.6-S3-1 review.

### Audit of full-buffer copy points (baseline for the Phase A work)
Primary hot paths (file:line, current behaviour, suggested fix):

- `internal/api/handlers.go:854` ‚Äî `handleGetObject`: range-request path
  `io.ReadAll(decryptedReader)` into `decryptedData` before slicing.
  Replace with a `bytes.Reader` slice on an already-chunked source;
  for legacy single-AEAD, keep the full read but cap via
  `Server.MaxLegacyCopySourceBytes` (reuse the V0.6-S3-1 knob).
- `internal/api/handlers.go:941` ‚Äî optimised range path
  `io.ReadAll(decryptedReader)` into `outputData` immediately before
  `w.Write(...)`. Replace with `io.CopyBuffer(w, decryptedReader, buf)`
  from the existing buffer pool.
- `internal/api/handlers.go:2618` + `:2675` ‚Äî `handleCopyObject`: double
  `io.ReadAll` (source plaintext ‚Üí destination ciphertext). Stream
  source‚Üídecrypt‚Üíencrypt‚Üíupload via a single `io.Pipe` chain; the AWS
  SDK accepts an `io.Reader` directly. Chunked sources/destinations
  remain correct because the chunked envelope is itself streaming.
- `internal/api/handlers.go:2242` ‚Äî `handleUploadPart`: `io.ReadAll(r.Body)`
  to satisfy the SDK's seekable-body contract. Replace with a
  seekable-body wrapper (`bytes.Reader` over a bounded pool buffer, or
  a retryable readSeeker built from `http.Request.Body`) so the part
  bytes never materialise in a new allocation.
- `internal/api/upload_part_copy.go:557` ‚Äî legacy UploadPartCopy
  fallback: `io.ReadAll(LimitReader(decryptedReader, cap+1))`. Streaming
  is non-trivial here because legacy is a single AEAD; leave as-is but
  document that the cap (default 256 MiB, per ADR 0006) bounds the
  allocation. No code change, just an explanatory comment linking to
  this task.
- `internal/crypto/engine.go:330,722,874,997,1087,1380,1500,1608,1701` ‚Äî
  `bytes.NewReader(plaintext)` inside `engine.Encrypt` / `Decrypt`
  compression branches. Compression already exposes streaming
  APIs (`crypto/compression.go:128,195,210` use `io.ReadAll` internally
  too ‚Äî those should be converted to streaming transforms: gzip/zstd
  readers/writers accept `io.Reader`/`io.Writer` directly).
- `internal/s3/proxy_client.go` ‚Äî `CopyObject` and `UploadPartCopy`
  already forward streaming, but the caller in `handleCopyObject`
  buffers; fixing the caller is sufficient here.

### Affected
- `internal/api/handlers.go` ‚Äî `handleGetObject`, `handleCopyObject`,
  `handleUploadPart`
- `internal/api/upload_part_copy.go` ‚Äî documentation comment only
- `internal/crypto/engine.go` ‚Äî compression branch refactors
- `internal/crypto/compression.go` ‚Äî convert gzip/zstd transforms to
  streaming
- `internal/s3/client.go` ‚Äî possibly a `ReadSeeker` helper for
  part-bodies

### Implementation Guide
- [ ] **Phase A ‚Äî Audit & baseline.** Capture `go test -bench` output for
  `BenchmarkEncrypt*`, `BenchmarkDecrypt*`, and add
  `BenchmarkGetObject_Large`, `BenchmarkCopyObject_Large`,
  `BenchmarkUploadPartCopy_*` fixtures (10 MiB, 100 MiB, 1 GiB payloads
  ‚Äî the 1 GiB runs gated on `-tags=large`). Record heap-in-use and
  CPU time as the pre-change baseline.
- [ ] **Phase B ‚Äî `handleGetObject` streaming.** Replace the two
  `io.ReadAll` sites (`:854`, `:941`) with `io.CopyBuffer` against the
  existing `h.metrics.RecordBufferPoolHit`-backed pool. Preserve
  Content-Length header emission (precompute length from the chunked
  manifest).
- [ ] **Phase C ‚Äî `handleCopyObject` streaming pipeline.** Replace the
  `ReadAll` pair at `:2618,:2675` with an `io.Pipe` chain:
  `source ‚Üí Decrypt ‚Üí Encrypt ‚Üí PutObject`. The SDK's `PutObject`
  accepts an `io.Reader` without seekability when `ContentLength` is
  supplied; compute length from the destination chunked manifest
  ahead of time (same formula already at `handlers.go:2217-2232`).
- [ ] **Phase D ‚Äî `handleUploadPart` seekable-body wrapper.** Add a
  pooled `*bytes.Reader` helper in `internal/s3/` that re-reads from
  a bounded buffer; replace the `io.ReadAll(r.Body)` at `:2242`.
  Bound by `Server.MaxPartBuffer` (new knob, default 64 MiB ‚Äî smaller
  than the 5 GiB S3 per-part cap because larger parts are rare and
  should stream directly; reject or chunk them with a clear error).
- [ ] **Phase E ‚Äî Compression streaming.** Convert
  `crypto/compression.go:{128,195,210}` from `io.ReadAll` to the
  `gzip.Reader` / `gzip.Writer` (and zstd equivalents) streaming
  APIs. Most of the rewrite is replacing the loop, not redesigning
  the contract ‚Äî the public
  `CompressionEngine.Compress/Decompress` signatures already return
  `io.Reader`.
- [ ] **Phase F ‚Äî Engine compression-branch refactors.** Replace the
  `bytes.NewReader(compressedData)` assignments in `engine.go`
  `Encrypt`/`Decrypt` with direct pipe of the streaming compression
  reader into the AEAD writer. Must preserve the
  `x-amz-meta-encryption-original-size` metadata emission.
- [ ] **Phase G ‚Äî Audit trail.** Add a runtime assertion (test-only) that
  exercises each changed path with a tracking `io.Reader` wrapper
  asserting peak in-flight buffered bytes ‚â§
  `2 √ó crypto.DefaultChunkSize + pool_buffer_size` (~130 KiB). Fails
  the test if any path regresses back to full-buffer behaviour.

### Affected configuration
- New: `Server.MaxPartBuffer` (Phase D). Default 64 MiB. Parts larger
  than the cap are either streamed directly (if the backend SDK
  supports it without seek) or rejected with a clear
  `501 NotImplemented` pointing at chunked encryption / client-side
  chunking.
- Reused: `Server.MaxLegacyCopySourceBytes` (introduced in V0.6-S3-1;
  remains the authoritative cap for legacy-AEAD objects that cannot
  be streamed).

### Tests
- [ ] Benchmarks added in Phase A; re-run post-change showing
  **‚â• 50% reduction in allocated bytes/op** and **‚â• 30% reduction in
  wall time** on the 100 MiB fixture. Threshold justifies calling
  this "zero-copy"; anything less and the change is premature.
- [ ] `TestHandleGetObject_Streaming_BoundedHeap` ‚Äî heap assertion.
- [ ] `TestHandleCopyObject_Streaming_BoundedHeap` ‚Äî heap assertion.
- [ ] `TestHandleUploadPart_Streaming_BoundedHeap` ‚Äî heap assertion.
- [ ] Integration (`-tags=integration`) against MinIO with 1 GiB
  objects ‚Äî GET, PUT, CopyObject, UploadPartCopy round-trips match
  byte-for-byte pre/post-refactor.
- [ ] Existing range-request and chunked-decryption suites pass
  unchanged (authenticity is preserved).
- [ ] `-race` clean; `-tags=fips` and `-tags=hsm` builds clean.

### Risks & Mitigations
| Risk | Mitigation |
|---|---|
| Streaming Encrypt before the caller has committed can produce partial writes if the source errors mid-stream. | Use `io.Pipe` with a cancellable context; on source error, propagate via `PipeWriter.CloseWithError`; destination PUT returns a failure and is retried by the client. Existing single-AEAD path already handled this implicitly via `ReadAll`; the pipe must propagate cleanly. |
| Incorrect Content-Length header if precomputation disagrees with actual stream length. | Add a `countingWriter` in front of the backend PUT that asserts bytes written == declared length; fail closed. |
| SDK seekability requirement for certain backends. | Audit: AWS SDK V2 `PutObject` and `UploadPart` accept non-seekable readers when `ContentLength` is set; this is documented and relied on by `minio-go` as well. Non-compliant backends fall back to the current buffer path via capability flag. |
| Compression streaming changes the byte layout if any caller depends on "compressed then AEAD'd in one pass". | The AEAD wrap happens over a `Writer` regardless; streaming only changes where the compression boundary sits. Covered by existing round-trip tests. |

### Definition of Done
- [ ] All copy points in the audit list above either rewritten to
  streaming or explicitly documented as intentionally buffered with
  a capped justification.
- [ ] Benchmark delta recorded in the PR body: ‚â• 50% allocated

- [ ] Integration tests with 1 GiB payloads pass byte-for-byte.

- [ ] No regression in existing range-request or chunked-decryption

## [V0.6-QA-3] Fix Data Race in `TestConfigReloader_FileWatching`
- Owner: QA / Config
- Priority: P1
- Labels: `milestone:v0.6`, `area:testing`, `area:config`, `bug`

### Summary
Fix a pre-existing data race in `TestConfigReloader_FileWatching` that
causes intermittent `-race` failures in the full test suite. The race is
on the `firstCallbackOld` / `firstCallbackNew` captures shared between a
background goroutine (the reload callback) and the main test goroutine
(the assertions).

### Symptom
`go test ./... -count=1 -race -short` fails intermittently with:

```
==================
WARNING: DATA RACE
...
Goroutine 16 (running) created at:
  github.com/kenneth/s3-encryption-gateway/internal/config.TestConfigReloader_FileWatching()
      internal/config/config_reloader_test.go:82 +0x68a
...
--- FAIL: TestConfigReloader_FileWatching (0.31s)
    testing.go:1712: race detected during execution of test
```

The test passes in isolation (`go test ./internal/config/... -race`)
because no scheduling pressure exists. Under the full suite, the
background fsnotify callback goroutine races the assertion thread.

### Root Cause
`internal/config/config_reloader_test.go:70-79`:

```go
var callbackCalled int64
var firstCallbackOld, firstCallbackNew *Config  // ‚Üê unsynchronised
reloader.SetOnReloadCallback(func(old, new *Config) error {
    callCount := atomic.AddInt64(&callbackCalled, 1)
    if callCount == 1 {
        firstCallbackOld = old     // ‚Üê written from callback goroutine
        firstCallbackNew = new     // ‚Üê written from callback goroutine
    }
    return nil
})

go reloader.Start()
// ...
time.Sleep(200 * time.Millisecond)

// Read from main goroutine WITHOUT synchronisation:
assert.NotNil(t, firstCallbackOld)       // ‚Üê races the write above
assert.NotNil(t, firstCallbackNew)       // ‚Üê races the write above
assert.Equal(t, "info", firstCallbackOld.LogLevel)
assert.Equal(t, "debug", firstCallbackNew.LogLevel)
```

The `atomic.AddInt64` on `callbackCalled` synchronises the counter but
NOT the two pointer captures ‚Äî they need their own synchronisation.

### Affected
- `internal/config/config_reloader_test.go` ‚Äî primary fix
- Possibly `TestConfigReloader_SIGHUP` at `:117` ‚Äî same pattern; audit

### Implementation Guide
- [ ] Replace the shared `*Config` captures with a channel-based
  handoff, OR
- [ ] Guard the captures with a `sync.Mutex`, OR
- [ ] Use `sync.Once` + a struct for the first-call capture (cleanest).

Recommended shape (third option):

```go
var firstCall struct {
    sync.Once
    old, new *Config
    done     chan struct{}
}
firstCall.done = make(chan struct{})

reloader.SetOnReloadCallback(func(old, new *Config) error {
    atomic.AddInt64(&callbackCalled, 1)
    firstCall.Do(func() {
        firstCall.old = old
        firstCall.new = new
        close(firstCall.done)
    })
    return nil
})

// In the assertion phase, wait deterministically rather than time.Sleep:
select {
case <-firstCall.done:
case <-time.After(2 * time.Second):
    t.Fatal("timed out waiting for reload callback")
}

assert.NotNil(t, firstCall.old)
assert.Equal(t, "info", firstCall.old.LogLevel)
```

This also removes the `time.Sleep(200*time.Millisecond)` timing
assumption, which is a separate (smaller) flake vector.

- [ ] Apply the same pattern to `TestConfigReloader_SIGHUP` if it
  exhibits the same race on audit.
- [ ] Run `go test ./... -count=100 -race -short` (or `-count=50` if
  100 is too slow) to confirm the flake is eliminated ‚Äî historically
  this test was a 1-in-~20 flake, so 50 iterations is the minimum
  statistically meaningful threshold.

### Tests
- [ ] The existing test keeps its behavioural assertions (config
  values change as expected) ‚Äî this is a fix for the test harness,
  not for production code.
- [ ] Stress run: `go test ./internal/config/... -count=100 -race`
  passes.
- [ ] Full suite: `go test ./... -count=5 -race -short` passes with
  no `DATA RACE` warnings.

### Definition of Done
- [ ] `TestConfigReloader_FileWatching` passes under `-race` in the
  full-suite execution ‚â• 50 consecutive times.
- [ ] `TestConfigReloader_SIGHUP` audited and fixed if it shows the
  same pattern.
- [ ] `time.Sleep`-based waits replaced with deterministic channel
  synchronisation where practical.
- [ ] No production-code changes required (or if they are, they are
  scoped separately with a design review).

### Cross-references
- V0.6-S3-1 implementation review ‚Äî surfaced this flake consistently
  during the full-suite race runs; the fix is a prerequisite for
  reliable CI on the v0.6 milestone.
