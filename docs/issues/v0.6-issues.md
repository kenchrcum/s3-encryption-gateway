# Milestone v0.6 ŌĆö Issue Tracker and Implementation Guide

This document tracks all issues for v0.6 with implementation guides, tests, and acceptance criteria derived from `docs/ROADMAP.md`.

## Scope and Goals
- Security/Crypto: Pluggable KeyManager, optional FIPS-compliant crypto build profile
- S3 Compatibility: Multipart copy; Object Lock/Retention passthrough guidance
- Performance: Zero-copy streaming paths where feasible; retry policy customization
- Observability: pprof endpoints gated behind admin flag; profiling recipes
- Operations & Helm/K8s: Blue/green and canary examples; values validation/schema
- Config & UX: Admin API for safe key rotation (drain + cutover)
- Testing & Quality: Performance baselines; coverage gate >=80%; mutation testing on crypto-free critical code
- Docs: Migration guide for enabling KMS mode from single-password

## Milestone Acceptance Criteria
- Coverage gate (>=80%) and mutation testing on critical crypto-free code.
- Helm values validation (schema/JSONSchema) and default overlays.

## [V0.6-PERF-1] Zero-Copy Streaming Where Feasible
- Owner: Perf
- Priority: P1
- Labels: `milestone:v0.6`, `area:perf`, `area:crypto`, `area:s3`

### Summary
Audit and eliminate full-object in-memory buffers on the hot data paths
(GET, PUT, CopyObject, UploadPart, UploadPartCopy). Replace with streaming
pipelines using `io.Pipe` / `io.CopyBuffer` while preserving AEAD
authenticity guarantees and existing chunked-envelope correctness.

### Motivation
Several handlers currently perform `io.ReadAll` over whole-object payloads
and then re-wrap the bytes in `bytes.NewReader` before emitting them to
the next stage. For large objects this materialises two or more
full-object copies in heap, dominates GC pressure, and caps practical
throughput well below what the crypto primitives sustain. The most
egregious paths were documented as follow-ups during V0.6-S3-1 review.

### Audit of full-buffer copy points (baseline for the Phase A work)
Primary hot paths (file:line, current behaviour, suggested fix):

- `internal/api/handlers.go:854` ŌĆö `handleGetObject`: range-request path
  `io.ReadAll(decryptedReader)` into `decryptedData` before slicing.
  Replace with a `bytes.Reader` slice on an already-chunked source;
  for legacy single-AEAD, keep the full read but cap via
  `Server.MaxLegacyCopySourceBytes` (reuse the V0.6-S3-1 knob).
- `internal/api/handlers.go:941` ŌĆö optimised range path
  `io.ReadAll(decryptedReader)` into `outputData` immediately before
  `w.Write(...)`. Replace with `io.CopyBuffer(w, decryptedReader, buf)`
  from the existing buffer pool.
- `internal/api/handlers.go:2618` + `:2675` ŌĆö `handleCopyObject`: double
  `io.ReadAll` (source plaintext ŌåÆ destination ciphertext). Stream
  sourceŌåÆdecryptŌåÆencryptŌåÆupload via a single `io.Pipe` chain; the AWS
  SDK accepts an `io.Reader` directly. Chunked sources/destinations
  remain correct because the chunked envelope is itself streaming.
- `internal/api/handlers.go:2242` ŌĆö `handleUploadPart`: `io.ReadAll(r.Body)`
  to satisfy the SDK's seekable-body contract. Replace with a
  seekable-body wrapper (`bytes.Reader` over a bounded pool buffer, or
  a retryable readSeeker built from `http.Request.Body`) so the part
  bytes never materialise in a new allocation.
- `internal/api/upload_part_copy.go:557` ŌĆö legacy UploadPartCopy
  fallback: `io.ReadAll(LimitReader(decryptedReader, cap+1))`. Streaming
  is non-trivial here because legacy is a single AEAD; leave as-is but
  document that the cap (default 256 MiB, per ADR 0006) bounds the
  allocation. No code change, just an explanatory comment linking to
  this task.
- `internal/crypto/engine.go:330,722,874,997,1087,1380,1500,1608,1701` ŌĆö
  `bytes.NewReader(plaintext)` inside `engine.Encrypt` / `Decrypt`
  compression branches. Compression already exposes streaming
  APIs (`crypto/compression.go:128,195,210` use `io.ReadAll` internally
  too ŌĆö those should be converted to streaming transforms: gzip/zstd
  readers/writers accept `io.Reader`/`io.Writer` directly).
- `internal/s3/proxy_client.go` ŌĆö `CopyObject` and `UploadPartCopy`
  already forward streaming, but the caller in `handleCopyObject`
  buffers; fixing the caller is sufficient here.

### Affected
- `internal/api/handlers.go` ŌĆö `handleGetObject`, `handleCopyObject`,
  `handleUploadPart`
- `internal/api/upload_part_copy.go` ŌĆö documentation comment only
- `internal/crypto/engine.go` ŌĆö compression branch refactors
- `internal/crypto/compression.go` ŌĆö convert gzip/zstd transforms to
  streaming
- `internal/s3/client.go` ŌĆö possibly a `ReadSeeker` helper for
  part-bodies

### Implementation Guide
- [ ] **Phase A ŌĆö Audit & baseline.** Capture `go test -bench` output for
  `BenchmarkEncrypt*`, `BenchmarkDecrypt*`, and add
  `BenchmarkGetObject_Large`, `BenchmarkCopyObject_Large`,
  `BenchmarkUploadPartCopy_*` fixtures (10 MiB, 100 MiB, 1 GiB payloads
  ŌĆö the 1 GiB runs gated on `-tags=large`). Record heap-in-use and
  CPU time as the pre-change baseline.
- [ ] **Phase B ŌĆö `handleGetObject` streaming.** Replace the two
  `io.ReadAll` sites (`:854`, `:941`) with `io.CopyBuffer` against the
  existing `h.metrics.RecordBufferPoolHit`-backed pool. Preserve
  Content-Length header emission (precompute length from the chunked
  manifest).
- [ ] **Phase C ŌĆö `handleCopyObject` streaming pipeline.** Replace the
  `ReadAll` pair at `:2618,:2675` with an `io.Pipe` chain:
  `source ŌåÆ Decrypt ŌåÆ Encrypt ŌåÆ PutObject`. The SDK's `PutObject`
  accepts an `io.Reader` without seekability when `ContentLength` is
  supplied; compute length from the destination chunked manifest
  ahead of time (same formula already at `handlers.go:2217-2232`).
- [ ] **Phase D ŌĆö `handleUploadPart` seekable-body wrapper.** Add a
  pooled `*bytes.Reader` helper in `internal/s3/` that re-reads from
  a bounded buffer; replace the `io.ReadAll(r.Body)` at `:2242`.
  Bound by `Server.MaxPartBuffer` (new knob, default 64 MiB ŌĆö smaller
  than the 5 GiB S3 per-part cap because larger parts are rare and
  should stream directly; reject or chunk them with a clear error).
- [ ] **Phase E ŌĆö Compression streaming.** Convert
  `crypto/compression.go:{128,195,210}` from `io.ReadAll` to the
  `gzip.Reader` / `gzip.Writer` (and zstd equivalents) streaming
  APIs. Most of the rewrite is replacing the loop, not redesigning
  the contract ŌĆö the public
  `CompressionEngine.Compress/Decompress` signatures already return
  `io.Reader`.
- [ ] **Phase F ŌĆö Engine compression-branch refactors.** Replace the
  `bytes.NewReader(compressedData)` assignments in `engine.go`
  `Encrypt`/`Decrypt` with direct pipe of the streaming compression
  reader into the AEAD writer. Must preserve the
  `x-amz-meta-encryption-original-size` metadata emission.
- [ ] **Phase G ŌĆö Audit trail.** Add a runtime assertion (test-only) that
  exercises each changed path with a tracking `io.Reader` wrapper
  asserting peak in-flight buffered bytes Ōēż
  `2 ├Ś crypto.DefaultChunkSize + pool_buffer_size` (~130 KiB). Fails
  the test if any path regresses back to full-buffer behaviour.

### Affected configuration
- New: `Server.MaxPartBuffer` (Phase D). Default 64 MiB. Parts larger

- Reused: `Server.MaxLegacyCopySourceBytes` (introduced in V0.6-S3-1;
  remains the authoritative cap for legacy-AEAD objects that cannot

### Tests

  wall time** on the 100 MiB fixture. Threshold justifies calling
  this "zero-copy"; anything less and the change is premature.
- [ ] `TestHandleGetObject_Streaming_BoundedHeap` ŌĆö heap assertion.
- [ ] `TestHandleCopyObject_Streaming_BoundedHeap` ŌĆö heap assertion.
- [ ] `TestHandleUploadPart_Streaming_BoundedHeap` ŌĆö heap assertion.
- [ ] Integration (`-tags=integration`) against MinIO with 1 GiB

- [ ] Existing range-request and chunked-decryption suites pass

- [ ] `-race` clean; `-tags=fips` and `-tags=hsm` builds clean.

### Risks & Mitigations
| Risk | Mitigation |
|---|---|
| Streaming Encrypt before the caller has committed can produce partial writes if the source errors mid-stream. | Use `io.Pipe` with a cancellable context; on source error, propagate via `PipeWriter.CloseWithError`; destination PUT returns a failure and is retried by the client. Existing single-AEAD path already handled this implicitly via `ReadAll`; the pipe must propagate cleanly. |
| Incorrect Content-Length header if precomputation disagrees with actual stream length. | Add a `countingWriter` in front of the backend PUT that asserts bytes written == declared length; fail closed. |
| SDK seekability requirement for certain backends. | Audit: AWS SDK V2 `PutObject` and `UploadPart` accept non-seekable readers when `ContentLength` is set; this is documented and relied on by `minio-go` as well. Non-compliant backends fall back to the current buffer path via capability flag. |
| Compression streaming changes the byte layout if any caller depends on "compressed then AEAD'd in one pass". | The AEAD wrap happens over a `Writer` regardless; streaming only changes where the compression boundary sits. Covered by existing round-trip tests. |

- [ ] Benchmark delta recorded in the PR body: Ōēź 50% allocated
  bytes/op reduction and Ōēź 30% wall-time reduction on the 100 MiB
  fixture.
- [ ] Heap-bound assertion tests added for GET, CopyObject,
  UploadPart.

## [V0.6-QA-3] Fix Data Race in `TestConfigReloader_FileWatching`
- Owner: QA / Config
- Priority: P1
- Labels: `milestone:v0.6`, `area:testing`, `area:config`, `bug`

### Summary
Fix a pre-existing data race in `TestConfigReloader_FileWatching` that
causes intermittent `-race` failures in the full test suite. The race is
on the `firstCallbackOld` / `firstCallbackNew` captures shared between a
background goroutine (the reload callback) and the main test goroutine
(the assertions).

### Symptom
`go test ./... -count=1 -race -short` fails intermittently with:

```
==================
WARNING: DATA RACE
...
Goroutine 16 (running) created at:
  github.com/kenneth/s3-encryption-gateway/internal/config.TestConfigReloader_FileWatching()
      internal/config/config_reloader_test.go:82 +0x68a
...
--- FAIL: TestConfigReloader_FileWatching (0.31s)
    testing.go:1712: race detected during execution of test
```

The test passes in isolation (`go test ./internal/config/... -race`)
because no scheduling pressure exists. Under the full suite, the
background fsnotify callback goroutine races the assertion thread.

### Root Cause
`internal/config/config_reloader_test.go:70-79`:

```go
var callbackCalled int64
var firstCallbackOld, firstCallbackNew *Config  // ŌåÉ unsynchronised
reloader.SetOnReloadCallback(func(old, new *Config) error {
    callCount := atomic.AddInt64(&callbackCalled, 1)
    if callCount == 1 {
        firstCallbackOld = old     // ŌåÉ written from callback goroutine
        firstCallbackNew = new     // ŌåÉ written from callback goroutine
    }
    return nil
})

go reloader.Start()
// ...
time.Sleep(200 * time.Millisecond)

// Read from main goroutine WITHOUT synchronisation:
assert.NotNil(t, firstCallbackOld)       // ŌåÉ races the write above
assert.NotNil(t, firstCallbackNew)       // ŌåÉ races the write above
assert.Equal(t, "info", firstCallbackOld.LogLevel)
assert.Equal(t, "debug", firstCallbackNew.LogLevel)
```

The `atomic.AddInt64` on `callbackCalled` synchronises the counter but
NOT the two pointer captures ŌĆö they need their own synchronisation.

### Affected
- `internal/config/config_reloader_test.go` ŌĆö primary fix
- Possibly `TestConfigReloader_SIGHUP` at `:117` ŌĆö same pattern; audit

### Implementation Guide
- [ ] Replace the shared `*Config` captures with a channel-based
  handoff, OR
- [ ] Guard the captures with a `sync.Mutex`, OR
- [ ] Use `sync.Once` + a struct for the first-call capture (cleanest).

Recommended shape (third option):

```go
var firstCall struct {
    sync.Once
    old, new *Config
    done     chan struct{}
}
firstCall.done = make(chan struct{})

reloader.SetOnReloadCallback(func(old, new *Config) error {
    atomic.AddInt64(&callbackCalled, 1)
    firstCall.Do(func() {
        firstCall.old = old
        firstCall.new = new
        close(firstCall.done)
    })
    return nil
})

// In the assertion phase, wait deterministically rather than time.Sleep:
select {
case <-firstCall.done:
case <-time.After(2 * time.Second):
    t.Fatal("timed out waiting for reload callback")
}

assert.NotNil(t, firstCall.old)
assert.Equal(t, "info", firstCall.old.LogLevel)
```

This also removes the `time.Sleep(200*time.Millisecond)` timing
assumption, which is a separate (smaller) flake vector.

- [ ] Apply the same pattern to `TestConfigReloader_SIGHUP` if it
  exhibits the same race on audit.
- [ ] Run `go test ./... -count=100 -race -short` (or `-count=50` if
  100 is too slow) to confirm the flake is eliminated ŌĆö historically
  this test was a 1-in-~20 flake, so 50 iterations is the minimum
  statistically meaningful threshold.

### Tests
- [ ] The existing test keeps its behavioural assertions (config
  values change as expected) ŌĆö this is a fix for the test harness,
  not for production code.
- [ ] Stress run: `go test ./internal/config/... -count=100 -race`
  passes.
- [ ] Full suite: `go test ./... -count=5 -race -short` passes with
  no `DATA RACE` warnings.

### Definition of Done
- [ ] `TestConfigReloader_FileWatching` passes under `-race` in the
  full-suite execution Ōēź 50 consecutive times.
- [ ] `TestConfigReloader_SIGHUP` audited and fixed if it shows the
  same pattern.
- [ ] `time.Sleep`-based waits replaced with deterministic channel
  synchronisation where practical.
- [ ] No production-code changes required (or if they are, they are
  scoped separately with a design review).

### Cross-references
- V0.6-S3-1 implementation review ŌĆö surfaced this flake consistently
  during the full-suite race runs; the fix is a prerequisite for
  reliable CI on the v0.6 milestone.
