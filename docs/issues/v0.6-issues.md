# Milestone v0.6 — Issue Tracker and Implementation Guide

This document tracks all issues for v0.6 with implementation guides, tests, and acceptance criteria derived from `docs/ROADMAP.md`.

## Scope and Goals
- Security/Crypto: Pluggable KeyManager, optional FIPS-compliant crypto build profile
- S3 Compatibility: Multipart copy; Object Lock/Retention passthrough guidance
- Performance: Zero-copy streaming paths where feasible; retry policy customization
- Observability: pprof endpoints gated behind admin flag; profiling recipes
- Operations & Helm/K8s: Blue/green and canary examples; values validation/schema
- Config & UX: Admin API for safe key rotation (drain + cutover)
- Testing & Quality: Performance baselines; coverage gate >=80%; mutation testing on crypto-free critical code
- Docs: Migration guide for enabling KMS mode from single-password

## Milestone Acceptance Criteria
- Coverage gate (>=80%) and mutation testing on critical crypto-free code.
- Helm values validation (schema/JSONSchema) and default overlays.

Labels: `milestone:v0.6`, `area:kms`, `area:s3`, `area:perf`, `area:observability`, `area:helm`, `area:testing`.

## [V0.6-SEC-3] Encrypted Multipart Uploads (Per-Upload DEK + Finalization Manifest) ✅ COMPLETE
- Owner: Crypto / API
- Priority: P0
- Status: **Implemented** — 2026-04-20
- Labels: `milestone:v0.6`, `area:crypto`, `area:s3`, `area:security`
- Plan: `docs/plans/V0.6-SEC-3-plan.md`
- ADR: `docs/adr/0009-encrypted-multipart-uploads.md`
- Depends on: V0.6-SEC-1 (pluggable KeyManager — complete),
  V0.6-S3-1 (UploadPartCopy dispatch — complete),
  V0.6-CFG-1 (admin API for key rotation — complete).

### Summary
Close the plaintext-at-rest hole for S3 multipart uploads documented in
ADR 0002 and flagged in `README.md:700`. Every part of every multipart
upload is currently stored **unencrypted** on the backend because
`internal/api/handlers.go:2211-2295` buffers and forwards the part body
verbatim. AWS SDK / CLI / boto3 auto-multipart kicks in at 5–16 MiB,
so any object ≥ ~8 MiB under default client behaviour lands
unencrypted — a headline defect for a gateway whose value proposition
is transparent encryption at rest.

Adopt the per-upload DEK + HKDF-derived per-part IV + finalization
manifest design (ADR 0009). State lives in **Valkey** (Redis-
protocol compatible, BSD-3 licensed) keyed
`mpu:<base64url(sha256(uploadId))>` with a 7-day TTL refreshed on
every `UploadPart`. Valkey is a required infrastructure dependency
when `EncryptMultipartUploads=true`; deployments that cannot run
Valkey use `Server.DisableMultipartUploads=true` (5 GiB single-PUT
ceiling, no state needed). Opt-in per bucket via
`PolicyConfig.EncryptMultipartUploads` in v0.6; default flip to
`true` in v0.7 after a soak.

### Motivation
1. **Plaintext at rest for the default client path.** AWS CLI / boto3
   / `aws-sdk-go-v2` / `minio-go` all auto-multipart ≥ 5–16 MiB. A
   gateway that only encrypts single PUTs is silently leaking the
   majority of real-world data.
2. **The existing fail-closed option breaks defaults.**
   `DisableMultipartUploads=true` caps objects at 5 GiB and makes
   every mainstream client's default multipart flow fail. Not a
   real solution for production.
3. **Every primitive needed is already in-tree.** Chunked AEAD
   (`internal/crypto/chunked.go`), envelope encryption
   (`KeyManager.WrapKey/UnwrapKey`), fallback-pointer metadata
   (`engine.go:961+`), ranged decrypt
   (`engine.DecryptRange`, engine.go:1263) — the gap is wiring, not
   crypto research.
4. **The rollout is safe.** Opt-in per-bucket, feature-flagged,
   byte-for-byte backward-compat when the flag is off. The
   `DisableMultipartUploads` fail-closed escape hatch is preserved
   unchanged for FIPS-restricted deployments.

### Implementation Guide (7 phases — full detail in plan) ✅ COMPLETE
- [x] **Phase A — Crypto primitives.** `mpu_iv.go`
  (HKDF-SHA256 → 12-byte IV), `mpu_manifest.go` (schema, round-trip,

- [ ] **Phase B — Valkey state store.** `StateStore` interface,

- [ ] **Phase C — Handler wiring (happy path).** Thread DEK + state

- [ ] **Phase F — Metrics, audit, admin.** Six Prometheus metrics,

### Security-critical properties (non-negotiable)
- [ ] **IV uniqueness** structurally guaranteed by
  `HKDF-Expand(DEK, salt=sha256(uploadId),
  info=iv_prefix || be32(part) || be32(chunk))`. No counter reuse
  possible across parts. Per AES-GCM nonce-uniqueness requirement
  (NIST SP 800-38D §8.2;
  *Serious Cryptography* 2nd ed. §9.3, 9.5).
- [ ] **Part reorder / truncate attacks** produce AEAD tag
  verification failures → 500 InternalError + audit event; never
  silent mis-decryption.
- [ ] **Wrapped DEK in Valkey is ciphertext only.** Recovering
  plaintext still requires compromising the `KeyManager` (Cosmian
  KMIP / HSM). Valkey-read-only access discloses no plaintext.
  TLS 1.3 required on the wire.
- [ ] **Manifest written atomically with Complete** — supplied as
  metadata to the backend `CompleteMultipartUpload` call, so no
  inter-call crash window between manifest persistence and object
  finalization.
- [ ] **Opt-in flag snapshot captured at Create** — policy flips
  mid-upload do not affect in-flight MPUs; test asserts this.
- [ ] **FIPS-approved primitives only** — AES-256-GCM +
  HKDF-SHA256 (ADR 0005 profile unchanged).

### Known Limitations (documented, not blockers)
- **Valkey is a required infrastructure dependency** when
  `EncryptMultipartUploads=true`. Deployments that cannot run
  Valkey use `DisableMultipartUploads=true` instead (5 GiB
  single-PUT ceiling).
- **Valkey AOF loss window ≤ 1 s** — deterministic IV schedule
  makes client retries byte-identical; `Complete` validation
  against per-part `etag` catches and recovers from missing
  state rows.
- **Cryptographic rollback protection for Valkey state** is a
  v0.7 hardening (noted in ADR 0009 §Security Considerations and
  §Future Considerations #5).
- **Default `EncryptMultipartUploads=false`** in v0.6; operators
  opt-in per bucket. v0.7 flips default after soak.
- **Minimum-storage-duration backends** (Wasabi, AWS Glacier /
  Standard-IA / One Zone-IA) are the reason backend-object state
  was rejected; Valkey avoids the cost. See ADR 0009
  §Alternatives E and §Reference 13.

## [V0.6-PERF-1] Zero-Copy Streaming Where Feasible ✓ DONE
- Owner: Perf
- Priority: P1
- Labels: `milestone:v0.6`, `area:perf`, `area:crypto`, `area:s3`
- Status: **Done** — implemented by Sonnet 4.6; audited and gap-filled 2026-04-23.

### Summary
Audit and eliminate full-object in-memory buffers on the hot data paths
(GET, PUT, CopyObject, UploadPart, UploadPartCopy). Replace with streaming
pipelines using `io.Pipe` / `io.CopyBuffer` while preserving AEAD
authenticity guarantees and existing chunked-envelope correctness.

### Motivation
Several handlers currently perform `io.ReadAll` over whole-object payloads
and then re-wrap the bytes in `bytes.NewReader` before emitting them to
the next stage. For large objects this materialises two or more
full-object copies in heap, dominates GC pressure, and caps practical
throughput well below what the crypto primitives sustain. The most
egregious paths were documented as follow-ups during V0.6-S3-1 review.

### Audit of full-buffer copy points (baseline for the Phase A work)
Primary hot paths (file:line, current behaviour, suggested fix):

- `internal/api/handlers.go:854` — `handleGetObject`: range-request path
  `io.ReadAll(decryptedReader)` into `decryptedData` before slicing.
  Replace with a `bytes.Reader` slice on an already-chunked source;
  for legacy single-AEAD, keep the full read but cap via
  `Server.MaxLegacyCopySourceBytes` (reuse the V0.6-S3-1 knob).
- `internal/api/handlers.go:941` — optimised range path
  `io.ReadAll(decryptedReader)` into `outputData` immediately before
  `w.Write(...)`. Replace with `io.CopyBuffer(w, decryptedReader, buf)`
  from the existing buffer pool.
- `internal/api/handlers.go:2618` + `:2675` — `handleCopyObject`: double
  `io.ReadAll` (source plaintext → destination ciphertext). Stream
  source→decrypt→encrypt→upload via a single `io.Pipe` chain; the AWS
  SDK accepts an `io.Reader` directly. Chunked sources/destinations
  remain correct because the chunked envelope is itself streaming.
- `internal/api/handlers.go:2242` — `handleUploadPart`: `io.ReadAll(r.Body)`
  to satisfy the SDK's seekable-body contract. Replace with a
  seekable-body wrapper (`bytes.Reader` over a bounded pool buffer, or
  a retryable readSeeker built from `http.Request.Body`) so the part
  bytes never materialise in a new allocation.
- `internal/api/upload_part_copy.go:557` — legacy UploadPartCopy
  fallback: `io.ReadAll(LimitReader(decryptedReader, cap+1))`. Streaming
  is non-trivial here because legacy is a single AEAD; leave as-is but
  document that the cap (default 256 MiB, per ADR 0006) bounds the
  allocation. No code change, just an explanatory comment linking to
  this task.
- `internal/crypto/engine.go:330,722,874,997,1087,1380,1500,1608,1701` —
  `bytes.NewReader(plaintext)` inside `engine.Encrypt` / `Decrypt`
  compression branches. Compression already exposes streaming
  APIs (`crypto/compression.go:128,195,210` use `io.ReadAll` internally
  too — those should be converted to streaming transforms: gzip/zstd
  readers/writers accept `io.Reader`/`io.Writer` directly).
- `internal/s3/proxy_client.go` — `CopyObject` and `UploadPartCopy`
  already forward streaming, but the caller in `handleCopyObject`
  buffers; fixing the caller is sufficient here.

### Affected
- `internal/api/handlers.go` — `handleGetObject`, `handleCopyObject`,
  `handleUploadPart`
- `internal/api/upload_part_copy.go` — documentation comment only
- `internal/crypto/engine.go` — compression branch refactors

  streaming
- `internal/s3/client.go` — possibly a `ReadSeeker` helper for

| Compression streaming changes the byte layout if any caller depends on "compressed then AEAD'd in one pass". | The AEAD wrap happens over a `Writer` regardless; streaming only changes where the compression boundary sits. Covered by existing round-trip tests. |

1. **Process-global runtime singleton contention.**

   most one writer per OS process, but each provider started its own
   gateway with its own per-handler semaphore (cap 4). When rustfs,
   garage, seaweedfs, and minio hit `/debug/pprof/profile?seconds=1`

   without a mutex while `Shutdown` read them, producing a race detector
   failure whenever a test's `t.Cleanup` fired Shutdown before Start
   finished binding the listener. Fixed by introducing `Server.mu
   sync.Mutex` that guards both fields, with Start publishing atomically
   and Shutdown loading under the lock. Shutdown is now safe to call
   concurrently with Start.

## [V0.6-QA-3] Fix Data Race in `TestConfigReloader_FileWatching` ✅ DONE
- Owner: QA / Config
- Priority: P1
- Status: **Completed** (2026-04-23, verified by 50-run `-race` stress test)
- Labels: `milestone:v0.6`, `area:testing`, `area:config`, `bug`

### Summary
Fix a pre-existing data race in `TestConfigReloader_FileWatching` that
causes intermittent `-race` failures in the full test suite. The race is
on the `firstCallbackOld` / `firstCallbackNew` captures shared between a
background goroutine (the reload callback) and the main test goroutine
(the assertions).

### Symptom
`go test ./... -count=1 -race -short` fails intermittently with:

```
==================
WARNING: DATA RACE
...
Goroutine 16 (running) created at:
  github.com/kenneth/s3-encryption-gateway/internal/config.TestConfigReloader_FileWatching()
      internal/config/config_reloader_test.go:82 +0x68a
...
--- FAIL: TestConfigReloader_FileWatching (0.31s)
    testing.go:1712: race detected during execution of test
```

The test passes in isolation (`go test ./internal/config/... -race`)
because no scheduling pressure exists. Under the full suite, the
background fsnotify callback goroutine races the assertion thread.

### Root Cause
`internal/config/config_reloader_test.go:70-79`:

```go
var callbackCalled int64
var firstCallbackOld, firstCallbackNew *Config  // ← unsynchronised
reloader.SetOnReloadCallback(func(old, new *Config) error {
    callCount := atomic.AddInt64(&callbackCalled, 1)
    if callCount == 1 {
        firstCallbackOld = old     // ← written from callback goroutine
        firstCallbackNew = new     // ← written from callback goroutine
    }
    return nil
})

go reloader.Start()
// ...
time.Sleep(200 * time.Millisecond)

// Read from main goroutine WITHOUT synchronisation:
```

```go
    old, new *Config
    done     chan struct{}
}

    firstCall.Do(func() {
        firstCall.old = old
        firstCall.new = new
        close(firstCall.done)

})

// In the assertion phase, wait deterministically rather than time.Sleep:
select {
case <-firstCall.done:
case <-time.After(2 * time.Second):

assert.Equal(t, "info", firstCall.old.LogLevel)
```

The test suite has accumulated three generations of conventions on top
of each other and is no longer trustworthy as a release gate. The
headline failure is `make test-comprehensive` step 3: `TestProvider_*`
and `TestGateway_ProviderIntegration` all hit `NoSuchBucket` on MinIO
because the "provider" code path seeds a bucket name that was never
created. Below that headline are deeper problems:

- **Fractured taxonomy.** Unit tests, integration tests,
  end-to-end tests, chaos tests, load tests, interop smoke tests, and
  provider-matrix tests all live in the same `test/` package with
  different harnesses, different discovery rules, and different
  opinions on how to start a backend. Some use `//go:build integration`,
  some use `testing.Short()`, some use `RUN_INTEGRATION_TESTS=1` env
  gates, some use `-run TestFoo` naming conventions, and at least one

  exercised by any `make` target at all.
- **Backend fixtures are inconsistent.** MinIO is reached via a

  (`test/minio.go:68-76`); Garage is a `pkill && exec.Command`
  auto-bootstrap from a local binary (`test/garage.go:42-69`); Cosmian
  KMIP starts its own in-process server. There is no shared
  `Provider` interface for a backend-under-test — each file rolls its
  own setup/teardown and leaks state across tests (the
  `minioProviderServer` global at `test/minio.go:37-40` with its

  most visible symptom).
- **MinIO is de-facto dead upstream.** Since early 2025 MinIO AGPL
  releases strip the web console, remove features, and the project has
  been reorganising toward "AIStor." Keeping MinIO as our only local
  reference backend makes the suite dependent on a project whose
  open-source trajectory is now unclear. We need to be running against
  at least one actively maintained alternative (Garage, RustFS) and
  keep MinIO for compatibility only.
- **No provider-matrix guarantee.** `TestProvider_CoreFlows` iterates
  a slice of providers but only MinIO is ever populated locally;
  Garage has its own parallel test (`TestS3Gateway_Garage_EndToEnd`)
  that runs a different flow. There is no single contract-test matrix
  guaranteeing "every feature works on every supported provider."
- **`make test-comprehensive` is failing.** See
  `comprehensive_step_3.log` lines 220-265. Root cause:
  `StartSharedMinIOServerForProvider` generates a bucket name like
  `test-provider-bucket-<nanos>` (`test/minio.go:112`) but never
  creates it in MinIO. Subsequent PUTs fail with `NoSuchBucket`. This
  is a symptom of the fractured fixtures — every harness re-invents
  "make a bucket" and at least one does it wrong.
- **Flaky scaffolding.** The TTFB value of `39.5s` on a 10s window in

### Motivation
1. **Trust.** A failing `make test-comprehensive` on main is a release
   blocker on principle; we ship a gateway whose value proposition is
   trust, and a red tree erodes reviewer trust in every subsequent
   change. *Software Engineering at Google* ch. 11 "Testing Overview"
   and ch. 14 "Larger Testing" are explicit that a flaky suite is
   worse than no suite.
2. **Provider coverage is the product.** The gateway's whole point is
   to sit in front of *any* S3-compatible backend. If we only test
   against MinIO (even when it works), we cannot claim compatibility
   with AWS / Wasabi / Hetzner / B2 / Garage / RustFS with a straight
   face. Contract-style testing across a provider matrix is the
   documented pattern here (*Contract Testing in Action* ch. 2;
   *Microservices Patterns* ch. 9.5 "Testing component tests").
3. **Maintainability.** 34 Go files in `test/`, ~10 500 LoC, five
   harness conventions. Every new test today requires the author to
   pick one of five setups and hope they picked the right one. This
   is exactly the failure mode *Full Stack Testing* ch. 3 warns about:
   "the test suite becomes a liability when the cost of writing a
   correct test exceeds the cost of writing the feature."
4. **MinIO-dead-open-source risk.** Relying on MinIO as our only
   always-on local reference means a future MinIO AGPL change could
   force us to rewrite the entire fixture layer under time pressure.
   De-risking now by adding Garage and RustFS to the daily path
   decouples us.
5. **Four mutually-inconsistent deployment paths for local
   dependencies.** Today, bringing up a full test environment
   requires all four of:
   - `docker-compose -f test/docker-compose.yml up -d` to start
     MinIO + Valkey (manual, pre-test, developer must remember).
   - A local `garage` binary on `PATH`, started via
     `exec.Command("garage", …)` + a `pkill` safety net
     (`test/garage.go:42-69`).
   - A Cosmian KMIP in-process Go server (different lifecycle rules
     entirely).
   - Whatever the developer has running on ports 9000/9001/6379/
     5696/9998 from unrelated work.

   Four lifecycle models × two process types (container vs. binary)
   × two start-time conventions (pre-test vs. test-start) = the
   combinatorial mess that produces bugs like
   "`TestS3Gateway_Garage_EndToEnd` leaves orphan garage processes
   after a `SIGINT`-interrupted run." The testing surface must
   collapse to **exactly one** deployment model: Testcontainers-Go
   starts every dependency at test start, and every dependency is
   torn down at test end, even when the test panics or the suite
   is `SIGINT`-ed. No `docker-compose up` prerequisite, no binary
   fallback, no in-process hybrid, no manual cleanup.

### Scope
This task covers **the entire testing environment**, not just the
failing comprehensive step. Splitting it up per-file would just create
another generation layered on top — the point is a clean single
architecture. The output is:

1. A single test-tier taxonomy with explicit discovery rules (what
   runs where, what gates what, how to opt in).
2. A single `Provider` interface + factory that every backend
   (MinIO, Garage, RustFS locally; plus AWS / Wasabi / Hetzner / B2
   / any future public S3 vendor when creds are supplied)
   implements. **The interface is explicitly a plug-in surface**:
   adding a new public S3 vendor is a single-file drop-in under
   `test/provider/` registered via `provider.Register(...)` in
   `init()` — no edits to the conformance suite, no edits to CI,
   no edits to the Makefile.

   against every `Provider` the environment can supply. "Every
   functionality must be tested through all providers" — per the
   user's direction — is enforced mechanically: adding a new
   provider wires it into one slice and the full conformance runs
   against it automatically.
4. A repaired `make test-comprehensive` that is green on a clean
   checkout on a developer machine, and a CI workflow that runs the

5. **Docker is the only infrastructure prerequisite.** Every local

   gains a container image) is launched at test start via

### Proposed test-tier taxonomy
Per *Full Stack Testing* ch. 2 "Manual Exploratory Testing" → ch. 3
"Automated Functional Testing" → ch. 6 "Performance Testing" pyramid,
and *Software Engineering at Google* ch. 11 "Small / Medium / Large"
nomenclature, we adopt **three** tiers (not five):

| Tier | Google size | Go build tags | Runs where | Time budget | What counts |
|---|---|---|---|---|---|
| **1. Unit** | Small | *none* (default) | Every `go test ./...`, every PR, every push | < 60 s wall-clock | In-process only, no Docker, no network, no filesystem beyond `t.TempDir()`. `go test -race -short` MUST pass and stays ≤ 60 s. |

| **3. Soak / Load / Chaos** | Large | `//go:build soak`, `//go:build load`, `//go:build chaos` | `make test-soak`, `make test-load`, on-demand | 10-60 min each | Long-running resource exhaustion, throughput baselines, fault injection. Not a PR gate. |

### Pluggable architecture for new public S3 vendors
The `Provider` interface is the stable extension point. Adding a new
public S3 vendor (e.g. DigitalOcean Spaces, Cloudflare R2, OVHcloud,
Scaleway, IDrive e2, Linode Object Storage, Tigris, Storj Gateway,
etc.) is a single-file contribution:

```go
// test/provider/digitalocean.go
package provider

import "os"

func init() {
    if os.Getenv("DO_SPACES_ACCESS_KEY") != "" {
        Register(&externalProvider{
            name:     "digitalocean",
            endpoint: "https://" + envOr("DO_SPACES_REGION", "fra1") +
                      ".digitaloceanspaces.com",
            region:   envOr("DO_SPACES_REGION", "fra1"),
            keyEnv:   "DO_SPACES_ACCESS_KEY",
            secretEnv: "DO_SPACES_SECRET_KEY",
            bucketEnv: "DO_SPACES_BUCKET",
            caps:     CapMultipartUpload | CapMultipartCopy |
                      CapObjectTagging | CapPresignedURL,
            cleanup:  CleanupPolicyDelete, // DO charges for storage
        })
    }
}
```

Required:
- One `init()` that calls `provider.Register` when the vendor's
  required env vars are set.
- A `Capabilities()` bitmap declaring which optional features are
  supported (Object Lock, Versioning, Object Tagging, conditional
  writes, presigned URLs, etc.).
- A `CleanupPolicy` declaring whether objects should be deleted
  (free-delete backends) or left for the provider's lifecycle
  policy (minimum-storage-duration backends).

Not required (and not allowed):
- **No** edits to the conformance suite.
- **No** edits to the Makefile.
- **No** edits to CI matrices (the matrix auto-expands via
  `provider.All()`).
- **No** `if providerName == "digitalocean"` branches anywhere
  outside the provider file.

The plug-in contract is enforced by
`test/conformance/matrix_selftest.go` (tier 1, no Docker
required): it instantiates a mock provider with every capability
bit set and asserts the conformance suite runs end-to-end against
it with zero provider-specific branches.

### Docker-only deployment model (non-negotiable)
One infrastructure prerequisite: **Docker Engine (or a compatible
runtime) on the host**. That is the entire prerequisite list. In
particular:

- **No pre-start step required.** The developer does NOT run
  `docker-compose up` before `make test`. The harness starts what
  it needs, when it needs it, and names containers uniquely per
  run so concurrent `go test -p 4` invocations do not collide.
- **No binary fallback.** `hasMinIOBinary()` (removed),
  `hasGarageBinary()` (removed), `exec.LookPath("garage")`
  (removed), `exec.Command("garage", ...)` (removed),
  `exec.Command("pkill", "garage")` (removed).
- **No host-port dependency.** Testcontainers allocates ephemeral
  ports; the harness reads the mapped port via
  `container.MappedPort(ctx, "9000/tcp")` and passes it to the
  gateway config. The old hard-coded `127.0.0.1:9000`,
  `127.0.0.1:3900`, `127.0.0.1:6379`, `127.0.0.1:5696` assumptions
  are all deleted.
- **No in-process hybrid.** The Cosmian KMIP in-process server
  used today in `TestEncryptionEngineWithCosmianKMIP` is retained
  *only* for pure-unit (tier 1) tests of the KMIP client. Any
  conformance or integration test that involves KMS uses a
  Cosmian KMS container (either the `ghcr.io/cosmian/kms` image or
  an equivalent) via Testcontainers.
- **Deterministic teardown.** Testcontainers-Go registers a
  Ryuk reaper container on first use; Ryuk receives a heartbeat
  from the test process and kills every labelled container if
  the heartbeat stops — so `SIGKILL` on the test process, a panic
  in the test function, or a `go test -timeout` fire all result
  in a clean container wipe. This replaces the `StopForce()`
  calls on `test/minio.go:87-92` and the `pkill garage` safety net
  on `test/garage.go:43-45`, both of which are known to leak under
  interruption.
- **Docker unavailability is a `t.Skip`, not a build error.**
  Tier 2/3 tests call `testcontainers.SkipIfProviderIsNotHealthy(t)`
  at the top of each provider's `Start`. Tier 1 is unaffected.
- **The existing `test/docker-compose.yml` is deleted.** A
  `docker/dev-compose.yml` survives *only* for developers who
  want a long-running local environment for manual smoke
  sessions; no test reads from it.

This collapses the four current deployment paths (docker-compose,
garage binary fallback, in-process KMIP, developer-started
containers) to one. Per *Modern REST API Development in Go*
ch. 10: one deployment path, one lifecycle model, one failure
mode to reason about.

### Non-goals (explicit)
- **Rewriting the gateway** to test cleanly. The fractured test
  layer is the problem; the gateway's public HTTP surface is stable.
- **Mocking** the backend for conformance. We specifically want real
  containers; mocks already live under unit tests.
- **Switching test framework.** `testing` + `testify` is fine; we are
  not introducing Ginkgo. (Per *Learning Go, 2nd Edition* ch. 15
  "Writing Tests": stdlib `testing` + table-driven remains the
  idiomatic path.)
- **Parallelism at the backend level.** Testcontainers gives us
  uniquely-named containers per test-run; we will not try to share
  one backend across many `t.Run` subtests for perf. Isolation >
  speed.
- **Solving V0.6-QA-1 (per-provider perf baselines).** That task
  *consumes* the `Provider` interface this task creates. QA-1 comes
  after.

### Known Limitations (documented, not blockers)
- **Testcontainers-Go requires Docker.** Developers without Docker
  get tier 1 only; tier 2 skips cleanly with a one-line explanation.
  (Per *Modern REST API Development in Go* ch. 10
  "Integration Testing": this is the accepted tradeoff.)
- **RustFS is young.** Currently 1.0.x; we pin a specific tag and
  re-evaluate quarterly.
- **Garage's S3 compatibility is not 100%.** Known gaps
  (e.g. Object Lock limited to governance mode) are encoded in
  `Provider.Capabilities()` and the relevant conformance tests skip
  via the bitmap — not via runtime feature-detection inside the
  test, which would mask regressions.
- **External provider tests are best-effort.** Network flakes
  against real AWS / Wasabi / etc. are noted, not counted against
  the PR gate.
