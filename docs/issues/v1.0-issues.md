## [V1.0-SEC-2] HKDF-based Chunk-IV Derivation (non-breaking migration)
- Owner: Security/Crypto
- Priority: P1
- Labels: `area:crypto`, `area:security`, `enhancement`

### Summary
`deriveChunkIV` in `internal/crypto/chunked.go` derives per-chunk IVs by XOR-ing the chunk index into the last 4 bytes of the random `baseIV`. This is functionally correct (each object has a random 96-bit `baseIV`; birthday-bound collisions between objects are not possible), but cryptographically inconsistent with `internal/crypto/mpu_iv.go`, which correctly uses HKDF.

The migration is **fully backward-compatible** via a metadata flag with a dual-read window: new objects are written with HKDF and marked accordingly; existing objects (without the flag) continue to be decrypted via the XOR path. The XOR path is retained for **at least two major versions** (v1.x and v2.x) and will only be removed once the migration tool (â†’ V1.0-MAINT-1) has confirmed all objects are migrated.

### Affected
- `internal/crypto/chunked.go` â€” `deriveChunkIV` (both methods), new constant
- `internal/crypto/engine.go` â€” set metadata flag on the encrypt path

### Migration strategy
```
Encrypt (new):  HKDF  â†’ MetaIVDerivation = "hkdf-sha256"  set
Decrypt (new):  flag present  â†’ HKDF path
                flag absent   â†’ XOR path (legacy, kept until v3.0)

Migration tool (V1.0-MAINT-1): reads objects (XOR decryption),
re-writes them (HKDF encryption) â†’ flag is then set.
XOR code can be removed in v3.0 once all objects are confirmed migrated.
```

### Implementation Guide
- [ ] Introduce constant `MetaIVDerivation = "x-amz-meta-enc-iv-deriv"`.
  - Value `"hkdf-sha256"` = new path; absent/empty field = XOR legacy.
- [ ] Implement `deriveChunkIVHKDF` as a separate function (add, do not replace):
  ```go
  func deriveChunkIVHKDF(baseIV []byte, chunkIndex int) []byte {
      info := make([]byte, 8+4)
      copy(info, "chunk-iv")
      binary.BigEndian.PutUint32(info[8:], uint32(chunkIndex))
      r := hkdf.Expand(sha256.New, baseIV, info)
      iv := make([]byte, len(baseIV))
      io.ReadFull(r, iv)
      return iv
  }
  ```
- [ ] `chunkedEncryptReader`: use `deriveChunkIVHKDF` going forward; set `MetaIVDerivation = "hkdf-sha256"` in metadata.
- [ ] `chunkedDecryptReader`: read `MetaIVDerivation` from the manifest/metadata:
  - `"hkdf-sha256"` â†’ `deriveChunkIVHKDF`
  - absent/empty â†’ existing `deriveChunkIV` (XOR, leave unchanged, do not delete)
- [ ] Keep the existing `deriveChunkIV` (XOR) with a `// Deprecated: used for objects without MetaIVDerivation flag. Remove no earlier than v3.0.` comment.
- [ ] `ChunkManifest` struct: add optional field `IVDerivation string` (JSON tag `omitempty`).

### Tests
- [ ] KAT: `deriveChunkIVHKDF(baseIV, 0)` and `deriveChunkIVHKDF(baseIV, 1)` produce stable, distinct, deterministically reproducible IVs.
- [ ] Round-trip (HKDF): encrypt with new path, decrypt, plaintext matches.
- [ ] Round-trip (XOR legacy): encrypt with old code (no flag), decrypt with new code â†’ fallback to XOR path; plaintext matches.
- [ ] Cross-version: an object with the HKDF flag **cannot** be decrypted via the XOR path (negative test).
- [ ] Regression: the full existing test suite passes without modifications.

### Definition of Done
- [ ] New HKDF path is active for all newly encrypted objects.
- [ ] Dual-read window works; no existing encrypted object breaks.
- [ ] XOR path is explicitly marked deprecated.
- [ ] All tests pass under `-race`.
- [ ] Consistent with `mpu_iv.go`.
- [ ] CHANGELOG entry: non-breaking; migration via V1.0-MAINT-1 available.

### Deprecation timeline
| Version | Status |
|---|---|
| v1.0 | HKDF is written; XOR can be read |
| v1.xâ€“v2.x | XOR read support retained; migration tool available |
| v3.0 | XOR path removed (after confirmed full migration) |

## [V1.0-KMS-4] Self-Contained Envelope Encryption Provider
- Owner: Crypto
- Priority: P1
- Labels: `area:kms`, `area:crypto`, `enhancement`

### Summary
Implement a self-contained envelope encryption mode that does not require an external KMS or KMIP server. Today, all envelope encryption (DEK wrap/unwrap) is delegated to an external KeyManager (Cosmian KMIP, HSM). This is the right architecture for enterprise and regulated deployments, but it creates a barrier for operators who want strong encryption without managing a KMS dependency.

A self-contained provider wraps the DEK using a locally-held Key Encryption Key (KEK) â€” either an AES-256 master key or an RSA key pair â€” without any network call. This gives every deployment strong envelope security (unique DEK per object, KEK never touches the data) with zero external dependencies.

### Motivation
- Operators running the gateway in air-gapped or resource-constrained environments have no path to envelope encryption today; they must use password-derived keys (PBKDF2) or forgo encryption.
- Envelope encryption is cryptographically superior to password-derived keys: a compromised object's ciphertext reveals no information about the KEK, and rotating the KEK does not require re-encrypting object data.
- A self-contained mode reduces the operational barrier to entry for secure deployments, without weakening the existing pluggable `KeyManager` interface.

### Security Considerations
- The KEK must never be logged or included in error messages; enforce with linter rule or code review.
- Zeroize the KEK from memory on `Close()` and after each `WrapKey`/`UnwrapKey` operation (consistent with existing `zeroBytes` convention in `internal/crypto/engine.go`).
- If the KEK is lost, all wrapped DEKs are irrecoverable â€” document this clearly and recommend key backup procedures.
- RSA variant: use OAEP with SHA-256 (not PKCS#1 v1.5); enforce 2048-bit minimum.
- AES variant: derive a separate HMAC key from the KEK via HKDF for any integrity checks; do not reuse the encryption key for authentication.
- Support environment-variable-only key injection to avoid keys on disk in container deployments.

## [V1.0-MAINT-1] Offline Migration Tool (Re-Encrypt & Format Migration)
- Owner: Ops/DevEx
- Priority: P1
- Labels: `area:ops`, `area:crypto`, `area:ecosystem`, `tool`

### Summary
A CLI tool for batch re-encryption of all objects in a bucket or prefix. Primary use cases:

1. **Hard key rotation**: all objects encrypted with the old key/KEK are re-encrypted with the new key â†’ the old key can then be safely removed. Complements the existing online key rotation (drain-and-cutover) with a complete offline path.
2. **Format migration**: objects encrypted with the XOR-IV derivation (legacy) are re-encrypted with HKDF (â†’ V1.0-SEC-2). Once fully migrated, the XOR path can be removed in v3.0.
3. **Algorithm change**: migration from AES-256-GCM to ChaCha20-Poly1305 or vice versa.

The tool operates **read-decrypt-reencrypt-write** directly on the S3 backend without involving the gateway process. It is **idempotent** and **resumable**: it stores its progress in a state file and can be continued after interruption.
