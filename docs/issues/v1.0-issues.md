## [V1.0-SEC-2] HKDF-based Chunk-IV Derivation (non-breaking migration) Ō£ģ DONE
- Owner: Security/Crypto
- Priority: P1
- Status: **Ō£ģ VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:security`, `enhancement`

> **Verified:** 2026-04-29 ŌĆö `deriveChunkIVHKDF` implemented in `internal/crypto/chunked.go` (line 122); `chunkedEncryptReader.deriveChunkIV` delegates to HKDF path; `ChunkManifest.IVDerivation` field set to `"hkdf-sha256"` on all new objects; `chunkedDecryptReader.deriveChunkIV` dispatches on `manifest.IVDerivation` ŌĆö HKDF for new objects, legacy XOR for old objects (deprecated comment present); `MetaIVDerivation = "hkdf-sha256"` written into object metadata by both `encryptChunked` and `encryptChunkedWithMetadataFallback` paths; `range_decrypt.go` also updated with same dispatch logic. All tests pass under `-race`.

### Summary
`deriveChunkIV` in `internal/crypto/chunked.go` derives per-chunk IVs by XOR-ing the chunk index into the last 4 bytes of the random `baseIV`. This is functionally correct (each object has a random 96-bit `baseIV`; birthday-bound collisions between objects are not possible), but cryptographically inconsistent with `internal/crypto/mpu_iv.go`, which correctly uses HKDF.

The migration is **fully backward-compatible** via a metadata flag with a dual-read window: new objects are written with HKDF and marked accordingly; existing objects (without the flag) continue to be decrypted via the XOR path. The XOR path is retained for **at least two major versions** (v1.x and v2.x) and will only be removed once the migration tool (ŌåÆ V1.0-MAINT-1) has confirmed all objects are migrated.

### Affected
- `internal/crypto/chunked.go` ŌĆö `deriveChunkIV` (both methods), new constant
- `internal/crypto/engine.go` ŌĆö set metadata flag on the encrypt path

### Migration strategy
```
Encrypt (new):  HKDF  ŌåÆ MetaIVDerivation = "hkdf-sha256"  set
Decrypt (new):  flag present  ŌåÆ HKDF path
                flag absent   ŌåÆ XOR path (legacy, kept until v3.0)

Migration tool (V1.0-MAINT-1): reads objects (XOR decryption),
re-writes them (HKDF encryption) ŌåÆ flag is then set.
XOR code can be removed in v3.0 once all objects are confirmed migrated.
```

### Implementation Guide
- [ ] Introduce constant `MetaIVDerivation = "x-amz-meta-enc-iv-deriv"`.
  - Value `"hkdf-sha256"` = new path; absent/empty field = XOR legacy.
- [ ] Implement `deriveChunkIVHKDF` as a separate function (add, do not replace):
  ```go
  func deriveChunkIVHKDF(baseIV []byte, chunkIndex int) []byte {
      info := make([]byte, 8+4)
      copy(info, "chunk-iv")
      binary.BigEndian.PutUint32(info[8:], uint32(chunkIndex))
      r := hkdf.Expand(sha256.New, baseIV, info)
      iv := make([]byte, len(baseIV))
      io.ReadFull(r, iv)
      return iv
  }
  ```
- [ ] `chunkedEncryptReader`: use `deriveChunkIVHKDF` going forward; set `MetaIVDerivation = "hkdf-sha256"` in metadata.
- [ ] `chunkedDecryptReader`: read `MetaIVDerivation` from the manifest/metadata:
  - `"hkdf-sha256"` ŌåÆ `deriveChunkIVHKDF`
  - absent/empty ŌåÆ existing `deriveChunkIV` (XOR, leave unchanged, do not delete)
- [ ] Keep the existing `deriveChunkIV` (XOR) with a `// Deprecated: used for objects without MetaIVDerivation flag. Remove no earlier than v3.0.` comment.
- [ ] `ChunkManifest` struct: add optional field `IVDerivation string` (JSON tag `omitempty`).

### Tests
- [x] KAT: `deriveChunkIVHKDF(baseIV, 0)` and `deriveChunkIVHKDF(baseIV, 1)` produce stable, distinct, deterministically reproducible IVs.
- [x] Round-trip (HKDF): encrypt with new path, decrypt, plaintext matches.
- [x] Round-trip (XOR legacy): encrypt with old code (no flag), decrypt with new code ŌåÆ fallback to XOR path; plaintext matches.
- [x] Cross-version: an object with the HKDF flag **cannot** be decrypted via the XOR path (negative test).
- [x] Regression: the full existing test suite passes without modifications.

### Definition of Done
- [x] New HKDF path is active for all newly encrypted objects.
- [x] Dual-read window works; no existing encrypted object breaks.
- [x] XOR path is explicitly marked deprecated.
- [x] All tests pass under `-race`.
- [x] Consistent with `mpu_iv.go`.
- [x] CHANGELOG entry: non-breaking; migration via V1.0-MAINT-1 available.

### Deprecation timeline
| Version | Status |
|---|---|
| v1.0 | HKDF is written; XOR can be read |
| v1.xŌĆōv2.x | XOR read support retained; migration tool available |
| v3.0 | XOR path removed (after confirmed full migration) |

### Cross-References
- V1.0-MAINT-1 (migration tool) ŌĆö enables full migration of existing objects

## [V1.0-SEC-4] Restrict AAD fallback to explicitly marked legacy objects Ō£ģ DONE
- Owner: Security/Crypto
- Priority: P1
- Status: **Ō£ģ VERIFIED COMPLETE**
- Labels: `area:security`, `area:crypto`, `bug`

> **Verified:** 2026-04-29 ŌĆö `MetaLegacyNoAAD = "x-amz-meta-enc-legacy-no-aad"` constant defined in `internal/crypto/engine.go` (line 54); the `gcm.Open(nil, iv, ciphertext, nil)` fallback in `Decrypt` is guarded by `expandedMetadata[MetaLegacyNoAAD] == "true"` (line 719); the same guard applied to the key-resolver alternate-key path (line 745); constant is registered in `isEncryptionMetadata` (line 1734); new objects written by `Encrypt` never set this flag. Tests `TestEncrypt_DoesNotSetLegacyNoAADFlag`, `TestAADFallback_NewObjectTamperedFails`, `TestAADFallback_LegacyObjectWithFlagSucceeds`, `TestAADFallback_LegacyObjectWithoutFlagFails` all pass under `-race`.

### Summary
`internal/crypto/engine.go` automatically retries AEAD decryption without AAD (`gcm.Open(nil, iv, ciphertext, nil)`) when decryption with AAD fails. The retry exists for backward compatibility with objects encrypted before AAD was introduced. Without being restricted to explicitly marked legacy objects, this allows an attacker with backend write access to use tampered metadata to bypass the AAD integrity check.

### Affected
- `internal/crypto/engine.go` ŌĆö Decrypt function (AAD fallback path)
- `internal/crypto/engine.go` ŌĆö Encrypt function (set legacy flag)

### Implementation Guide
- [ ] Introduce metadata constant `MetaLegacyNoAAD = "x-amz-meta-enc-legacy-no-aad"`.
- [ ] Encrypt path: new objects do **not** receive this flag. Only legacy objects (encrypted without AAD) have it.
- [ ] Decrypt path: the `gcm.Open(nil, iv, ciphertext, nil)` fallback is executed **only** when `MetaLegacyNoAAD == "true"` is set in the metadata.
  ```go
  if openErr != nil && metadata[MetaLegacyNoAAD] == "true" {
      if pt, err2 := gcm.Open(nil, iv, ciphertext, nil); err2 == nil {
          plaintext = pt
          openErr = nil
      }
  }
  ```
- [ ] Document in the changelog: objects without this flag will fail to decrypt if they were encrypted without AAD (breaking change for very old objects ŌĆö migration required).
- [ ] Consider a migration tool or admin API endpoint for back-tagging old objects with `MetaLegacyNoAAD`.

### Tests
- [x] Test: new object encrypted with AAD, tampered metadata ŌåÆ decryption fails (no fallback).
- [x] Test: legacy object (no AAD, flag set) ŌåÆ decryption succeeds via fallback.
- [x] Test: legacy object without flag ŌåÆ decryption fails (no blind fallback).

### Definition of Done
- [x] AAD fallback only runs when `MetaLegacyNoAAD == "true"`.
- [x] New objects do not receive the flag.
- [x] All three test scenarios implemented and passing.
- [x] Breaking change documented in CHANGELOG.

## [V1.0-OPS-2] Security-Scanning in CI & Dependency Management
- Owner: Ops/Security
- Priority: P0
- Labels: `area:ops`, `area:security`, `area:ci`

### Summary

Dependabot is commented out in `.github/dependabot.yml`. There is no `SECURITY.md`. These gaps must be closed for v1.0.

### Affected
- `.github/workflows/` ŌĆö new security job steps
- Renovate configuration (already active; no Dependabot needed)
- `SECURITY.md` (new)
- `Makefile` / dev tooling ŌĆö local security scan targets

### Implementation Guide

#### Phase 1 ŌĆö Dependency Vulnerability Scanning
- [ ] Add `govulncheck` as a CI step (after `go build`, before tests):
  ```yaml
  - name: govulncheck
    run: go run golang.org/x/vuln/cmd/govulncheck@latest ./...
  ```
- [ ] Priority packages: `golang.org/x/crypto` and `github.com/aws/aws-sdk-go-v2` ŌĆö both have historically received security-relevant patches.
- [ ] Dependency updates run via **Renovate** (already active) ŌĆö no Dependabot required.

#### Phase 2 ŌĆö Static Analysis
- [ ] Add `gosec` as a CI step:
  ```yaml
  - name: gosec
    uses: securego/gosec@master
    with:
      args: '-exclude=G104 ./...'
  ```
- [ ] Run `gosec` locally and triage initial findings: fix real issues; suppress false positives with `#nosec G<rule>` and a justification comment.
- [ ] No new code may introduce untriaged `gosec` HIGH findings (PR gate).

#### Phase 3 ŌĆö Container Image Scanning

  ```yaml

  ```
- [ ] CRITICAL findings block the release; HIGH findings produce a warning (non-blocking, but visible in the release summary).

#### Phase 4 ŌĆö SECURITY.md & Responsible Disclosure
- [ ] Create `SECURITY.md` in the repository root:
  - Supported versions (v0.6.x, v1.0.x)
  - Contact method for vulnerability reports (email or GitHub Private Vulnerability Reporting)
  - Expected response timeline (acknowledge: 48 h, triage: 7 days, fix: depends on severity)
  - Out of scope (e.g. DoS via legitimately large objects)
- [ ] Enable GitHub Private Vulnerability Reporting in repository settings.

### Tests
- [ ] CI fails when `govulncheck` reports a known vulnerability (verifiable by temporarily pinning a known-vulnerable version in a feature branch).
- [ ] `gosec` step is listed as a required check in branch protection rules.

## [V1.0-KMS-4] Self-Contained Envelope Encryption Provider
- Owner: Crypto
- Priority: P1
- Labels: `area:kms`, `area:crypto`, `enhancement`

### Summary
Implement a self-contained envelope encryption mode that does not require an external KMS or KMIP server. Today, all envelope encryption (DEK wrap/unwrap) is delegated to an external KeyManager (Cosmian KMIP, HSM). This is the right architecture for enterprise and regulated deployments, but it creates a barrier for operators who want strong encryption without managing a KMS dependency.

A self-contained provider wraps the DEK using a locally-held Key Encryption Key (KEK) ŌĆö either an AES-256 master key or an RSA key pair ŌĆö without any network call. This gives every deployment strong envelope security (unique DEK per object, KEK never touches the data) with zero external dependencies.

### Motivation
- Operators running the gateway in air-gapped or resource-constrained environments have no path to envelope encryption today; they must use password-derived keys (PBKDF2) or forgo encryption.
- Envelope encryption is cryptographically superior to password-derived keys: a compromised object's ciphertext reveals no information about the KEK, and rotating the KEK does not require re-encrypting object data.
- A self-contained mode reduces the operational barrier to entry for secure deployments, without weakening the existing pluggable `KeyManager` interface.

### Affected
- `internal/crypto/keymanager_selfcontained.go` (new) ŌĆö AES-256-GCM KEK wrap + RSA-OAEP KEK wrap implementations
- `internal/crypto/keymanager.go` ŌĆö register new provider types
- `internal/config/config.go` ŌĆö new `self_contained` KMS provider section
- `docs/KMS_COMPATIBILITY.md` ŌĆö document new provider
- `docs/adr/ADR-XXXX-self-contained-envelope.md` (new)

### Implementation Guide

#### Phase 1 ŌĆö AES-KEK Provider
- [ ] Implement `AESKEKManager` satisfying `KeyManager` interface.

  - `Close`: zeroize KEK from memory.
- [ ] Accept KEK as base64-encoded 256-bit key from config or environment variable reference (`${ENV_VAR}` syntax consistent with existing config).

## [V1.0-SEC-12] Key Padding Creates Catastrophically Weak Keys Ō£ģ DONE
- Owner: Security/Crypto
- Priority: P0
- Status: **Ō£ģ VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:security`, `bug`
- Source: `docs/issues/v1.0-deep-analysis-findings.md` DAF-02 / V1.0-SEC-12 (CVSS 7.5)

### Summary
When a key is shorter than `keySize`, the engine "pads" it by repeating the key prefix:
```go
if len(key) < keySize {
    copy(adjustedKey[len(key):], key[:keySize-len(key)])
}
```
A 16-byte key padded to 32 bytes becomes `key[:16] + key[:16]` ŌĆö a key with perfect 16-byte periodicity, catastrophically weak for AES-256. This pattern exists at **8+ locations** in `internal/crypto/engine.go` (lines 430-437, 648-655, 873-892, 1002-1021, 1180-1199, 1353-1372, 1476-1483, 1590-1597).

**Breaking-change scope is narrow ŌĆö see analysis below.**

### Root Cause Analysis

The padding block appears in two structurally different code paths, and only one is reachable in practice:

**KMS path (`e.kmsManager != nil`):** After `UnwrapKey`, if the KMS returns a wrong-sized key the padding block silently corrects it. This is the **real vulnerability** ŌĆö a misconfigured or buggy KMS returning a 16-byte key produces silently weakened ciphertext with no operator-visible error.

**Password/PBKDF2 path (`e.kmsManager == nil`):** After `deriveKey`, which calls `pbkdf2.Key(sha256.New, password, salt, 100000, aesKeySize)` with `aesKeySize = 32` hardcoded. PBKDF2 is a PRF that always returns **exactly** the requested number of bytes ŌĆö `len(key) != keySize` is **provably unreachable** here. The padding block in this branch is dead code. The same applies to `generateDataKey` on the encrypt path, which allocates exactly `keySize` bytes from `crypto/rand`.

**Consequence:** this fix has zero behavioural impact on password-only (non-KMS) deployments. It is a breaking change only when KMS is enabled and the KMS was silently returning wrong-sized keys ŌĆö which was already producing silently corrupted (weakened) ciphertext. Hardening that broken path is the correct outcome.

### Affected
- `internal/crypto/engine.go` ŌĆö all `adjustedKey` padding blocks (lines 430, 648, 873, 885, 1002, 1014, 1180, 1192, 1353, 1365, 1476, 1590)

### Implementation Guide

Apply a **split fix** by path, not a blanket removal:

**KMS branch (after `UnwrapKey`) ŌĆö all encrypt and decrypt functions:**
- [x] Replace the padding block with a hard error:
  ```go
  if len(key) != keySize {
      zeroBytes(key)
      return nil, nil, fmt.Errorf("key size mismatch: KMS returned %d bytes, want %d ŌĆö refusing to pad", len(key), keySize)
  }
  ```
- [x] This applies in: `Encrypt`, `Decrypt`, `encryptChunked`, `decryptChunked`, `DecryptRange`, and all other KMS branches. Note: `Decrypt` (line 639) already has this check ŌĆö verify it is not redundantly followed by another padding block.

**Password/PBKDF2 branch (after `deriveKey`) ŌĆö all functions:**
- [x] Remove the unreachable padding block entirely (it is dead code ŌĆö PBKDF2 always returns exactly `aesKeySize` bytes). No error handling needed; add a comment explaining why:
  ```go
  // deriveKey always returns exactly aesKeySize bytes via PBKDF2;
  // the following size check is a defensive assertion only.
  if len(key) != keySize {
      zeroBytes(key)
      return nil, nil, fmt.Errorf("internal: PBKDF2 returned unexpected key size %d (want %d)", len(key), keySize)
  }
  ```
  Using an assertion-style error rather than padding makes future refactors safe without silently degrading.

**`generateDataKey` (encrypt path):**
- [x] Same: `generateDataKey` always returns exactly `keySize` bytes. Remove the subsequent padding block; replace with a defensive assertion error if present.

- [x] Run a global search for `copy(adjustedKey[len(key):]` to confirm all instances are addressed.

### Security Considerations
- Padding repeating key bytes reduces effective key entropy: a 16-byte key padded to 32 bytes is trivially identifiable via periodicity even without knowing the password.
- The fix surfaces KMS misconfiguration at operation time rather than after a breach.
- Password-only deployments are entirely unaffected; no data is at risk from this fix.

### Breaking Change Assessment
| Deployment mode | Behaviour change | Data at risk? |
|---|---|---|
| Password-only (no KMS) | None ŌĆö dead code removed | No |
| KMS, correct key size | None ŌĆö happy path unchanged | No |
| KMS, returning undersized key | Now returns an error instead of padding | Already corrupted ŌĆö fix is correct |

**This is not a meaningful breaking change for any correctly-functioning deployment.**

### Tests
- [x] Unit: inject a 16-byte key via mock KMS into `Encrypt`, `Decrypt`, `encryptChunked`, `decryptChunked`, `DecryptRange` ŌåÆ each returns an error, padding is never applied.
- [x] Unit: inject a 32-byte key via mock KMS ŌåÆ no error; happy path unchanged (regression).
- [x] Unit: password-only mode (no KMS), any object size ŌåÆ encrypt/decrypt succeeds; no code path enters the former padding block.
- [x] Unit: `deriveKey` returns exactly 32 bytes (assert in test, confirm no padding path reachable).

### Definition of Done
- [x] All `copy(adjustedKey[len(key):]` padding blocks replaced with hard errors (KMS branch) or defensive assertions (PBKDF2 branch).
- [x] No silent key padding remains anywhere in the engine.

## [V1.0-SEC-29] UploadPartCopy Unbounded `io.ReadAll` on Chunked Sources Ō£ģ DONE
- Owner: Security/API
- Priority: P2
- Labels: `area:security`, `area:api`, `bug`
- Status: **Ō£ģ VERIFIED COMPLETE**

> **Verified:** 2026-04-30 ŌĆö Two unbounded `io.ReadAll` sites replaced with `io.LimitReader(reader, maxCopyPartRangeBytes+1)` guards; all four already-bounded sites annotated with consistent `V1.0-SEC-29` rationale comments; regression tests `TestSEC29_ChunkedSourceBufferCap_uploadPartCopyChunked` and `TestSEC29_ChunkedSourceBufferCap_ReencryptMPU` pass under `-race`.

### Summary
`internal/api/upload_part_copy.go` contained six `io.ReadAll` calls across three functions that handle the `UploadPartCopy` S3 operation. Two of these calls ŌĆö one in `uploadPartCopyChunked` and one in the `SourceClassChunked` arm of `uploadPartCopyReencryptMPU` ŌĆö read from a `DecryptRange` output with no upper bound. An attacker crafting a cross-bucket part copy against a very large chunked-encrypted source could cause unbounded heap allocation, enabling a remote memory-exhaustion DoS.

The four remaining `io.ReadAll` sites were already guarded by `io.LimitReader(reader, maxLegacyCap+1)` (enforcing `Server.MaxLegacyCopySourceBytes`, default 256 MiB) or by a documented structural invariant (`encLen` from the upstream MPU schedule). These sites were not vulnerable but lacked consistent rationale comments.

### Affected
- `internal/api/upload_part_copy.go`
  - `uploadPartCopyChunked` (line ~532) ŌĆö `io.ReadAll(decryptedReader)` **unbounded**
  - `uploadPartCopyReencryptMPU` / `SourceClassChunked` arm (line ~771) ŌĆö `io.ReadAll(decR)` **unbounded**
  - `uploadPartCopyLegacy` (line ~587) ŌĆö `io.ReadAll(limited)` already bounded by `maxLegacyCap`
  - `uploadPartCopyReencryptMPU` / `SourceClassPlaintext` arm (line ~730) ŌĆö bounded by `maxLegacyCap`
  - `uploadPartCopyReencryptMPU` / `SourceClassLegacy` arm (line ~790) ŌĆö bounded by `maxLegacyCap`
  - `uploadPartCopyReencryptMPU` / re-encrypt output (line ~818) ŌĆö bounded by `encLen` invariant

### Implementation Guide
- [x] `uploadPartCopyChunked`: replace `io.ReadAll(decryptedReader)` with:
  ```go
  io.ReadAll(io.LimitReader(decryptedReader, maxCopyPartRangeBytes+1))
  ```
  followed by a post-read length check returning an error when `len(decryptedBytes) > maxCopyPartRangeBytes`.
- [x] `uploadPartCopyReencryptMPU` / `SourceClassChunked`: apply the same `LimitReader` guard around `io.ReadAll(decR)`.
- [x] Update comments on all six `io.ReadAll` sites with consistent `V1.0-SEC-29` rationale explaining bound mechanism and why the guard is correct.
- [x] The limit used is `maxCopyPartRangeBytes` (5 GiB) ŌĆö the S3-spec per-part maximum. `DecryptRange` already emits only the requested plaintext slice, so a well-formed request never hits this guard.

### Security Considerations
- Reading one byte past the limit (`maxCopyPartRangeBytes+1`) distinguishes "exactly at limit" from "over limit" without a second read. An extra byte never causes a meaningful allocation.
- A well-formed request by a conforming client will never hit the guard: `x-amz-copy-source-range` is validated to Ōēż 5 GiB at the handler level (lines ~152ŌĆō165) before dispatch; `DecryptRange` clips output to the requested range.
- The guard protects against: (a) a corrupt/tampered source object whose plaintext metadata claims a smaller size than the actual ciphertext; (b) a future regression where the range calculation produces an over-size range.
