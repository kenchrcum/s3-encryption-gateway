# Milestone v1.0 — Issue Tracker and Implementation Guide

This document tracks issues for v1.0 with implementation guides and validation criteria derived from `docs/ROADMAP.md`.

## Scope and Goals
- Security/Crypto: Full KMS production readiness (HA, caching, failure modes); sensitive data zeroization and constant-time comparisons audit
- S3 Compatibility: Broad feature parity (tagging, ACLs, lifecycle headers passthrough); SDK/tool compatibility certification matrix
- Performance & Scalability: Horizontal scale guidance; tuned HPA/SLOs; high-throughput benchmarks
- Observability: Turnkey dashboards (Grafana) and alerting rules
- Operations: Multi-arch images, SBOMs, provenance (SLSA); Helm chart on Artifact Hub with automated checks
- Ecosystem: Additional backends via shims (GCS, Azure Blob, filesystem); CLI for local testing (presign, quick put/get)

## Milestone Acceptance Criteria
- Compatibility certification matrix completed for targeted SDKs/tools.
- Published dashboards and alerting rules; documented SLOs and autoscaling guidance.
- Supply-chain artifacts for releases (multi-arch, SBOM, provenance) available.

## [V1.0-OPS-2] Security-Scanning in CI & Dependency Management
- Owner: Ops/Security
- Priority: P0
- Labels: `area:ops`, `area:security`, `area:ci`

### Summary
A security product that does not scan its own dependencies for vulnerabilities is a contradiction. Currently `govulncheck`, `gosec`, and container scanning are entirely absent from the CI pipeline. Dependabot is commented out in `.github/dependabot.yml`. There is no `SECURITY.md`. These gaps must be closed for v1.0.

### Affected
- `.github/workflows/` — new security job steps
- Renovate configuration (already active; no Dependabot needed)
- `SECURITY.md` (new)
- `Makefile` / dev tooling — local security scan targets

### Implementation Guide

#### Phase 1 — Dependency Vulnerability Scanning
- [ ] Add `govulncheck` as a CI step (after `go build`, before tests):
  ```yaml
  - name: govulncheck
    run: go run golang.org/x/vuln/cmd/govulncheck@latest ./...
  ```
- [ ] Priority packages: `golang.org/x/crypto` and `github.com/aws/aws-sdk-go-v2` — both have historically received security-relevant patches.
- [ ] Dependency updates run via **Renovate** (already active) — no Dependabot required.

#### Phase 2 — Static Analysis
- [ ] Add `gosec` as a CI step:
  ```yaml
  - name: gosec
    uses: securego/gosec@master
    with:
      args: '-exclude=G104 ./...'
  ```
- [ ] Run `gosec` locally and triage initial findings: fix real issues; suppress false positives with `#nosec G<rule>` and a justification comment.
- [ ] No new code may introduce untriaged `gosec` HIGH findings (PR gate).

#### Phase 3 — Container Image Scanning
- [ ] Add a Trivy container scan to the release workflow:
  ```yaml
  - name: trivy image scan
    uses: aquasecurity/trivy-action@master
    with:
      image-ref: ghcr.io/${{ github.repository }}:${{ github.sha }}
      severity: HIGH,CRITICAL
      exit-code: '1'
  ```
- [ ] CRITICAL findings block the release; HIGH findings produce a warning (non-blocking, but visible in the release summary).

#### Phase 4 — SECURITY.md & Responsible Disclosure
- [ ] Create `SECURITY.md` in the repository root:
  - Supported versions (v0.6.x, v1.0.x)
  - Contact method for vulnerability reports (email or GitHub Private Vulnerability Reporting)
  - Expected response timeline (acknowledge: 48 h, triage: 7 days, fix: depends on severity)
  - Out of scope (e.g. DoS via legitimately large objects)
- [ ] Enable GitHub Private Vulnerability Reporting in repository settings.

### Tests
- [ ] CI fails when `govulncheck` reports a known vulnerability (verifiable by temporarily pinning a known-vulnerable version in a feature branch).
- [ ] `gosec` step is listed as a required check in branch protection rules.

### Definition of Done
- [ ] `govulncheck` runs on every PR and blocks on findings.
- [ ] `gosec` runs on every PR; HIGH findings are triaged.
- [ ] Trivy container scan in the release workflow; CRITICAL findings block.
- [ ] Renovate (already active) covers automatic dependency updates — no separate action needed.
- [ ] `SECURITY.md` present and complete.
- [ ] No known CVE in direct or transitive dependencies at the time of the v1.0 release.

## [V1.0-KMS-4] Self-Contained Envelope Encryption Provider
- Owner: Crypto
- Priority: P1
- Labels: `area:kms`, `area:crypto`, `enhancement`

### Summary
Implement a self-contained envelope encryption mode that does not require an external KMS or KMIP server. Today, all envelope encryption (DEK wrap/unwrap) is delegated to an external KeyManager (Cosmian KMIP, HSM). This is the right architecture for enterprise and regulated deployments, but it creates a barrier for operators who want strong encryption without managing a KMS dependency.

A self-contained provider wraps the DEK using a locally-held Key Encryption Key (KEK) — either an AES-256 master key or an RSA key pair — without any network call. This gives every deployment strong envelope security (unique DEK per object, KEK never touches the data) with zero external dependencies.

### Motivation
- Operators running the gateway in air-gapped or resource-constrained environments have no path to envelope encryption today; they must use password-derived keys (PBKDF2) or forgo encryption.
- Envelope encryption is cryptographically superior to password-derived keys: a compromised object's ciphertext reveals no information about the KEK, and rotating the KEK does not require re-encrypting object data.
- A self-contained mode reduces the operational barrier to entry for secure deployments, without weakening the existing pluggable `KeyManager` interface.

### Affected
- `internal/crypto/keymanager_selfcontained.go` (new) — AES-256-GCM KEK wrap + RSA-OAEP KEK wrap implementations
- `internal/crypto/keymanager.go` — register new provider types
- `internal/config/config.go` — new `self_contained` KMS provider section
- `docs/KMS_COMPATIBILITY.md` — document new provider
- `docs/adr/ADR-XXXX-self-contained-envelope.md` (new)

### Implementation Guide

#### Phase 1 — AES-KEK Provider
- [ ] Implement `AESKEKManager` satisfying `KeyManager` interface.
  - `WrapKey`: generate random DEK → wrap with AES-256-GCM using the KEK → store `{nonce, ciphertext}` in `KeyEnvelope.Ciphertext`.
  - `UnwrapKey`: decrypt `KeyEnvelope.Ciphertext` with KEK → return plaintext DEK.
  - `HealthCheck`: validate KEK is loaded and correct length (32 bytes).
  - `ActiveKeyVersion`: return configured key version integer.
  - `Close`: zeroize KEK from memory.
- [ ] Accept KEK as base64-encoded 256-bit key from config or environment variable reference (`${ENV_VAR}` syntax consistent with existing config).
- [ ] Support multiple versioned KEKs (map of version → key) to allow dual-read rotation without re-encrypting objects.

#### Phase 2 — RSA-KEK Provider
- [ ] Implement `RSAKEKManager` satisfying `KeyManager` interface.
  - `WrapKey`: wrap DEK with RSA-OAEP (SHA-256) using the RSA public key.
  - `UnwrapKey`: decrypt envelope using RSA private key.
  - `HealthCheck`: validate key pair consistency (modulus match).
  - Minimum key size: 2048 bits; recommended: 4096 bits.
- [ ] Accept PEM-encoded public + private key from config or environment variable references.
- [ ] Support key versioning via fingerprint (SHA-256 of public key modulus).

#### Phase 3 — Key Rotation Support
- [ ] Implement `RotatableKeyManager` for AES-KEK provider: `PrepareRotation` / `PromoteActiveVersion`.
- [ ] Document rotation runbook: add new KEK version → promote → re-wrap objects on next write (lazy rotation).
- [ ] Admin API endpoint to trigger rotation (reuse existing `internal/admin` rotation infrastructure).

#### Phase 4 — Configuration & Docs
- [ ] Add `self_contained` section to config schema:
  ```yaml
  kms:
    provider: self_contained
    self_contained:
      type: aes        # or rsa
      key_version: 1
      keys:
        - version: 1
          aes_key: "${AES_KEK_KEY}"   # base64 256-bit
  ```
- [ ] Update `docs/KMS_COMPATIBILITY.md` with provider matrix entry.
- [ ] Write ADR documenting the design rationale, security properties, and comparison with external KMS.
- [ ] Add getting-started example to `README.md`.

### Security Considerations
- The KEK must never be logged or included in error messages; enforce with linter rule or code review.
- Zeroize the KEK from memory on `Close()` and after each `WrapKey`/`UnwrapKey` operation (consistent with existing `zeroBytes` convention in `internal/crypto/engine.go`).
- If the KEK is lost, all wrapped DEKs are irrecoverable — document this clearly and recommend key backup procedures.
- RSA variant: use OAEP with SHA-256 (not PKCS#1 v1.5); enforce 2048-bit minimum.
- AES variant: derive a separate HMAC key from the KEK via HKDF for any integrity checks; do not reuse the encryption key for authentication.
- Support environment-variable-only key injection to avoid keys on disk in container deployments.

### Tests
- Unit: AES-KEK wrap/unwrap round-trip; wrong KEK returns `ErrUnwrapFailed`; Close zeroizes; version mismatch returns `ErrKeyNotFound`.
- Unit: RSA-KEK wrap/unwrap round-trip; key-pair mismatch detected at init; minimum key size enforced.
- Unit: dual-version read (old KEK version wraps, new version cannot unwrap, old version can).
- Integration (Tier 2): full encrypt/decrypt cycle via engine with AES-KEK provider against MinIO.
- Integration: key rotation promotion — objects written with v1 still decrypt after v2 promoted as active.
- Fuzz: `WrapKey`/`UnwrapKey` with malformed `KeyEnvelope.Ciphertext`.

### Definition of Done
- [ ] `AESKEKManager` and `RSAKEKManager` implement `KeyManager` interface and pass all unit tests.
- [ ] `AESKEKManager` implements `RotatableKeyManager`; rotation path tested end-to-end.
- [ ] Tier 2 conformance tests pass with `self_contained` provider against MinIO and Garage.
- [ ] Config schema updated; environment-variable injection works.
- [ ] ADR authored and merged.
- [ ] `docs/KMS_COMPATIBILITY.md` updated with provider entry.
- [ ] No KEK material appears in any log output (verified by log-scraping test).

### Known Limitations
- RSA key rotation requires generating a new key pair and re-distributing the public key; no automated rotation path.
- Self-contained mode offers no HSM boundary — key material lives in process memory. Operators requiring HSM-grade isolation should use the existing Cosmian KMIP or HSM adapter.

### Cross-References
- ADR-0007 (RotatableKeyManager interface)
- `internal/crypto/keymanager_cosmian.go` — reference implementation of `KeyManager`
- `V1.0-KMS-2` (AWS KMS Adapter), `V1.0-KMS-3` (Vault Transit) — complementary external KMS adapters

## [V1.0-CRYPTO-1] argon2id KDF Support (non-FIPS)
- Owner: Crypto
- Priority: P2 (post V1.0-SEC-H03)
- Labels: `area:crypto`, `area:security`, `enhancement`

### Summary
Add argon2id as an optional, user-selectable KDF for non-FIPS deployments. argon2id is memory-hard (PHC 2015 winner) and OWASP's #1 password-hashing recommendation. PBKDF2 remains the FIPS-compliant default. The `kdf_argon2_default.go` stub and `MetaKDFParams` wire format introduced by V1.0-SEC-H03 provide the full infrastructure; this issue tracks the actual implementation.

### Why Separate from V1.0-SEC-H03
- No FIPS 140-3 validated implementation exists for argon2id → FIPS builds must continue using PBKDF2.
- The `//go:build !fips` compile-time gate (matching the existing ChaCha20 pattern) is already in place by V1.0-SEC-H03.
- Benefit over 600k PBKDF2 is meaningful but not blocking; the immediate security gap (100k → 600k) is addressed by V1.0-SEC-H03.

### Conditions for Activation
- Operator sets `encryption.kdf.algorithm: argon2id` in config (new field in `KDFConfig`, introduced alongside `KDFConfig.PBKDF2`)
- argon2id parameters live at `encryption.kdf.argon2id.time`, `encryption.kdf.argon2id.memory`, `encryption.kdf.argon2id.threads`
- FIPS build (`-tags fips`) rejects `algorithm: argon2id` at startup with `ErrAlgorithmNotApproved`
- Migration: `s3eg-migrate --filter=kdf` handles PBKDF2→argon2id and argon2id→PBKDF2 because the algorithm is encoded in `MetaKDFParams`

### Implementation Guide
- [ ] Replace stub body in `internal/crypto/kdf_argon2_default.go` with `golang.org/x/crypto/argon2.IDKey`
  - Recommended parameters (OWASP 2024): `m=19456 (19 MiB), t=2, p=1`
- [ ] Add `Algorithm string` field to `KDFConfig` in `internal/config/config.go` (yaml: `encryption.kdf.algorithm`; default `"pbkdf2-sha256"`)
- [ ] Add `Argon2id Argon2idConfig` field to `KDFConfig` (yaml: `encryption.kdf.argon2id`) with `Time uint32`, `Memory uint32`, `Threads uint8`
- [ ] Config validation: reject `algorithm: argon2id` in FIPS builds at startup
- [ ] Wire `cfg.Encryption.KDF.Algorithm` into engine to select KDF on encrypt; `ParseKDFParams` / `FormatKDFParams` already handle the `argon2id:t:m:p` wire format on decrypt
- [ ] Add argon2id KAT test in `internal/crypto/kdf_test.go`
- [ ] Update `docs/MIGRATION.md` with argon2id migration runbook
- [ ] Update Helm `values.yaml` and `values.schema.json` with `encryption.kdf.argon2id.*` entries

### Tests
- [ ] Non-FIPS: encrypt with argon2id; decrypt; plaintext matches
- [ ] FIPS: `encryption.kdf.algorithm: argon2id` rejected at startup with `ErrAlgorithmNotApproved`
- [ ] Migration tool: `s3eg-migrate --filter=kdf` handles PBKDF2→argon2id cross-algorithm re-encryption
- [ ] Conformance: `testKDF_Argon2id_RoundTrip` and `testKDF_Argon2id_LegacyRead` added to `test/conformance/kdf_test.go`

### Definition of Done
- [ ] Non-FIPS build: full encrypt/decrypt cycle with argon2id (unit + Tier 2 conformance)
- [ ] FIPS build: `encryption.kdf.algorithm: argon2id` rejected; PBKDF2 continues to work
- [ ] Migration path between PBKDF2 and argon2id demonstrated end-to-end
- [ ] `docs/MIGRATION.md` updated; Helm schema updated

## [V1.0-CRYPTO-2] Valkey At-Rest Encryption for Multipart Upload State
- Owner: Crypto / Security
- Priority: P1
- Labels: `area:crypto`, `area:security`, `enhancement`

### Summary
Add optional at-rest encryption for the full multipart-upload state blob persisted in Valkey.

Today, `UploadState` (including the wrapped DEK envelope, IV prefix, bucket/key names,
and per-part metadata) is stored in Valkey as plaintext JSON. The DEK itself is already
wrapped by the `KeyManager` before being written, so an attacker with Valkey read access
cannot recover the object-encryption key without also compromising the KMS/HSM or gateway
password. However, the remainder of the state — bucket names, object keys, IV prefixes,
and part-level metadata — is visible to anyone with access to Valkey memory dumps, RDB/AOF
files, or replication streams. This is an at-rest disclosure gap that TLS on the wire does
not close.

### Threat Model
- **Attacker with Valkey read access** (e.g., compromised Valkey replica, stolen backup,
  insider with `ACL` access) can enumerate in-flight multipart uploads, learn bucket/key
  names, and collect wrapped DEK envelopes for later offline analysis.
- **Attacker with Valkey write access** can tamper with part records or swap wrapped DEKs
  (existing tamper-detection in `UnwrapKey` AAD catches some variants, but not all).

Encrypting the entire state blob at rest removes this read-disclosure vector entirely.

### Affected
- `internal/mpu/state.go` — encrypt before `HSET`, decrypt after `HGET`
- `internal/config/config.go` — new `ValkeyEncryptionPasswordEnv` config field
- `internal/crypto/` — new `aesgcm.Seal` / `aesgcm.Open` helper or reuse existing
  `passwordKeyManager` machinery
- `docs/adr/0009-encrypted-multipart-uploads.md` — update threat-model and security
  considerations
- `docs/plans/V1.0-CRYPTO-2-plan.md` (new) — detailed implementation plan

### Implementation Guide

#### Phase 1 — Encryption Layer Design
- [ ] Define a versioned ciphertext format for the at-rest blob:
  ```
  version(1 byte) || nonce(12 bytes) || ciphertext(... ) || tag(16 bytes)
  ```
  Version starts at `0x01` (AES-256-GCM). Leave room for future AEAD algorithms.
- [ ] Derive a dedicated AEAD key from the configured password via HKDF-SHA256:
  ```
  key = HKDF-SHA256(salt="s3eg-mpu-state-v1", ikm=password)
  ```
  This isolates the Valkey encryption key from object-encryption keys.
- [ ] Add `EncryptState([]byte) ([]byte, error)` and `DecryptState([]byte) ([]byte, error)`
  methods on `ValkeyStateStore`. Encrypt the full JSON blob before `HSET`; decrypt
  after `HGET` / `HGETALL`.

#### Phase 2 — Configuration
- [ ] Add `EncryptionPasswordEnv string` to `ValkeyConfig` in `internal/config/config.go`:
  ```yaml
  multipart_state:
    valkey:
      # ... existing fields ...
      # Optional: dedicated password for at-rest encryption of multipart state in Valkey.
      # If unset, falls back to the gateway's main encryption password.
      # Recommended in production to achieve isolation between object-encryption keys
      # and Valkey storage encryption.
      encryption_password_env: "VALKEY_ENCRYPTION_PASSWORD"
  ```
- [ ] Env var: `VALKEY_ENCRYPTION_PASSWORD` (optional). If absent, derive the AEAD key
  from `ENCRYPTION_PASSWORD` (gateway main password) with a distinct HKDF info string
  so the two key hierarchies never collide.
- [ ] Config validation: if `EncryptionPasswordEnv` is set but the env var is empty,
  fail closed at startup with a clear error.

#### Phase 3 — Backwards Compatibility
- [ ] **Unencrypted read path:** On `Get`/`List`, if decryption fails with an AEAD tag
  error, attempt to parse the blob as raw JSON (legacy unencrypted state). If that
  succeeds, treat it as unencrypted and log a one-time migration warning.
- [ ] **Migration strategy:** Existing deployments can be migrated lazily:
  - Old state reads transparently (fallback path above).
  - New state writes are encrypted.
  - Orphaned old state expires via TTL naturally.
- [ ] Add a config flag `valkey.encrypt_state` (default `true` in v1.0; `false` preserves
  backwards compatibility for operators who need time to rotate). Deprecate `false` in
  v1.1 and remove in v2.0.

#### Phase 4 — Metrics & Observability
- [ ] Prometheus counter `gateway_mpu_state_encrypted_writes_total`
- [ ] Prometheus counter `gateway_mpu_state_legacy_reads_total` (tracks fallback reads)
- [ ] Log warning on first legacy read per gateway process: "Unencrypted Valkey state
  detected — enable valkey.encrypt_state=true"

#### Phase 5 — Documentation
- [ ] Update ADR-0009 "Security Considerations" to reflect that at-rest encryption is
  now available and recommended.
- [ ] Update runbook: `docs/RUNBOOK.md` — add "Valkey state at-rest encryption" section.
- [ ] Update deployment docs: mention `VALKEY_ENCRYPTION_PASSWORD` in secret management
  checklist.

### Tests
- [ ] Unit: round-trip encrypt/decrypt of `UploadState` JSON; ciphertext is non-deterministic
  (new nonce per write); tampered ciphertext returns error.
- [ ] Unit: legacy unencrypted blob parses correctly via fallback path.
- [ ] Unit: `HKDF` derivation produces different keys for `ENCRYPTION_PASSWORD` vs
  `VALKEY_ENCRYPTION_PASSWORD` even when the literal password is identical.
- [ ] Unit: missing `VALKEY_ENCRYPTION_PASSWORD` env var fails startup when
  `EncryptionPasswordEnv` is configured.
- [ ] Integration (miniredis): full `Create` → `Get` → `AppendPart` → `Delete` cycle with
  encryption enabled; raw hash value does not contain plaintext bucket/key names.
- [ ] Fuzz: malformed ciphertext headers, wrong versions, truncated blobs — all rejected.

### Security Considerations
- **Key isolation:** Using a distinct `VALKEY_ENCRYPTION_PASSWORD` (recommended) ensures
  that a compromise of the Valkey encryption key does not automatically compromise
  object-encryption keys, and vice-versa.
- **Nonce uniqueness:** AES-GCM nonce is 96-bit random per write. Probability of collision
  is negligible (< 2^-32) for the expected lifetime state-write volume.
- **No key-derivation on hot path:** Derive the AEAD key once at `ValkeyStateStore`
  construction and cache it in the struct; do not re-run HKDF on every read/write.
- **Zeroize:** Clear the derived AEAD key from memory on `Close()` using `zeroBytes()`.
- **Version byte:** Future algorithms can be introduced by bumping the version byte;
  old gateways with a newer version will fail closed with a clear error rather than
  silently mis-decrypt.

### Definition of Done
- [ ] `UploadState` blobs are encrypted at rest when `valkey.encrypt_state=true`.
- [ ] Fallback read path supports legacy unencrypted state with a warning.
- [ ] `VALKEY_ENCRYPTION_PASSWORD` is supported and documented.
- [ ] All unit and integration tests pass.
- [ ] ADR-0009 and runbook updated.
- [ ] No plaintext bucket/key/IV metadata visible in Valkey `HGETALL` output when
