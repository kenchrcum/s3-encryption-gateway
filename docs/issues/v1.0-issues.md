## [V1.0-SEC-2] HKDF-based Chunk-IV Derivation (non-breaking migration)
- Owner: Security/Crypto
- Priority: P1
- Labels: `area:crypto`, `area:security`, `enhancement`

### Summary
`deriveChunkIV` in `internal/crypto/chunked.go` derives per-chunk IVs by XOR-ing the chunk index into the last 4 bytes of the random `baseIV`. This is functionally correct (each object has a random 96-bit `baseIV`; birthday-bound collisions between objects are not possible), but cryptographically inconsistent with `internal/crypto/mpu_iv.go`, which correctly uses HKDF.

The migration is **fully backward-compatible** via a metadata flag with a dual-read window: new objects are written with HKDF and marked accordingly; existing objects (without the flag) continue to be decrypted via the XOR path. The XOR path is retained for **at least two major versions** (v1.x and v2.x) and will only be removed once the migration tool (→ V1.0-MAINT-1) has confirmed all objects are migrated.

### Affected
- `internal/crypto/chunked.go` — `deriveChunkIV` (both methods), new constant
- `internal/crypto/engine.go` — set metadata flag on the encrypt path

### Migration strategy
```
Encrypt (new):  HKDF  → MetaIVDerivation = "hkdf-sha256"  set
Decrypt (new):  flag present  → HKDF path
                flag absent   → XOR path (legacy, kept until v3.0)

Migration tool (V1.0-MAINT-1): reads objects (XOR decryption),
re-writes them (HKDF encryption) → flag is then set.
XOR code can be removed in v3.0 once all objects are confirmed migrated.
```

### Implementation Guide
- [ ] Introduce constant `MetaIVDerivation = "x-amz-meta-enc-iv-deriv"`.
  - Value `"hkdf-sha256"` = new path; absent/empty field = XOR legacy.
- [ ] Implement `deriveChunkIVHKDF` as a separate function (add, do not replace):
  ```go
  func deriveChunkIVHKDF(baseIV []byte, chunkIndex int) []byte {
      info := make([]byte, 8+4)
      copy(info, "chunk-iv")
      binary.BigEndian.PutUint32(info[8:], uint32(chunkIndex))
      r := hkdf.Expand(sha256.New, baseIV, info)
      iv := make([]byte, len(baseIV))
      io.ReadFull(r, iv)
      return iv
  }
  ```
- [ ] `chunkedEncryptReader`: use `deriveChunkIVHKDF` going forward; set `MetaIVDerivation = "hkdf-sha256"` in metadata.
- [ ] `chunkedDecryptReader`: read `MetaIVDerivation` from the manifest/metadata:
  - `"hkdf-sha256"` → `deriveChunkIVHKDF`
  - absent/empty → existing `deriveChunkIV` (XOR, leave unchanged, do not delete)
- [ ] Keep the existing `deriveChunkIV` (XOR) with a `// Deprecated: used for objects without MetaIVDerivation flag. Remove no earlier than v3.0.` comment.
- [ ] `ChunkManifest` struct: add optional field `IVDerivation string` (JSON tag `omitempty`).

### Tests
- [ ] KAT: `deriveChunkIVHKDF(baseIV, 0)` and `deriveChunkIVHKDF(baseIV, 1)` produce stable, distinct, deterministically reproducible IVs.
- [ ] Round-trip (HKDF): encrypt with new path, decrypt, plaintext matches.
- [ ] Round-trip (XOR legacy): encrypt with old code (no flag), decrypt with new code → fallback to XOR path; plaintext matches.
- [ ] Cross-version: an object with the HKDF flag **cannot** be decrypted via the XOR path (negative test).
- [ ] Regression: the full existing test suite passes without modifications.

### Definition of Done
- [ ] New HKDF path is active for all newly encrypted objects.
- [ ] Dual-read window works; no existing encrypted object breaks.
- [ ] XOR path is explicitly marked deprecated.
- [ ] All tests pass under `-race`.
- [ ] Consistent with `mpu_iv.go`.
- [ ] CHANGELOG entry: non-breaking; migration via V1.0-MAINT-1 available.

### Deprecation timeline
| Version | Status |
|---|---|
| v1.0 | HKDF is written; XOR can be read |
| v1.x–v2.x | XOR read support retained; migration tool available |
| v3.0 | XOR path removed (after confirmed full migration) |

## [V1.0-SEC-4] Restrict AAD fallback to explicitly marked legacy objects
- Owner: Security/Crypto
- Priority: P1
- Labels: `area:security`, `area:crypto`, `bug`

### Summary
`internal/crypto/engine.go` automatically retries AEAD decryption without AAD (`gcm.Open(nil, iv, ciphertext, nil)`) when decryption with AAD fails. The retry exists for backward compatibility with objects encrypted before AAD was introduced. Without being restricted to explicitly marked legacy objects, this allows an attacker with backend write access to use tampered metadata to bypass the AAD integrity check.

### Affected
- `internal/crypto/engine.go` — Decrypt function (AAD fallback path)
- `internal/crypto/engine.go` — Encrypt function (set legacy flag)

### Implementation Guide
- [ ] Introduce metadata constant `MetaLegacyNoAAD = "x-amz-meta-enc-legacy-no-aad"`.
- [ ] Encrypt path: new objects do **not** receive this flag. Only legacy objects (encrypted without AAD) have it.
- [ ] Decrypt path: the `gcm.Open(nil, iv, ciphertext, nil)` fallback is executed **only** when `MetaLegacyNoAAD == "true"` is set in the metadata.
  ```go
  if openErr != nil && metadata[MetaLegacyNoAAD] == "true" {
      if pt, err2 := gcm.Open(nil, iv, ciphertext, nil); err2 == nil {
          plaintext = pt
          openErr = nil
      }
  }
  ```
- [ ] Document in the changelog: objects without this flag will fail to decrypt if they were encrypted without AAD (breaking change for very old objects — migration required).
- [ ] Consider a migration tool or admin API endpoint for back-tagging old objects with `MetaLegacyNoAAD`.

### Tests
- [ ] Test: new object encrypted with AAD, tampered metadata → decryption fails (no fallback).
- [ ] Test: legacy object (no AAD, flag set) → decryption succeeds via fallback.
- [ ] Test: legacy object without flag → decryption fails (no blind fallback).

### Definition of Done

- [ ] New objects do not receive the flag.

## [V1.0-KMS-4] Self-Contained Envelope Encryption Provider
- Owner: Crypto
- Priority: P1
- Labels: `area:kms`, `area:crypto`, `enhancement`

### Summary
Implement a self-contained envelope encryption mode that does not require an external KMS or KMIP server. Today, all envelope encryption (DEK wrap/unwrap) is delegated to an external KeyManager (Cosmian KMIP, HSM). This is the right architecture for enterprise and regulated deployments, but it creates a barrier for operators who want strong encryption without managing a KMS dependency.

A self-contained provider wraps the DEK using a locally-held Key Encryption Key (KEK) — either an AES-256 master key or an RSA key pair — without any network call. This gives every deployment strong envelope security (unique DEK per object, KEK never touches the data) with zero external dependencies.

### Motivation
- Operators running the gateway in air-gapped or resource-constrained environments have no path to envelope encryption today; they must use password-derived keys (PBKDF2) or forgo encryption.
- Envelope encryption is cryptographically superior to password-derived keys: a compromised object's ciphertext reveals no information about the KEK, and rotating the KEK does not require re-encrypting object data.
- A self-contained mode reduces the operational barrier to entry for secure deployments, without weakening the existing pluggable `KeyManager` interface.

- Owner: Ops/DevEx
- Priority: P1
- Labels: `area:ops`, `area:crypto`, `area:ecosystem`, `tool`

### Summary

- Parallelism via configurable worker count
- Progress state in a local JSON file (resume after interruption)
- Metrics: objects total / migrated / skipped / failed

## [V1.0-SEC-10] Rate Limiter Timing Side-Channel Mitigation ✅ DONE
- Owner: Security
- Priority: P3
- Status: **✅ VERIFIED COMPLETE**
- Labels: `area:security`, `area:api`, `enhancement`
- Source: `docs/security/SECURITY_ANALYSIS_v0.6.md` finding #6 (CVSS 2.7)

### Summary
`RateLimiter.Allow` in `internal/middleware/security.go` has multiple early-return paths (new bucket, window reset, tokens available, tokens exhausted) with measurably different execution times. A client sending many requests could use timing measurements to infer their current token count or when a window will reset — a low-value information-disclosure side-channel. ### Affected
- `internal/middleware/security.go` — `RateLimiter.Allow` (lines 98–132)

### Implementation Guide
**Chosen: Option A — Constant-time padding.**

Add a deferred minimum-execution-time guarantee:
```go
func (rl *RateLimiter) Allow(key string) bool {
    start := time.Now()
    defer func() {
        if elapsed := time.Since(start); elapsed < minAllowTime {
            // Use a precise spin-wait instead of time.Sleep because
            // time.Sleep has coarse granularity (~1 ms on standard Linux
            // tickless kernels) which would defeat the purpose of a
            // microsecond-level constant-time guarantee.
            deadline := start.Add(minAllowTime)
            for time.Now().Before(deadline) {
            }
        }
    }()
    // ... existing logic unchanged ...
}
const minAllowTime = 50 * time.Microsecond
```

- [x] Decision recorded: Option A implemented.
- [x] `minAllowTime` constant added; benchmark verifies P99 request latency impact < 0.1 ms.

- The practical exploitability of this side-channel is very low: the timing difference is sub-microsecond, requires many samples, and reveals only rate-limit state (already visible via 429).
- Option A's sleep introduces a DoS-amplification concern if the rate limiter is called very frequently; cap `minAllowTime` conservatively.

### Tests
- [x] Benchmark `Allow` with 10 000 iterations; P99 latency ≤ `minAllowTime + 20µs`.
- [x] Existing rate-limiter tests pass without modification.

### Definition of Done

## [V1.0-SEC-12] Key Padding Creates Catastrophically Weak Keys ✅ DONE
- Owner: Security/Crypto
- Priority: P0
- Status: **✅ VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:security`, `bug`
- Source: `docs/issues/v1.0-deep-analysis-findings.md` DAF-02 / V1.0-SEC-12 (CVSS 7.5)

### Summary
When a key is shorter than `keySize`, the engine "pads" it by repeating the key prefix:
```go
if len(key) < keySize {
    copy(adjustedKey[len(key):], key[:keySize-len(key)])
}
```
A 16-byte key padded to 32 bytes becomes `key[:16] + key[:16]` — a key with perfect 16-byte periodicity, catastrophically weak for AES-256. This pattern exists at **8+ locations** in `internal/crypto/engine.go` (lines 430-437, 648-655, 873-892, 1002-1021, 1180-1199, 1353-1372, 1476-1483, 1590-1597).

**Breaking-change scope is narrow — see analysis below.**

### Root Cause Analysis

The padding block appears in two structurally different code paths, and only one is reachable in practice:

**KMS path (`e.kmsManager != nil`):** After `UnwrapKey`, if the KMS returns a wrong-sized key the padding block silently corrects it. This is the **real vulnerability** — a misconfigured or buggy KMS returning a 16-byte key produces silently weakened ciphertext with no operator-visible error.

**Password/PBKDF2 path (`e.kmsManager == nil`):** After `deriveKey`, which calls `pbkdf2.Key(sha256.New, password, salt, 100000, aesKeySize)` with `aesKeySize = 32` hardcoded. PBKDF2 is a PRF that always returns **exactly** the requested number of bytes — `len(key) != keySize` is **provably unreachable** here. The padding block in this branch is dead code. The same applies to `generateDataKey` on the encrypt path, which allocates exactly `keySize` bytes from `crypto/rand`.

**Consequence:** this fix has zero behavioural impact on password-only (non-KMS) deployments. It is a breaking change only when KMS is enabled and the KMS was silently returning wrong-sized keys — which was already producing silently corrupted (weakened) ciphertext. Hardening that broken path is the correct outcome.

### Affected
- `internal/crypto/engine.go` — all `adjustedKey` padding blocks (lines 430, 648, 873, 885, 1002, 1014, 1180, 1192, 1353, 1365, 1476, 1590)

### Implementation Guide

Apply a **split fix** by path, not a blanket removal:

**KMS branch (after `UnwrapKey`) — all encrypt and decrypt functions:**
- [x] Replace the padding block with a hard error:
  ```go
  if len(key) != keySize {
      zeroBytes(key)
      return nil, nil, fmt.Errorf("key size mismatch: KMS returned %d bytes, want %d — refusing to pad", len(key), keySize)
  }
  ```
- [x] This applies in: `Encrypt`, `Decrypt`, `encryptChunked`, `decryptChunked`, `DecryptRange`, and all other KMS branches. Note: `Decrypt` (line 639) already has this check — verify it is not redundantly followed by another padding block.

**Password/PBKDF2 branch (after `deriveKey`) — all functions:**
- [x] Remove the unreachable padding block entirely (it is dead code — PBKDF2 always returns exactly `aesKeySize` bytes). No error handling needed; add a comment explaining why:
  ```go
  // deriveKey always returns exactly aesKeySize bytes via PBKDF2;
  // the following size check is a defensive assertion only.
  if len(key) != keySize {
      zeroBytes(key)
      return nil, nil, fmt.Errorf("internal: PBKDF2 returned unexpected key size %d (want %d)", len(key), keySize)
  }
  ```
  Using an assertion-style error rather than padding makes future refactors safe without silently degrading.

**`generateDataKey` (encrypt path):**
- [x] Same: `generateDataKey` always returns exactly `keySize` bytes. Remove the subsequent padding block; replace with a defensive assertion error if present.

- [x] Run a global search for `copy(adjustedKey[len(key):]` to confirm all instances are addressed.

### Security Considerations
- Padding repeating key bytes reduces effective key entropy: a 16-byte key padded to 32 bytes is trivially identifiable via periodicity even without knowing the password.
- The fix surfaces KMS misconfiguration at operation time rather than after a breach.
- Password-only deployments are entirely unaffected; no data is at risk from this fix.

### Breaking Change Assessment
| Deployment mode | Behaviour change | Data at risk? |
|---|---|---|
| Password-only (no KMS) | None — dead code removed | No |
| KMS, correct key size | None — happy path unchanged | No |
| KMS, returning undersized key | Now returns an error instead of padding | Already corrupted — fix is correct |

**This is not a meaningful breaking change for any correctly-functioning deployment.**

### Tests
- [x] Unit: inject a 16-byte key via mock KMS into `Encrypt`, `Decrypt`, `encryptChunked`, `decryptChunked`, `DecryptRange` → each returns an error, padding is never applied.
- [x] Unit: inject a 32-byte key via mock KMS → no error; happy path unchanged (regression).
- [x] Unit: password-only mode (no KMS), any object size → encrypt/decrypt succeeds; no code path enters the former padding block.
- [x] Unit: `deriveKey` returns exactly 32 bytes (assert in test, confirm no padding path reachable).

### Definition of Done
- [x] All `copy(adjustedKey[len(key):]` padding blocks replaced with hard errors (KMS branch) or defensive assertions (PBKDF2 branch).
- [x] No silent key padding remains anywhere in the engine.
- [x] CHANGELOG entry: no breaking change for password-only mode; KMS deployments with misconfigured key sizes now fail loudly (correct behaviour).
- [x] All tests pass under `-race`.
