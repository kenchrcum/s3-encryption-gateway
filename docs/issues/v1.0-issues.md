## [V1.0-KMS-4] Self-Contained Envelope Encryption Provider
- Owner: Crypto
- Priority: P1
- Labels: `area:kms`, `area:crypto`, `enhancement`

### Summary
Implement a self-contained envelope encryption mode that does not require an external KMS or KMIP server. Today, all envelope encryption (DEK wrap/unwrap) is delegated to an external KeyManager (Cosmian KMIP, HSM). This is the right architecture for enterprise and regulated deployments, but it creates a barrier for operators who want strong encryption without managing a KMS dependency.

A self-contained provider wraps the DEK using a locally-held Key Encryption Key (KEK) â€” either an AES-256 master key or an RSA key pair â€” without any network call. This gives every deployment strong envelope security (unique DEK per object, KEK never touches the data) with zero external dependencies.

### Motivation
- Operators running the gateway in air-gapped or resource-constrained environments have no path to envelope encryption today; they must use password-derived keys (PBKDF2) or forgo encryption.
- Envelope encryption is cryptographically superior to password-derived keys: a compromised object's ciphertext reveals no information about the KEK, and rotating the KEK does not require re-encrypting object data.
- A self-contained mode reduces the operational barrier to entry for secure deployments, without weakening the existing pluggable `KeyManager` interface.

### Affected
- `internal/crypto/keymanager_selfcontained.go` (new) â€” AES-256-GCM KEK wrap + RSA-OAEP KEK wrap implementations
- `internal/crypto/keymanager.go` â€” register new provider types
- `internal/config/config.go` â€” new `self_contained` KMS provider section
- `docs/KMS_COMPATIBILITY.md` â€” document new provider
- `docs/adr/ADR-XXXX-self-contained-envelope.md` (new)

### Implementation Guide

#### Phase 1 â€” AES-KEK Provider
- [ ] Implement `AESKEKManager` satisfying `KeyManager` interface.
  - `WrapKey`: generate random DEK â†’ wrap with AES-256-GCM using the KEK â†’ store `{nonce, ciphertext}` in `KeyEnvelope.Ciphertext`.
  - `UnwrapKey`: decrypt `KeyEnvelope.Ciphertext` with KEK â†’ return plaintext DEK.
  - `HealthCheck`: validate KEK is loaded and correct length (32 bytes).
  - `ActiveKeyVersion`: return configured key version integer.
  - `Close`: zeroize KEK from memory.
- [ ] Accept KEK as base64-encoded 256-bit key from config or environment variable reference (`${ENV_VAR}` syntax consistent with existing config).
- [ ] Support multiple versioned KEKs (map of version â†’ key) to allow dual-read rotation without re-encrypting objects.

#### Phase 2 â€” RSA-KEK Provider
- [ ] Implement `RSAKEKManager` satisfying `KeyManager` interface.
  - `WrapKey`: wrap DEK with RSA-OAEP (SHA-256) using the RSA public key.
  - `UnwrapKey`: decrypt envelope using RSA private key.
  - `HealthCheck`: validate key pair consistency (modulus match).
  - Minimum key size: 2048 bits; recommended: 4096 bits.
- [ ] Accept PEM-encoded public + private key from config or environment variable references.
- [ ] Support key versioning via fingerprint (SHA-256 of public key modulus).

#### Phase 3 â€” Key Rotation Support
- [ ] Implement `RotatableKeyManager` for AES-KEK provider: `PrepareRotation` / `PromoteActiveVersion`.
- [ ] Document rotation runbook: add new KEK version â†’ promote â†’ re-wrap objects on next write (lazy rotation).
- [ ] Admin API endpoint to trigger rotation (reuse existing `internal/admin` rotation infrastructure).

#### Phase 4 â€” Configuration & Docs
- [ ] Add `self_contained` section to config schema:
  ```yaml
  kms:
    provider: self_contained
    self_contained:
      type: aes        # or rsa
      key_version: 1
      keys:
        - version: 1
          aes_key: "${AES_KEK_KEY}"   # base64 256-bit
  ```
- [ ] Update `docs/KMS_COMPATIBILITY.md` with provider matrix entry.
- [ ] Write ADR documenting the design rationale, security properties, and comparison with external KMS.
- [ ] Add getting-started example to `README.md`.

### Security Considerations
- The KEK must never be logged or included in error messages; enforce with linter rule or code review.
- Zeroize the KEK from memory on `Close()` and after each `WrapKey`/`UnwrapKey` operation (consistent with existing `zeroBytes` convention in `internal/crypto/engine.go`).
- If the KEK is lost, all wrapped DEKs are irrecoverable â€” document this clearly and recommend key backup procedures.
- RSA variant: use OAEP with SHA-256 (not PKCS#1 v1.5); enforce 2048-bit minimum.
- AES variant: derive a separate HMAC key from the KEK via HKDF for any integrity checks; do not reuse the encryption key for authentication.
- Support environment-variable-only key injection to avoid keys on disk in container deployments.

### Tests
- Unit: AES-KEK wrap/unwrap round-trip; wrong KEK returns `ErrUnwrapFailed`; Close zeroizes; version mismatch returns `ErrKeyNotFound`.
- Unit: RSA-KEK wrap/unwrap round-trip; key-pair mismatch detected at init; minimum key size enforced.
- Unit: dual-version read (old KEK version wraps, new version cannot unwrap, old version can).
- Integration (Tier 2): full encrypt/decrypt cycle via engine with AES-KEK provider against MinIO.
- Integration: key rotation promotion â€” objects written with v1 still decrypt after v2 promoted as active.
- Fuzz: `WrapKey`/`UnwrapKey` with malformed `KeyEnvelope.Ciphertext`.

### Definition of Done
- [ ] `AESKEKManager` and `RSAKEKManager` implement `KeyManager` interface and pass all unit tests.
- [ ] `AESKEKManager` implements `RotatableKeyManager`; rotation path tested end-to-end.
- [ ] Tier 2 conformance tests pass with `self_contained` provider against MinIO and Garage.
- [ ] Config schema updated; environment-variable injection works.
- [ ] ADR authored and merged.
- [ ] `docs/KMS_COMPATIBILITY.md` updated with provider entry.
- [ ] No KEK material appears in any log output (verified by log-scraping test).

### Known Limitations
- RSA key rotation requires generating a new key pair and re-distributing the public key; no automated rotation path.
- Self-contained mode offers no HSM boundary â€” key material lives in process memory. Operators requiring HSM-grade isolation should use the existing Cosmian KMIP or HSM adapter.

## [V1.0-CRYPTO-2] Valkey At-Rest Encryption for Multipart Upload State
- Owner: Crypto / Security
- Priority: P1
- Labels: `area:crypto`, `area:security`, `enhancement`

### Summary
Add optional at-rest encryption for the full multipart-upload state blob persisted in Valkey.

Today, `UploadState` (including the wrapped DEK envelope, IV prefix, bucket/key names,
and per-part metadata) is stored in Valkey as plaintext JSON. The DEK itself is already
wrapped by the `KeyManager` before being written, so an attacker with Valkey read access
cannot recover the object-encryption key without also compromising the KMS/HSM or gateway
password. However, the remainder of the state â€” bucket names, object keys, IV prefixes,
and part-level metadata â€” is visible to anyone with access to Valkey memory dumps, RDB/AOF
files, or replication streams. This is an at-rest disclosure gap that TLS on the wire does
not close.

### Threat Model
- **Attacker with Valkey read access** (e.g., compromised Valkey replica, stolen backup,
  insider with `ACL` access) can enumerate in-flight multipart uploads, learn bucket/key
  names, and collect wrapped DEK envelopes for later offline analysis.
- **Attacker with Valkey write access** can tamper with part records or swap wrapped DEKs
  (existing tamper-detection in `UnwrapKey` AAD catches some variants, but not all).

Encrypting the entire state blob at rest removes this read-disclosure vector entirely.

### Affected
- `internal/mpu/state.go` â€” encrypt before `HSET`, decrypt after `HGET`
- `internal/config/config.go` â€” new `ValkeyEncryptionPasswordEnv` config field
- `internal/crypto/` â€” new `aesgcm.Seal` / `aesgcm.Open` helper or reuse existing
  `passwordKeyManager` machinery
- `docs/adr/0009-encrypted-multipart-uploads.md` â€” update threat-model and security
  considerations
- `docs/plans/V1.0-CRYPTO-2-plan.md` (new) â€” detailed implementation plan

### Implementation Guide

#### Phase 1 â€” Encryption Layer Design
- [ ] Define a versioned ciphertext format for the at-rest blob:
  ```
  version(1 byte) || nonce(12 bytes) || ciphertext(... ) || tag(16 bytes)
  ```
  Version starts at `0x01` (AES-256-GCM). Leave room for future AEAD algorithms.
- [ ] Derive a dedicated AEAD key from the configured password via HKDF-SHA256:
  ```
  key = HKDF-SHA256(salt="s3eg-mpu-state-v1", ikm=password)
  ```
  This isolates the Valkey encryption key from object-encryption keys.
- [ ] Add `EncryptState([]byte) ([]byte, error)` and `DecryptState([]byte) ([]byte, error)`
  methods on `ValkeyStateStore`. Encrypt the full JSON blob before `HSET`; decrypt
  after `HGET` / `HGETALL`.

#### Phase 2 â€” Configuration

### Security Considerations
- **Key isolation:** Using a distinct `VALKEY_ENCRYPTION_PASSWORD` (recommended) ensures
  that a compromise of the Valkey encryption key does not automatically compromise
  object-encryption keys, and vice-versa.
- **Nonce uniqueness:** AES-GCM nonce is 96-bit random per write. Probability of collision
  is negligible (< 2^-32) for the expected lifetime state-write volume.
- **No key-derivation on hot path:** Derive the AEAD key once at `ValkeyStateStore`
  construction and cache it in the struct; do not re-run HKDF on every read/write.
- **Zeroize:** Clear the derived AEAD key from memory on `Close()` using `zeroBytes()`.
- **Version byte:** Future algorithms can be introduced by bumping the version byte;
  old gateways with a newer version will fail closed with a clear error rather than
  silently mis-decrypt.

## [V1.0-AUTH-1] Gateway-Managed Credential Store & Unified Authentication Refactor
- Owner: Security / API
- Priority: P0
- Labels: `milestone:v1.0`, `area:security`, `area:api`, `area:config`, `breaking-change`

### Summary

Replace the current dual-mode (single-bucket / multi-bucket) authentication architecture with a unified model: the gateway maintains its own list of known access-key/secret-key pairs. Every request â€” whether it uses AWS Signature V4 (header or presigned URL) or AWS Signature V2/query-parameter style â€” must present a credential that the gateway recognises and can validate. Requests with no credentials, or with an unknown access key, are rejected before any backend interaction occurs.

The gateway's configured backend credentials (`backend.access_key` / `backend.secret_key`) determine the backend scope. When they grant access to a single bucket, the gateway naturally becomes single-bucket; when they grant access to multiple buckets, it is multi-bucket. The `proxied_bucket` option becomes a non-breaking optional filter (enforced if set, ignored if absent) rather than a mode switch. The `backend.use_client_credentials` passthrough mode is removed.

This resolves **CWE-306 (Missing Authentication for Critical Function)** identified in the data-plane security audit.

### Motivation

- Any client reachable at the network level can currently issue a bare `GET /bucket/key` â€” no credentials needed â€” and receive decrypted object content. The entire security perimeter is the network firewall.
- The original "network is the perimeter" design is now considered a liability, particularly as the gateway is increasingly deployed behind load-balancers and in multi-tenant Kubernetes clusters where pod-to-pod routes are less tightly controlled.
- The dual-mode model (single-bucket vs. multi-bucket) introduces unnecessary complexity and code that can be collapsed into a single, cleaner auth path.

### Affected Components

| Component | Change |
|-----------|--------|
| `internal/config/config.go` | Add `AuthConfig.Credentials []GatewayCredential`; add env/valueFrom support; deprecate `Backend.UseClientCredentials` |
| `internal/api/auth.go` | Add `CredentialStore` interface + `StaticCredentialStore` implementation |
| `internal/api/handlers.go` | Rewrite `getS3Client` to always validate against credential store (fail-closed) |
| `internal/api/credentials.go` | No structural change; `ExtractCredentials` is reused as-is |

| `cmd/server/main.go` | Wire `CredentialStore` into `Handler` at startup |

## [V1.0-OPS-3] MPU Manifest Orphan Cleanup on Object Deletion
- Owner: Ops / API
- Priority: P2
- Labels: `area:ops`, `area:s3`, `bug`

### Summary
When an encrypted multipart upload (MPU) object is deleted â€” either directly or via S3 lifecycle rotation â€” the corresponding `.mpu-manifest` metadata file is not removed from the backend. Over time, these orphaned manifest files accumulate in the bucket, consuming storage and polluting object listings.

### Background
Each MPU encrypted object is written alongside a companion manifest file stored at `<key>.mpu-manifest` in the same bucket. This manifest is used by the gateway during decryption to resolve DEK envelopes and part boundaries. The `DeleteObject` and `DeleteObjects` handlers in the gateway proxy the delete request to the backend but do not issue a corresponding delete for the manifest file.

### Affected
- `internal/api/handlers.go` â€” `DeleteObject` and `DeleteObjects` handlers
- `internal/s3/` â€” backend client delete calls
- Tests: `internal/api/` â€” delete handler tests

### Implementation Guide
- [ ] In the `DeleteObject` handler: after a successful backend delete of the primary object, issue a second backend delete for `<key>.mpu-manifest`. Treat a `404` on the manifest delete as a no-op (non-MPU objects have no manifest).
- [ ] In the `DeleteObjects` (batch delete) handler: for each successfully deleted object, collect manifest keys and issue a follow-up batch delete for the manifest files.
- [ ] Log manifest cleanup at `debug` level; log unexpected errors (non-404) at `warn`.
- [ ] Ensure the manifest cleanup does not cause the primary delete to appear failed if only the manifest delete errors.

### Tests
- [ ] Unit: `DeleteObject` for an MPU-encrypted key also deletes `<key>.mpu-manifest`.
- [ ] Unit: `DeleteObject` for a non-MPU key does not error when manifest 404s.
- [ ] Unit: `DeleteObjects` batch cleans up all manifest files for deleted MPU objects.
- [ ] Integration (Tier 2): upload encrypted MPU object â†’ delete via gateway â†’ verify manifest is absent from backend bucket.

### Definition of Done
- [ ] `DeleteObject` and `DeleteObjects` handlers clean up companion `.mpu-manifest` files.
- [ ] Manifest cleanup failures (non-404) are logged as warnings but do not propagate as primary delete errors.
- [ ] All unit and Tier 2 integration tests pass.
