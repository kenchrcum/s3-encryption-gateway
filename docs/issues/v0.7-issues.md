## [V0.7-SEC-1] HKDF-based Chunk-IV Derivation (V1.0-SEC-2) âś… DONE
- Owner: Security/Crypto
- Priority: P1
- Status: **âś… VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:security`, `enhancement`

> **Verified:** 2026-04-29

### Summary
Per-chunk IV derivation switched from XOR to HKDF-SHA256. New objects carry
`x-amz-meta-enc-iv-deriv="hkdf-sha256"`; objects without the flag continue to
decrypt via the legacy XOR path (retained until v3.0).

### Breaking change notice
**Write-side only.** New chunked objects are written with HKDF; old objects
still decrypt transparently. The XOR read path is deprecated and will be
removed in v3.0. Operators should run `s3eg-migrate --migration-class sec2`
to re-encrypt legacy objects.

## [V0.7-SEC-2] Restrict AAD Fallback to Explicitly Marked Legacy Objects (V1.0-SEC-4) âś… DONE
- Owner: Security/Crypto
- Priority: P1
- Status: **âś… VERIFIED COMPLETE**
- Labels: `area:security`, `area:crypto`, `bug`

> **Verified:** 2026-04-29

### Summary
The blind `gcm.Open(..., nil)` AAD fallback is now gated behind the metadata
flag `x-amz-meta-enc-legacy-no-aad="true"`. New objects never receive this flag.

### Breaking change notice
Objects encrypted without AAD that do **not** carry the legacy flag will now
fail to decrypt. For operators with pre-AAD objects, two-step recovery is
available:
1. `s3eg-migrate backfill-legacy-no-aad` â€” adds the marker in-place (CopyObject,
   no re-encryption).
2. `s3eg-migrate --migration-class sec4` â€” full re-encrypt with AAD (removes the
   marker permanently).

- Owner: Crypto/Performance

- Status: **âś… VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:performance`, `bug`

> **Re-verified:** 2026-04-30 â€” supersedes the 0.6.4 partial fix.

### Summary
Eliminated the redundant outer `aead.Seal` from
`encryptChunkedWithMetadataFallback`. The new v2 on-disk format is a streaming

decrypt such objects. Mitigation: do not roll back after writes; or disable the
fallback path (`total_header_limit: 0`) during rollback. Legacy objects written
before the upgrade remain readable on all versions.

## [V1.0-SEC-5] Integer Overflow in Encrypted Range Calculation âś… DONE

- Status: **âś… VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:security`, `bug`

### Summary
`calculateEncryptedByteRange` in `internal/crypto/range_optimization.go` performs `int` arithmetic before casting to `int64`. The expression `chunkSize + tagSize` and `endChunk + 1` can silently overflow a 32-bit `int` on 32-bit platforms or when input values are near `math.MaxInt32`, producing incorrect byte-range offsets or panics. The existing guard (`endChunk < startChunk`) does not cover `endChunk < 0`.

### Affected
- `internal/crypto/range_optimization.go` â€” `calculateEncryptedByteRange` (lines 34â€“44)

### Implementation Guide
- [x] Add missing input guard: `endChunk < 0` alongside the existing `startChunk < 0` check.
- [x] Detect pre-cast overflow before `int64` promotion:
  ```go
  if endChunk > math.MaxInt32-1 || chunkSize > math.MaxInt32-tagSize {
      return 0, 0, fmt.Errorf("range parameters exceed safe limits")
  }
  ```
- [x] Change the return signature to `(encryptedStart, encryptedEnd int64, err error)`.
- [x] Add a post-calculation sanity check: `if encryptedEnd < encryptedStart { return 0, 0, fmt.Errorf("range calculation overflow") }`.
- [x] Update all call sites to handle the new `error` return value.

### Security Considerations
- With adversarially crafted object metadata a caller could supply out-of-range chunk indices; the overflow would yield incorrect byte-range offsets, potentially causing an out-of-bounds read or exposing a wrong ciphertext slice.

### Tests

## [V1.0-SEC-12] Key Padding Creates Catastrophically Weak Keys âś… DONE
- Owner: Security/Crypto
- Priority: P0
- Status: **âś… VERIFIED COMPLETE**
- Labels: `area:crypto`, `area:security`, `bug`
- Source: `docs/issues/v1.0-deep-analysis-findings.md` DAF-02 / V1.0-SEC-12 (CVSS 7.5)

### Summary
When a key is shorter than `keySize`, the engine "pads" it by repeating the key prefix:
```go
if len(key) < keySize {
    copy(adjustedKey[len(key):], key[:keySize-len(key)])
}
```
A 16-byte key padded to 32 bytes becomes `key[:16] + key[:16]` â€” a key with perfect 16-byte periodicity, catastrophically weak for AES-256. This pattern exists at **8+ locations** in `internal/crypto/engine.go` (lines 430-437, 648-655, 873-892, 1002-1021, 1180-1199, 1353-1372, 1476-1483, 1590-1597).

**Breaking-change scope is narrow â€” see analysis below.**

### Root Cause Analysis

The padding block appears in two structurally different code paths, and only one is reachable in practice:

**KMS path (`e.kmsManager != nil`):** After `UnwrapKey`, if the KMS returns a wrong-sized key the padding block silently corrects it. This is the **real vulnerability** â€” a misconfigured or buggy KMS returning a 16-byte key produces silently weakened ciphertext with no operator-visible error.

**Password/PBKDF2 path (`e.kmsManager == nil`):** After `deriveKey`, which calls `pbkdf2.Key(sha256.New, password, salt, 100000, aesKeySize)` with `aesKeySize = 32` hardcoded. PBKDF2 is a PRF that always returns **exactly** the requested number of bytes â€” `len(key) != keySize` is **provably unreachable** here. The padding block in this branch is dead code. The same applies to `generateDataKey` on the encrypt path, which allocates exactly `keySize` bytes from `crypto/rand`.

**Consequence:** this fix has zero behavioural impact on password-only (non-KMS) deployments. It is a breaking change only when KMS is enabled and the KMS was silently returning wrong-sized keys â€” which was already producing silently corrupted (weakened) ciphertext. Hardening that broken path is the correct outcome.

### Affected
- `internal/crypto/engine.go` â€” all `adjustedKey` padding blocks (lines 430, 648, 873, 885, 1002, 1014, 1180, 1192, 1353, 1365, 1476, 1590)

### Implementation Guide

Apply a **split fix** by path, not a blanket removal:

**KMS branch (after `UnwrapKey`) â€” all encrypt and decrypt functions:**
- [x] Replace the padding block with a hard error:
  ```go
  if len(key) != keySize {
      zeroBytes(key)
      return nil, nil, fmt.Errorf("key size mismatch: KMS returned %d bytes, want %d â€” refusing to pad", len(key), keySize)
  }
  ```
- [x] This applies in: `Encrypt`, `Decrypt`, `encryptChunked`, `decryptChunked`, `DecryptRange`, and all other KMS branches. Note: `Decrypt` (line 639) already has this check â€” verify it is not redundantly followed by another padding block.

**Password/PBKDF2 branch (after `deriveKey`) â€” all functions:**
- [x] Remove the unreachable padding block entirely (it is dead code â€” PBKDF2 always returns exactly `aesKeySize` bytes). No error handling needed; add a comment explaining why:
  ```go
  // deriveKey always returns exactly aesKeySize bytes via PBKDF2;
  // the following size check is a defensive assertion only.
  if len(key) != keySize {
      zeroBytes(key)
      return nil, nil, fmt.Errorf("internal: PBKDF2 returned unexpected key size %d (want %d)", len(key), keySize)
  }
  ```
  Using an assertion-style error rather than padding makes future refactors safe without silently degrading.

**`generateDataKey` (encrypt path):**
- [x] Same: `generateDataKey` always returns exactly `keySize` bytes. Remove the subsequent padding block; replace with a defensive assertion error if present.

- [x] Run a global search for `copy(adjustedKey[len(key):]` to confirm all instances are addressed.

### Security Considerations
- Padding repeating key bytes reduces effective key entropy: a 16-byte key padded to 32 bytes is trivially identifiable via periodicity even without knowing the password.
- The fix surfaces KMS misconfiguration at operation time rather than after a breach.
- Password-only deployments are entirely unaffected; no data is at risk from this fix.

### Breaking Change Assessment
| Deployment mode | Behaviour change | Data at risk? |
|---|---|---|
| Password-only (no KMS) | None â€” dead code removed | No |
| KMS, correct key size | None â€” happy path unchanged | No |
| KMS, returning undersized key | Now returns an error instead of padding | Already corrupted â€” fix is correct |

**This is not a meaningful breaking change for any correctly-functioning deployment.**

### Tests
- [x] Unit: inject a 16-byte key via mock KMS into `Encrypt`, `Decrypt`, `encryptChunked`, `decryptChunked`, `DecryptRange` â†’ each returns an error, padding is never applied.
- [x] Unit: inject a 32-byte key via mock KMS â†’ no error; happy path unchanged (regression).
- [x] Unit: password-only mode (no KMS), any object size â†’ encrypt/decrypt succeeds; no code path enters the former padding block.
- [x] Unit: `deriveKey` returns exactly 32 bytes (assert in test, confirm no padding path reachable).

### Definition of Done
- [x] All `copy(adjustedKey[len(key):]` padding blocks replaced with hard errors (KMS branch) or defensive assertions (PBKDF2 branch).
- [x] No silent key padding remains anywhere in the engine.
- [x] CHANGELOG entry: no breaking change for password-only mode; KMS deployments with misconfigured key sizes now fail loudly (correct behaviour).
